SOLID 원칙

SOLID는 변화에 강하고 유지보수하기 쉬운 객체지향 설계(OOP)를 위한 5가지 원칙의 머리글자입니다 — SRP, OCP, LSP, ISP, DIP. 각 원칙은 결합을 줄이고 응집을 높이는 서로 다른 각도의 가이드이며, 이를 따르면 자연스럽게 테스트하기 쉽고 확장 가능한 코드가 됩니다.

구체적인 개념 설명

객체지향 설계의 가장 큰 적은 "변경할 때 영향이 어디까지 퍼지는지 모르는 코드"입니다. SOLID는 그 영향 범위를 좁히는 기술 — 책임을 작게 나누고(SRP), 변경 대신 확장으로(OCP), 부모 약속을 지키고(LSP), 필요 없는 의존을 잘라내고(ISP), 추상에 의존하도록(DIP) — 입니다.

SOLID는 주문처럼 "이 줄만 외우면 좋은 코드"가 아닙니다. 각 원칙은 다른 원칙을 받쳐주고, 하나만 과하게 적용하면 다른 원칙이 깨집니다. 그래서 항상 "이 변경을 쉽게 만들기 위해 어떤 원칙이 도움 되는가"의 시선으로 봐야 합니다.

비유 — 회사의 부서 분할

한 사람이 모든 일을 맡으면 본인이 자리를 비우면 회사 전체가 멈추고, 한 가지 변경이 다른 모든 일에 영향을 줍니다. 부서를 나누고 인터페이스(요청서·결재선)로만 협력하게 만들면 부서 안 변경이 다른 부서에 거의 영향을 주지 않죠. SOLID는 그런 부서 설계 가이드입니다.

왜 필요한가?

  • 변경 비용 낮추기 — 한 곳을 고칠 때 영향이 그 안에 머물도록.
  • 테스트 용이성 — 의존을 인터페이스로 끊으면 테스트에서 가짜로 대체하기 쉽다.
  • 재사용 — 책임이 작고 의존이 적은 모듈은 다른 곳에 그대로 옮기기 쉽다.
  • 설계 의도 가시화 — "이 클래스가 무슨 일을 하는가"가 이름과 인터페이스에서 드러난다.

어떻게 동작하는가?

5가지 원칙은 "결합을 줄이고 응집을 높인다"는 같은 목표를 다른 각도로 풀어낸다.

1) SRP — Single Responsibility Principle

"한 클래스는 한 가지 변경 이유만 가져야 한다." 보통 "한 가지 일만 한다"로 단순화하지만, 더 정확한 표현은 "한 클래스는 한 가지 액터(stakeholder)를 위해서만 변경되어야 한다"입니다. 

// 위반: 한 클래스가 보고서 데이터 조회·포맷·이메일 발송을 모두 담당
class ReportService {
    public Data load() { ... }
    public String formatHtml(Data d) { ... }
    public void sendEmail(String html, String to) { ... }
}

// 준수: 책임을 분리 — 데이터 수집, 포맷, 발송이 각자 변경 이유가 다르다
class ReportRepository { Data load(); }
class ReportFormatter  { String formatHtml(Data d); }
class EmailSender      { void send(String html, String to); }

2) OCP — Open/Closed Principle

"확장에는 열려 있고, 수정에는 닫혀 있어야 한다." 새 기능을 추가할 때 기존 코드를 고치지 않고 새 코드를 더하는 식으로 변경할 수 있어야 합니다. 다형성과 인터페이스를 통해 자연스럽게 달성됩니다. 

// 위반: 결제 수단이 추가될 때마다 if-else가 늘어남
class Checkout {
    void pay(String type, long amount) {
        if (type.equals("CARD"))      { /* ... */ }
        else if (type.equals("TOSS")) { /* ... */ }
        // 새 수단 추가하려면 여기를 또 수정해야 한다
    }
}

// 준수: 새 결제 수단은 새 클래스로 추가만 하면 된다
interface PaymentGateway { void charge(long amount); }
class CardGateway   implements PaymentGateway { ... }
class TossGateway   implements PaymentGateway { ... }
class KakaoGateway  implements PaymentGateway { ... }   // 새로 추가만

class Checkout {
    private final PaymentGateway gateway;
    Checkout(PaymentGateway g) { this.gateway = g; }
    void pay(long amount) { gateway.charge(amount); }
}

3) LSP — Liskov Substitution Principle

"자식 객체는 부모 객체로 그대로 대체될 수 있어야 한다." 부모를 받는 코드가 자식을 받았을 때 예기치 않게 깨지면 안 됩니다. 즉 자식은 부모가 약속한 계약(사전조건은 강화하지 않고, 사후조건은 약화하지 않으며, 불변식을 지킴)을 만족해야 합니다. 

// 고전적 위반: Square가 Rectangle을 상속하면서 setW/setH가 동시에 바뀌게 함
class Rectangle { int w, h; void setW(int v){w=v;} void setH(int v){h=v;} int area(){return w*h;} }
class Square extends Rectangle {
    @Override void setW(int v) { w = v; h = v; }   // 사후조건이 부모와 다름
    @Override void setH(int v) { w = v; h = v; }
}
// 부모 인자로 Rectangle을 받는 코드가 Square를 받으면 expectations이 깨짐 → LSP 위반

// 해결: 상속 대신 별도 추상화로 분리
interface Shape { int area(); }
record Rectangle(int w, int h) implements Shape { public int area(){return w*h;} }
record Square(int side)        implements Shape { public int area(){return side*side;} }

4) ISP — Interface Segregation Principle

"클라이언트는 자신이 사용하지 않는 메서드에 의존하지 않아야 한다." 뚱뚱한 인터페이스 하나보다 역할별로 잘게 나뉜 인터페이스 여러 개가 낫습니다. 

// 위반: 모든 일을 한 인터페이스에 욱여넣음. 프린터만 쓰고 싶은데 fax/scan까지 의존
interface MultiFunctionDevice {
    void print(Doc d);
    void scan(Doc d);
    void fax(Doc d);
}

// 준수: 역할별로 분리. 클라이언트는 자기에게 필요한 것만 구현/사용
interface Printer { void print(Doc d); }
interface Scanner { void scan(Doc d); }
interface Fax     { void fax(Doc d); }

class SimplePrinter implements Printer { public void print(Doc d) { ... } }
class AllInOne     implements Printer, Scanner, Fax { ... }

5) DIP — Dependency Inversion Principle

"고수준 모듈이 저수준 모듈에 의존해서는 안 되며, 둘 다 추상화에 의존해야 한다." 즉 비즈니스 로직이 구체적인 DB·외부 API에 직접 의존하지 말고, 인터페이스를 사이에 두라는 원칙입니다. 

// 위반: 비즈니스 로직이 MySQL 구현에 직접 의존 → 다른 DB로 바꾸기 어렵고 테스트 어려움
class OrderService {
    private final MySqlOrderRepository repo = new MySqlOrderRepository();
}

// 준수: 추상에 의존, 구현은 외부에서 주입
interface OrderRepository { void save(Order o); Order findById(long id); }

class OrderService {
    private final OrderRepository repo;          // 인터페이스에 의존
    public OrderService(OrderRepository r) { this.repo = r; }
}

class MySqlOrderRepository  implements OrderRepository { ... }   // 운영
class InMemoryOrderRepository implements OrderRepository { ... } // 테스트용
고수준과 저수준이 서로 직접 의존하지 않고 추상에 함께 의존한다.

6) 원칙들 사이의 관계

  • SRP가 흐트러지면 OCP가 따라 흐트러진다(여러 변경 이유가 한 클래스에 모이면 새 변경마다 그 클래스를 수정해야 함).
  • OCP는 보통 다형성 + DIP로 달성된다(추상에 의존해야 새 구현으로 확장 가능).
  • LSP가 깨지면 다형성을 못 쓰므로 OCP도 의미가 사라진다.
  • ISP는 SRP의 인터페이스 버전 — 인터페이스도 작게.

7) 안티패턴 — SOLID를 잘못 적용한 경우

  • 인터페이스 폭발 — DIP를 과하게 적용해 모든 클래스에 인터페이스를 두는 것. 가짜 추상화가 생긴다.
  • 지나친 분해 — SRP 이름으로 너무 잘게 나눠 클래스가 수십 개로 폭발하고 흐름을 추적하기 어렵게 됨.
  • 전략 패턴 남용 — if 두 줄을 갈아끼우려고 인터페이스+구현체 3개를 만드는 식.
  • 억지 추상 — 한 번도 갈아끼우지 않을 의존에 인터페이스를 두는 것. YAGNI 위반.

시간/공간 복잡도 또는 trade-off

관점SOLID 충실위반왜 그런가?
변경 영향 범위좁음넓음책임이 분리되어 변경이 한곳에 머묾
테스트 용이성높음낮음의존을 인터페이스로 끊어 가짜로 대체
초기 설계 비용높음낮음클래스·인터페이스 분리에 시간 투자
코드 양많음(파일 수↑)적음분리는 불가피하게 양을 늘린다
흐름 추적성낮을 수 있음높음분산된 클래스를 따라가야 함
도구 도움IDE jump가 핵심덜 필요분리된 코드는 IDE 의존도가 올라감

실무에서 어디에 쓰이는가?

  • Spring 기반 개발 — DI/IoC가 DIP의 직접 구현. Repository·Service 분리는 SRP/OCP의 적용.
  • 플러그인/어댑터 — 결제 수단·알림 채널·외부 시스템 통합이 OCP+ISP의 전형.
  • 도메인 모델 — DDD의 애그리거트·도메인 서비스 분리도 결국 SRP/LSP 위에서 동작.
  • 레거시 리팩터링 — God Object 분해, 인터페이스 추출, 의존 주입 도입이 대부분 SOLID 적용.

면접에 나올 질문과 답변

SRP의 "한 가지 책임"은 어떻게 해석해야 하나요?
"한 가지 일만 한다"보다 더 정확한 표현은 "한 가지 변경 이유만 가진다"입니다. 같은 클래스를 수정해야 할 이유가 여러 액터에서 오면(예: 회계팀 변경 vs 마케팅팀 변경) 그 클래스는 SRP를 어긴 것입니다. 변경 이유를 액터 단위로 묶어 클래스를 나누는 것이 핵심입니다.
OCP는 어떻게 달성하나요?
다형성과 추상화로 달성합니다. 변할 가능성이 있는 부분을 인터페이스로 추출하고, 새 동작은 새 구현체로 추가하면 기존 코드를 수정할 필요가 없습니다. "변할 가능성"을 모든 곳에서 가정하면 인터페이스 폭발이 일어나므로, 실제 변화가 관찰되거나 명확한 도메인 변동성이 있을 때 적용합니다.
LSP가 깨졌다는 신호는 무엇인가요?
대표 신호는 "부모 타입을 받는 코드가 자식을 받았을 때 예외나 다른 동작을 보이는 것"입니다. 예: 자식이 부모의 메서드를 호출했을 때 UnsupportedOperationException을 던지거나, 부모가 보장하던 사후조건을 약화하는 경우입니다. 보통 잘못된 상속 관계의 신호이므로 합성으로 바꾸거나 인터페이스를 분리해야 합니다.
ISP와 SRP의 차이는?
SRP는 클래스 차원에서 책임을 작게 나누는 원칙이고, ISP는 같은 아이디어를 인터페이스에 적용한 원칙입니다. 하나의 거대한 인터페이스를 만들면 그것을 구현하는 클래스는 자기에게 필요 없는 메서드까지 구현하게 되고, 그 인터페이스를 사용하는 클라이언트도 필요 없는 의존을 갖게 됩니다. 역할별로 인터페이스를 작게 쪼개는 것이 ISP입니다.
DIP와 DI(의존성 주입)는 같은가요?
다릅니다. DIP는 설계 원칙("추상에 의존하라"), DI는 구현 기법("의존을 외부에서 넣어준다")입니다. DI는 DIP를 실현하는 가장 흔한 도구이지만, DI 컨테이너 없이도 생성자 주입으로 DIP를 지킬 수 있습니다. 반대로 DI를 쓰더라도 구체 클래스에 의존하면 DIP는 깨진 것입니다.
SOLID를 너무 충실히 따르면 어떤 문제가 생기나요?
클래스·파일 수가 폭증해 흐름을 따라가기 어려워지고, 한 번도 갈아끼우지 않을 곳에 인터페이스가 생겨 가짜 추상화가 만들어집니다. SOLID는 변경이 자주 일어나거나 다양한 구현이 필요한 영역에 우선 적용하고, 단순한 영역은 그대로 두는 것이 현명합니다. "필요할 때까지는 추상화하지 말 것(YAGNI)"이 SOLID의 좋은 동반자입니다.
