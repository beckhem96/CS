CI / CD 파이프라인

CI(Continuous Integration)는 코드 변경을 자주 합쳐 자동으로 빌드·테스트하는 관행이고, CD는 그 검증된 산출물을 자동으로 배포 가능 상태(Continuous Delivery)로 두거나, 더 나아가 운영에 자동 배포(Continuous Deployment)하는 관행입니다. 파이프라인 단계와 배포 전략(Blue-Green, Canary, Rolling)을 함께 설계해야 안전한 자동화가 됩니다.

구체적인 개념 설명

옛날에는 한 달에 한 번 큰 릴리스를 위해 며칠씩 통합·테스트를 했습니다. 이러면 통합 시점에 충돌이 폭발하고 배포 위험도 커지죠. CI/CD는 변경을 자주, 작게, 자동화된 검증과 함께 합치는 흐름으로 이 위험을 분산시킵니다. "자주 배포할수록 안전하다"는 다소 역설적인 명제가 성립합니다 — 변경 단위가 작아 문제가 생겼을 때 원인을 빠르게 추적할 수 있기 때문입니다.

CI는 "합치는 자동화", CD는 "내보내는 자동화"입니다. 그리고 이 둘 사이에 자동화된 테스트가 들어가야 비로소 의미가 있습니다. CI/CD는 단순한 도구가 아니라 팀의 작업 흐름과 코드 품질 게이트를 함께 설계하는 일입니다.

비유 — 컨베이어 벨트

공장의 컨베이어 벨트는 부품이 정해진 위치에서 검수받고, 조립되고, 포장되어 출고됩니다. CI/CD는 코드가 빌드 → 테스트 → 보안 스캔 → 패키징 → 배포의 단계를 자동으로 통과하는 컨베이어이고, 각 단계의 게이트가 실패하면 그 자리에서 멈춥니다.

왜 필요한가?

  • 통합 위험 분산 — 큰 통합 한 번 대신 작은 통합 여럿으로.
  • 피드백 가속 — 푸시 직후 빌드/테스트 결과를 받아 빠르게 고친다.
  • 릴리스 자신감 — 자동 검증을 통과한 산출물은 손으로 만든 것보다 신뢰할 수 있다.
  • 롤백 용이성 — 배포가 자동화되면 롤백도 자동으로 만들 수 있다.
  • 표준화 — 사람마다 다른 빌드 환경이 사라진다("내 컴퓨터에선 됐는데" 박멸).

어떻게 동작하는가?

1) 파이프라인 단계 한눈에

단계마다 게이트가 있어 통과한 것만 다음 단계로. 실패는 즉시 알림.

2) CI 단계 — 합치는 자동화

  • 트리거 — push, PR open, 스케줄, 수동.
  • 빌드 — 컴파일·의존성 해결·패키징(JAR/Docker 이미지).
  • 테스트 — 단위/통합. 피라미드 위쪽은 별도 잡으로 분리.
  • 품질 게이트 — 린트, 정적 분석(SonarQube), 커버리지 임계값.
  • 보안 스캔 — 의존성 취약점(SCA: Snyk, Dependabot), SAST(코드 정적 분석), 시크릿 누출 검사.
  • 아티팩트 산출 — 레지스트리에 이미지/패키지 푸시. 태그는 git SHA + 시맨틱 버전.

3) CD 단계 — Delivery vs Deployment

구분Continuous DeliveryContinuous Deployment
운영 배포버튼 한 번 (수동 승인)완전 자동
전제 조건자동화 + 게이트+ 강한 자동 검증·관측·롤백
리스크승인 단계가 안전판관측·롤백이 안전판이어야

4) 배포 전략 — Blue-Green

두 환경을 운영하다가 한 번에 스위치. 빠른 롤백이 가능하지만 자원 비용이 크다.

5) 배포 전략 — Canary

소수 사용자에게만 새 버전을 보여주고 메트릭이 정상이면 점진적으로 비율을 늘린다.

6) 배포 전략 — Rolling

여러 인스턴스를 일정 비율씩 새 버전으로 교체합니다. 자원 사용은 적지만 배포 중 두 버전이 동시에 도는 시간이 길어, DB 스키마 변경처럼 두 버전이 공존할 때 깨지는 경우는 마이그레이션 전략을 신중히 잡아야 합니다(Backward-compatible deploy).

7) Feature Flag와 Dark Launch

배포는 하지만 기능은 끈 채로 운영에 올린 뒤 플래그로 점진적으로 켜는 방법입니다. 배포와 릴리스를 분리하므로 빠른 롤백·A/B 테스트·유저 그룹 단위 릴리스가 가능해집니다(LaunchDarkly, GrowthBook 같은 도구).

8) 자동 롤백과 SLO 게이트

Canary 도중 에러율·지연·비즈니스 메트릭이 임계치를 넘으면 자동으로 이전 버전으로 되돌리는 메커니즘이 핵심입니다. "자동 배포"의 안전판은 결국 관측(Observability)과 자동 롤백이며, 둘이 없으면 자동 배포는 위험합니다.

9) 인프라/보안 자동화

  • IaC(Infrastructure as Code) — Terraform·Pulumi로 인프라도 PR/리뷰/CI로 관리.
  • GitOps — 배포 상태를 git에 선언하고, 컨트롤러가 운영을 일치시킨다(Argo CD, Flux).
  • 아티팩트 서명 — 누가 만든 빌드인지 검증(Sigstore, cosign).
  • SBOM — 소프트웨어 자재 명세로 의존성 추적·취약점 대응.

10) GitHub Actions 예시

# .github/workflows/ci.yml
name: CI
on: [push, pull_request]
jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-java@v4
        with: { distribution: 'temurin', java-version: '21' }
      - name: Build & unit tests
        run: ./gradlew build
      - name: Integration tests
        run: ./gradlew integrationTest
      - name: Static analysis
        run: ./gradlew sonarqube
      - name: Build Docker image
        run: docker build -t myapp:${{ github.sha }} .
      - name: Push to registry
        if: github.ref == 'refs/heads/main'
        run: |
          echo "${{ secrets.REGISTRY_TOKEN }}" | docker login -u ci --password-stdin
          docker push myapp:${{ github.sha }}

시간/공간 복잡도 또는 trade-off

전략롤백 속도자원 비용두 버전 공존적합 상황
Blue-Green매우 빠름(스위치)큼(2배)없음큰 변경, 빠른 롤백 필요
Canary빠름중있음실 사용자 기반 검증
Rolling중간작음오랫동안일반적, 호환되는 변경
Recreate느림(다운타임)작음없음다운타임 허용 시
Feature Flag즉시(toggle)—코드 안에서 분기점진적 릴리스 / A-B

실무에서 어디에 쓰이는가?

  • 모든 PR에 CI — 빌드·테스트·정적 분석을 거치지 않으면 머지 불가.
  • main 브랜치 자동 배포 — staging 환경까지 자동, 운영은 승인 또는 SLO 게이트.
  • 스키마 변경 시 두 단계 배포 — 호환되는 마이그레이션 → 코드 배포 → 정리. Rolling/Canary 호환을 위해.
  • 관측성 + 자동 롤백 — Prometheus/Grafana + Argo Rollouts·Spinnaker 같은 도구로 SLO 위반 시 자동 되돌림.
  • 모바일 앱 — staged rollout(점진 배포), 강제 업데이트 정책.

면접에 나올 질문과 답변

CI와 CD의 정확한 차이는?
CI는 자주 합쳐 자동으로 빌드·테스트하는 관행이고, CD는 그 결과를 자동으로 배포 가능 상태로 만드는(Delivery) 또는 운영에 자동 배포하는(Deployment) 관행입니다. CI는 합치는 자동화, CD는 내보내는 자동화로 기억하면 좋습니다.
Blue-Green과 Canary 중 무엇을 선택하나요?
Blue-Green은 두 환경을 동시에 운영해 한 번에 스위치 — 빠른 롤백이 강점이고 자원 비용이 큽니다. Canary는 일부 사용자에게만 새 버전을 보여주며 점진적으로 비율을 늘려 — 실 사용자 기반 검증에 강하지만 두 버전이 공존하는 시간이 깁니다. 리스크가 큰 큰 변경은 Blue-Green, 점진적 검증이 필요한 변경은 Canary가 흔한 선택입니다.
Continuous Deployment의 안전판은 무엇인가요?
① 강한 자동화된 테스트 피라미드, ② 관측(Observability)과 SLO 기반 알림, ③ 자동 롤백 메커니즘입니다. 세 가지 중 하나라도 약하면 자동 배포는 위험해지므로, 보통 Continuous Delivery(승인 단계 유지)에서 시작해 셋이 모두 갖춰진 뒤 Continuous Deployment로 넘어갑니다.
DB 스키마 변경은 CI/CD에서 어떻게 다루나요?
호환되는 마이그레이션을 우선 적용하는 두 단계(또는 세 단계) 배포가 표준입니다. ① 새 컬럼/테이블을 옛 코드와 호환되게 추가, ② 새 코드를 배포해 새 컬럼을 사용, ③ 옛 컬럼/코드 정리. Rolling/Canary로 옛 버전과 새 버전이 공존해도 깨지지 않도록 하는 것이 핵심입니다(backward-compatible deploys).
Feature Flag의 단점은?
① 분기 코드가 늘어 복잡도와 테스트 조합이 폭발할 수 있고, ② 끄지 못한 플래그가 누적되면 dead 코드가 쌓이며, ③ 잘못된 플래그 토글은 즉각적인 운영 사고로 이어질 수 있습니다. 주기적으로 플래그를 청소하고, 토글 권한과 변경 로그를 관리해야 합니다.
"내 컴퓨터에선 됐는데" 문제를 CI/CD가 어떻게 해결하나요?
① 빌드/테스트가 깨끗한 컨테이너 환경에서 실행되어 환경 차이를 제거합니다. ② 의존성 버전이 lock 파일로 고정되어 동일하게 설치됩니다. ③ 통합 테스트가 Testcontainers 같은 도구로 실제 의존(DB·MQ)을 띄워 운영과 가까운 환경을 재현합니다. 결과적으로 "환경 변수 차이로 운영에서만 실패"가 줄어듭니다.
