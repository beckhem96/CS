테스트 전략

테스트 전략은 "무엇을 어떤 비율로 자동화할 것인가"를 정하는 가이드입니다. 테스트 피라미드를 기준으로 빠르고 격리된 단위 테스트를 많이, 통합 테스트를 적당히, 느린 E2E 테스트를 소수만 두는 것이 일반적인 균형입니다. 여기에 테스트 더블(Mock·Stub·Spy·Fake)과 TDD가 도구로 들어옵니다.

구체적인 개념 설명

테스트는 단순히 버그를 찾는 도구가 아니라 변경에 대한 안전망입니다. 좋은 테스트가 있으면 리팩터링·라이브러리 업그레이드·새 기능 추가가 자신감 있게 진행되고, 없으면 변경 자체가 두려워집니다.

모든 코드를 같은 방식으로 테스트할 필요는 없습니다. 외부 의존이 없는 순수 로직은 단위 테스트로, 여러 컴포넌트가 협력하는 흐름은 통합 테스트로, 사용자 시나리오는 E2E로 — 층마다 책임이 다릅니다.

비유 — 자동차 점검

엔진 부속을 따로 시험하는 작업(단위), 엔진과 변속기를 합쳐 시험하는 작업(통합), 그리고 실제로 도로를 달려보는 시험(E2E)이 있습니다. 도로 시험 한 번이 가장 현실적이지만 가장 비싸고 느려서, 부속 단계를 미리 충분히 시험해 도로 시험에서 큰 문제가 안 나오게 만드는 것이 합리적입니다.

왜 필요한가?

  • 리그레션 방어 — 새 변경이 기존 동작을 깨지 않는다는 증거.
  • 설계 피드백 — 테스트가 어렵다 = 결합이 강하다는 신호.
  • 문서 — 테스트는 의도를 가장 정확하게 보여주는 살아있는 명세.
  • 배포 자신감 — CI에서 자동 검증이 통과해야 배포가 풀린다.

어떻게 동작하는가?

1) 테스트 피라미드

아래로 갈수록 많고 빠르고 격리되어 있고, 위로 갈수록 적지만 사용자에 가깝다.

단위 테스트(Unit Test)

  • 한 클래스/메서드의 순수 로직을 검증.
  • 외부 의존(DB, HTTP, 시계, 랜덤)은 가짜로 대체.
  • 밀리초 단위로 실행. 수천 개라도 수 초 안에 끝나야 한다.

통합 테스트(Integration Test)

  • 여러 컴포넌트가 함께 동작하는 경계를 검증(예: 컨트롤러 + 서비스 + 리포지토리 + DB).
  • Testcontainers로 진짜 DB·MQ를 띄워 환경 재현성을 높임.
  • 단위보다 느리지만 운영과 가까운 자신감을 준다.

E2E(End-to-End) 테스트

  • 사용자 시나리오 전체를 브라우저/디바이스 단으로 검증.
  • flaky해지기 쉬워(타이밍·환경) 핵심 시나리오만 골라야 한다.
  • 도구: Playwright, Cypress, Selenium.

2) 좋은 단위 테스트 — F.I.R.S.T 원칙

  • Fast — 빠르다(밀리초).
  • Isolated — 격리되어 있다(다른 테스트의 영향을 받지 않는다).
  • Repeatable — 어디서 몇 번 돌려도 같은 결과.
  • Self-validating — pass/fail이 자동으로 결정된다.
  • Timely — 코드를 작성할 때 함께 작성한다.

3) 테스트 더블(Test Double) 5종

Dummy/Stub/Spy/Mock/Fake — 모두 "테스트 더블"이라는 한 가족이지만 의도가 다르다.
  • Dummy — 메서드 시그니처를 만족시키기 위해 넘기는 가짜. 실제로 호출되지 않는다.
  • Stub — 미리 정해진 값을 반환한다. "이 메서드를 호출하면 X를 줄게"의 가짜.
  • Spy — 진짜처럼 동작하면서 호출 기록을 남겨 나중에 검증.
  • Mock — "이 메서드가 N번 호출되어야 한다"처럼 행동을 기대하고 그 기대를 검증.
  • Fake — 단순화된 진짜 구현(예: in-memory DB 리포지토리). 통합 테스트의 실용적 대안.
// JUnit 5 + Mockito — Stub과 Mock 차이
@Test
void 결제_성공시_주문상태_변경() {
    PaymentGateway gateway = mock(PaymentGateway.class);
    when(gateway.charge(any())).thenReturn(true);          // ← Stub: 정해진 값 반환

    OrderRepository repo = mock(OrderRepository.class);
    OrderService svc = new OrderService(gateway, repo);

    svc.checkout(orderOf(10_000));

    verify(repo).save(argThat(o -> o.status() == PAID));   // ← Mock: 행동 검증
}

4) TDD(Test-Driven Development)

"테스트 → 코드 → 리팩터" 사이클을 짧게 반복합니다.

  1. RED — 실패하는 테스트를 먼저 작성한다(아직 코드 없음).
  2. GREEN — 테스트가 통과하도록 가장 단순한 코드를 작성한다.
  3. REFACTOR — 중복 제거·이름 개선. 테스트는 통과를 유지.

TDD의 진짜 가치는 "테스트가 많아진다"가 아니라 "테스트가 가능한 설계가 자연스럽게 만들어진다"입니다. 외부 의존이 명확히 분리되고 SRP·DIP가 자연스럽게 따라옵니다.

5) BDD와 Given-When-Then

행위 중심 개발(BDD)은 비즈니스 시나리오를 자연어에 가까운 형식으로 표현합니다. "Given 조건 / When 행위 / Then 결과"의 3단 구조는 단위 테스트에서도 가독성을 높이는 유용한 템플릿입니다. 

@Test
void 잔액부족이면_출금실패한다() {
    // Given
    Account a = new Account("A", 1_000);
    // When & Then
    assertThatThrownBy(() -> a.withdraw(2_000))
        .isInstanceOf(IllegalStateException.class)
        .hasMessageContaining("잔액 부족");
}

6) 무엇을 테스트하지 말 것인가

  • 구현 디테일 — private 메서드, 내부 자료구조. 리팩터링에 깨진다.
  • 표준 라이브러리·프레임워크 — 이미 검증된 영역.
  • getter/setter — 의미 있는 행동이 없다.
  • 외부 시스템 그 자체 — 우리 서비스의 경계를 모킹/페이크로 끊고, 외부와의 계약은 별도 컨트랙트 테스트로.

7) 추가 테스트 종류

  • Snapshot 테스트 — UI/JSON의 출력 변화를 자동 비교.
  • Property-based 테스트 — 입력을 무작위로 생성해 불변식이 항상 성립하는지(예: jqwik).
  • Mutation 테스트 — 코드 일부를 변형해도 테스트가 잡아내는지(테스트의 테스트).
  • Contract 테스트 — 서비스 간 API 계약을 양쪽이 동시에 검증(예: Pact).
  • 부하/성능 테스트 — k6, Gatling 등.
  • 보안 테스트 — SAST/DAST, 의존성 취약점 스캔.

시간/공간 복잡도 또는 trade-off

층속도비용자신감왜 그런가?
단위매우 빠름저좁음(부분만)외부 의존을 가짜로 대체
통합중중중간실제 컴포넌트 묶음 검증
E2E느림고높음(사용자 시점)전체 환경을 띄워야
피라미드 역전(아이스크림 콘)——비효율위쪽이 두꺼우면 깨지기 쉽고 느려진다

실무에서 어디에 쓰이는가?

  • CI 파이프라인 — PR마다 단위·통합 테스트 자동 실행, 일정 커버리지 미만이면 빌드 실패.
  • 코드 리뷰 체크리스트 — "테스트가 동반되었는가"가 표준 항목.
  • 레거시 보강 — 변경 전에 특성화 테스트(characterization test)로 현재 동작을 잠궈둔 뒤 리팩터링.
  • 릴리스 게이트 — E2E 핵심 시나리오를 staging에서 통과해야 배포 진행.

면접에 나올 질문과 답변

테스트 피라미드와 아이스크림 콘 안티패턴의 차이는?
피라미드는 단위 테스트가 가장 많고 위로 갈수록 줄어드는 모양입니다. 빠르고 안정적입니다. 아이스크림 콘은 그 반대로 E2E가 많고 단위가 적은 모양으로, 테스트가 느리고 flaky해 변경 시 자신감이 떨어집니다. 피라미드를 지향하고, E2E는 핵심 시나리오에만 한정해야 합니다.
Mock과 Stub의 차이는?
Stub은 "호출되면 무엇을 반환할지"만 설정하는 가짜이고, Mock은 "어떤 호출이 일어나야 하는지"의 행동을 기대하고 검증합니다. Mockito의 when(...).thenReturn(...)은 stub 동작이고, verify(...)는 mock 검증입니다. 과도한 mock은 구현에 강하게 결합되어 리팩터링을 어렵게 합니다.
TDD의 진짜 효과는 무엇인가요?
커버리지 증가가 아니라 테스트 가능한 설계로 자연스럽게 유도되는 것입니다. 외부 의존이 명확히 분리되고 책임이 작게 쪼개지면서 SRP·DIP가 따라옵니다. 결과적으로 변경에 강한 코드와 회귀 방어망을 동시에 얻습니다.
private 메서드는 어떻게 테스트하나요?
직접 테스트하지 말고, 그 메서드를 호출하는 public 진입점을 통해 검증합니다. private을 직접 테스트하고 싶다는 충동이 든다면 보통은 그 책임을 별도 클래스로 분리할 신호입니다. 분리하면 그 책임은 새 클래스의 public 메서드로 노출되어 자연스럽게 테스트할 수 있습니다.
통합 테스트에서 진짜 DB를 띄우는 게 좋은가요?
대부분 좋습니다. H2 같은 가짜 DB는 운영 DB와 SQL 방언·트랜잭션 동작이 다를 수 있어 통합 테스트의 자신감을 떨어뜨립니다. Testcontainers로 운영과 같은 PostgreSQL/MySQL을 컨테이너로 띄우면 운영과 거의 같은 환경에서 검증할 수 있습니다. 실행 시간은 늘어나지만, 운영 장애를 막는 가치가 보통 더 큽니다.
테스트가 자주 깨져 무시되는 상황은 어떻게 해결하나요?
깨지는 원인이 flaky(타이밍/환경)인지 의미 있는 회귀인지를 먼저 분리해야 합니다. flaky 테스트는 격리하고 원인을 추적해 안정화하거나 다른 층으로 옮깁니다(E2E를 통합으로 등). 끄는 것이 가장 나쁜 선택 — 깨진 창문 효과로 다른 테스트도 신뢰를 잃습니다.
