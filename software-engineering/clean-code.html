클린 코드 · 리팩터링 · 코드 리뷰

클린 코드는 동작뿐 아니라 의도가 한눈에 드러나는 코드, 리팩터링은 동작을 바꾸지 않고 구조만 다듬는 작업, 코드 리뷰는 그 두 가지를 팀이 함께 지키도록 만드는 협업 활동입니다. 셋은 분리된 기법이 아니라 "유지보수 가능한 코드"라는 같은 목표를 향한 세 가지 통로입니다.

구체적인 개념 설명

코드의 진짜 비용은 처음 짤 때가 아니라 3개월 뒤 다른 사람이(또는 미래의 내가) 읽고 고칠 때에 발생합니다. 클린 코드는 그 미래의 비용을 미리 줄이는 활동입니다. 변수 이름을 의도가 드러나게 짓고, 함수를 한 가지 일만 하게 자르고, 매직 넘버에 이름을 붙이는 것 — 이런 작은 결정들이 쌓여 "고치기 두려운 코드"와 "안심하고 고치는 코드"를 가릅니다.

리팩터링은 그렇게 다듬는 행위 자체에 이름을 붙인 것입니다. 핵심은 "동작을 바꾸지 않는다"는 약속입니다. 그래서 리팩터링은 항상 "녹색 테스트"가 뒷받침되어야 하고, 코드 리뷰는 그 약속이 잘 지켜졌는지 팀이 함께 확인하는 자리입니다.

한 줄 요약 — "고치는 비용을 미리 깎는 활동"

  • 클린 코드 — 의도가 드러나는 작은 단위로 쪼갠다.
  • 리팩터링 — 동작은 그대로 두고 구조만 다듬는다(테스트가 안전망).
  • 코드 리뷰 — 미래의 자기를 도와줄 기회를 팀이 함께 만든다.

왜 필요한가?

  • 읽는 시간이 쓰는 시간보다 훨씬 길다 — 코드는 한 번 쓰이고 수십 번 읽힙니다. 읽기 비용을 줄이는 것이 가장 큰 절약입니다.
  • 변경 비용을 낮춘다 — 좋은 구조는 새 요구사항을 작은 변경으로 흡수합니다. 나쁜 구조는 한 줄 수정에 며칠을 쓰게 만듭니다.
  • 버그를 줄인다 — 의도가 명확한 코드는 잘못 가정하기 어렵고, 잘 분리된 코드는 한 변경이 다른 곳을 망가뜨리지 않습니다.
  • 지식 공유 — 리뷰를 통해 시니어의 판단 기준이 팀에 퍼지고, 새 멤버가 도메인을 빠르게 익힙니다.
  • 심리적 안전감 — "고쳐도 괜찮다"는 확신이 있어야 개선이 누적됩니다. 클린 코드 + 테스트 + 리뷰가 그 확신의 기반입니다.

어떻게 동작하는가?

1) 좋은 이름 — 가장 싸고 강력한 도구

변수·함수·클래스 이름은 그 자체로 작은 문서입니다. 좋은 이름은 주석을 거의 없게 만들고, 잘못된 이름은 어떤 주석으로도 만회할 수 없습니다. 좋은 이름의 두 가지 기준은 "의도가 드러나는가"와 "잘못된 가정을 하지 않게 하는가"입니다. 

// 나쁨 — 의도가 드러나지 않음
List<User> list = userRepo.findAll().stream()
    .filter(u -> u.getDays() > 30)
    .toList();

// 좋음 — 무엇을 고르는지가 이름에 드러남
List<User> inactiveUsers = userRepo.findAll().stream()
    .filter(this::isInactive)
    .toList();

private boolean isInactive(User user) {
    return user.daysSinceLastLogin() > INACTIVE_THRESHOLD_DAYS;
}
  • 의미가 들어가는 이름 — data, tmp, handle 같은 이름은 책임을 숨깁니다.
  • 일관된 어휘 — 같은 개념엔 같은 단어 (fetch·get·load 섞지 않기).
  • 거짓말하지 않기 — userList가 실제로는 Set이라면 이름을 바꾸거나 자료구조를 바꾸기.
  • 매직 넘버에 이름 — if (age > 19) → if (age > ADULT_AGE).

2) 함수는 작게, 한 가지 일만

함수가 길다는 건 "한 함수가 너무 많은 일을 한다"는 신호입니다. 작은 함수는 이름이 곧 설명이 되고, 테스트가 쉽고, 재사용도 자연스럽습니다. "이 함수의 이름을 한 단계 위 추상화로 부를 수 있는가"가 좋은 분해의 신호입니다. 

// 나쁨 — 한 함수가 검증·변환·저장·알림을 다 함
public void registerUser(SignupRequest req) {
    if (req.email() == null || !req.email().contains("@")) throw ...
    if (req.password().length() < 8) throw ...
    User user = new User(req.email(), hash(req.password()), Instant.now());
    userRepo.save(user);
    mailer.send(user.email(), "환영합니다");
}

// 좋음 — 단계별 함수로 분해
public void registerUser(SignupRequest req) {
    validate(req);
    User user = createUser(req);
    userRepo.save(user);
    notifyWelcome(user);
}

3) 코드 스멜 — 무엇을 보고 의심할 것인가

스멜증상대처
긴 함수한 함수가 50줄 넘음, 들여쓰기 4단 이상기능 단위로 추출 (Extract Function)
큰 클래스필드·메서드가 너무 많고 책임이 섞임책임별로 클래스 분리 (Extract Class)
중복 코드같은 패턴이 3곳 이상공통 함수/클래스로 추출, 단 첫 두 번은 참고 안 해도 됨
긴 매개변수 목록인자 5개 이상, 같이 다니는 값들객체로 묶기 (Introduce Parameter Object)
Feature Envy한 클래스가 다른 클래스의 데이터만 만지작그 메서드를 데이터가 있는 쪽으로 옮기기 (Move Method)
주석 의존코드 자체가 안 읽혀 주석으로 변명이름 개선 + 함수 추출로 주석을 사라지게
의미 없는 이름doIt, process, util역할이 드러나는 이름으로 변경
거대 if/switch분기가 길고 새 분기 추가가 두려움다형성·전략 패턴으로 대체

4) 안전한 리팩터링 절차

리팩터링은 "작은 변경 → 테스트 → 커밋"의 짧은 사이클을 반복하는 것이다.
  • 특성화 테스트(characterization test) — 레거시 코드는 명세가 없으니 현재 동작을 그대로 적어 두는 테스트부터 만든다.
  • 한 번에 한 가지 — 이름 바꾸기와 함수 추출을 같은 커밋에 섞지 말 것. 문제가 생겼을 때 어디서 깨졌는지 알 수 없다.
  • 되돌릴 수 있는 단위로 커밋 — 작게 자주 커밋하면 실패해도 한 단계만 되감으면 된다.
  • IDE 자동 리팩터링 우선 사용 — Rename·Extract Method 등은 IDE가 안전하게 처리한다. 손으로 하면 실수 위험이 크다.

5) 효과적인 코드 리뷰 — 무엇을 보고 어떻게 말하는가

코드 리뷰의 목적은 결함 잡기뿐이 아닙니다. 지식 공유 · 도메인 합의 · 일관된 스타일 · 후배 성장이 함께 일어나는 자리입니다. 효과적인 리뷰의 핵심은 "사람이 아닌 코드를 비평한다"와 "검토 가능한 작은 PR"입니다.

  • PR을 작게 (300줄 이하 권장) — 리뷰어의 집중력에는 한계가 있다. 큰 PR은 표면적 리뷰로 끝난다.
  • "왜"를 PR 설명에 적기 — 코드는 "무엇"을 보여 주지만 "왜"는 보여 주지 않는다.
  • "It vs You" — "이 함수는 책임이 두 개로 보입니다"가 "당신은 SRP를 모릅니다"보다 낫다.
  • 강한 의견은 근거와 함께 — "이건 안 됩니다" 대신 "이 경우 X가 깨질 것 같은데 어떻게 생각하시나요?"
  • nit·suggestion·blocker 구분 — 머지 차단 의견인지 단순 제안인지 명확히.
  • 자동화로 줄일 것은 자동화 — 포매터·린터·정적 분석이 잡을 일을 사람이 잡지 않기.

6) 주석 — 언제 쓰고 언제 쓰지 말 것인가

좋은 주석은 코드가 못하는 말을 한다: 왜 이렇게 결정했는지, 어떤 제약이 숨어 있는지, 어떤 비명백한 가정 위에 서 있는지. 나쁜 주석은 코드가 이미 하는 말을 반복한다. 

// 나쁨 — 코드가 이미 말하고 있음
i = i + 1;  // i를 1 증가시킨다

// 좋음 — 코드가 말하지 못하는 "왜"를 적음
// retry-after 헤더가 음수일 수 있어 0으로 보정 (서버 시계 오차 워크어라운드)
delay = Math.max(0, parseRetryAfter(headers));
  • 피해야 할 주석 — 코드 반복, 죽은 코드 보존, 변경 이력(VCS가 한다), 거짓이 된 옛 주석.
  • 가치 있는 주석 — "왜" 설명, 외부 명세 인용, 비명백한 trade-off, 임시 해결책의 만료 조건.
  • TODO에는 사람·기한 — // TODO(name, 2026-06): 캐시 일관성 재검토처럼 책임자와 시점을 적기.

trade-off

설계 결정장점비용
매우 작은 함수로 분해이름이 곧 설명, 테스트·재사용 쉬움호출 그래프가 깊어지면 흐름 추적 비용 ↑
리팩터링을 자주코드가 항상 깨끗, 변경 비용 낮음피처 진척이 느려 보임 (관리·일정 압박)
큰 PR로 일괄 리뷰맥락 한 번에 파악리뷰 집중력 저하, 결함 누락 ↑
작은 PR로 자주리뷰 품질 ↑, 머지 빠름맥락 단절, 잦은 컨텍스트 스위칭
주석 많이 달기의도 보존옛 주석이 거짓말이 되면 더 위험
자동 리팩터링 도구만 사용안전, 빠름도구 한계로 못 하는 변환은 손으로 다시

실무에서 어디에 쓰이는가?

  • PR 기반 워크플로 — GitHub/GitLab 위에서 코드 리뷰 → CI 통과 → 머지가 표준입니다.
  • 린터·포매터 — Prettier, Black, ESLint, Spotless가 스타일 논쟁을 코드 리뷰에서 배제합니다.
  • 정적 분석 — SonarQube, ErrorProne, ruff가 흔한 버그·스멜을 자동 검출.
  • 레거시 모더나이제이션 — 특성화 테스트 → 작은 리팩터링 → 점진적 교체 (Strangler fig).
  • 온보딩 — 신규 멤버의 첫 PR이 리뷰를 거치며 팀 규칙·도메인을 학습.
  • 회고·코드 표준 문서 — 리뷰에서 반복되는 의견은 문서로 옮겨 다음 PR이 바로 학습하도록.

면접에 나올 질문과 답변

"클린 코드"가 무엇이라고 생각하나요?
동작뿐 아니라 의도가 한눈에 드러나, 다음 사람이 변경할 때 두려움이 없는 코드입니다. 의미가 드러나는 이름, 한 가지 일만 하는 작은 함수, 일관된 어휘, 죽은 코드가 없는 상태가 그 형태입니다. 클린 코드는 미적인 취향이 아니라 미래의 변경 비용을 미리 깎는 활동이며, 테스트·리팩터링·리뷰와 함께 작동할 때 효과가 가장 큽니다.
리팩터링을 안전하게 진행하려면 어떻게 해야 하나요?
먼저 동작을 보호할 테스트를 확보하고(없으면 현재 동작을 그대로 적는 특성화 테스트부터 만듭니다), 한 번에 한 가지 변환만 적용한 뒤 즉시 테스트를 돌려 녹색을 확인하고 작은 단위로 커밋합니다. IDE의 자동 리팩터링(Rename·Extract Method)을 적극 활용하면 손으로 하다가 발생하는 실수를 줄일 수 있고, 문제가 생기면 즉시 되돌릴 수 있을 만큼 작게 자르는 것이 핵심입니다.
코드 스멜을 몇 가지 들고 어떻게 대처하시나요?
대표적으로 긴 함수(→ 함수 추출), 중복 코드(→ 공통 추출), 긴 매개변수 목록(→ 매개변수 객체로 묶기), Feature Envy(→ 메서드를 데이터가 있는 쪽으로 이동), 거대 if/switch(→ 다형성·전략 패턴) 등이 있습니다. 스멜은 즉시 대처해야 하는 버그가 아니라 신호이므로, 변경이 잦은 영역에서 만나면 그때 다듬는 식으로 점진적으로 정리합니다.
효과적인 코드 리뷰의 원칙은 무엇인가요?
PR을 작게 유지하고(300줄 이하 권장), PR 설명에 "왜"를 적으며, 코멘트는 사람이 아닌 코드에 향하게 합니다. 강한 의견엔 근거를 함께 제시하고, blocker·suggestion·nit를 구분해 머지 차단인지 단순 제안인지 명확히 합니다. 스타일은 포매터·린터에 맡기고, 사람의 시간은 도메인 의사결정과 설계에 집중시키는 것이 핵심입니다.
주석은 언제 쓰고 언제 쓰지 말아야 하나요?
코드가 말하지 못하는 "왜"를 설명할 때만 가치가 있습니다 — 왜 이런 결정을 했는지, 어떤 외부 제약이 있는지, 어떤 비명백한 가정 위에 서 있는지가 그 예입니다. 반대로 코드가 이미 말하는 "무엇"을 반복하는 주석, 죽은 코드 보존용 주석, 거짓이 된 옛 주석은 오히려 해롭습니다. 주석을 추가하기 전에 "이름이나 함수 추출로 코드 자체가 이 말을 하게 만들 수 없는가?"를 먼저 묻습니다.
리뷰에서 의견 충돌이 생기면 어떻게 해결하시나요?
먼저 의견을 데이터로 환원하려 합니다 — "이 변경이 어떤 시나리오에서 어떤 결과를 만드는가"를 같이 짚습니다. 그래도 결정이 안 되면 팀 규칙·기존 패턴·외부 가이드라인을 근거로 잡고, 그 사례에 한해 결정을 하나 내린 뒤 결과를 회고에서 점검합니다. 중요한 건 그 결정이 다음에 또 충돌하지 않도록 팀 코드 표준 문서로 옮기는 것입니다.
