디자인 패턴 입문(GoF 23)

디자인 패턴은 자주 등장하는 설계 문제에 대한 이름 붙은 해법 목록입니다. GoF(Gang of Four)가 정리한 23개 패턴은 목적에 따라 생성(Creational) · 구조(Structural) · 행위(Behavioral) 세 묶음으로 나뉘며, 모두 OOP의 다형성·합성·인터페이스를 활용해 변화에 강한 코드를 만드는 도구입니다.

구체적인 개념 설명

비슷한 설계 문제를 만나는 사람들이 비슷한 해법을 반복적으로 떠올리다 보니, "이 상황에는 보통 이렇게 푸는 것이 좋더라"가 패턴으로 굳어졌습니다. 패턴은 정답이 아니라 이름 붙은 어휘입니다 — 같은 어휘를 쓰면 팀이 더 빨리 같은 그림을 공유할 수 있습니다.

처음 패턴을 배울 때 흔한 함정은 "패턴을 적용하기 위한 코드"를 짜는 것입니다. 패턴은 문제가 보일 때 꺼내드는 도구이지, 모든 코드를 패턴 모양으로 다듬을 이유는 없습니다. 그래서 "어떤 변화가 자주 발생하는가"를 먼저 보고 그에 맞는 패턴을 고르는 순서가 옳습니다.

비유 — 요리 레시피북

"재료가 이렇게 있을 때 보통 이런 요리를 하더라"의 모음이 레시피북이라면, 디자인 패턴은 코드 설계의 레시피북입니다. 레시피를 안다고 모든 음식을 만들 필요는 없지만, 비슷한 상황을 만났을 때 처음부터 고민하지 않아도 되죠.

왜 필요한가?

  • 공통 어휘 — "Strategy로 빼자", "Decorator 한 겹 추가" 한 마디로 의도가 전달된다.
  • 검증된 구조 — 비슷한 문제에서 반복적으로 잘 동작한 형태.
  • SOLID 적용 도구 — OCP·DIP를 코드에서 자연스럽게 만들도록 도와준다.
  • 리팩터링 목적지 — 안티패턴을 가진 코드를 어디로 옮길지를 알려준다.

어떻게 동작하는가?

1) 세 카테고리 한눈에

생성=만들기, 구조=합치기, 행위=협력. 이 세 줄로 23개를 분류해 외운다.

2) 생성(Creational) — 객체 생성을 캡슐화

패턴의도대표 예시
Singleton한 클래스에 인스턴스 하나만 보장로그 매니저, 설정 객체
Factory Method구체 클래스 결정을 서브클래스로 위임List.of(), Calendar.getInstance()
Abstract Factory관련된 객체군을 함께 생성UI 룩&필 (Mac/Windows 위젯 가족)
Builder복잡한 객체를 단계적으로 조립StringBuilder, Lombok @Builder
Prototype기존 객체를 복제해 새 객체 생성JS의 prototype, Object.clone()
 
// Builder — 큰 객체를 가독성 있게 조립
HttpRequest req = HttpRequest.newBuilder()
        .uri(URI.create("https://api.example.com"))
        .header("Authorization", "Bearer ...")
        .timeout(Duration.ofSeconds(5))
        .GET()
        .build();

3) 구조(Structural) — 객체를 합치는 형태

패턴의도대표 예시
Adapter호환되지 않는 인터페이스를 변환레거시 API 래퍼
Decorator객체에 책임을 동적으로 덧붙임Java I/O BufferedInputStream
Proxy접근 제어·지연 로딩 등 대리자JPA의 lazy loading, RPC stub
Facade복잡한 서브시스템에 단순한 입구Spring의 JdbcTemplate
Composite부분-전체 트리 구조를 동일하게UI 트리, 파일 시스템
Bridge추상과 구현을 분리해 독립 변경JDBC API ↔ 드라이버
Flyweight다수의 작은 객체를 공유로 절약String 풀, 폰트 글리프 캐시
 
// Decorator — 기능을 한 겹씩 덧입히기
Reader r = new BufferedReader(
              new InputStreamReader(
                 new FileInputStream("data.txt"),
                 StandardCharsets.UTF_8));
// File 읽기 → 인코딩 변환 → 버퍼링이 차례로 추가

4) 행위(Behavioral) — 객체 사이의 협력 방식

패턴의도대표 예시
Strategy알고리즘을 갈아끼울 수 있게정렬 비교자, 결제 수단
Observer상태 변화를 구독자에게 통지UI 이벤트, 발행/구독
Command요청을 객체로 캡슐화Undo/Redo, 작업 큐
State객체의 상태에 따라 동작이 변함주문 상태머신
Template Method알고리즘 골격은 부모, 일부는 자식Spring JdbcTemplate hook
Iterator내부 표현 노출 없이 순회Java Iterator
Chain of Responsibility요청을 처리자 체인으로 전달Servlet Filter, 미들웨어
Mediator객체 사이 통신을 중재자로 모음채팅 룸 서버
Memento상태 스냅샷을 저장/복원편집기의 undo 스택
Visitor구조를 그대로 두고 동작을 추가AST 위 코드 분석기
Interpreter도메인 언어의 문법 트리 해석정규식 엔진, SQL 파서
 
// Strategy — 알고리즘 갈아끼우기
interface SortStrategy { void sort(int[] a); }
class QuickSort implements SortStrategy { public void sort(int[] a) { ... } }
class MergeSort implements SortStrategy { public void sort(int[] a) { ... } }

class Sorter {
    private final SortStrategy s;
    Sorter(SortStrategy s) { this.s = s; }
    void run(int[] a) { s.sort(a); }
}

// Observer — 발행/구독
interface Observer { void onEvent(Event e); }
class Topic {
    private final List<Observer> subs = new ArrayList<>();
    public void subscribe(Observer o) { subs.add(o); }
    public void publish(Event e) { subs.forEach(s -> s.onEvent(e)); }
}

5) "Strategy"와 "State"의 차이

구조는 거의 같지만 의도가 다릅니다. Strategy는 외부에서 골라주는 알고리즘이고, State는 객체 내부 상태에 따라 자동으로 바뀌는 동작입니다. Strategy는 클라이언트가 의도적으로 갈아끼우고, State는 객체가 스스로 상태 전이를 관리합니다.

6) 패턴을 어떻게 고를까 — 변화의 축

"코드의 어느 부분이 자주 바뀌는가?"를 먼저 보고, 그 변화 축에 맞는 패턴을 고릅니다.

  • 알고리즘만 바뀐다 → Strategy
  • 객체 생성 방식이 다양하다 → Factory / Builder
  • 기능을 누적해 덧붙이고 싶다 → Decorator
  • 요청 자체를 저장·재생하고 싶다 → Command
  • 상태에 따라 다르게 동작한다 → State
  • 외부 시스템 인터페이스가 다르다 → Adapter / Facade
  • 접근/생성 비용을 제어 → Proxy / Flyweight

7) 모던 자바와 패턴

  • 람다 / 메서드 레퍼런스로 Strategy·Command·Observer를 인터페이스 1메서드로 간단히.
  • 레코드 / sealed class가 Visitor·State 패턴과 잘 어울림(switch 표현식).
  • Stream이 Iterator를 한 단계 추상화.
  • Optional이 Null Object 패턴의 변형.
  • Spring 같은 IoC 컨테이너가 Factory·Singleton·Strategy 결합을 대신 처리.

시간/공간 복잡도 또는 trade-off

  • 가독성 vs 유연성 — 패턴은 유연성을 주지만 흐름 추적이 어려워질 수 있다. 단순한 곳엔 안 쓰는 게 낫다.
  • 클래스 수 폭증 — 인터페이스 + 구현체 + 팩토리 + 컨텍스트 등 파일이 늘어남.
  • 추상화 비용 — 가상 호출, 객체 생성 비용이 약간 증가.
  • 학습 곡선 — 팀이 같은 패턴 어휘를 공유하지 않으면 오히려 의사소통 비용이 늘어난다.

실무에서 어디에 쓰이는가?

  • Spring — Factory(BeanFactory), Singleton(빈 스코프), Proxy(@Transactional, AOP), Template Method(JdbcTemplate).
  • Java I/O — Decorator(Stream/Reader 체이닝), Adapter(InputStreamReader).
  • UI 프레임워크 — Composite(트리), Observer(이벤트), Strategy(layout, 정렬).
  • 정적 도구 — Visitor(AST 분석), Interpreter(파서), Iterator(컬렉션 순회).
  • 메시징/이벤트 — Observer/Mediator로 약한 결합 유지.

면접에 나올 질문과 답변

디자인 패턴 카테고리 3가지와 각 카테고리의 의도는?
생성(Creational)은 객체 생성 방식을 캡슐화해 변화에 대비하는 패턴(Factory, Builder, Singleton 등), 구조(Structural)는 객체를 결합해 더 큰 구조를 만들거나 인터페이스를 변환하는 패턴(Adapter, Decorator, Proxy 등), 행위(Behavioral)는 객체 간 책임 분배와 협력 방식을 다루는 패턴(Strategy, Observer, State 등)입니다.
Strategy와 State 패턴의 차이는?
구조는 비슷하지만 의도가 다릅니다. Strategy는 외부에서 알고리즘을 선택해 주입하는 패턴이고, State는 객체 내부 상태에 따라 동작이 자동으로 바뀌며 객체 스스로 상태 전이를 관리하는 패턴입니다. "내가 원하는 알고리즘으로 갈아끼우는가" vs "객체의 상태에 따라 동작이 자연스럽게 바뀌는가"로 구분합니다.
Singleton의 단점은?
① 전역 상태가 되어 테스트 격리를 어렵게 합니다. ② 멀티스레드 환경에서 잘못 구현하면 race condition이 발생합니다(이중 검증 락, 정적 홀더, enum 등으로 해결). ③ 의존을 숨겨 결합을 늘립니다(필요한 곳에 명시적으로 주입하는 게 보통 더 낫다). 실무에선 Spring의 빈 스코프가 사실상 Singleton 역할을 안전하게 대신해 줍니다.
Decorator와 Proxy의 차이는?
구조는 같지만 목적이 다릅니다. Decorator는 기능을 동적으로 추가하는 데 초점이 있고, Proxy는 접근 제어·지연 로딩·원격 호출 같이 객체 접근을 통제하는 데 초점이 있습니다. Java I/O의 BufferedReader는 Decorator의 예, JPA의 lazy proxy는 Proxy의 예입니다.
디자인 패턴을 외워서 다 적용하면 좋은가요?
아닙니다. 패턴은 변화가 자주 일어나는 곳에 꺼내 드는 도구이지, 모든 코드를 패턴 모양으로 다듬어야 하는 건 아닙니다. 과도하게 적용하면 클래스 수가 폭발하고 흐름이 흐려집니다. "이 변화 축이 실제로 자주 발생하는가"를 먼저 보고 적용하세요.
현대 자바·프레임워크에서 사라진/덜 보이는 패턴이 있나요?
함수형 인터페이스와 람다가 들어오면서 Strategy·Command·Observer는 명시적 클래스 없이 람다로 표현되는 경우가 많아졌습니다. DI 컨테이너가 Factory·Singleton 결합을 대신 처리해 직접 작성하는 일도 줄었습니다. sealed class와 pattern matching이 들어오면서 Visitor의 자리도 줄어드는 추세입니다.
