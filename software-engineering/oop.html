OOP(Object-Oriented Programming)

OOP는 데이터(상태)와 그것을 다루는 동작(메서드)을 객체로 묶고, 객체끼리 메시지를 주고받으며 협력하도록 만드는 프로그래밍 패러다임입니다. 네 기둥인 캡슐화·상속·다형성·추상화로 변화에 강한 코드를 만드는 도구를 제공합니다.

구체적인 개념 설명

절차적 코드는 데이터와 그 데이터를 처리하는 함수가 떨어져 있어, 같은 데이터를 만지는 함수가 늘어날수록 영향 범위를 추적하기가 어려워집니다. OOP는 데이터와 그것을 다루는 동작을 한 객체로 묶고 외부에서는 정해진 통로로만 만지게 해, 변경의 영향을 객체 안에 가둡니다.

OOP의 핵심은 "클래스"가 아니라 "객체 사이의 책임 분배"입니다. 어떤 객체가 어떤 책임을 가지고, 다른 객체에 무엇을 부탁할지를 결정하는 것이 OOP 설계의 본질입니다.

비유 — 식당과 주방

손님(클라이언트)은 직접 주방에 들어가지 않고 점원에게 주문(메시지)을 합니다. 주방의 내부 절차는 바뀌어도 점원의 인터페이스(메뉴)만 같으면 손님은 아무 영향이 없죠. 객체와 메시지는 정확히 이 그림입니다.

왜 필요한가?

  • 변경의 국소화 — 어떤 데이터의 처리 방식이 바뀌어도 그 객체 내부만 바뀌면 된다.
  • 재사용 — 잘 정의된 객체와 인터페이스는 다른 곳에서 그대로 쓸 수 있다.
  • 유연한 확장 — 새로운 종류의 동작을 기존 코드를 거의 안 건드리고 추가(다형성).
  • 모델링의 자연스러움 — 현실 세계의 "역할/책임"을 코드에 그대로 옮기기 쉽다.

어떻게 동작하는가?

1) 클래스와 객체

클래스(Class)는 설계도, 객체(Object)는 그 설계도로 만든 인스턴스입니다. 클래스는 어떤 상태(필드)와 동작(메서드)이 있을지 정의하고, 객체는 실제 메모리에 그 상태를 가진 채로 존재합니다. 

class Account {
    private final String id;       // 상태(필드)
    private long balance;

    public Account(String id) {     // 생성자
        this.id = id;
        this.balance = 0;
    }
    public void deposit(long amount) { // 동작(메서드)
        if (amount <= 0) throw new IllegalArgumentException();
        balance += amount;
    }
    public long getBalance() { return balance; }
}

Account a = new Account("A001");   // 객체 생성(인스턴스화)
a.deposit(10_000);

2) 네 기둥(Four Pillars)

네 기둥은 서로를 받쳐 변화에 강한 구조를 만든다.

① 캡슐화(Encapsulation)

상태를 객체 안에 숨기고(private), 외부에서는 메서드를 통해서만 접근하도록 만듭니다. 불변식(예: 잔액은 음수가 될 수 없음)을 객체가 직접 지키므로 잘못된 상태가 만들어지지 않습니다. 

// 잘못된 예: 상태를 외부에 그대로 노출 → 누구나 잔액을 직접 바꿔버림
public class BadAccount { public long balance; }

// 좋은 예: 상태는 숨기고, 검증된 동작만 노출
public class Account {
    private long balance;
    public void withdraw(long amount) {
        if (amount > balance) throw new IllegalStateException("잔액 부족");
        balance -= amount;
    }
}

② 상속(Inheritance)

공통 속성과 행위를 부모 클래스로 추출하고, 자식 클래스가 이를 물려받아 확장하거나 특수화합니다. 코드 재사용에 유용하지만 강한 결합을 만들기 때문에 남용하면 변경이 어려워집니다 — 그래서 "상속보다 합성(composition)"이 자주 권장됩니다. 

abstract class Shape { abstract double area(); }
class Circle    extends Shape { double r; double area() { return Math.PI * r * r; } }
class Rectangle extends Shape { double w, h; double area() { return w * h; } }

③ 다형성(Polymorphism)

같은 메시지(메서드 호출)에 대해 객체의 실제 타입에 따라 다른 동작이 일어납니다. 덕분에 클라이언트 코드는 구체 타입에 의존하지 않고도 다양한 객체를 다룰 수 있습니다.

  • 서브타입 다형성 — 부모 타입 변수에 자식 객체를 담고 메서드 호출.
  • 매개변수 다형성 — 제네릭(List<T>).
  • 오버로딩 — 같은 이름, 다른 시그니처(컴파일타임).
  • 오버라이딩 — 부모 메서드를 자식이 재정의(런타임).
List<Shape> shapes = List.of(new Circle(...), new Rectangle(...));
for (Shape s : shapes) {
    System.out.println(s.area());   // 같은 호출, 객체에 따라 다른 결과
}

④ 추상화(Abstraction)

세부를 숨기고 핵심만 인터페이스로 드러냅니다. 추상 클래스나 인터페이스가 도구이며, "이 객체에게 무엇을 부탁할 수 있는가"를 정의하는 것이 곧 추상화입니다. 

interface PaymentGateway {
    boolean charge(String customerId, long amount);
}
// 구현체는 Stripe·Toss 등 다양하지만, 호출자는 PaymentGateway만 안다.

3) 객체와 메시지

OOP의 본질은 클래스가 아니라 객체 간 협력입니다. 객체는 자기 책임만 지고 다른 책임은 다른 객체에 부탁합니다(메시지). Smalltalk를 만든 앨런 케이의 표현으로 "OOP는 메시지에 관한 것이지 객체에 관한 게 아니다"라고 하기도 합니다.

4) is-a vs has-a — 상속 vs 합성

"하위 종류"는 상속, "구성 요소를 가짐"은 합성. 잘못 매핑하면 결합이 부풀어 변경 비용이 커진다.

"X는 Y의 일종이다"가 자연스러우면 상속, "X는 Y를 가진다"가 자연스러우면 합성입니다. 현대 OOP는 합성을 우선하라고 권장합니다 — 상속은 컴파일 시점에 부모와 강하게 묶이지만, 합성은 런타임에 실제 협력자를 갈아끼울 수 있어 유연합니다.

5) 추상 클래스 vs 인터페이스 (Java)

추상 클래스인터페이스
의도"공통 부모""능력/계약"
다중 구현단일 상속다중 구현 가능
상태(필드)가질 수 있음상수만(public static final)
구현 코드일부 구현 포함default/static 메서드 가능
대표 사례Template MethodStrategy, Comparable, Runnable

6) 다형성을 활용한 OCP — 확장에 열려 있고 변경에 닫혀 있다

// 새 결제 수단을 추가해도 Checkout은 안 바뀐다 — OCP
interface PaymentGateway { void charge(long amount); }
class CardGateway   implements PaymentGateway { ... }
class TossGateway   implements PaymentGateway { ... }
class KakaoGateway  implements PaymentGateway { ... }   // 새로 추가

class Checkout {
    private final PaymentGateway gateway;
    public Checkout(PaymentGateway g) { this.gateway = g; }
    public void pay(long amount) { gateway.charge(amount); }
}

7) OOP의 함정

  • 상속 남용 — 단지 코드 재사용이 목적이라면 합성을 먼저 검토.
  • Anemic Domain Model — 객체가 데이터 그릇이 되고 로직은 외부 서비스에 흩어져 결국 절차적 코드가 되는 안티패턴.
  • God Object — 모든 책임을 떠안은 거대한 클래스. SRP 위반의 전형.
  • 깊은 상속 트리 — 변경이 트리 전체로 퍼져 LSP 위반이 생기기 쉬움.

시간/공간 복잡도 또는 trade-off

  • 구조화 vs 단순함 — 객체를 잘게 나누면 변경에 강하지만 작은 코드는 오히려 복잡해 보일 수 있다.
  • 다형성의 비용 — 가상 호출(virtual dispatch)은 직접 호출보다 약간 느리고 인라이닝을 막을 수 있다(JIT가 대체로 잘 처리).
  • 상속과 결합 — 부모를 바꾸면 자식 모두에 영향. 깊을수록 위험.
  • 도메인과 OOP의 거리 — 데이터 처리 중심 도메인(분석·ML)에서는 함수형이 더 적합할 수 있다.

실무에서 어디에 쓰이는가?

  • 도메인 모델링 — 엔티티와 값 객체로 비즈니스 규칙을 표현(DDD).
  • 플러그인/전략 — 결제 수단, 알림 채널, 외부 API 어댑터.
  • UI 컴포넌트 — 위젯·뷰의 상태와 동작을 묶음.
  • 프레임워크 확장점 — Spring의 BeanPostProcessor, Servlet Filter 등 인터페이스 기반 확장.

면접에 나올 질문과 답변

OOP의 네 기둥을 설명해 주세요.
캡슐화는 상태와 동작을 객체에 묶고 외부 접근을 제어해 불변식을 지키는 것, 상속은 공통을 부모로 추출해 자식이 확장/특수화하는 관계, 다형성은 같은 메시지에 대해 객체의 실제 타입에 따라 다르게 동작하는 능력, 추상화는 세부를 숨기고 인터페이스로 핵심만 드러내는 설계 도구입니다.
왜 "상속보다 합성"이라고 하나요?
상속은 컴파일 시점에 부모와 강하게 묶여 부모 변경이 자식 모두에 영향을 주고, 행위 변경이 런타임에 어렵습니다. 합성은 협력 객체를 필드로 가지므로 인터페이스만 같다면 런타임에 다른 구현으로 갈아끼울 수 있어 변경에 강합니다. 그래서 "is-a"가 명확하지 않다면 합성으로 시작하는 것이 안전합니다.
오버로딩과 오버라이딩의 차이는?
오버로딩은 같은 이름의 메서드를 시그니처(파라미터 타입/개수)만 다르게 여러 개 두는 것으로 컴파일 시점에 결정됩니다. 오버라이딩은 부모 메서드를 자식이 재정의하는 것으로, 실제 호출은 객체의 런타임 타입에 따라 결정됩니다(동적 디스패치).
추상 클래스와 인터페이스 중 무엇을 쓸지 어떻게 결정하나요?
"공통 부모로 묶고 일부 구현을 공유하고 싶다"면 추상 클래스, "능력/계약을 정의하고 그 능력을 가진 누구나 구현하게 하고 싶다"면 인터페이스를 씁니다. Java는 다중 상속이 안 되지만 인터페이스는 여러 개 구현 가능하므로, 다양한 객체에 같은 능력을 부여할 때는 인터페이스가 적합합니다.
캡슐화는 단순히 private 필드를 두는 건가요?
아닙니다. 진짜 캡슐화는 객체가 자기 불변식을 스스로 지키는 것입니다. 필드를 private으로 만들고 setter를 무작정 열면 결국 외부에서 마음대로 상태를 바꿔 캡슐화는 사라집니다. 대신 의미 있는 동작(예: withdraw())을 노출해 그 안에서 검증을 해야 합니다.
OOP가 항상 정답인가요?
아닙니다. 데이터 변환 중심의 작업(파이프라인·분석·ML)은 함수형이 더 명료할 수 있고, 시스템 프로그래밍은 절차적/구조적 접근이 적합한 경우가 많습니다. OOP는 도메인의 책임을 객체로 잘 매핑할 수 있을 때 빛납니다. 다른 패러다임과 섞어 쓰는 것이 현실적입니다.
