Java 언어 기본 — JVM·JDK·JRE와 컴파일·실행 흐름

Java는 JVM(Java Virtual Machine) 위에서 실행되는 정적 타입 언어로, 소스 코드를 중간 표현인 바이트코드(bytecode)로 컴파일한 뒤, 어떤 OS에서든 그 OS에 맞는 JVM이 바이트코드를 해석·실행합니다. "Write Once, Run Anywhere"라는 슬로건이 이 두 단계 컴파일·실행 모델에서 나옵니다.

기준: Java 21 (LTS). 별도 표시가 없으면 OpenJDK / HotSpot JVM을 가정합니다.

구체적인 개념 설명

Java가 다른 언어와 다른 결정적 차이는 "한 번 컴파일하면 어디서든 같은 결과를 낸다"는 점입니다. C/C++는 OS·CPU별로 따로 컴파일해 네이티브 실행 파일을 만들어야 하지만, Java는 .java를 .class(바이트코드)로 한 번만 만들면 윈도우/리눅스/맥의 JVM이 알아서 자기 OS에 맞게 실행합니다.

또 하나의 기둥은 관리되는 런타임(managed runtime)이라는 점입니다. JVM이 메모리 할당과 회수(GC), 보안 검증, 동적 최적화(JIT)를 책임지므로 개발자는 "객체를 만들고 참조를 끊으면 끝"이라는 단순한 모델로 일할 수 있습니다. 단, 이 편의는 GC 일시 정지·메모리 사용량·기동 시간이라는 비용으로 돌아옵니다.

비유 — 항공권과 환승 공항

소스 코드는 "한국에서 발권한 항공권(.java)", 바이트코드는 "어느 나라에서든 통하는 표준 e-티켓(.class)", JVM은 "각 공항의 게이트(OS별 JVM)"입니다. 티켓 양식만 표준이면 게이트(OS) 종류가 달라도 환승이 가능합니다.

왜 필요한가?

  • 플랫폼 독립성 — OS·CPU가 바뀌어도 같은 .class가 그대로 동작.
  • 안전성 — 바이트코드 검증기(verifier)가 잘못된 명령(잘못된 캐스팅·스택 오버/언더플로 등)을 실행 전에 거부.
  • 동적 최적화 — JIT 컴파일러가 핫 메서드를 런타임에 네이티브 코드로 컴파일해 C에 가까운 속도를 낸다.
  • 관리되는 메모리 — GC가 객체 수명을 알아서 추적하므로 메모리 누수·이중 해제 같은 클래식한 버그가 줄어든다.
  • 강한 정적 타입 — 컴파일 시점에 타입 오류를 잡아 런타임에서 노출되는 버그를 줄인다.

어떻게 동작하는가?

1) JDK / JRE / JVM의 관계

입문자가 가장 먼저 헷갈리는 세 약어부터 정리합니다. 크기 관계는 JDK ⊃ JRE ⊃ JVM이고, 각자 책임이 다릅니다.

JDK는 "개발하려면 필요한 모든 것", JRE는 "실행만 하면 되는 환경", JVM은 "바이트코드를 실제로 돌리는 엔진"이다.
  • JVM — .class를 메모리에 로딩, 검증, 실행. GC와 JIT가 여기서 동작.
  • JRE — JVM + 표준 라이브러리(java.lang, java.util, java.io …). "남이 만든 jar를 돌리기만 하면 되는" 환경.
  • JDK — JRE + 컴파일러(javac) · 디버거(jdb) · 문서 도구(javadoc) · jlink(커스텀 런타임 만들기) 등 개발 도구.

참고 — 최근 변화

Java 11 이후로 Oracle은 JRE 단독 배포를 사실상 중단했고, 보통 JDK만 받아서 그 안의 java 명령으로 실행합니다. 대신 jlink로 "내 앱이 쓰는 모듈만 골라 만든 미니 JRE"를 빌드해 배포 크기를 줄이는 방식이 권장됩니다.

2) 컴파일·실행 흐름 — .java에서 화면에 "Hello"가 뜨기까지

소스 → 바이트코드 → JVM 내부에서 로딩·검증·실행. 같은 .class가 OS만 다른 JVM 위에서 그대로 돈다.

실제 명령으로는 다음 흐름입니다.

// Hello.java
public class Hello {
    public static void main(String[] args) {
        System.out.println("Hello, " + (args.length > 0 ? args[0] : "world"));
    }
}

// 1) 컴파일: 소스를 바이트코드로
$ javac Hello.java          // Hello.class 생성

// 2) 실행: JVM이 .class를 읽어 실행
$ java Hello Java
Hello, Java

// (Java 11+) 단일 파일이면 컴파일 단계를 생략하고 바로 실행도 가능
$ java Hello.java Java

바이트코드는 사람이 읽기 어렵지만 도구로 들여다볼 수 있습니다. javap -c Hello를 실행하면 다음과 비슷한 JVM 명령(니모닉)이 보입니다. 

// javap -c Hello (발췌)
public static void main(java.lang.String[]);
  Code:
     0: getstatic  #2 // Field java/lang/System.out:Ljava/io/PrintStream;
     3: ldc        #3 // String Hello, world
     5: invokevirtual #4 // Method java/io/PrintStream.println:(Ljava/lang/String;)V
     8: return

스택 머신에서 동작하는 단순한 명령 집합이라, 어느 OS의 JVM이든 같은 의미로 해석할 수 있습니다.

3) 클래스 로딩 단계 — Loading → Linking → Initialization

JVM은 .class를 한 번에 메모리로 다 올리지 않고, 처음 쓸 때 게으르게(lazy) 가져옵니다. 이때 거치는 단계를 알면 "왜 static 필드가 그 시점에 초기화되는가?" 같은 질문에 답할 수 있습니다.

"준비"에서는 static 필드가 기본값(0/null)으로만 자리잡고, 실제 대입은 "초기화"에서 일어난다 — 면접 단골.

클래스 로더는 위임 모델(parent-first delegation)로 동작합니다. 자식 로더는 직접 찾지 않고 부모에게 먼저 물어보며, 이로써 java.lang.String 같은 핵심 클래스를 누구도 위조할 수 없게 만듭니다(부트스트랩 로더가 먼저 응답).

4) Interpreter + JIT — "느리다"는 오해

JVM은 처음에는 바이트코드를 한 줄씩 해석(interpret)해 빠르게 시작합니다. 그러다 같은 메서드가 자주 호출되면(=hot) JIT(Just-In-Time) 컴파일러가 그 메서드를 네이티브 코드로 컴파일해 캐시합니다. 다음 호출부터는 컴파일된 네이티브 코드가 실행돼 C에 가까운 속도가 납니다.

  • C1(Client) — 빠르게 컴파일하지만 최적화 약함. 워밍업 단계용.
  • C2(Server) — 컴파일이 느린 대신 인라이닝·escape analysis 같은 강한 최적화로 정상 가속.
  • Tiered Compilation — Java 8+ 기본. C1 → C2 단계적으로 올리는 방식.

이 때문에 Java 프로그램은 시작 직후 1~2초가 가장 느리고("warm-up"), 시간이 지나면 점점 빨라집니다. 마이크로벤치마크를 짤 때 첫 1만 회를 버려야 하는 이유입니다.

5) 정적 타입과 기본 자료형 vs 참조형

Java는 변수에 타입을 명시하고(var는 추론), 컴파일러가 타입 호환성을 검사합니다. 자료형은 두 갈래로 나뉩니다.

  • 기본 자료형(Primitive) — boolean / byte / short / int / long / float / double / char. 값 자체를 스택이나 객체 필드에 직접 저장. 8개로 끝.
  • 참조형(Reference) — 클래스·배열·인터페이스. 변수에는 객체의 주소(참조)가 들어가고, 객체 본체는 힙에 저장.
int  a = 10;        // 기본형: a라는 메모리에 10이 직접 들어감
String s = "hi";    // 참조형: s에는 힙의 String 객체를 가리키는 참조가 들어감

Integer boxed = 10; // auto-boxing: int → Integer (힙 객체)
int unboxed = boxed;// auto-unboxing: Integer → int

// 같은 값처럼 보여도 박싱은 객체이므로 == 비교에 주의
Integer x = 1000, y = 1000;
System.out.println(x == y);        // false! (캐시 범위 밖)
System.out.println(x.equals(y));   // true

Integer 같은 박스 타입은 -128~127 범위만 캐시하기 때문에 그 밖의 값은 새 객체가 만들어져 == 비교가 깨집니다 — 단골 면접 함정.

6) 진입점 — main 메서드와 args

public class App {
    public static void main(String[] args) {  // 이 시그니처를 JVM이 찾는다
        // ...
    }
}

JVM은 클래스 로드 후 정확히 public static void main(String[]) 시그니처의 메서드를 찾아 호출합니다. 시그니처가 다르면 "Main method not found". Java 21의 미리보기 기능에서는 클래스/한정자 없이 void main()만 써도 되도록 단순화하는 방향이 검토되고 있습니다.

7) 패키지·클래스패스 — 같은 이름이 충돌하지 않게

클래스명만으로는 전 세계 코드와 충돌하므로, Java는 도메인 역순 패키지명(com.example.shop)으로 네임스페이스를 만듭니다. JVM이 클래스를 찾는 경로가 classpath(또는 Java 9+의 module path)이며, 이 경로 안에 패키지 디렉터리 구조 그대로 .class가 놓여 있어야 합니다. 

// src/main/java/com/example/shop/Order.java
package com.example.shop;

public class Order { ... }

// 사용처
import com.example.shop.Order;
// 또는: import com.example.shop.*;

시간/공간 복잡도 또는 trade-off

특성이득비용
바이트코드 + JVM플랫폼 독립, 안전성네이티브보다 시작이 느리고 메모리 더 씀
JIT 동적 최적화실행 정보 기반의 강한 최적화워밍업 시간, 컴파일 스레드 부담

관리되는 런타임 전반보안·관측성 좋고 생태계 거대경량 CLI나 단명 프로세스에 무거움(GraalVM AOT로 보완 가능)

실무에서 어디에 쓰이는가?

  • 웹 백엔드 — Spring Boot, Quarkus, Micronaut. 대형 트래픽 서버의 주력.
  • 안드로이드 — Android Runtime(ART)은 JVM과 다르지만 Java/Kotlin이 1차 언어.
  • 데이터/인프라 — Hadoop, Kafka, Spark, Elasticsearch, Cassandra가 모두 JVM 위에서 동작.
  • 금융·통신·공공 시스템 — 호환성과 안정성 때문에 수십 년 운영되는 코드 베이스가 많음.
  • 빌드/도구 — Gradle, Maven, IntelliJ IDEA 자체도 JVM 앱.

면접에 나올 질문과 답변

JDK, JRE, JVM의 차이를 설명해 주세요.
JVM은 바이트코드를 로딩·검증·실행하는 가상 머신이고, JRE는 JVM에 표준 라이브러리를 더해 "Java 앱을 실행만 하기 위한 환경"입니다. JDK는 JRE에 javac·jdb·javadoc 같은 개발 도구를 더해 "개발자가 컴파일·디버그까지 할 수 있는 키트"입니다. 포함 관계는 JDK ⊃ JRE ⊃ JVM입니다.
"Write Once, Run Anywhere"가 가능한 이유는?
Java 컴파일러는 OS·CPU에 종속적인 네이티브 코드 대신 표준화된 바이트코드를 생성합니다. 이 바이트코드를 각 OS용으로 만든 JVM이 해석·실행하므로 같은 .class를 윈도우/리눅스/맥에서 그대로 돌릴 수 있습니다. 즉, 플랫폼 차이를 JVM이라는 한 단계가 흡수합니다. 다만 OS 콜·파일 경로 같은 외부 의존이 있으면 코드 레벨에서도 호환에 신경 써야 합니다.
바이트코드를 직접 본 적 있나요? 어떻게 보나요?
javap -c -p Foo.class로 바이트코드 디스어셈블 결과를 볼 수 있습니다. JVM은 스택 머신이라 iload, iconst, invokevirtual 같은 피연산자가 거의 없는 단순 명령으로 표현됩니다. 이 단순함 덕에 검증기가 안전성을 빠르게 검사할 수 있고, 어떤 JVM이든 같은 의미로 해석할 수 있습니다.
클래스 로딩의 단계를 설명해 주세요.
Loading(클래스 파일을 읽어 Class 객체 생성) → Linking(Verify·Prepare·Resolve) → Initialization(static 필드의 실제 값 대입과 static {} 블록 실행)입니다. Prepare 단계에서 static 필드가 기본값(0/null)으로만 잡히고 실제 대입은 Initialization에서 이뤄지는 점이 핵심입니다. 그리고 Initialization은 클래스가 "처음 능동적으로 사용될 때" 일어나므로, 단순히 클래스가 메모리에 로드된다고 static이 즉시 실행되는 건 아닙니다.
JIT 컴파일러란?
실행 중에 자주 호출되는 메서드(hot method)를 런타임에 네이티브 코드로 컴파일해 두고 다음부터 그 네이티브 버전을 실행하는 기술입니다. Java 8+의 HotSpot은 C1(빠른 컴파일·약한 최적화)과 C2(느린 컴파일·강한 최적화)를 결합한 Tiered Compilation을 기본으로 씁니다. 덕분에 Java는 시작은 인터프리터처럼 빠르고, 정상 운영 시점에는 네이티브에 가까운 속도를 냅니다. 단, 시작 직후 워밍업 비용이 있어 JMH 같은 벤치마크 도구가 워밍업 라운드를 따로 둡니다.
Integer를 ==로 비교하면 왜 결과가 들쭉날쭉한가요?
Integer는 객체이므로 ==는 참조(주소) 비교입니다. JVM은 -128~127 범위의 박싱 값을 캐시해 같은 객체를 재사용하므로 그 범위에서는 ==도 true가 나오지만, 그 밖에서는 새 객체가 매번 생성돼 false가 됩니다. 값 비교는 항상 equals() 또는 intValue()로 해야 합니다.
Java가 느리다는 인식이 아직도 맞나요?
20년 전 인터프리터 위주의 초기 JVM 인상이 남아 있는 것으로, 지금의 Java는 강한 JIT와 정교한 GC 덕분에 정상 운영 상태에서 C/C++에 근접한 처리량을 냅니다. 다만 두 가지 약점은 남아 있습니다 — (1) 시작 시간과 메모리 풋프린트가 큼, (2) 워밍업 직후 첫 호출이 느림. 서버리스나 CLI처럼 시작이 잦은 워크로드에는 GraalVM Native Image(AOT 컴파일)나 CRaC 같은 보조 기술이 쓰입니다.
