컬렉션 프레임워크 — List·Set·Map·Queue

컬렉션 프레임워크(Java Collections Framework, JCF)는 객체 그룹을 다루는 자료구조와 알고리즘을 통일된 인터페이스로 묶어 둔 표준 라이브러리입니다. 모든 구현은 Iterable 또는 Map 인터페이스에 뿌리를 두며, 어떤 자료구조 위에서 동작하든 같은 메서드 이름과 의미를 보장합니다.

구체적인 개념 설명

이 프레임워크가 만들어 준 가장 큰 가치는 "인터페이스로 받고, 구현체는 갈아끼운다"는 코딩 습관을 표준화한 것입니다. 메서드 시그니처를 List<Order>로 받으면 호출자는 ArrayList·LinkedList·CopyOnWriteArrayList 등 어떤 구현이든 넘길 수 있고, 본문은 그대로입니다.

각 구현체는 동일한 의미를 다른 방식으로 달성하므로 성능 특성이 다릅니다. ArrayList.get(i)은 O(1)이지만 LinkedList.get(i)은 O(n)이고, HashMap.get은 평균 O(1)이지만 TreeMap.get은 O(log n)입니다. 같은 인터페이스를 구현해도 "어떻게 동작하는가"는 자료구조 그 자체이므로, 알고리즘 페이지에서 다룬 내용을 여기서는 Java 관점으로 다시 봅니다.

자료구조 자체에 대한 설명은 자료구조 → 배열·연결 리스트·해시 테이블를 함께 보세요. 이 페이지는 Java 구현체의 차이와 함정에 초점을 둡니다.

왜 필요한가?

  • 표준 인터페이스 — 어디서든 같은 사용법으로 자료를 다룰 수 있다.
  • 구현체 교체 자유 — 시그니처가 인터페이스라면 성능·동시성 요구가 바뀌어도 한 줄로 교체.
  • 풍부한 알고리즘 — Collections.sort, Arrays.binarySearch처럼 표준 알고리즘이 같이 옴.
  • 제네릭 통합 — 컴파일 타임 타입 안전성, 런타임 캐스팅 제거.

어떻게 동작하는가?

1) 전체 구조 — 두 뿌리(Iterable, Map)

Map은 Collection이 아니라 별도 계층 — 키로 값을 찾는 도구라 "그룹"의 의미가 다르기 때문이다.

2) List — ArrayList vs LinkedList

연산ArrayListLinkedList
get(i)O(1)O(n)
add(e) — 끝에 추가amortized O(1)O(1)
add(0, e) — 앞에 추가O(n) (전부 한 칸씩 밀어야)O(1)
remove(i)O(n)O(n) (탐색 + 제거)
iteration매우 빠름 (캐시 친화)느림 (포인터 추적)
메모리요소당 1 슬롯요소당 노드 객체 + 양방향 포인터

이름이 주는 인상과 달리 실무에서는 거의 항상 ArrayList가 정답입니다. "삽입/삭제가 많으니 LinkedList"라는 말은 옛날 이야기고, 현대 CPU는 캐시 친화적인 연속 메모리 배치에 워낙 유리해서 LinkedList의 노드 추적 비용이 크기 때문입니다. LinkedList는 큐/덱(Deque)이 필요할 때만 쓰는 게 좋고, 그마저도 ArrayDeque가 보통 더 빠릅니다. 

// ArrayList의 내부 동작 — 동적 배열
List<Integer> list = new ArrayList<>();      // 기본 capacity 0, 첫 add에서 10
list.add(1); list.add(2); ... list.add(11);   // capacity 부족 → 1.5배로 확장 후 복사

// 미리 크기를 알면 초기 capacity를 주면 resize 비용 제거
List<Integer> list = new ArrayList<>(10_000);

// 불변 리스트
List<Integer> immut = List.of(1, 2, 3);       // add 시 UnsupportedOperationException

3) HashMap — 충돌·resize·treeify

HashMap은 Java 코드 어디서나 등장합니다. 핵심 메커니즘 세 가지를 알아야 합니다.

기본 16 버킷 + 0.75 load factor + 8개 이상 충돌 시 트리화 — Java 8 HashMap의 3대 마법.
  • 버킷 배열 — hash(key) % capacity 위치에 entry 저장.
  • 충돌 처리 — Separate Chaining(연결 리스트). 한 버킷에 8개 이상 쌓이고 capacity가 64 이상이면 Red-Black Tree로 변환해 최악 O(n) → O(log n).
  • Resize — size > capacity × loadFactor(0.75)이 되면 capacity를 2배로 늘리고 모든 entry를 재해싱. O(n) 비용이라 큰 맵에선 일시적 latency 폭증 원인.
// 미리 크기를 알면 capacity를 직접 지정 → resize 회피
Map<String, User> m = new HashMap<>(/* expected */ 10_000, 0.75f);

// hash() — Java 8부터 상위 16비트와 XOR로 분포 개선
static final int hash(Object key) {
    int h = key.hashCode();
    return h ^ (h >>> 16);
}

치명적 함정 — equals/hashCode

키로 쓰는 객체는 equals()와 hashCode()를 둘 다 일관되게 재정의해야 합니다. 한 쪽만 재정의하면 같은 키로 put한 값이 get에서 안 나옵니다. record는 자동 생성되므로 안전하고, 클래스라면 IDE 자동 생성을 쓰세요. 그리고 키가 mutable 객체라면 put 후 필드를 바꾸면 안 됩니다 — hash가 바뀌어 영영 못 찾는 "버킷 미아"가 됩니다.

4) TreeMap·LinkedHashMap — 순서가 필요할 때

구현순서get/put특징
HashMap없음 (보장 X)평균 O(1)가장 빠름. 디폴트 선택.
LinkedHashMap삽입 순서 (또는 access 순서)평균 O(1)이중 연결 리스트로 순서 유지. LRU 캐시 구현에 적합.
TreeMap키 정렬 순서O(log n)Red-Black Tree. range query, floor/ceiling 가능.
ConcurrentHashMap없음평균 O(1)동시성 안전. 락 분할(bin lock)로 전체 락 회피.
 
// LinkedHashMap으로 LRU 캐시 한 줄
class LRU<K, V> extends LinkedHashMap<K, V> {
    private final int capacity;
    LRU(int capacity) {
        super(16, 0.75f, /* accessOrder */ true);  // get할 때 맨 뒤로 이동
        this.capacity = capacity;
    }
    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        return size() > capacity;                  // 초과 시 가장 오래된 키 제거
    }
}

// TreeMap의 range
NavigableMap<Integer, String> tm = new TreeMap<>();
tm.floorKey(50);    // 50 이하 가장 큰 키
tm.subMap(10, 30);  // 10 <= k < 30

5) Set — Map의 키만 쓰는 친구

HashSet·LinkedHashSet·TreeSet은 내부적으로 각각 HashMap·LinkedHashMap·TreeMap의 키를 쓰는 얇은 래퍼입니다. 그래서 성능 특성도 동일합니다 — 중복을 막고 싶을 때 가장 자연스러운 선택입니다.

6) Queue·Deque — 양방향 큐

Deque는 양 끝에서 모두 추가·제거 가능한 큐입니다. 스택과 큐가 모두 필요하면 무조건 ArrayDeque — 옛 Stack은 동기화된 Vector 기반이라 느리고, LinkedList는 캐시 비친화적입니다. 

Deque<Integer> stack = new ArrayDeque<>();
stack.push(1); stack.push(2);
stack.pop();   // 2

Deque<Integer> queue = new ArrayDeque<>();
queue.offer(1); queue.offer(2);
queue.poll();  // 1

PriorityQueue<Integer> pq = new PriorityQueue<>();   // 최소 힙
pq.offer(3); pq.offer(1); pq.offer(2);
pq.poll();   // 1

7) Iterator와 fail-fast

Java의 컬렉션 대부분은 fail-fast iterator를 사용합니다. 순회 도중 컬렉션이 외부에서 수정되면 즉시 ConcurrentModificationException을 던져 일찍 실패합니다. 자주 발생하는 함정 — 순회 중 remove()는 반드시 iterator의 메서드로 해야 합니다. 

List<Integer> list = new ArrayList<>(List.of(1, 2, 3, 4));

// ❌ ConcurrentModificationException
for (Integer x : list) {
    if (x % 2 == 0) list.remove(x);
}

// ✅ Iterator의 remove
Iterator<Integer> it = list.iterator();
while (it.hasNext()) {
    if (it.next() % 2 == 0) it.remove();
}

// ✅ Java 8+ 더 깔끔
list.removeIf(x -> x % 2 == 0);

ConcurrentHashMap·CopyOnWriteArrayList의 iterator는 fail-safe로, 순회 시점의 스냅샷을 보거나 약한 일관성(weakly consistent)을 보장합니다. 예외는 안 나지만 새로 들어온 변경이 보일 수도 안 보일 수도 있다는 의미입니다.

9) 불변(immutable) 컬렉션

// Java 9+ — 불변 컬렉션 팩토리
List<Integer> l = List.of(1, 2, 3);
Set<String>  s = Set.of("a", "b");
Map<String, Integer> m = Map.of("a", 1, "b", 2);

// 기존 컬렉션의 불변 뷰 (수정 시도 시 예외)
List<Integer> view = Collections.unmodifiableList(list);

// copyOf — 입력의 변경과 무관한 진짜 복사본 (Java 10+)
List<Integer> copy = List.copyOf(list);

불변 컬렉션은 스레드 안전이 자동으로 보장되고, 방어적 복사가 필요 없으며, hashCode를 캐시할 수 있어 키로 쓰기에도 적합합니다. 외부에 컬렉션을 노출할 땐 가능한 한 불변으로 두세요.

실무에서 어디에 쓰이는가?

  • 일반 리스트 → ArrayList. 의심하지 말고 시작.
  • 중복 제거 → HashSet (순서 무관) / LinkedHashSet (입력 순서 유지).
  • 키-값 매핑 → HashMap (대부분), TreeMap (정렬·범위 쿼리), ConcurrentHashMap (동시 접근).
  • LRU 캐시 → LinkedHashMap의 accessOrder + removeEldestEntry.
  • 스택·큐·덱 → ArrayDeque.
  • 우선순위 작업 큐 → PriorityQueue / PriorityBlockingQueue.
  • 읽기 위주 + 외부 노출 → List.of / Map.of로 불변.

면접에 나올 질문과 답변

ArrayList와 LinkedList 중 무엇을 쓰나요?
대부분 ArrayList입니다. 인덱스 접근이 O(1)이고, 캐시 친화적인 연속 메모리 배치라 순회·검색이 압도적으로 빠릅니다. LinkedList는 양 끝 삽입/삭제가 O(1)이지만 노드마다 객체가 따로 만들어져 메모리·캐시 미스가 많고, 인덱스 접근이 O(n)입니다. 스택·큐·덱이 필요하면 LinkedList보다 ArrayDeque가 더 빠르므로, LinkedList의 자리는 사실상 거의 없습니다.
HashMap에 키를 넣었는데 get으로 못 찾는 경우가 있다는데?
equals와 hashCode의 일관성이 깨져 있을 때 발생합니다. 한쪽만 재정의했거나, mutable 키 객체를 put한 뒤 필드를 바꿔 hashCode가 변한 경우가 대표적입니다. record는 자동 생성이라 안전하고, 일반 클래스는 IDE의 generate equals/hashCode를 항상 함께 쓰세요.
HashMap이 한 버킷에 키가 많이 쌓이면 어떻게 되나요?
Java 8부터는 한 버킷의 충돌 항목 수가 8 이상이고 전체 capacity가 64 이상이면 그 버킷을 Red-Black Tree로 변환합니다(treeify). 이로써 최악 시간 복잡도가 O(n)에서 O(log n)으로 떨어져 의도적인 hash 충돌 공격(예: 오래전의 요청 파라미터 해시 충돌)에도 견디게 됐습니다. 충돌이 다시 6 이하로 줄면 untreeify로 다시 리스트가 됩니다.
HashMap의 resize는 언제 일어나고 왜 비용이 큰가요?
size > capacity × loadFactor(기본 0.75)가 되면 capacity를 2배로 늘립니다. 단순 메모리 확장이 아니라 모든 entry를 새 버킷으로 재해싱해야 해서 O(n)이 들고, 그 순간 모든 put이 그 비용을 부담하므로 큰 맵에선 latency 스파이크가 됩니다. 미리 크기를 알면 new HashMap<>(expectedSize / loadFactor + 1)처럼 capacity를 지정해 resize 자체를 피하는 게 좋습니다.
ConcurrentHashMap은 어떻게 동시성을 처리하나요?
Java 7까지는 16개 segment로 나눠 segment 단위 락을 잡았지만, Java 8부터는 segment를 없애고 버킷(bin) 단위 락 + CAS로 동작합니다. 없는 키 추가는 CAS로 락 없이 시도하고, 충돌이 나면 그 노드만 짧게 락을 잡습니다. iterator는 fail-safe라 순회 도중 다른 스레드가 수정해도 예외가 안 나며, 약한 일관성(weakly consistent) 스냅샷을 봅니다.
List를 순회하면서 요소를 지우려면?
for-each + list.remove()는 fail-fast iterator가 ConcurrentModificationException을 던집니다. 정답은 (1) Iterator를 직접 얻어 it.remove()를 쓰거나, (2) Java 8+의 list.removeIf(predicate)를 쓰는 것입니다. 후자가 가장 깔끔합니다. ArrayList의 removeIf는 내부적으로 한 번에 압축하므로 성능도 우수합니다.
LRU 캐시를 직접 구현해 보세요.
가장 간단한 방법은 LinkedHashMap을 accessOrder=true로 만들고 removeEldestEntry를 오버라이드하는 것입니다 — get할 때마다 맨 뒤로 옮겨지고, 크기를 넘으면 가장 오래 안 쓰인 키가 자동 제거됩니다. 면접에서 손코딩이 필요하면 HashMap + 이중 연결 리스트로 직접 구현하는 형태도 자주 나옵니다(get/put 모두 O(1) 보장).
불변 컬렉션과 unmodifiable 뷰의 차이는?
Collections.unmodifiableList는 래퍼 뷰라 원본을 수정하면 뷰에서도 보입니다. 즉 "이 핸들로는 수정 못하지만 원본이 바뀌면 노출"되는 셈입니다. 반면 List.copyOf나 List.of는 진짜 불변 객체로 원본 변경에 영향을 받지 않습니다. 외부에 안전하게 노출하고 싶다면 copyOf 쪽이 정답입니다.
