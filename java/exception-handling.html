예외 처리 — checked vs unchecked, try-with-resources

예외(Exception)는 "정상 흐름으로는 처리할 수 없는 비정상 상황"을 표현하는 객체입니다. Java는 예외를 일급 객체로 두고 메서드 시그니처에 던질 수 있는 예외를 명시(throws)하게 해, 호출자에게 "이 예외는 신경 써야 한다"는 신호를 컴파일 타임에 전달합니다.

구체적인 개념 설명

예외는 단순한 에러 코드가 아니라 객체이므로 메시지·원인 예외·스택 트레이스 같은 풍부한 정보를 담고, 스택을 풀면서(stack unwinding) 적절한 핸들러까지 전파됩니다. 이 메커니즘이 핵심입니다 — 호출 깊이가 100단계여도 어디선가 catch하기만 하면 그 사이의 코드는 자동으로 정리되고 제어가 핸들러로 점프합니다.

Java가 다른 언어와 가장 다른 점은 checked exception이라는 컴파일러 강제 검증입니다. "이 메서드는 IOException을 던질 수 있어"라고 시그니처에 적어 두면, 호출자는 try-catch로 잡거나 자기 시그니처에도 throws를 적어야만 컴파일됩니다. 강력한 안전망인 동시에 많은 비판도 받는 설계 결정입니다.

왜 필요한가?

  • 정상/예외 흐름 분리 — if 반환 코드 체크가 산만하게 흩어지지 않도록.
  • 호출자에게 강제 알림 — checked 예외로 "이 상황을 다뤄야 한다"는 의무를 컴파일러가 검사.
  • 자원 정리 보장 — finally / try-with-resources로 정상·예외 어느 경로든 자원 해제.
  • 맥락 보존 — 스택 트레이스로 "어디서 시작해 어디서 터졌는지" 정확히 추적.

어떻게 동작하는가?

1) Throwable 계층 — Error vs Exception vs RuntimeException

Error는 잡지 않고, Exception은 의무적으로 다루며, RuntimeException은 검사 대상이 아니다.
  • Error — JVM 자체의 비정상 상태(OutOfMemoryError, StackOverflowError). 보통 잡으려 시도하지 않는다.
  • Checked Exception (Exception 직속, RuntimeException 제외) — 컴파일러가 try-catch 또는 throws를 강제. 외부 자원/IO 같은 "회복 가능한" 실패를 표현.
  • Unchecked Exception (RuntimeException 계열) — 컴파일러가 검사하지 않음. 대부분 프로그래머의 실수(NPE, 잘못된 인자, 인덱스 범위 초과)를 나타냄.

2) try / catch / finally의 흐름

정상이든 예외든, 심지어 return·throw 도중에도 finally는 실행된다. 자원 해제의 마지막 보루. 
try {
    int x = parse(input);    // 예외 가능 지점
    use(x);
} catch (NumberFormatException e) {
    log.warn("형식 오류: {}", input, e);
    return DEFAULT;
} catch (IllegalStateException e) {
    log.error("상태 오류", e);
    throw e;                  // 다시 던지기
} finally {
    cleanup();                // 정상/예외 모두 실행
}

catch 블록은 위에서부터 매칭되며, 한 블록에 여러 타입을 묶는 multi-catch도 가능합니다. 

try { ... }
catch (IOException | SQLException e) {  // 둘 다 같은 식으로 처리
    log.error("외부 자원 실패", e);
}

3) try-with-resources — finally의 진화형

파일·DB 커넥션·소켓 같은 자원은 사용 후 반드시 닫아야 하는데, finally로 직접 쓰면 코드가 장황하고 버그도 생기기 쉽습니다. Java 7+의 try-with-resources는 AutoCloseable을 구현한 자원을 자동으로 닫아 줍니다. 

// 옛날 방식 — 5단계 들여쓰기, NPE 위험, 예외 가림
InputStream in = null;
try {
    in = Files.newInputStream(path);
    return read(in);
} finally {
    if (in != null) {
        try { in.close(); } catch (IOException ignored) {}
    }
}

// 현대 방식 — 자동 해제, 깔끔
try (InputStream in = Files.newInputStream(path)) {
    return read(in);
}
// in.close()가 try 종료 시 자동 호출됨

// 여러 자원도 OK — 선언 역순으로 닫힘
try (InputStream in = Files.newInputStream(src);
     OutputStream out = Files.newOutputStream(dst)) {
    in.transferTo(out);
}

try-with-resources의 또 하나의 장점은 예외 가림(suppressed exception) 처리입니다. 본문에서 IOException이 나고 close()에서도 또 예외가 나면 옛날 코드는 두 번째가 첫 번째를 덮어 쓰는데, try-with-resources는 두 번째를 e.getSuppressed()에 담아 둘 다 살려 줍니다.

4) checked vs unchecked — 어떻게 선택할까?

구분Checked ExceptionUnchecked Exception
상속Exception (RuntimeException 제외)RuntimeException 또는 Error
컴파일러처리/선언 강제검사 안 함
의도"호출자가 회복 가능""버그 또는 회복 불가"
예시IOException, SQLExceptionNPE, IllegalArgumentException
람다 친화도나쁨 (선언 안 된 함수형 인터페이스에 못 던짐)좋음

현대 Java 커뮤니티(Spring 포함)는 대부분의 비즈니스/도메인 예외를 unchecked로 두는 쪽으로 기울었습니다. checked는 다음과 같은 단점이 있기 때문입니다.

  • 람다·Stream에서 일급으로 다루기 어려움 (Function은 checked를 던질 수 없음).
  • 호출 체인 모두에 throws를 전염시킴 → 인터페이스 변경 비용이 큼.
  • "회복 가능한" 예외라는 설계 의도가 현실에서 거의 지켜지지 않음.

Spring은 SQLException(checked)을 DataAccessException(unchecked)으로 변환해 호출자에게 부담을 주지 않는 사례가 대표적입니다.

5) 예외 설계 원칙

① 예외에 의미 있는 이름과 정보를 담기

// 나쁨 — 무슨 일인지 모름
throw new RuntimeException("error");

// 좋음 — 도메인 의미가 드러남
public class InsufficientBalanceException extends RuntimeException {
    public InsufficientBalanceException(String accountId, long requested, long available) {
        super("계좌 %s: %d 요청 / %d 보유".formatted(accountId, requested, available));
    }
}

② 원인 예외(cause) 체이닝

낮은 계층의 기술 예외를 높은 계층의 도메인 예외로 변환할 때, 원인을 항상 보존하세요. 그래야 스택 트레이스에 "어디서 진짜로 터졌는지"가 남습니다. 

try {
    return jdbc.query(sql, ...);
} catch (SQLException e) {
    // 원인을 두 번째 인자로 넘김 — getCause()로 추적 가능
    throw new OrderRepositoryException("주문 조회 실패", e);
}

③ 빈 catch는 거의 항상 버그

try { ... }
catch (Exception e) { }   // ❌ 무엇이 어디서 사라졌는지 영영 알 수 없음

// 최소한 로깅 + 의미를 담은 처리
catch (Exception e) {
    log.warn("부가 작업 실패 — 본 흐름은 계속", e);
}

④ 너무 넓은 catch는 진짜 버그를 가린다

// ❌ 모든 예외를 한 묶음으로 — NPE까지 같은 처리
try { ... } catch (Exception e) { return null; }

// ✅ 다루는 의도에 맞는 타입만
try { ... } catch (IOException | TimeoutException e) { ... }

⑤ 예외를 흐름 제어로 쓰지 않기

// ❌ 예외로 분기 — 느리고 의도 불명
try {
    map.get(key);
} catch (NullPointerException e) { ... }

// ✅ 정상 흐름으로
if (map.containsKey(key)) { ... }
// 또는
Optional.ofNullable(map.get(key)).ifPresent(...);

6) 스택 트레이스 읽기

예외 메시지보다 스택 트레이스 자체를 읽는 능력이 더 중요합니다. 핵심 패턴 두 가지를 보세요. 

java.lang.NullPointerException: Cannot invoke "User.getName()" because "user" is null
    at com.example.app.OrderService.notify(OrderService.java:42)    ← 진짜 터진 곳
    at com.example.app.OrderService.place(OrderService.java:28)
    at com.example.app.WebController.create(WebController.java:55)
    ... 12 more
Caused by: java.sql.SQLTimeoutException: query timeout                ← 원인 예외
    at org.postgresql.jdbc.PgStatement.execute(PgStatement.java:478)
    ... 18 more
  • 맨 위가 가장 안쪽(예외가 던져진 지점), 아래로 내려갈수록 호출자.
  • "Caused by"가 진짜 원인을 보여줌 — 위 트레이스의 결론은 "DB 쿼리 타임아웃 → user가 null로 흘러옴 → NPE".
  • ... N more는 윗 트레이스와 겹치는 프레임이 N개 생략됐다는 뜻.
  • Java 14+의 Helpful NullPointerException이 켜져 있으면(JDK 기본) "which 변수가 null인지"까지 알려준다.

7) finally의 함정 — return은 finally에서 덮어쓰지 말 것

int risky() {
    try { return 1; }
    finally { return 2; }   // ❌ try의 return을 덮어버림 → 항상 2 반환
}

finally에서 return·throw를 하면 try의 결과를 조용히 덮어 버립니다. 진단도 안 되고 의도도 흐려지므로 finally는 정리 작업만 두세요.

시간/공간 복잡도 또는 trade-off

  • 예외는 비싸다 — new Exception() 자체가 스택 트레이스를 채우는 작업이라 일반 메서드 호출의 수십~수백 배. 흐름 제어로 쓰면 안 되는 이유.
  • checked의 경직성 — 인터페이스에 throws X가 새로 생기면 모든 구현·호출자에 영향.
  • 너무 넓은 try — 진단을 흐리고 catch 범위에 의해 의도치 않은 예외까지 삼킴.
  • 너무 좁은 try — 자원 해제 영역이 분산돼 누수 위험.

실무에서 어디에 쓰이는가?

  • 외부 자원 호출 경계 — DB·HTTP·파일 IO에서 try-with-resources + 예외 변환.
  • 도메인 예외 — 비즈니스 규칙 위반(잔액 부족, 중복 주문)을 의미 있는 unchecked로 던지고 컨트롤러에서 매핑.
  • 전역 예외 핸들러 — Spring의 @ControllerAdvice에서 도메인 예외 → HTTP 상태 코드 변환.
  • 관측성 — 예외와 함께 컨텍스트(요청 ID, 사용자 ID)를 MDC에 넣고 구조화 로그에 기록.

면접에 나올 질문과 답변

checked 예외와 unchecked 예외의 차이는?
checked는 Exception 하위(RuntimeException 제외)이며 컴파일러가 try-catch 또는 throws를 강제합니다. "호출자가 회복할 수 있는 외부 자원 실패"를 표현하기 위해 도입됐습니다. unchecked는 RuntimeException 계열로 컴파일러가 검사하지 않으며, NPE처럼 대부분 프로그래머 실수를 표현합니다. 현대 Java는 람다·Stream과의 충돌, 의무 throws의 전염성 때문에 도메인 예외를 unchecked로 두는 경향이 강합니다.
try-with-resources는 어떻게 동작하나요?
리소스가 AutoCloseable을 구현해야 하며, try 블록이 정상 종료되든 예외가 발생하든 컴파일러가 자동으로 close()를 호출하는 코드를 삽입합니다. 여러 자원은 선언 역순으로 닫히고, 본문 예외와 close 예외가 동시에 발생하면 본문 예외가 우선이고 close 예외는 getSuppressed()에 담겨 모두 보존됩니다. finally를 직접 작성하던 시절의 NPE 위험과 예외 가림 문제를 한꺼번에 해결합니다.
finally는 항상 실행되나요?
try나 catch에서 return·throw가 있어도 finally는 실행됩니다. 예외는 (1) JVM이 System.exit()으로 종료하거나, (2) 데몬 스레드가 죽거나, (3) 하드웨어 결함·OS 강제 종료처럼 JVM 자체가 멈출 때 정도입니다. 다만 finally에서 return/throw하면 try의 결과를 덮어 버려 진단을 어렵게 하므로, finally는 정리 작업만 두세요.
예외를 다시 던질 때(rethrow) 주의할 점은?
원인 예외(cause)를 반드시 보존해야 합니다 — throw new MyException("msg", e)처럼 원인을 넘겨 주면 스택 트레이스에 "Caused by"로 진짜 시작 지점이 남습니다. 또한 단순히 throw e로 다시 던질 거면 catch는 사실상 의미가 없으므로, 로깅·자원 정리·예외 변환 같은 추가 처리가 있을 때만 잡으세요.
왜 빈 catch 블록은 위험한가요?
예외가 발생했지만 아무도 모르는 상태가 됩니다. 같은 코드는 정상으로 보이지만 결과 데이터는 부분적으로 누락되거나 잘못된 상태일 수 있고, 운영 중 문제 추적이 거의 불가능해집니다. 최소한 로그라도 남기고, 처리할 수 없다면 다시 던지거나 도메인 예외로 변환해 의도를 명확히 해야 합니다.
예외를 흐름 제어로 쓰면 왜 안 되나요?
(1) 성능 — 예외 객체 생성과 스택 트레이스 채우기는 일반 호출 대비 수십~수백 배 비싸고, JIT 인라이닝도 깨집니다. (2) 가독성 — "정상이지만 예외 객체로 표현된 분기"는 코드 의도를 모호하게 만들어 리뷰어가 진짜 예외와 구분하지 못합니다. (3) 디버깅 — 진짜 비정상이 일어나도 로그가 묻혀 발견이 늦어집니다. 흐름 분기는 if·Optional·Result 타입으로 표현해야 합니다.
스택 트레이스를 어떻게 읽나요?
맨 윗줄이 예외가 실제로 던져진 곳이고 아래로 내려갈수록 호출자입니다. "Caused by"가 보이면 그 아래가 원인 예외의 트레이스로, 보통 그 가장 윗 줄이 진짜 시작점입니다. 라이브러리·프레임워크 프레임은 무시하고 본인 패키지(com.example.app...)만 따라 내려가면 빠르게 원인을 좁힐 수 있습니다. Java 14+에서는 NPE 메시지에 어떤 변수가 null이었는지 직접 적혀 나오므로(Helpful NPE), 메시지를 먼저 읽으세요.
throws Exception처럼 가장 상위 예외를 선언하면 안 되나요?
문법상 가능하지만 호출자에게 "어떤 예외든 나올 수 있다"는 무책임한 신호를 줍니다. catch 측에서도 모든 예외를 다 잡아야 하므로 핵심 예외와 일반 예외를 구분할 수 없게 됩니다. 가능한 한 의미가 분명한 가장 좁은 타입을 선언하고, 정말 불특정 다수를 던질 가능성이 있다면 도메인 예외로 변환해 호출자가 한 의미로 다룰 수 있게 하세요.
