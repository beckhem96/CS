제네릭(Generic) — 타입 소거·bounded wildcard·PECS

제네릭(Generics)은 클래스·인터페이스·메서드를 타입 매개변수로 일반화하는 기능입니다. 덕분에 컴파일 타임에 타입 안전성을 보장하면서도 캐스팅 코드를 없앨 수 있습니다. Java 제네릭은 런타임에 타입 정보가 사라지는(타입 소거, type erasure) 방식으로 구현돼, 이 점에서 다른 언어와 큰 차이가 납니다.

구체적인 개념 설명

제네릭이 없던 시절 Java 컬렉션은 모든 원소를 Object로 다뤘습니다. 꺼낼 때마다 캐스팅이 필요했고, list.add(integer) 후 (String) list.get(0)처럼 잘못 꺼내면 런타임에 ClassCastException이 터졌습니다. 제네릭의 본질은 "이 컬렉션은 어떤 타입을 담는다"는 약속을 시그니처에 박아 두고 컴파일러가 검사하게 하는 것입니다.

Java 제네릭의 가장 큰 특이점은 타입 소거(type erasure)입니다. 컴파일러는 타입 매개변수를 사용해 검사를 마친 뒤, 바이트코드에는 그 정보를 거의 남기지 않고 Object로 바꿔 둡니다. 호환성 — 즉 제네릭이 추가된 Java 5 이전의 코드와 같은 바이트코드 위에서 동작해야 한다는 제약 때문입니다. 이로 인해 new T[]가 안 되고, list instanceof List<String>도 안 되는 등의 한계가 생깁니다.

왜 필요한가?

  • 컴파일 타임 타입 안전성 — 잘못된 타입 추가를 컴파일러가 즉시 거부.
  • 캐스팅 제거 — (String) list.get(i) 같은 boilerplate 사라짐.
  • 재사용성 — 한 번 정의한 자료구조/알고리즘을 여러 타입에 그대로 사용.
  • 도구·문서 친화 — IDE 자동 완성, 리팩토링, javadoc이 정확한 타입을 안다.

어떻게 동작하는가?

1) 기본 형태 — 클래스/메서드 제네릭

// 제네릭 클래스 — T는 타입 매개변수
public class Box<T> {
    private T value;
    public void put(T v) { this.value = v; }
    public T   get()     { return value; }
}

Box<Integer> b1 = new Box<>();   // T = Integer
b1.put(42);
int x = b1.get();                 // 캐스팅 불필요

// 제네릭 메서드 — 클래스가 제네릭이 아니어도 메서드 단위로 도입 가능
public static <T> T firstOrDefault(List<T> list, T fallback) {
    return list.isEmpty() ? fallback : list.get(0);
}
String s = firstOrDefault(List.of("a", "b"), "");  // T = String 추론

관용적 타입 매개변수 이름 — T(Type), E(Element), K/V(Key/Value), R(Result), N(Number).

2) 타입 소거(Type Erasure) — 제네릭의 가장 중요한 사실

컴파일러가 제네릭으로 검사한 다음 바이트코드를 만들 때 타입 매개변수를 모두 지웁니다.

컴파일 타임에는 타입 안전, 런타임에는 raw List만 남는다 — 캐스팅은 컴파일러가 알아서 삽입.

이로 인해 다음과 같은 제약들이 생깁니다.

// ❌ T로 instanceof 못 함
if (obj instanceof List<String>) { ... }   // 컴파일 에러

// ❌ T로 배열 생성 못 함
T[] arr = new T[10];                       // 컴파일 에러
// 우회: Object[] 만들고 캐스팅 (경고 + 안전성 책임은 개발자)
T[] arr = (T[]) new Object[10];

// ❌ static 멤버에 클래스 타입 매개변수 사용 못 함
class Box<T> {
    static T defaultValue;                 // 컴파일 에러
}

// ❌ 같은 시그니처처럼 보여 오버로드 불가
void m(List<Integer> xs) { }
void m(List<String>  xs) { }              // erasure 후 같은 시그니처

3) Bounded Type Parameter — "T는 X의 하위여야 함"

<T extends Number>처럼 상한을 둘 수 있습니다. 인터페이스도 가능하고 여러 개도 가능합니다. 

public static <T extends Number> double sum(List<T> nums) {
    double s = 0;
    for (T n : nums) s += n.doubleValue();   // Number의 메서드 호출 가능
    return s;
}

// 다중 bound — class 먼저, interface 나중
<T extends Number & Comparable<T>>
T max(List<T> xs) { ... }

4) Wildcard — ?, ? extends T, ? super T

메서드 매개변수에 "어떤 타입 파라미터든 받을 수 있다"고 표현할 때 와일드카드를 씁니다. 핵심은 가변성(variance)입니다.

형태의미읽기쓰기
List<?>"무언가의 리스트" (unbounded)Object만null만
List<? extends Number>Number의 어떤 하위 타입의 리스트Number로 읽기 OK거의 불가 (소비 못 함)
List<? super Integer>Integer의 어떤 상위 타입의 리스트Object만Integer 쓰기 OK

왜 그런지 — 가장 흔한 함정과 함께 보면:

List<Integer> ints = new ArrayList<>();
List<? extends Number> nums = ints;     // OK — 하위 타입의 리스트로 받음

Number n = nums.get(0);                  // 읽기 OK — Number 보장
nums.add(1);                             // ❌ 컴파일 에러
// 왜? nums가 List<Double>일 수도 있어 컴파일러는 안전을 보장 못 함

List<? super Integer> sink = new ArrayList<Number>();
sink.add(42);                            // ✅ 쓰기 OK — Integer 또는 그 하위만 들어가도 무조건 안전
Object x = sink.get(0);                  // 읽으면 Object 정도만 보장

5) PECS — Producer Extends, Consumer Super

외울 한 줄: "내가 데이터를 가져다 쓸 거면 extends, 데이터를 넣을 거면 super".

Joshua Bloch의 PECS 원칙은 와일드카드를 "어느 쪽으로 데이터가 흐르는가"로 결정하라는 가이드입니다. Java 표준 라이브러리의 Collections.copy 시그니처가 정확히 이 원칙을 따릅니다. 

// 표준 라이브러리의 PECS 적용
public static <T> void copy(List<? super T> dest,        // T를 받아 넣을 곳 → super
                            List<? extends T> src) {     // T를 꺼내 줄 곳 → extends
    for (int i = 0; i < src.size(); i++) {
        dest.set(i, src.get(i));
    }
}

// 호출 — 다양한 조합 허용
List<Number>  dest = new ArrayList<>(...);
List<Integer> src  = List.of(1, 2, 3);
Collections.copy(dest, src);   // T = Number; src는 ? extends Number, dest는 ? super Number

Stream API의 Function<? super T, ? extends R>도 같은 원칙입니다 — T를 받아(consume = super) R을 만든다(produce = extends).

6) Raw Type — 절대 새로 쓰지 마라

List처럼 타입 인자를 안 적은 형태가 raw type입니다. Java 5에서 제네릭이 추가될 때 이전 코드와의 호환을 위해만 남겨 둔 것이라 새 코드에선 쓰면 안 됩니다. 

List rawList = new ArrayList();      // ⚠️ raw type 경고
rawList.add("hello");
rawList.add(42);                      // 어떤 타입이든 들어감 — 타입 안전성 사라짐

// 라이브러리에서 raw List를 받을 때
List<String> safe = (List<String>) rawList;   // 'unchecked' 경고
String s = safe.get(0);                        // OK
String t = safe.get(1);                        // ClassCastException at runtime!

raw type을 쓰면 그 영향이 전염됩니다 — raw List에서 꺼내 쓰는 다른 코드의 타입 검사도 약화돼서 컴파일러가 발견했을 버그를 놓치게 됩니다. "어떤 타입이든 받겠다"가 의도라면 raw 대신 List<?>를 쓰세요. 와일드카드는 타입 안전성을 유지합니다.

7) 제네릭과 가변 인자(varargs)

// "Heap pollution" 경고 — 제네릭 타입의 배열은 만들 수 없는데 varargs는 배열로 변환됨
public static <T> List<T> asList(T... items) {
    return Arrays.asList(items);
}

// Java 7+: 안전하다고 명시
@SafeVarargs
public static <T> List<T> asList(T... items) {
    return Arrays.asList(items);
}

8) 제네릭 타입과 상속·할당 — 불공변(invariant)

List<Integer>는 List<Number>의 하위 타입이 아닙니다. 직관과 다른 이 결정이 와일드카드의 존재 이유입니다.

참고: 배열은 공변(Integer[]은 Number[]) — 그래서 Number[] arr = new Integer[1]; arr[0] = 1.0;이 ArrayStoreException을 던진다.

9) 제네릭과 리플렉션 — TypeToken 트릭

소거 때문에 List<String>.class는 안 됩니다. 그러나 클래스의 상속 관계에 박혀 있는 제네릭 정보는 리플렉션으로 살아 있습니다. 이 점을 이용한 것이 Jackson TypeReference·Spring ParameterizedTypeReference입니다. 

// Jackson — 익명 자식 클래스로 제네릭 정보 잡아두기
List<User> users = mapper.readValue(json, new TypeReference<List<User>>() {});
//                                          ^^^^^^^^^^^^^^^^^^^^^^^^^^^^
// 익명 클래스가 List<User>를 superclass에 박아 두므로 reflection으로 추출 가능

시간/공간 복잡도 또는 trade-off

  • 런타임 비용 없음 — 타입 소거 덕분에 제네릭 사용 자체의 런타임 오버헤드가 없다(컴파일 타임에 모두 처리).
  • 표현력 제한 — new T[]·T.class·instanceof T 같은 자연스러운 연산이 막힘.
  • 호환성 vs 일관성 — Java 5 이전 코드 호환을 위해 reified generics(C#처럼 런타임에 살아 있는 타입 정보)를 포기.
  • API 가독성 — 와일드카드와 PECS는 처음 보면 어려움. 그러나 잘 쓴 시그니처는 호출자에게 매우 친절함.

실무에서 어디에 쓰이는가?

  • 컬렉션·Stream API — 모든 컨테이너의 기본 도구.
  • Repository / DAO — Spring Data의 JpaRepository<T, ID>처럼 엔티티 타입을 매개로.
  • 유틸리티 메서드 — <T> Optional<T> find(...)처럼 호출자가 타입을 결정.
  • 이벤트/메시지 핸들러 — EventHandler<T extends Event>로 안전한 디스패치.
  • JSON·HTTP 라이브러리 — TypeReference 패턴으로 제네릭 deserialization.

면접에 나올 질문과 답변

Java의 타입 소거(type erasure)란 무엇이고 왜 그렇게 됐나요?
컴파일 타임에 타입 매개변수로 안전성을 검사한 뒤, 바이트코드에서는 그 매개변수를 Object(또는 상한)로 치환해 정보를 지우는 방식입니다. 이유는 호환성 — Java 5에서 제네릭이 추가될 때 그 이전 raw 코드와 같은 바이트코드 위에서 동작해야 했기 때문입니다. 부작용으로 new T[]·T.class·instanceof T 같은 연산이 불가능해졌고, List<String>과 List<Integer>가 런타임에 같은 Class가 됩니다.
PECS가 무엇인가요?
Producer Extends, Consumer Super의 약자로, 와일드카드를 어떻게 쓸지 결정하는 원칙입니다. 제네릭 매개변수가 데이터를 꺼내 주는 쪽(Producer)이면 ? extends T, 데이터를 받아 넣는 쪽(Consumer)이면 ? super T를 씁니다. 예를 들어 Collections.copy(dest, src)는 src에서 읽어 dest에 쓰므로 List<? extends T> src, List<? super T> dest가 정답입니다.
List<Integer>가 List<Number>의 하위 타입이 아닌 이유는?
만약 하위 타입이라면 List<Number> n = new ArrayList<Integer>();가 가능해지고, 그 다음 n.add(3.14)가 호출되어 Integer 리스트에 Double이 들어가 ClassCastException이 생길 수 있습니다. 이를 막기 위해 제네릭은 불공변(invariant)으로 설계됐습니다. 다양한 하위 타입을 받고 싶을 때는 와일드카드(List<? extends Number>)를 쓰면 됩니다.
raw type을 쓰면 안 되는 이유는?
타입 안전성이 사라져 컴파일러가 잡을 수 있던 잘못된 타입 추가를 런타임까지 가져갑니다. 더 큰 문제는 raw type에서 꺼낸 값을 다른 코드에 넘기는 순간 그 코드의 타입 검사도 약해지는 전염이 일어난다는 점입니다. "어떤 타입이든 받겠다"가 진짜 의도라면 raw 대신 List<?>를 쓰면 안전성을 유지할 수 있습니다.
제네릭으로 배열을 만들 수 없는 이유는?
배열은 런타임에 자기 원소 타입을 알고 있어 ArrayStoreException으로 자기를 보호하는데, 제네릭은 소거되어 런타임에 타입을 모릅니다. 따라서 new T[10]을 허용하면 그 배열이 자기 타입을 검증할 수 없게 되어 안전성이 깨집니다. 실무에서는 (T[]) new Object[n]로 우회하고 unchecked 경고를 책임지거나, 배열 대신 ArrayList<T>를 쓰는 것이 안전합니다.
List<?>와 List<Object>의 차이는?
List<Object>는 "Object 리스트만" 받습니다 — List<String>은 받을 수 없습니다(불공변). List<?>는 "원소 타입을 모르는 어떤 리스트든" 받지만, 안전을 위해 null 외에는 add할 수 없습니다. 즉 ?는 "읽기 전용 리뷰", Object는 "Object를 담는 컨테이너"라는 다른 의도를 표현합니다.
제네릭 메서드와 제네릭 클래스의 차이는?
제네릭 클래스는 클래스 레벨에 타입 매개변수가 있어 인스턴스마다 결정됩니다(new Box<Integer>()). 제네릭 메서드는 호출마다 컴파일러가 추론하거나 명시한 타입으로 결정됩니다. 클래스가 비제네릭이라도 메서드는 제네릭일 수 있고, 그 반대도 가능합니다. 또 static 메서드는 클래스의 타입 매개변수를 쓸 수 없으므로 자기 타입 매개변수를 따로 선언해야 합니다.
Jackson에서 List<User>를 어떻게 역직렬화하나요?
List<User>.class는 소거 때문에 안 되므로, TypeReference의 익명 자식 클래스를 만들어 그 superclass에 제네릭 정보를 박아 둡니다. Jackson은 그 클래스의 getGenericSuperclass()로 정보를 추출합니다. 이 패턴은 Spring의 ParameterizedTypeReference, Google Guice의 TypeLiteral 등에서도 동일하게 쓰입니다.
