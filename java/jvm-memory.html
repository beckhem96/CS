JVM 메모리 구조 — heap·stack·metaspace, GC 개요

JVM은 실행 중 메모리를 용도별로 다섯 개 영역(Heap, Stack, Metaspace, PC Register, Native Method Stack)으로 나누어 관리합니다. 이 중 객체 인스턴스가 사는 Heap은 GC의 무대이고, 메서드 호출과 지역 변수가 쌓이는 Stack은 스레드별로 따로 존재합니다. "어떤 객체가 어디에 살고 누가 정리해 주는가"를 아는 것이 OOM·메모리 누수·GC 튜닝의 출발점입니다.

구체적인 개념 설명

C/C++ 같은 언어는 메모리를 코드 / 데이터 / 힙 / 스택으로 나누고 개발자가 malloc/free로 직접 관리합니다. JVM은 같은 그림 위에 관리되는 런타임을 얹어 — 객체 할당은 자동(new), 회수는 GC가 책임 — 메모리 누수와 이중 해제 같은 클래식한 버그군을 거의 없앴습니다. 대신 우리는 "왜 이 객체는 GC가 안 가져가지?", "왜 GC가 자주 멈추지?" 같은 새로운 종류의 문제를 다뤄야 합니다.

메모리 영역을 이해하면 다음 질문에 답할 수 있습니다 — "static 필드는 어디 사는가?", "지역 변수는 GC 대상인가?", "왜 거대한 List를 잡고 있으면 OOM이 나는가?", "Metaspace란 옛날 PermGen과 뭐가 다른가?".

왜 필요한가?

  • 안전성 — 자동 메모리 관리로 누수·dangling pointer 위험 감소.
  • 예측 가능한 모델 — 영역별 책임이 정해져 있어 진단·튜닝이 가능.
  • 스레드 격리 — Stack은 스레드별이라 지역 변수는 자동으로 thread-safe.
  • 성능 — 영역을 분리해 GC가 객체 수명에 맞는 알고리즘(generational)을 쓸 수 있다.

어떻게 동작하는가?

1) JVM의 5개 메모리 영역

힙·메타스페이스는 모든 스레드가 공유, 스택·PC·네이티브 스택은 스레드마다 독립.
  • Heap — 모든 객체와 배열이 사는 곳. GC의 무대. Young/Old로 나뉨.
  • JVM Stack — 메서드 호출마다 프레임(지역 변수 + 피연산자 스택)이 push/pop. 스레드별로 독립.
  • Metaspace — 클래스 메타데이터(클래스 정보, 메서드 바이트코드, static 필드)가 위치. Java 8부터 PermGen을 대체. 네이티브 메모리에 있어 힙 한도와 별개.
  • PC Register — 스레드마다 자기가 실행 중인 바이트코드 명령의 주소를 보관. 컨텍스트 스위칭에 필요.
  • Native Method Stack — JNI(Java Native Interface)로 호출되는 C/C++ 코드가 쓰는 스택.

2) Heap의 세부 구조 — Generational

"대부분의 객체는 금방 죽는다"는 weak generational hypothesis가 이 분리의 근거.

Generational Hypothesis — "대부분의 객체는 짧게 살고, 늙은 객체일수록 더 오래 산다." 이 경험칙을 받아들이면 GC 알고리즘은 두 부분에 다른 전략을 쓸 수 있습니다.

  • Minor GC — Young만 청소. 빠르고 자주(보통 수 ms). Eden이 가득 차면 발동.
  • Major GC / Full GC — Old(또는 전체)를 정리. 느리고 드물지만 길게 멈춤.
  • Promotion — Survivor에서 N번 살아남은(-XX:MaxTenuringThreshold) 객체는 Old로 승격.

3) 객체 생성 흐름 — new 키워드 한 줄이 일으키는 일

  1. JVM이 클래스가 로드/초기화되었는지 확인. 아니면 클래스 로딩 단계 진입.
  2. 객체 크기를 계산(필드 + 정렬 + 헤더). 보통 객체 헤더는 12~16 바이트.
  3. Eden에 공간을 잡음 — 대부분의 JVM은 TLAB(Thread-Local Allocation Buffer)로 락 없이 빠르게 포인터 증가만으로 할당.
  4. 객체 메모리를 0으로 초기화.
  5. 객체 헤더 세팅(클래스 포인터, hash, lock).
  6. 생성자 호출 → 필드 대입.
  7. 참조를 호출자에게 반환.
// 한 줄에 7단계가 다 일어남
User u = new User("alice");

// JIT의 escape analysis가 "이 객체는 메서드 밖으로 안 나가" 판단하면
// 힙이 아니라 스택에 잡히기도 함 (scalar replacement)

4) Stack — 메서드 프레임의 푸시/팝

메서드를 호출할 때마다 프레임이 스택에 푸시됩니다. 프레임 안에는 (1) 지역 변수 배열, (2) 피연산자 스택, (3) 상수 풀에 대한 참조가 들어 있습니다. 메서드가 끝나면 그 프레임이 그대로 빠지므로 지역 변수는 자동으로 정리되고, GC가 따로 처리할 필요가 없습니다.

호출 깊이 = 프레임 수. 무한 재귀는 StackOverflowError, 지나치게 깊으면 -Xss로 늘릴 수 있다.

핵심 규칙: 변수 자체는 스택, 객체 본체는 힙입니다. User u = new User()에서 u는 스택에 있는 4/8바이트 참조이고, User 객체는 힙에 있습니다. 그래서 메서드가 끝나면 참조 u는 스택에서 사라지고, 이 시점에 더는 가리키는 게 없으면 객체는 다음 GC에서 회수됩니다.

5) Metaspace — 옛 PermGen의 후계자

Java 7까지는 클래스 메타데이터가 힙 안의 PermGen(고정 크기)에 저장돼 동적으로 클래스를 많이 로드하면 PermGen Space OOM이 자주 발생했습니다. Java 8부터 PermGen이 사라지고 Metaspace가 그 자리를 대신합니다.

  • Heap이 아니라 네이티브 메모리에 위치 → 기본 한도가 사실상 OS 메모리.
  • -XX:MaxMetaspaceSize로 한도 설정 가능. 무제한으로 두면 폭주 시 OS가 OOM-kill.
  • 스프링 부트 같은 프레임워크가 동적 프록시·CGLIB로 클래스를 많이 만들면 여기에 부담이 간다 → 클래스 로더 누수가 가장 흔한 OOM 원인.
  • static 필드가 가리키는 객체는 여전히 힙에 있고, Metaspace에는 그 필드의 메타정보만 들어간다.

6) 참조의 4단계 — strong / soft / weak / phantom

Java는 GC와 협력하기 위해 4단계 참조를 제공합니다. "이 참조가 살아 있을 때 GC가 어떻게 행동해야 하나"를 표현합니다.

참조 종류GC 동작대표 사용처
Strong참조가 있는 한 절대 회수 X일반 변수 — User u = ...
Soft메모리 부족할 때 회수메모리 민감한 캐시
Weak다음 GC에서 즉시 회수WeakHashMap, ThreadLocal 키
Phantom이미 회수 후, 정리 작업 트리거네이티브 자원 정리(Cleaner API)
 
// WeakReference — 캐시 키가 사라지면 entry도 사라지길 원할 때
WeakReference<LargeImage> ref = new WeakReference<>(loadImage());
LargeImage img = ref.get();   // null일 수 있음

// WeakHashMap — 키가 다른 곳에서 더 이상 참조 안 되면 entry 자동 제거
Map<Session, UserData> cache = new WeakHashMap<>();

7) GC 개요 — Mark / Sweep / Compact

세부 알고리즘(Serial / Parallel / G1 / ZGC)은 심화 페이지에서 다루고, 여기서는 모든 GC가 공통으로 하는 일을 봅니다.

Mark → Sweep → (선택적) Compact. 모든 현대 GC가 이 패턴의 변형이다.

GC Root는 "이 위치에서 시작해 도달할 수 있는 객체는 살아 있다"고 판정하는 시작점들입니다.

  • 현재 활성 스레드 스택의 지역 변수와 피연산자 스택.
  • 모든 클래스의 static 필드.
  • JNI 핸들(네이티브 코드가 잡고 있는 참조).
  • 모니터(synchronized 락) 객체.

이 루트에서 시작해 그래프 탐색(BFS/DFS)으로 mark가 진행됩니다. 한 객체에서 다른 객체로 갈 수 없으면 dead로 판정해 회수합니다.

8) Stop-The-World

Mark 단계에서는 객체 그래프가 일관되어야 하므로 모든 애플리케이션 스레드를 잠깐 멈춥니다. 이 일시 정지를 STW pause라 부릅니다. 예전 GC들은 STW가 수백 ms~수초였지만, G1·ZGC·Shenandoah 같은 현대 GC는 동시 마킹·동시 압축으로 대부분의 작업을 STW 밖에서 수행해 ms 단위까지 줄였습니다.

9) OOM과 메모리 누수의 진단

OutOfMemoryError는 한 종류가 아닙니다. 메시지를 잘 봐야 어디가 터졌는지 압니다.

메시지원인대응
Java heap space힙 부족 — 객체가 GC 안 됨힙 dump → MAT로 retainer 분석
GC overhead limit exceededGC가 시간의 98%를 쓰는데 회수율 2% 미만위와 동일
Metaspace클래스 너무 많이 로드 (클래스 로더 누수)핫 리로드·동적 프록시 점검
unable to create new native threadOS 스레드 한도 초과스레드 풀 도입, ulimit 조정
Direct buffer memory네이티브 buffer 누수 (NIO, Netty)direct memory 한도·릴리스 점검

대표적인 누수 패턴:

  • static 컬렉션에 무한히 추가 — 캐시처럼 쓰이지만 만료가 없음.
  • ThreadLocal 미정리 — Tomcat 같은 컨테이너에서 스레드가 재사용되면 ThreadLocal이 영원히 남는다. remove() 필수.
  • Listener / Callback 등록 후 미해제 — 익명 이너 클래스가 외부 인스턴스를 잡고 있다.
  • 비정적 inner class의 외부 참조 — 호출자가 죽어도 외부 인스턴스가 못 죽는다.

10) 자주 쓰는 JVM 옵션

// 힙 크기 (시작·최대를 같게 두면 resize 비용 회피)
-Xms2g -Xmx2g

// 메타스페이스 한도
-XX:MaxMetaspaceSize=256m

// GC 알고리즘 선택
-XX:+UseG1GC          // Java 9+ 기본 (균형형)
-XX:+UseZGC           // 짧은 pause (Java 15+)
-XX:+UseShenandoahGC  // 짧은 pause (OpenJDK 12+)

// 스레드 스택 크기
-Xss512k

// OOM 시 힙 덤프
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/log/dump.hprof

// GC 로그
-Xlog:gc*:file=gc.log:time,uptime,level,tags

시간/공간 복잡도 또는 trade-off

  • 처리량 vs 일시 정지 — Parallel GC는 처리량이 높지만 STW가 길고, ZGC는 STW가 짧지만 처리량이 약간 낮음.
  • 힙 크기 — 크면 GC 빈도는 줄지만 한 번 돌 때 길어진다. 작으면 빈번하지만 짧다.
  • generational — 대부분 빨리 죽는다는 가정이 깨지면(긴 수명 객체가 많으면) 효과가 낮다.
  • Compact 비용 — 단편화는 풀지만 살아 있는 객체를 옮기므로 비용이 크다.

실무에서 어디에 쓰이는가?

  • OOM 디버깅 — heap dump → Eclipse MAT의 "Leak Suspects"로 retainer 추적.
  • GC 튜닝 — 응답 시간이 중요한 서비스는 G1/ZGC, 처리량은 Parallel.
  • 캐시 설계 — soft/weak reference, Caffeine 같은 라이브러리.
  • 스레드 풀 설계 — 스택 크기와 OS 스레드 한도를 고려해 풀 크기 결정.
  • 관측성 — Micrometer로 jvm.gc.pause·jvm.memory.used 메트릭 수집.

면접에 나올 질문과 답변

JVM 메모리 영역을 설명해 주세요.
스레드 공유 영역은 (1) 모든 객체와 배열이 사는 Heap, (2) 클래스 메타데이터·static 필드 메타정보가 들어가는 Metaspace입니다. 스레드별 영역은 (1) 메서드 프레임이 push/pop되는 JVM Stack, (2) 현재 실행 중인 바이트코드 주소를 담는 PC Register, (3) JNI 호출용 Native Method Stack입니다. 지역 변수는 스택에, 객체 본체는 힙에 — 이 분리가 가장 중요한 그림입니다.
PermGen과 Metaspace의 차이는?
Java 7까지는 클래스 메타데이터가 힙 내부의 PermGen에 저장됐고, 크기가 고정이라 동적으로 클래스를 많이 로드하면 PermGen Space OOM이 흔했습니다. Java 8부터는 PermGen이 사라지고 Metaspace가 그 자리를 대신합니다 — 네이티브 메모리에 위치하므로 사실상 OS 메모리까지 자랄 수 있고, -XX:MaxMetaspaceSize로 한도를 설정합니다. 다만 한도를 안 두면 클래스 로더 누수로 호스트 메모리가 폭주할 수 있습니다.
Young Gen과 Old Gen으로 나누는 이유는?
"대부분의 객체는 짧게 살고, 늙은 객체일수록 더 오래 산다"는 weak generational hypothesis 때문입니다. 이 가정이 맞으면 Young만 자주 청소(Minor GC)해 짧고 빠른 회수가 가능하고, Old는 가끔만 정리(Major/Full GC)해도 충분합니다. 살아남은 객체를 옮기는 비용이 죽은 객체를 회수하는 비용보다 훨씬 작으므로, 이 분리가 GC 처리량을 크게 끌어올립니다.
Stop-The-World pause를 줄이는 방법은?
가장 간단한 답은 현대 GC를 쓰는 것입니다 — G1(Java 9+ 기본), ZGC, Shenandoah는 marking과 compaction의 대부분을 동시(concurrent) 단계에 두어 STW를 ms 단위까지 줄였습니다. 그 외에는 (1) 힙 사이즈와 GC 알고리즘에 맞는 튜닝, (2) 큰 객체 할당 빈도 줄이기, (3) Old 승격을 줄이도록 Survivor 크기 조정, (4) 너무 큰 컬렉션·캐시를 GC 친화적인 자료구조로 교체 같은 접근이 있습니다.
"strong reference만 GC가 안 가져간다"는데, 다른 참조는요?
Soft는 메모리가 부족할 때만 회수되므로 메모리 민감한 캐시에 적합합니다. Weak는 다음 GC에서 즉시 회수돼 WeakHashMap이나 ThreadLocal의 키처럼 "원본이 살아 있을 때만 의미 있는" 참조에 씁니다. Phantom은 객체가 회수된 후에 트리거되어 네이티브 자원 해제 같은 후처리에 쓰입니다(Cleaner API). 일반 코드에서는 strong과 weak·soft만 알아도 충분합니다.
메모리 누수가 흔한 패턴 몇 가지는?
(1) static 컬렉션을 캐시처럼 쓰지만 만료가 없는 경우 — 무한히 자라며 절대 GC되지 않습니다. (2) ThreadLocal을 사용 후 remove()를 안 한 경우 — Tomcat처럼 스레드가 재사용되는 환경에서 ThreadLocal이 영원히 남습니다. (3) 익명 이너 클래스 / non-static inner class가 외부 인스턴스를 암묵 참조하는 경우 — 콜백 등록 후 해제 안 하면 외부 인스턴스가 GC되지 않습니다. (4) 클래스 로더 누수 — 핫 리로드 환경에서 옛 로더와 그 안의 클래스들이 살아 남아 Metaspace를 갉아먹습니다.
힙 덤프를 어떻게 분석하나요?
OOM이 났을 때 -XX:+HeapDumpOnOutOfMemoryError가 켜져 있으면 .hprof 파일이 생성됩니다. Eclipse MAT(Memory Analyzer Tool)로 열어 "Leak Suspects" 리포트를 보면 가장 큰 retainer가 무엇인지, 그것이 무엇을 잡고 있는지 그래프로 보여줍니다. 보통 비정상적으로 큰 ConcurrentHashMap·ArrayList나 누적된 ThreadLocal이 범인이며, dominator tree에서 retained heap이 큰 객체부터 봅니다.
스택과 힙 중 어느 쪽이 더 빠른가요?
스택이 압도적으로 빠릅니다. 메서드 호출/종료 시 포인터 증가/감소만으로 할당·해제가 끝나고, 캐시 친화적이며 thread-safe(스레드별)이기 때문입니다. 힙은 GC가 책임을 지지만 동시 접근 동기화·할당 시 메모리 탐색·일시 정지가 필요합니다. 그래서 JIT는 escape analysis로 "이 객체는 메서드 밖으로 안 나간다"고 판단되면 힙 대신 스택에 펼쳐 놓는 scalar replacement 최적화를 시도합니다.
