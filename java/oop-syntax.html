객체지향과 Java 문법 — class·interface·abstract·record·enum

Java는 OOP를 일급으로 지원하는 언어로, class·interface·abstract class·record·enum이라는 다섯 가지 종류의 형(type)을 제공합니다. 각각은 "이 타입은 어떤 책임을 가지는가"라는 의도를 표현하기 위한 도구이며, 접근 제어자·오버로딩·오버라이딩이 이들을 결합해 다형성을 만듭니다.

이 페이지는 OOP 패러다임 자체가 아니라 Java 문법 차원의 도구를 다룹니다. OOP의 4기둥(캡슐화·상속·다형성·추상화) 자체는 소프트웨어공학 → OOP 페이지를 함께 보세요.

구체적인 개념 설명

Java에서 타입은 곧 "내가 어떤 책임을 가질 수 있는지에 대한 약속"입니다. 같은 데이터를 담더라도 class로 두는지 record로 두는지에 따라 "이건 식별자가 있는 객체야" vs "이건 값일 뿐이야"라는 의도가 달라집니다. 컴파일러와 다른 개발자는 그 의도를 읽고 적절한 사용 방식을 선택합니다.

또한 Java는 단일 상속 + 다중 인터페이스 구현이라는 제한된 구조를 택했습니다. 이는 C++의 다중 상속이 만들어 낸 다이아몬드 문제·우선순위 모호성을 피하면서도, "이 객체가 가질 수 있는 능력"은 인터페이스로 자유롭게 조합하게 하는 절충안입니다.

왜 필요한가?

  • 의도의 분리 — "데이터 묶음(record)" / "기본 골격(abstract)" / "능력 계약(interface)"을 문법으로 구분.
  • 변경의 국소화 — private 필드와 패키지 단위 접근 제어로 외부 영향을 차단.
  • 다형성 — 인터페이스/상속과 오버라이딩을 통해 호출자는 구체 타입을 몰라도 된다.
  • 도구 친화 — 정적 타입 + 명시적 시그니처 → IDE의 리팩토링·자동 완성·find usages가 정확.

어떻게 동작하는가?

1) 다섯 가지 타입 한눈에

"무엇을 표현할 것인가"에 따라 도구를 고르면 코드가 자기 의도를 말해 준다.

2) class — 가장 기본 단위

public class Account {
    private final String id;        // 불변 필드
    private long balance;            // 가변 상태

    public Account(String id) {      // 생성자
        this.id = Objects.requireNonNull(id);
        this.balance = 0;
    }
    public void deposit(long amount) {
        if (amount <= 0) throw new IllegalArgumentException();
        balance += amount;
    }
    public long balance() { return balance; }
}

3) abstract class — "골격은 줄게, 구체는 자식이 채워"

abstract로 선언된 클래스는 인스턴스화할 수 없고, 일부 메서드는 구현 없이 시그니처만 둡니다. 자식이 그 추상 메서드를 채워야 인스턴스가 가능합니다. Template Method 패턴이 대표적인 사용처 — 흐름은 부모가 정하고 단계 일부만 자식이 바꿉니다. 

abstract class Report {
    // 흐름은 부모가 고정
    public final String render() {
        return header() + body() + footer();
    }
    protected String header() { return "=== 보고서 ===\n"; }
    protected abstract String body();           // 자식이 채워야 함
    protected String footer() { return "\n=== 끝 ===\n"; }
}
class SalesReport extends Report {
    protected String body() { return "이번 달 매출 ..."; }
}

4) interface — 능력/계약

"이걸 가진 누구나 X를 할 수 있어"라는 능력을 표현합니다. 다중 구현이 가능하므로 한 객체가 여러 능력을 동시에 지닐 수 있습니다. Java 8+에서는 default 메서드(기본 구현)와 static 메서드까지 둘 수 있어 추상 클래스와의 경계가 흐려졌습니다. 

interface Comparable<T> {
    int compareTo(T other);
}
interface Closeable {
    void close() throws IOException;
    // default·static 메서드도 가능
    static Closeable noop() { return () -> {}; }
}

// 다중 구현
class FileHandle implements Comparable<FileHandle>, Closeable {
    public int compareTo(FileHandle o) { ... }
    public void close() { ... }
}

그럼 abstract와 interface는 언제 어느 쪽?

  • "공통 부모로 묶고 일부 구현·상태를 공유" 하고 싶다 → abstract class.
  • "이 능력을 가진 누구든 X를 할 수 있어야 해"(다양한 객체에 부여) → interface.
  • 다중 능력 결합이 필요하면 무조건 interface (Java는 다중 상속 불가).

5) record — 데이터 클래스를 한 줄로

Java 16+에서 정식 추가된 record는 "이 객체는 불변 데이터의 묶음"이라는 의도를 표현합니다. 컴파일러가 final 필드, 생성자, 접근자(name()), equals/hashCode/toString을 자동 생성해 줍니다. 

// 단 한 줄
public record Point(double x, double y) { }

// 동등하게 펼치면 대략 이런 코드:
// public final class Point {
//     private final double x;
//     private final double y;
//     public Point(double x, double y) { ... }
//     public double x() { return x; }
//     public double y() { return y; }
//     public boolean equals(Object o) { ... 모든 컴포넌트 비교 ... }
//     public int hashCode() { ... }
//     public String toString() { return "Point[x=..., y=...]"; }
// }

// 컴팩트 생성자(검증/정규화)
public record Email(String value) {
    public Email {
        if (!value.contains("@")) throw new IllegalArgumentException();
        value = value.toLowerCase();   // 대입 가능
    }
}

record는 상속 불가(다른 클래스를 상속할 수도, 누가 record를 상속할 수도 없음)이지만 인터페이스는 구현할 수 있습니다. DTO·VO·이벤트·튜플 같은 "값으로 다루고 싶은" 자료에 쓰면 코드가 극적으로 줄어듭니다.

6) enum — 제한된 인스턴스의 집합

"이 타입의 인스턴스는 정해진 N개뿐"임을 표현하는 도구입니다. 단순 상수 나열을 넘어, 각 상수에 동작을 붙일 수 있어 Strategy 패턴의 가벼운 대안으로도 쓰입니다. 

enum Status {
    ACTIVE, SUSPENDED, CLOSED;
}

// 상수마다 동작 다르게 — 다형성을 enum이 직접 표현
enum Operation {
    PLUS  { public int apply(int a, int b) { return a + b; } },
    MINUS { public int apply(int a, int b) { return a - b; } };
    public abstract int apply(int a, int b);
}
int r = Operation.PLUS.apply(3, 5);  // 8

// switch 식(Java 14+)과 잘 어울림 — 모든 케이스를 컴파일러가 검사
String label = switch (status) {
    case ACTIVE    -> "활성";
    case SUSPENDED -> "정지";
    case CLOSED    -> "종료";
};

enum은 JVM이 한 인스턴스만 보장하므로 싱글톤 구현의 가장 안전한 방법으로도 알려져 있습니다(직렬화·리플렉션 공격에 강함).

7) 접근 제어자(Access Modifiers)

기본은 가장 좁게(private) 두고 필요한 만큼만 넓힌다 — 캡슐화의 출발점.

8) 오버로딩 vs 오버라이딩

이름은 비슷하지만 작동 시점이 다릅니다.

오버로딩(Overloading)오버라이딩(Overriding)
의미같은 이름, 다른 시그니처부모 메서드를 자식이 재정의
결정 시점컴파일 타임 (정적 디스패치)런타임 (동적 디스패치)
시그니처파라미터 타입/개수가 달라야 함완전히 같아야 함(공변 반환은 허용)
리턴 타입달라도 됨 (단, 그것만 다른 건 안 됨)같거나 자식 타입
대표 사례println(int) vs println(String)toString() 재정의
 
class Calculator {
    int  add(int a, int b)        { return a + b; }    // 오버로딩
    long add(long a, long b)      { return a + b; }    // 오버로딩
    double add(double a, double b){ return a + b; }    // 오버로딩
}

class Animal { String speak() { return "..."; } }
class Dog extends Animal {
    @Override
    String speak() { return "멍멍"; }                  // 오버라이딩
}

Animal a = new Dog();
System.out.println(a.speak());  // "멍멍" — 동적 디스패치

@Override 어노테이션은 항상 붙이세요. 시그니처를 잘못 써서 "오버라이딩한 줄 알았는데 새 메서드가 만들어지는" 사고를 컴파일러가 잡아 줍니다.

9) static과 final — 자주 묶이는 두 키워드

  • static — 인스턴스가 아니라 클래스에 속한 멤버. 모든 객체가 공유.
  • final — 변수: 재할당 금지 / 메서드: 오버라이딩 금지 / 클래스: 상속 금지.
  • static final — 클래스 단위 불변 상수. public static final int MAX = 100;
public class Config {
    public static final int DEFAULT_TIMEOUT_MS = 3000;
    private static int counter = 0;        // 전역 가변 상태(주의: 동시성!)
    public static int next() { return ++counter; }
}
// 호출
Config.next();

static 가변 상태는 멀티스레드 환경에서 race condition의 단골 원인이므로 가급적 피합니다.

10) 생성자와 초기화 순서

클래스 인스턴스화 시 실행 순서는 다음과 같습니다 — 면접 단골입니다.

  1. (부모 클래스가 아직 로드 안 됐으면) static 필드 + static {} 블록 실행 — 한 번만.
  2. 해당 클래스의 static 초기화도 동일.
  3. 부모 인스턴스 필드 초기화 → 부모 인스턴스 초기화 블록 → 부모 생성자.
  4. 자식 인스턴스 필드 → 자식 인스턴스 초기화 블록 → 자식 생성자.
class Parent {
    static  { System.out.println("1. parent static"); }
            { System.out.println("3. parent instance"); }
    Parent() { System.out.println("4. parent ctor"); }
}
class Child extends Parent {
    static  { System.out.println("2. child static"); }
            { System.out.println("5. child instance"); }
    Child() { System.out.println("6. child ctor"); }
}
// new Child(); → 1, 2, 3, 4, 5, 6

11) 중첩 클래스 — 4가지

종류특징대표 사례
static nested class바깥 인스턴스와 무관, 그냥 안에 들어 있는 클래스Builder, Map.Entry
inner class (non-static)바깥 인스턴스에 대한 암묵 참조 보유이벤트 리스너 (요즘은 람다로 대체)
local class메서드 안에 정의거의 안 씀
anonymous class이름 없이 즉석 정의 + 인스턴스화람다 등장 전의 callback

Java 8 이후엔 anonymous class 대부분이 람다로 대체됐습니다. 다만 비정적 inner class는 바깥 인스턴스를 잡고 있어 메모리 누수의 원인이 될 수 있어, 꼭 필요하지 않으면 static nested class로 두세요.

12) sealed class — 상속 가능한 자식을 못박기 (Java 17+)

public sealed interface Shape permits Circle, Rectangle, Triangle { }

public final class Circle    implements Shape { ... }
public final class Rectangle implements Shape { ... }
public final class Triangle  implements Shape { ... }

// 컴파일러가 모든 케이스가 처리됐는지 검사
String name = switch (shape) {
    case Circle c    -> "원";
    case Rectangle r -> "사각형";
    case Triangle t  -> "삼각형";
    // default 없어도 컴파일 OK — sealed라서 케이스 완전성 검증
};

"이 인터페이스는 정해진 몇 개만 구현할 수 있어"라는 폐쇄형 계층을 만듭니다. 패턴 매칭과 결합해 ADT(대수적 자료형)에 가까운 모델링이 가능해졌습니다.

시간/공간 복잡도 또는 trade-off

  • 상속 vs 합성 — 상속은 컴파일 타임에 강하게 묶이므로 변경에 약하다. "is-a"가 명확할 때만 사용.
  • final 클래스 — 상속 막아 안전하지만 테스트용 mock이 어려워질 수 있다(Mockito 3+에서는 가능).
  • record vs class — record는 가독성·equals 자동/불변이 장점이지만 가변 상태가 필요하면 부적합.
  • 공개 API의 확장성 — public 필드·메서드가 많을수록 호환성 부담. 처음엔 좁게 시작.

실무에서 어디에 쓰이는가?

  • 도메인 모델 — 엔티티는 class, 값 객체는 record, 상태는 enum.
  • Spring 빈/서비스 — 대부분 class, 의존성은 final 필드 + 생성자 주입.
  • 플러그인/전략 — 추상 골격 = abstract class, 능력 계약 = interface.
  • DTO·이벤트·메시지 — record로 한 줄 정의(불변, equals 자동).
  • 패턴 매칭(Java 21) — sealed + record + switch 식으로 "ADT처럼" 모델링.

면접에 나올 질문과 답변

interface와 abstract class의 차이를 설명하고, 어떻게 선택하나요?
interface는 "능력/계약"으로 다중 구현이 가능하고 상태(인스턴스 필드)를 가질 수 없습니다. abstract class는 "공통 부모"로 단일 상속이며 일부 구현과 인스턴스 필드를 공유할 수 있습니다. "다양한 종류의 객체에 같은 능력을 부여" → interface, "기본 골격을 공유하면서 자식이 일부만 채움" → abstract class. 다중 능력 결합이 필요하면 무조건 interface입니다.
record는 언제 쓰면 좋고 한계는 뭔가요?
DTO·값 객체·이벤트처럼 식별자가 아니라 값으로 다루고 싶은 불변 데이터에 쓰면 좋습니다. equals/hashCode/toString이 자동 생성되고 코드가 짧아집니다. 한계는 (1) 상속 불가 — 다른 클래스를 상속할 수 없고, 누가 record를 상속할 수도 없음, (2) 모든 컴포넌트가 final이라 가변 필드를 둘 수 없음, (3) ORM 엔티티처럼 프록시·지연 로딩 기반 객체에는 잘 맞지 않음입니다.
오버로딩과 오버라이딩의 차이는?
오버로딩은 같은 이름의 메서드를 시그니처(파라미터)만 다르게 여러 개 두는 것으로 컴파일 시점에 어떤 메서드를 호출할지 결정됩니다(정적 디스패치). 오버라이딩은 부모의 메서드를 자식이 재정의하는 것으로 런타임 시점에 객체의 실제 타입에 따라 호출이 결정됩니다(동적 디스패치). @Override를 붙이면 시그니처 실수를 컴파일러가 잡아 줍니다.
접근 제어자 4종을 좁은 순으로 말해 주세요.
private(같은 클래스) → 패키지 기본(같은 패키지) → protected(같은 패키지 + 자식 클래스) → public(어디서든)입니다. 기본은 가장 좁게 두고 필요한 만큼만 넓히는 것이 캡슐화의 기본입니다. 외부 의존이 늘어날수록 변경 비용이 커지기 때문입니다.
static 메서드를 오버라이딩할 수 있나요?
없습니다. static 메서드는 클래스에 속해 있어 컴파일 시점의 선언 타입으로 호출이 정해집니다. 자식이 같은 시그니처를 선언해도 그건 "오버라이딩"이 아니라 "메서드 숨김(hiding)"이라 동적 디스패치가 일어나지 않습니다. 다형성을 원한다면 인스턴스 메서드로 만들어야 합니다.
enum으로 싱글톤을 만들면 좋다는데 왜죠?
JVM이 enum 상수를 한 인스턴스만 보장하고, 직렬화·역직렬화·리플렉션을 통한 다중 인스턴스 생성도 차단하기 때문입니다. 반면 private static final INSTANCE 방식은 리플렉션이나 직렬화 우회로 깨질 수 있어 추가 방어가 필요합니다. 단, enum 싱글톤은 lazy 초기화가 되지 않으므로 시작 비용이 큰 객체에는 부적합합니다.
생성자에서 this()와 super()의 차이는?
this()는 같은 클래스의 다른 생성자를 호출, super()는 부모 클래스의 생성자를 호출합니다. 둘 다 생성자의 첫 줄에만 올 수 있고, 명시적으로 쓰지 않으면 컴파일러가 기본 super()를 자동으로 넣습니다. 그래서 부모에 인자 없는 생성자가 없으면 자식 생성자에서 명시적으로 super(...)를 호출해야 합니다.
non-static inner class의 함정은?
non-static inner class는 바깥 인스턴스에 대한 암묵 참조를 가집니다. 그래서 inner 인스턴스가 살아 있는 동안 바깥 인스턴스가 GC되지 않아 메모리 누수를 만들 수 있습니다. Android에서 액티비티 누수 사례가 대표적입니다. 외부 인스턴스에 의존하지 않는다면 항상 static nested class로 두는 것이 안전합니다.
