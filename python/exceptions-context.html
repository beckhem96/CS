예외와 컨텍스트 매니저

Python 예외는 정상 흐름으로 처리할 수 없는 상황을 호출자에게 전파하는 객체 기반 메커니즘입니다. 컨텍스트 매니저는 with 문으로 자원 획득과 해제를 안전하게 묶는 프로토콜이며, __enter__와 __exit__로 동작합니다.

구체적인 개념 설명

Python은 오류 상황에서 특수 반환값보다 예외를 적극적으로 사용합니다. 파일이 없거나, 키가 없거나, 타입이 맞지 않으면 예외 객체가 생성되고 호출 스택을 거슬러 올라가며 처리자를 찾습니다. 처리자가 없으면 프로그램은 traceback을 출력하고 종료합니다.

자원 관리는 예외와 떼어놓을 수 없습니다. 파일을 열고 작업하다가 중간에 예외가 나도 파일은 닫혀야 합니다. with 문은 이런 정리 작업을 자동화해, 성공/실패와 관계없이 __exit__가 호출되도록 보장합니다.

비유 — 비상벨과 퇴실 점검

예외는 건물 안에서 문제가 생겼을 때 울리는 비상벨입니다. 컨텍스트 매니저는 비상벨이 울려도 문 잠그기·전기 끄기 같은 퇴실 점검을 반드시 하게 만드는 체크리스트입니다.

왜 필요한가?

  • 오류 흐름 분리 — 정상 로직과 실패 처리를 분리해 읽기 쉽게 만든다.
  • 자원 누수 방지 — 파일, 소켓, 락, DB 커넥션을 반드시 해제.
  • 원인 추적 — traceback으로 실패 위치와 호출 경로를 확인.
  • 도메인 오류 표현 — custom exception으로 비즈니스 실패를 명확히 표현.

어떻게 동작하는가?

1) 예외 전파와 traceback

예외는 발생 지점에서 멈추지 않고 처리자를 찾을 때까지 호출자를 향해 올라간다.

2) try / except / else / finally

try:
    value = int(text)
except ValueError as e:
    print("숫자가 아닙니다", e)
else:
    print("변환 성공", value)   # 예외가 없을 때만
finally:
    print("항상 실행")          # 성공/실패와 무관
  • except — 특정 예외를 잡아 처리.
  • else — try 블록에서 예외가 없을 때만 실행.
  • finally — 예외 여부와 관계없이 정리 작업 수행.

3) 예외 계층과 잡는 범위

대부분의 애플리케이션 예외는 Exception 아래에 있습니다. BaseException에는 KeyboardInterrupt, SystemExit 같은 종료 신호도 있으므로 일반 코드에서 무작정 잡으면 안 됩니다.

일반적으로는 구체 예외를 먼저 잡고, 넓은 Exception은 경계 계층에서만 잡는다.

4) custom exception

class InsufficientBalanceError(Exception):
    pass

class Account:
    def withdraw(self, amount):
        if amount > self.balance:
            raise InsufficientBalanceError("잔액 부족")
        self.balance -= amount

도메인 예외를 만들면 호출자가 실패 원인을 명확히 구분할 수 있습니다. 단, 너무 많은 예외 클래스를 만들면 오히려 복잡해집니다.

5) with와 context manager protocol

컨텍스트 매니저는 예외가 있어도 정리 코드를 보장하는 구조다.
with open("data.txt", encoding="utf-8") as f:
    text = f.read()
# 여기서 자동 close()

6) 직접 만드는 컨텍스트 매니저

from contextlib import contextmanager

@contextmanager
def timer(name):
    start = time.perf_counter()
    try:
        yield
    finally:
        elapsed = time.perf_counter() - start
        print(f"{name}: {elapsed:.3f}s")

with timer("job"):
    run_job()

시간/공간 복잡도 또는 trade-off

기능장점주의점원칙
예외오류 흐름 분리정상 제어 흐름 남용 금지예외적인 상황에 사용
넓은 except경계에서 안정성버그 은폐구체 예외 우선
finally정리 보장return 남용 시 흐름 혼란정리만 수행
with자원 누수 방지프로토콜 이해 필요자원 획득/해제에 적극 사용

실무에서 어디에 쓰이는가?

  • 파일/소켓/DB 커넥션 — with로 안전하게 열고 닫기.
  • 트랜잭션 — 성공 시 commit, 예외 시 rollback.
  • 도메인 오류 — 결제 실패, 권한 없음, 재고 부족 등 custom exception.
  • CLI/웹 경계 — 최상위에서 예외를 잡아 사용자 친화 메시지로 변환.

면접에 나올 질문과 답변

except Exception은 안전한가요?
경계 계층에서는 유용하지만 내부 로직에서 남용하면 버그를 숨깁니다. 구체 예외를 먼저 잡고, 로깅 후 재발생하거나 명확한 오류로 변환하는 것이 좋습니다. BaseException은 KeyboardInterrupt/SystemExit까지 잡으므로 일반적으로 피합니다.
finally는 언제 실행되나요?
try 블록이 성공하든 예외가 나든, except에서 처리하든 아니든 항상 실행됩니다. 그래서 파일 닫기, 락 해제 같은 정리 작업에 적합합니다. 다만 with 문이 더 간결하고 안전합니다.
컨텍스트 매니저는 어떻게 동작하나요?
with에 진입할 때 __enter__가 호출되고, 본문이 끝나면 성공/예외 여부와 관계없이 __exit__(exc_type, exc, tb)가 호출됩니다. __exit__가 True를 반환하면 예외를 삼킬 수 있지만 남용하면 디버깅이 어려워집니다.
custom exception은 언제 만드나요?
호출자가 실패 원인을 구분해 다르게 처리해야 할 때 만듭니다. 예를 들어 재고 부족과 결제 실패는 HTTP 응답이나 재시도 정책이 다를 수 있으므로 별도 도메인 예외가 유용합니다.
EAFP와 LBYL의 차이는?
EAFP는 먼저 실행하고 실패하면 예외로 처리하는 방식이고, LBYL은 실행 전에 조건을 검사하는 방식입니다. Python은 EAFP 관용구가 많지만, 예외가 너무 자주 발생하는 정상 흐름이면 조건 검사가 더 명확할 수 있습니다.
traceback을 읽는 요령은?
아래쪽이 실제 예외 발생 지점이고 위쪽은 호출 경로입니다. 예외 타입과 메시지를 먼저 보고, 마지막 프레임의 파일/라인을 확인한 뒤 그 위 호출자들이 어떤 인자를 넘겼는지 거슬러 올라가면 됩니다.
