가상환경과 패키징

가상환경은 프로젝트마다 독립된 Python 실행 환경과 패키지 설치 공간을 제공하는 도구입니다. 패키징은 코드를 설치 가능한 배포 단위로 구성하는 방식이며, 현대 Python은 pyproject.toml을 중심으로 빌드 시스템과 의존성을 표현합니다.

구체적인 개념 설명

Python 프로젝트가 커지면 프로젝트마다 필요한 패키지 버전이 달라집니다. A 프로젝트는 Django 4, B 프로젝트는 Django 5를 요구할 수 있고, 전역 Python에 모두 설치하면 충돌이 납니다. 가상환경은 프로젝트별로 독립된 site-packages를 만들어 이런 충돌을 막습니다.

패키징은 내 코드를 다른 프로젝트에서 설치하거나 배포할 수 있게 만드는 구조입니다. 예전에는 setup.py 중심이었지만, 현재는 PEP 517/518 이후 pyproject.toml이 표준 진입점입니다. uv, poetry, hatch 같은 도구는 이 흐름을 더 편하게 해줍니다.

비유 — 프로젝트마다 별도 공구함

전역 Python은 집 전체의 공용 공구함이고, 가상환경은 프로젝트마다 따로 둔 공구함입니다. 한 프로젝트가 특정 버전 드라이버를 써도 다른 프로젝트의 공구함에는 영향을 주지 않습니다.

왜 필요한가?

  • 의존성 충돌 방지 — 프로젝트별 패키지 버전을 분리.
  • 재현 가능한 환경 — requirements/lock 파일로 같은 버전을 설치.
  • 배포 단위 구성 — 패키지 metadata, entry point, build backend 명시.
  • 보안/운영 — 의존성 목록을 추적하고 취약점 대응 가능.

어떻게 동작하는가?

1) venv 구조

venv는 완전한 VM이 아니라 Python 실행 경로와 패키지 설치 위치를 분리하는 가벼운 환경이다.
python -m venv .venv
source .venv/bin/activate     # Windows: .venv\Scripts\activate
python -m pip install requests
python -m pip freeze > requirements.txt

2) pip와 requirements.txt

pip는 PyPI에서 패키지를 설치하는 기본 도구입니다. requirements.txt는 설치할 패키지 목록을 기록합니다.

# requirements.txt
requests==2.32.3
fastapi==0.111.0
uvicorn[standard]==0.30.1

# 설치
python -m pip install -r requirements.txt

실무에서는 직접 pip보다 python -m pip 형태를 권장합니다. 현재 선택된 Python 인터프리터와 pip가 같은 환경인지 명확하기 때문입니다.

3) pyproject.toml

[project]
name = "my-app"
version = "0.1.0"
requires-python = ">=3.12"
dependencies = [
  "requests>=2.32",
]

[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

pyproject.toml은 프로젝트 metadata와 빌드 시스템을 선언합니다. 도구마다 추가 섹션을 넣을 수 있어 ruff, black, pytest 설정도 한 파일에 모을 수 있습니다.

4) wheel과 source distribution

wheel은 설치용 빌드 산출물이라 sdist보다 설치가 빠르고 재현성이 좋다.

5) uv / poetry 개요

  • uv — Rust로 작성된 빠른 패키지/환경 관리 도구. pip, venv, pip-tools 역할을 빠르게 대체.
  • poetry — 의존성 선언, lock, build/publish를 한 도구에서 관리.
  • pip-tools — requirements.in에서 pinning된 requirements.txt 생성.

어떤 도구를 쓰든 핵심은 의존성 선언 파일과 lock/pin 파일을 구분하고, CI/배포에서 같은 버전을 설치하게 만드는 것입니다.

6) 패키지 구조

my-project/
  pyproject.toml
  src/
    my_app/
      __init__.py
      main.py
  tests/
    test_main.py

src/ layout은 테스트가 로컬 소스 디렉터리를 우연히 import하는 문제를 줄이고, 설치된 패키지 기준으로 검증하게 만들어 배포 문제를 일찍 잡습니다.

시간/공간 복잡도 또는 trade-off

도구장점단점/주의적합 상황
venv + pip표준, 단순lock 관리 약함작은 프로젝트
requirements.txt배포 재현 쉬움직접 관리 번거로움앱 배포
pyproject.toml표준 metadata도구별 설정 차이라이브러리/앱 공통
poetry통합 관리도구 의존팀 프로젝트
uv매우 빠름생태계 변화 중새 프로젝트, CI 최적화

실무에서 어디에 쓰이는가?

  • 로컬 개발 — 프로젝트별 .venv로 의존성 분리.
  • CI — lock/pin 파일로 동일 버전 설치.
  • 라이브러리 배포 — pyproject metadata + wheel 빌드.
  • Docker 이미지 — requirements/lock 기반으로 재현 가능한 설치 계층 생성.

면접에 나올 질문과 답변

가상환경은 VM인가요?
아닙니다. OS 가상화가 아니라 Python 실행 파일 경로와 패키지 설치 위치를 분리하는 가벼운 환경입니다. 프로젝트마다 독립된 site-packages를 갖게 해 의존성 충돌을 막습니다.
왜 python -m pip를 권장하나요?
현재 실행 중인 Python 인터프리터에 연결된 pip를 확실히 사용하기 위해서입니다. 시스템에 Python이 여러 개 있으면 pip 명령이 다른 Python 환경을 가리킬 수 있습니다.
requirements.txt와 pyproject.toml의 차이는?
requirements.txt는 주로 설치할 패키지와 버전을 나열하는 파일이고, pyproject.toml은 프로젝트 metadata, 빌드 시스템, 의존성, 도구 설정을 담는 표준 파일입니다. 앱 배포에서는 pinning된 requirements/lock도 여전히 중요합니다.
wheel이란 무엇인가요?
Python의 설치용 빌드 배포 형식(.whl)입니다. 소스에서 매번 빌드하지 않고 바로 설치할 수 있어 빠르고, 네이티브 확장이 있는 패키지에서는 플랫폼별 wheel이 설치 편의성을 크게 높입니다.
lock 파일이 왜 필요한가요?
의존성 범위만 적으면 시간이 지나 새 버전이 설치되어 동작이 달라질 수 있습니다. lock 파일은 하위 의존성까지 정확한 버전을 고정해 로컬, CI, 운영에서 같은 환경을 재현하게 합니다.
src layout은 왜 쓰나요?
프로젝트 루트에 패키지를 바로 두면 테스트가 설치된 패키지가 아니라 로컬 소스 디렉터리를 우연히 import할 수 있습니다. src layout은 설치 과정을 거친 패키지 기준으로 테스트하게 해 배포 문제를 더 빨리 발견합니다.
