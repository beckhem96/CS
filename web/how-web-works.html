Web 동작 큰 그림 — 주소 입력부터 화면 렌더링까지

웹 브라우저에 https://example.com을 입력하고 엔터를 누르면 DNS·TCP·TLS·HTTP·HTML 파싱·CSS 적용·JS 실행·렌더링의 8개 단계가 차례로 일어납니다. 이 페이지는 그 흐름 전체를 한 장에 정리해 "어느 단계에서 무엇이 결정되는가"를 분명히 합니다.

구체적인 개념 설명

웹 개발자가 마주치는 거의 모든 문제는 이 흐름의 특정 단계에서 일어납니다. "왜 페이지가 안 뜨지?"는 DNS·네트워크·TLS·HTTP 응답 중 어디서 막혔는지의 문제이고, "왜 첫 화면이 느리지?"는 HTML 다운로드 → 파싱 → CSS·JS 차단(blocking) → 렌더링 사이의 어느 구간이 길어졌는지의 문제이며, "왜 button 클릭이 안 먹지?"는 DOM·이벤트·JS 실행 컨텍스트 어딘가의 문제입니다.

각 단계를 깊이 알기 위해 다른 페이지로 분리했지만, 이 페이지에선 모든 단계가 어떻게 맞물리는지를 봅니다. 각 단계의 깊은 내용은 인라인 링크를 따라가세요.

왜 필요한가?

  • 디버깅의 출발점 — 문제가 어느 계층에서 발생했는지 좁힐 수 있어야 한다.
  • 성능 진단 — DevTools의 Network·Performance 탭은 이 흐름 그대로의 timeline을 보여준다.
  • 면접 — "주소창에 URL을 치면 무슨 일이 일어나는지 설명하세요"는 백엔드·프론트엔드 공통 단골.
  • 설계 직관 — 캐싱·CDN·프리커넥트·번들링 같은 최적화의 의미를 알 수 있다.

어떻게 동작하는가?

1) 전체 단계 — 8단계 한눈에

한 번의 페이지 로드는 사실상 8개 작업의 직렬 실행. 각 단계의 latency가 합쳐져 사용자가 느끼는 "느림"이 된다.

2) ① URL 파싱 — 보이지 않는 첫 단계

https://www.example.com:443/products?id=42&sort=price#reviews
└┬─┘   └────────┬─────────┘└┬┘└──┬───┘└─────────┬───────┘└──┬───┘
 scheme         host       port  path           query        fragment

브라우저는 입력값이 URL인지 검색어인지 먼저 판정합니다. URL이라면 위와 같이 분해해 어떤 호스트의 몇 번 포트에 어떤 path를 요청할지를 결정합니다. fragment(#reviews)는 서버로 전송되지 않습니다 — 클라이언트만 사용합니다.

3) ② DNS — 도메인을 IP로

브라우저는 캐시 → OS 캐시 → ISP의 리졸버 → 루트/TLD/권한 네임서버 순으로 IP를 찾습니다. 자세한 흐름은 네트워크 → DNS에서 다룹니다.

  • 브라우저 자체 DNS 캐시(보통 1분 내외).
  • OS 캐시 + hosts 파일.
  • ISP 또는 사용자 지정 리졸버(8.8.8.8, 1.1.1.1).
  • 루트 → TLD → 권한 네임서버 재귀 질의.

성능 팁 — <link rel="dns-prefetch" href="//api.example.com">이나 preconnect를 두면 외부 도메인 호출이 시작되기 전에 DNS·TCP·TLS를 먼저 끝낼 수 있습니다.

4) ③ TCP 3-way handshake

SYN → SYN-ACK → ACK 세 패킷으로 연결을 만듭니다. 같은 도메인의 다음 요청들은 보통 같은 연결을 재사용(keep-alive)해 이 비용을 분산합니다. HTTP/2·3는 한 연결에서 여러 요청을 동시에 처리(멀티플렉싱)해 이 비용을 더 줄입니다.

5) ④ TLS 핸드셰이크 — HTTPS의 핵심

HTTPS는 TCP 위에 TLS를 한 번 더 얹습니다. 핵심은 (1) 서버 인증서 검증, (2) 키 교환(ECDHE), (3) 대칭 키로 채널 암호화입니다. TLS 1.3은 1-RTT(첫 통신 1회 왕복)에 끝나고, 같은 서버에 다시 접속할 때 0-RTT(이전 세션 키 재사용)로 즉시 데이터 전송이 가능합니다.

6) ⑤ HTTP 요청·응답 — 실제 데이터의 첫 왕복

GET /products?id=42 HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0 ...
Accept: text/html
Cookie: session=abc123

HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 12345
Cache-Control: public, max-age=600
Set-Cookie: csrf=...

<!DOCTYPE html>...

서버는 보통 첫 응답으로 HTML 문서를 보내고, 브라우저가 그 안의 <link>·<script>·<img>를 보고 추가 리소스를 요청합니다. 자세한 헤더와 캐싱은 네트워크 → HTTP/HTTPS 참조.

7) ⑥ HTML 파싱 → DOM, CSS 파싱 → CSSOM

브라우저는 응답 바이트를 받자마자 스트리밍 파싱을 시작합니다. 모두 다운로드되길 기다리지 않고 바이트 청크가 도착하는 대로 토큰화 → 트리 구성을 진행합니다.

HTML과 CSS는 동시에 파싱되어 각각 DOM·CSSOM이 되고, 둘이 합쳐져 Render Tree가 만들어진다.

8) JS의 차단 vs 비차단 — 첫 페이지가 느린 가장 큰 이유

<script>를 만나면 브라우저는 HTML 파싱을 멈추고 JS를 다운로드·실행한 뒤에야 파싱을 재개합니다(JS가 document.write로 DOM을 바꿀 수 있기 때문). 이를 피하기 위한 두 속성이 있습니다. 

<!-- 동기 — 파싱 차단 -->
<script src="app.js"></script>

<!-- async — 다운로드는 비동기, 도착 즉시 실행. 다른 스크립트 순서 무시 -->
<script src="analytics.js" async></script>

<!-- defer — 다운로드는 비동기, DOM 완성 후 순서대로 실행. 권장 -->
<script src="app.js" defer></script>

CSS도 기본적으로 렌더링 차단 리소스입니다. CSS가 다 도착해야 첫 페인트가 시작되므로, "스타일이 클 때 화면이 빈 상태로 한참 보이는" 현상이 여기서 나옵니다.

9) ⑦ Render Tree → Layout → Paint → Composite

자세한 단계는 브라우저 렌더링 파이프라인에서 다루고, 여기서는 한 줄 요약입니다.

  • Layout(reflow) — 각 박스의 위치와 크기를 계산. 가장 비싼 단계.
  • Paint — 픽셀(텍스트·색·그림자)을 채움.
  • Composite — 레이어를 GPU로 합성. transform·opacity는 이 단계에서 처리되어 매우 빠름.

10) ⑧ 인터랙션 — 클릭 한 번이 무슨 일을 일으키나

  1. OS가 마우스 이벤트를 브라우저로 전달 → JS 이벤트 큐에 push.
  2. Event Loop가 큐에서 꺼내 대상 DOM의 핸들러 호출(버블링/캡처링).
  3. 핸들러가 상태 업데이트 → DOM 변경 → 영향 받은 부분만 재 layout/paint.
  4. 같은 프레임 안(보통 16.6ms)에 끝나야 60fps 유지.

11) DevTools로 직접 보기

크롬 DevTools에서 위 흐름이 그대로 보입니다.

  • Network 탭 — DNS·TCP·TLS·request/response의 시간 분해(waterfall).
  • Performance 탭 — Parse HTML / Recalc Style / Layout / Paint / Composite의 시간을 millisecond 단위로 시각화.
  • Lighthouse — Core Web Vitals(LCP·INP·CLS) 점수.

시간/공간 복잡도 또는 trade-off

  • 왕복 횟수(RTT)가 지배적 — DNS·TCP·TLS는 각각 1~3 RTT. 같은 호스트라면 connection reuse로 한 번만 부담.
  • HTTPS 비용 — 보안의 대가로 첫 연결에서 1~2 RTT 추가. TLS 1.3·session resumption·HTTP/3로 상당 부분 회수.
  • JS 차단 vs 비차단 — defer/async를 모르면 첫 페인트가 늦어진다. 거의 항상 defer가 정답.
  • CSS 크기 — 작을수록 첫 페인트가 빠르다. critical CSS만 인라인하고 나머지는 비동기로.
  • HTML 응답 latency — 백엔드 처리시간(TTFB)도 사용자 인지 속도의 큰 부분.

실무에서 어디에 쓰이는가?

  • 장애 분석 — DNS 안 풀림? TLS 인증서 만료? 5xx 응답? 어느 단계인지 좁혀가기.
  • 성능 최적화 — preconnect·dns-prefetch·resource hints, defer/async, code splitting.
  • CDN — 사용자 근처에서 정적 자원을 응답해 RTT를 줄임.
  • SEO와 SSR — 검색 엔진 봇이 첫 HTML 안에 콘텐츠가 있어야 한다는 요구.

면접에 나올 질문과 답변

"주소창에 URL을 치고 엔터를 누르면 어떤 일이 일어나나요?"
URL 파싱 → DNS 질의로 IP 획득 → TCP 3-way handshake → (HTTPS면) TLS 핸드셰이크 → HTTP 요청 송신 → 서버 응답 수신 → HTML 스트리밍 파싱(DOM 생성) → CSS 파싱(CSSOM) → JS 다운로드·실행 → DOM·CSSOM이 합쳐져 Render Tree → Layout → Paint → Composite의 순서입니다. 각 단계는 RTT 또는 CPU 시간을 만들어 내고, 사용자가 느끼는 "로딩 시간"은 이들의 합입니다.
async와 defer의 차이는?
둘 다 다운로드를 비동기로 하지만 실행 시점이 다릅니다. async는 다운로드가 끝나는 즉시 HTML 파싱을 멈추고 실행하므로 스크립트 간 실행 순서가 보장되지 않습니다 — 광고·analytics처럼 독립적인 스크립트에 적합합니다. defer는 다운로드가 끝나도 HTML 파싱이 다 끝난 후에 작성된 순서대로 실행합니다. 일반 앱 스크립트는 거의 항상 defer가 정답입니다.
CSS가 렌더링을 차단한다고 들었는데, 정확히 어떻게 그렇죠?
브라우저는 CSSOM이 완성되기 전에는 첫 페인트를 시작하지 않습니다 — 스타일 없는 화면을 잠깐 보여줬다가 다시 그리면 끔찍한 FOUC(Flash of Unstyled Content)가 생기기 때문입니다. 그래서 <link rel="stylesheet">는 사실상 렌더링 차단 리소스이고, 큰 CSS 번들은 first paint를 직접 늦춥니다. 대응으로 critical CSS만 인라인 두고, 나머지는 media나 preload로 비동기화합니다.
TLS 핸드셰이크가 비싸다는데 얼마나 비싼가요?
TLS 1.2는 추가 2 RTT, TLS 1.3는 1 RTT가 기본입니다. session resumption(이전 세션 재개)은 TLS 1.3에서 0-RTT까지 줄여 줍니다. HTTPS 비용 대부분은 첫 연결에서 발생하므로, keep-alive로 같은 연결을 재사용하고, preconnect로 임계 도메인을 미리 연결하고, HTTP/2·3로 한 연결에서 다중 요청을 처리하는 것이 일반적인 대응입니다.
HTML이 다 도착하기 전에도 화면이 보이기 시작하는 이유는?
브라우저가 스트리밍 파싱을 하기 때문입니다. 응답 바이트가 청크로 도착하는 즉시 토큰화·트리 구성·렌더링 파이프라인이 진행됩니다. 그래서 백엔드에서 flush로 헤더와 일부 HTML을 먼저 보내면, 사용자는 본문이 다 도착하기 전에 헤더·네비게이션을 볼 수 있습니다(early flush). SSR 프레임워크의 streaming 옵션이 이 원리를 활용합니다.
URL의 fragment(#xxx)는 서버로 전송되나요?
아니요. fragment는 브라우저(클라이언트)만 사용하며 HTTP 요청 라인에 포함되지 않습니다. 그래서 같은 페이지 내 앵커 점프, 라우팅 라이브러리의 hash 모드(#/about)에 자주 쓰입니다. 다만 ?로 시작하는 query string은 서버로 전송되니 둘을 구분해야 합니다.
왜 DNS prefetch나 preconnect를 권장하나요?
외부 도메인을 처음 호출하는 시점에 DNS·TCP·TLS의 1~3 RTT가 한꺼번에 부담됩니다. <link rel="preconnect" href="//cdn.example.com">를 두면 그 도메인에 대한 연결을 먼저 만들어 둡니다. 이미지·폰트·API가 서로 다른 도메인에 흩어져 있는 사이트에서 이 hint가 첫 렌더 시점을 수백 ms 당기는 사례가 흔합니다.
