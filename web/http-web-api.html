HTTP와 Web API

Web API는 브라우저와 서버가 HTTP를 통해 데이터를 주고받는 인터페이스입니다. 브라우저에서는 fetch API로 요청을 보내고, 서버는 보통 REST 스타일의 URL·메서드·상태 코드·JSON으로 응답합니다. 인증과 상태 유지를 위해 쿠키, 토큰, 브라우저 저장소가 함께 쓰입니다.

구체적인 개념 설명

웹 페이지는 HTML만 받아 끝나지 않습니다. 화면이 뜬 뒤에도 JavaScript는 서버에 데이터를 요청하고, 사용자의 입력을 저장하고, 인증 상태를 확인합니다. 이때 브라우저와 서버 사이의 약속이 Web API입니다.

HTTP는 메서드(GET/POST/PUT/PATCH/DELETE), 상태 코드, 헤더, 바디로 의미를 표현합니다. 좋은 API는 이 의미를 자연스럽게 사용하고, 오류를 상태 코드와 구조화된 JSON으로 명확히 전달합니다.

비유 — 식당 주문서

클라이언트는 주문서(HTTP request)를 보내고, 서버는 요리와 결과표(HTTP response)를 돌려줍니다. 주문서에는 메뉴(GET/POST), 추가 요청사항(headers), 주문 내용(body)이 적혀 있습니다.

왜 필요한가?

  • 동적 데이터 — 페이지 새로고침 없이 사용자/상품/알림 정보를 가져온다.
  • 클라이언트-서버 분리 — 프론트엔드와 백엔드가 명확한 계약으로 협업한다.
  • 상태 관리 — 쿠키/토큰/스토리지로 로그인 상태나 설정을 유지한다.
  • 보안 경계 — CORS, credentials, SameSite 같은 브라우저 정책을 이해해야 한다.

어떻게 동작하는가?

1) fetch 요청/응답 흐름

fetch는 Promise를 반환하며, 응답 바디는 response.json() 같은 메서드로 한 번 더 읽는다.
const res = await fetch("/api/users/1", {
  headers: { "Accept": "application/json" },
});
if (!res.ok) {
  throw new Error(`HTTP ${res.status}`);
}
const user = await res.json();

2) REST 스타일

동작HTTP예시성질
목록 조회GET/userssafe, idempotent
단건 조회GET/users/1safe, idempotent
생성POST/usersnon-idempotent
전체 교체PUT/users/1idempotent
부분 수정PATCH/users/1보통 non-idempotent 가능
삭제DELETE/users/1idempotent

REST는 URL을 자원(resource) 중심으로 설계하고, 동작은 HTTP 메서드로 표현하는 스타일입니다. 모든 API가 완벽한 REST일 필요는 없지만, 메서드 의미를 지키면 캐시·재시도·문서화가 쉬워집니다.

3) 상태 코드와 에러 응답

  • 2xx — 성공. 200 OK, 201 Created, 204 No Content.
  • 3xx — 리다이렉션.
  • 4xx — 클라이언트 요청 문제. 400, 401, 403, 404, 409, 422.
  • 5xx — 서버 문제. 500, 502, 503.
{
  "code": "VALIDATION_ERROR",
  "message": "email 형식이 올바르지 않습니다.",
  "fields": { "email": "invalid format" }
}

4) CORS와 credentials

브라우저는 다른 Origin으로 요청할 때 CORS 정책을 적용합니다. 서버가 Access-Control-Allow-Origin 등 허용 헤더를 보내야 JS가 응답을 읽을 수 있습니다. 쿠키를 함께 보내려면 fetch의 credentials 옵션과 서버의 Access-Control-Allow-Credentials가 모두 필요합니다.

CORS 오류는 서버가 응답을 안 한 것이 아니라, 브라우저가 JS에게 응답을 숨긴 경우가 많다.

5) 쿠키 vs Web Storage

저장소자동 전송JS 접근용량대표 용도
Cookie요청마다 자동HttpOnly면 불가작음세션, 인증
localStorage자동 전송 X가능수 MB사용자 설정
sessionStorage자동 전송 X가능수 MB탭 단위 임시 상태
IndexedDB자동 전송 X가능큼오프라인 데이터

민감한 인증 토큰을 localStorage에 저장하면 XSS에 탈취될 수 있습니다. 쿠키를 쓴다면 HttpOnly, Secure, SameSite를 함께 고려해야 합니다.

6) AbortController와 timeout

const controller = new AbortController();
const timer = setTimeout(() => controller.abort(), 3000);
try {
  const res = await fetch("/api/search", { signal: controller.signal });
} finally {
  clearTimeout(timer);
}

fetch 자체에는 timeout 옵션이 없으므로 AbortController로 취소를 구현합니다. 화면 전환 시 이전 요청을 취소하지 않으면 늦게 도착한 응답이 최신 UI를 덮을 수 있습니다.

실무에서 어디에 쓰이는가?

  • SPA 데이터 로딩 — fetch/React Query/SWR로 API 호출.
  • 폼 제출 — JSON POST, validation error 표시.
  • 인증 — 쿠키 세션 또는 OAuth/OIDC 토큰 흐름.
  • 파일 업로드 — multipart/form-data, progress, abort 처리.
  • 캐싱 — HTTP Cache-Control, ETag, SWR 전략.

면접에 나올 질문과 답변

fetch에서 404가 나면 catch로 가나요?
아닙니다. fetch는 네트워크 오류나 CORS 차단처럼 요청 자체가 실패할 때 reject됩니다. HTTP 404/500 응답은 정상적으로 resolve되므로 response.ok나 status를 직접 확인해야 합니다.
GET과 POST의 차이는?
GET은 자원 조회에 사용하고 safe/idempotent 성질을 가져 캐시와 재시도에 적합합니다. POST는 서버 상태를 만들거나 처리하는 요청에 쓰며 일반적으로 idempotent하지 않습니다.
CORS는 왜 필요한가요?
브라우저에서 실행되는 임의의 JS가 다른 Origin의 응답을 마음대로 읽지 못하게 하기 위해 필요합니다. 서버가 허용한 Origin에 대해서만 브라우저가 JS에게 응답 접근을 허용합니다.
쿠키와 localStorage 중 인증 토큰 저장은 무엇이 안전한가요?
일반적으로 HttpOnly + Secure + SameSite 쿠키가 XSS로부터 토큰을 숨길 수 있어 더 안전한 선택입니다. localStorage는 JS가 읽을 수 있으므로 XSS가 발생하면 토큰이 쉽게 탈취됩니다. 다만 쿠키는 CSRF 방어를 함께 고려해야 합니다.
PUT과 PATCH의 차이는?
PUT은 자원 전체 교체를 의미하고 보통 idempotent합니다. PATCH는 자원의 일부 변경을 의미하며 문서화한 patch 형식에 따라 idempotent하지 않을 수도 있습니다.
AbortController는 언제 쓰나요?
요청 timeout, 검색어 변경 시 이전 요청 취소, 화면 unmount 시 네트워크 요청 취소에 사용합니다. 늦게 도착한 응답이 최신 상태를 덮는 race condition을 줄일 수 있습니다.
