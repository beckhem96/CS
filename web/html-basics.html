HTML 핵심 — 시맨틱·form·접근성·SEO

HTML(HyperText Markup Language)은 웹 페이지의 문서 구조를 표현하는 언어입니다. "이건 제목, 이건 본문, 이건 네비게이션"이라는 의미를 태그로 적으면, 브라우저·검색엔진·스크린리더가 그 구조를 이해해 각자에게 맞게 사용합니다. "그냥 div로 감싸도 보이긴 하잖아?"가 아니라 — 시맨틱 마크업은 접근성(a11y)·SEO·유지보수에 직접적으로 영향을 주는 설계 결정입니다.

구체적인 개념 설명

HTML 문서는 중첩된 요소(element)의 트리입니다. 각 요소는 시작 태그·내용·끝 태그로 이뤄지고, 속성(attribute)으로 메타정보를 더합니다. 이 트리가 곧 브라우저의 DOM이 되고, CSS·JS가 그것을 바탕으로 동작합니다.

현대 HTML(HTML5+)의 가장 중요한 변화는 시맨틱 요소의 도입입니다 — div·span으로만 짜이던 페이지가 header·nav·main·article·section·aside·footer 같은 의미 있는 요소로 표현되어, 문서가 자기 구조를 스스로 설명할 수 있게 됐습니다.

왜 필요한가?

  • 접근성 — 스크린리더는 시맨틱 요소를 통해 사용자에게 페이지 구조를 안내한다.
  • SEO — 검색 엔진은 h1·article·nav의 의미를 활용해 페이지를 이해한다.
  • 유지보수 — 의미가 드러나는 마크업은 다른 개발자(나 자신 포함)가 빠르게 이해할 수 있다.
  • 도구 친화 — 브라우저 reader mode, 자동 outline, 모바일 reader가 모두 시맨틱 정보를 활용.

어떻게 동작하는가?

1) HTML 문서의 골격

<!DOCTYPE html>
<html lang="ko">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>페이지 제목</title>
  <meta name="description" content="이 페이지의 한 문장 요약">
  <link rel="canonical" href="https://example.com/page">
  <link rel="stylesheet" href="style.css">
</head>
<body>
  <header>...</header>
  <main>...</main>
  <footer>...</footer>
  <script src="app.js" defer></script>
</body>
</html>
  • <!DOCTYPE html> — HTML5 표준 모드 활성화. 빠지면 quirks mode로 떨어져 옛 IE 호환 동작이 켜진다.
  • lang="ko" — 스크린리더의 발음·검색엔진의 언어 판정에 사용.
  • charset="UTF-8" — 한글 깨짐 방지의 첫 줄. head 안에서 가능한 한 일찍.
  • viewport — 모바일에서 가로 너비를 디바이스에 맞춤. 빠지면 데스크톱 폭으로 렌더 후 축소.

2) 시맨틱 요소 — 페이지의 지도

div의 바다보다 의미 있는 지도가 검색엔진·스크린리더·동료 개발자 모두에게 도움이 된다.
요소의미전형적 사용
<header>섹션의 머리말로고, 검색, 1차 네비게이션
<nav>주요 링크 모음전역 네비게이션, 목차
<main>페이지의 주요 콘텐츠 (한 페이지에 1개)본문 영역 전체
<article>자체 완결되는 콘텐츠 단위블로그 글, 뉴스 항목, 댓글 1개
<section>주제별 묶음 (보통 제목과 함께)"FAQ", "관련 상품"
<aside>본문과 간접 관련사이드바, 광고, 인용 정보
<footer>섹션의 마무리저작권, 보조 링크
<figure> + <figcaption>본문에서 참조되는 그림/표이미지 + 캡션

3) 제목 위계와 outline

h1~h6은 단순한 폰트 크기가 아니라 문서의 목차 트리입니다. 한 페이지에 보통 h1은 하나, 그 아래 h2·h3가 위계적으로 들어옵니다. 스크린리더 사용자는 헤딩만 점프하면서 페이지를 탐색하기 때문에 위계가 망가지면 페이지를 이해할 수 없게 됩니다. 

<!-- ❌ 시각적 크기만 보고 잘못 사용 -->
<h2>페이지 제목</h2>
<h4>섹션 제목</h4>       <!-- h3를 건너뜀 -->

<!-- ✅ 위계가 명확 -->
<h1>페이지 제목</h1>
  <h2>섹션 1</h2>
    <h3>세부 1-1</h3>
  <h2>섹션 2</h2>

4) form — 사용자 입력의 표준

Form은 단순한 입력 모음이 아니라 웹 표준이 보장하는 동작 묶음입니다 — Enter 키 제출, 자동 검증, 자동 완성, 모바일 키보드 변경, 접근성 라벨링이 거의 무료로 따라옵니다. 

<form action="/login" method="post">
  <label for="email">이메일</label>
  <input id="email" name="email" type="email" required autocomplete="email">

  <label for="pw">비밀번호</label>
  <input id="pw" name="password" type="password" required minlength="8"
         autocomplete="current-password">

  <button type="submit">로그인</button>
</form>
  • label의 for가 input의 id와 일치해야 라벨 클릭으로 포커스가 이동하고, 스크린리더가 라벨을 읽어 준다.
  • type이 의미를 만든다 — email·tel·url·number·date는 모바일 키보드와 검증을 다르게 한다.
  • autocomplete는 무시하지 말 것 — 비밀번호 매니저·1Password가 이 속성을 보고 동작한다.
  • required·minlength·pattern으로 클라이언트 검증. 단, 서버 검증은 따로 해야 한다.
  • 한 form 안에서 button의 기본 type은 submit이다 — 버튼이 의도치 않게 폼을 제출하지 않으려면 type="button" 명시.

5) 버튼 vs 링크 — 가장 자주 틀리는 한 줄

주소가 바뀌면 <a href>, 동작을 트리거하면 <button>입니다. 둘은 외형이 비슷해도 키보드·접근성에서 완전히 다릅니다. 

<!-- ❌ 안티패턴 -->
<div onclick="navigate()">상세 보기</div>     <!-- 키보드로 못 누름, 스크린리더가 읽지 않음 -->
<a href="#" onclick="doStuff()">삭제</a>     <!-- 링크인 척하지만 동작만, 새 탭 열기·미들 클릭이 깨짐 -->

<!-- ✅ 의도에 맞게 -->
<a href="/items/42">상세 보기</a>
<button type="button" onclick="doStuff()">삭제</button>

6) 이미지와 미디어

<!-- alt는 "이미지가 못 뜰 때 보이는 글"이 아니라 "스크린리더가 읽는 글" -->
<img src="cat.jpg" alt="갈색 줄무늬 고양이가 창틀에 앉아 있다"
     width="600" height="400" loading="lazy">

<!-- 장식용 이미지는 alt를 빈 문자열로 — "장식이라 안 읽어도 된다" -->
<img src="divider.svg" alt="">

<!-- 반응형 이미지: 화면 크기에 따라 다른 파일 제공 -->
<img src="hero.jpg"
     srcset="hero-400.jpg 400w, hero-800.jpg 800w, hero-1600.jpg 1600w"
     sizes="(max-width: 600px) 100vw, 800px">

<video controls preload="metadata" poster="thumb.jpg">
  <source src="clip.webm" type="video/webm">
  <source src="clip.mp4"  type="video/mp4">
  자막: <track kind="captions" src="ko.vtt" srclang="ko" label="한국어">
</video>

width·height를 적어 두면 브라우저가 이미지를 받기 전에 자리를 미리 잡아 둬서 CLS(Cumulative Layout Shift)를 막을 수 있습니다. loading="lazy"는 viewport 밖 이미지를 지연 로딩해 첫 페이지 로드를 빠르게 합니다.

7) 접근성(a11y)의 기본기

  • 의미 있는 요소를 먼저 — div onclick 대신 button. 시맨틱이 가장 큰 a11y 도구.
  • 이미지에 alt — 의미 있는 이미지엔 설명, 장식엔 빈 문자열.
  • label 연결 — 모든 입력 요소는 라벨이 있어야 한다.
  • 색상 대비 — WCAG AA 기준 최소 4.5:1 (작은 텍스트).
  • 키보드 탐색 — Tab으로 모든 인터랙션에 도달 가능, 포커스 표시 유지.
  • ARIA는 보조 도구 — 시맨틱 HTML로 안 되는 경우에만 ARIA. "잘못된 ARIA가 ARIA 없는 것보다 나쁘다".
<!-- 시맨틱이 우선, ARIA는 보충 -->
<nav aria-label="주요 메뉴">...</nav>       <!-- 페이지에 nav가 여러 개일 때 구분 -->
<button aria-expanded="false" aria-controls="menu">메뉴 열기</button>
<ul id="menu" hidden>...</ul>

<!-- skip link — Tab 첫 누름에서 본문으로 바로 점프 -->
<a class="skip-link" href="#main">본문으로 건너뛰기</a>

8) SEO 기본기

<head>
  <title>[페이지명] | 사이트명</title>
  <meta name="description" content="60자 내외, 검색결과에 노출됨">
  <link rel="canonical" href="https://example.com/products/42">

  <!-- Open Graph: SNS 공유 미리보기 -->
  <meta property="og:title"       content="...">
  <meta property="og:description" content="...">
  <meta property="og:image"       content="https://.../share.jpg">
  <meta property="og:type"        content="article">

  <!-- 구조화 데이터(JSON-LD) — 리치 결과에 사용 -->
  <script type="application/ld+json">
  {
    "@context": "https://schema.org",
    "@type": "Article",
    "headline": "...",
    "author": {"@type": "Person", "name": "..."},
    "datePublished": "2026-05-08"
  }
  </script>
</head>
  • title과 description은 검색결과 클릭률(CTR)에 직접 영향. 페이지마다 달라야 한다.
  • canonical은 같은 콘텐츠가 여러 URL로 접근될 때 "이게 정본"이라고 알려준다 — 페이지네이션·필터 URL 중복 SEO 패널티 회피.
  • 구조화 데이터로 검색결과의 별점·이벤트·FAQ 같은 리치 결과를 받을 수 있다.
  • 중요한 콘텐츠는 첫 HTML 응답에 포함되어야 검색 봇이 본다 — 클라이언트 사이드 렌더링(CSR)만 하는 SPA는 SSR/Prerender 고려.

9) HTML의 보이지 않는 동작 — 자동 보정

HTML 파서는 잘못된 마크업을 만나도 가능한 한 트리를 만들어 냅니다. 이 관대함이 웹의 호환성을 만들었지만, 때로 의도와 다른 결과를 낳습니다. 

<!-- 작성한 코드 -->
<p>Hello<div>World</div></p>

<!-- 브라우저가 만드는 DOM (p 안에 div 못 들어감 → p가 자동으로 닫힘) -->
<p>Hello</p>
<div>World</div>
<p></p>

이런 차이는 CSS·JS가 의도치 않게 동작하는 원인이 됩니다. 의심스러우면 DevTools Elements 탭에서 실제 DOM을 보세요.

시간/공간 복잡도 또는 trade-off

  • 시맨틱 vs 단순함 — div·span으로 "일단 빠르게"는 가능하지만 a11y·SEO 비용을 나중에 갚게 된다.
  • ARIA의 역설 — 잘못된 ARIA는 시맨틱 HTML보다 나쁘다. 가능하면 시맨틱 요소로 끝내고 ARIA는 보충.
  • HTML 크기 — 첫 HTML이 작을수록 TTFB·FCP가 빠르다. 그러나 SSR로 의미 있는 콘텐츠를 첫 응답에 담는 것이 SEO에 유리.
  • 인라인 vs 외부 — critical CSS 인라인은 첫 페인트를 빠르게 하지만 캐시 활용도가 떨어짐.

실무에서 어디에 쓰이는가?

  • 접근성 감사 — Lighthouse·axe로 시맨틱·라벨·대비 검증.
  • SEO — 메타 태그·구조화 데이터·정본 URL 관리. 콘텐츠 사이트의 트래픽 결정.
  • 이메일 HTML — 옛 HTML4 호환성을 요구하는 메일 클라이언트(Outlook 등) 때문에 별도 규칙.
  • print 스타일 — @media print와 시맨틱 마크업으로 인쇄 친화적인 페이지.

면접에 나올 질문과 답변

시맨틱 HTML이 왜 중요한가요?
세 가지 이유입니다 — (1) 접근성: 스크린리더가 시맨틱 요소를 통해 페이지 구조를 사용자에게 안내합니다. (2) SEO: 검색 엔진은 h1·article·nav의 의미를 활용해 페이지를 이해하고 색인합니다. (3) 유지보수성: 의미가 드러나는 마크업은 동료(나 자신 포함)가 코드를 빠르게 이해할 수 있게 합니다. 시맨틱은 비용이 거의 없는 가장 큰 ROI 결정입니다.
div와 button의 차이는 단순히 외형 아닌가요?
외형은 같게 만들 수 있지만 동작이 완전히 다릅니다. button은 (1) Tab으로 포커스 가능, (2) Enter/Space로 활성화, (3) 스크린리더가 "버튼"으로 읽어 줌, (4) form 안에서 자동으로 submit이 됨. div는 이 중 어느 것도 자동으로 안 됩니다. 그래서 클릭 핸들러를 단 div는 키보드 사용자와 보조기기 사용자에게 사실상 보이지 않는 컴포넌트가 됩니다.
label의 for와 input의 id는 왜 연결해야 하나요?
(1) 라벨 클릭 시 자동으로 입력 요소에 포커스가 갑니다(특히 모바일에서 작은 체크박스 누르기 편해짐). (2) 스크린리더가 입력 요소에 포커스가 갈 때 라벨 텍스트를 읽어 줍니다. (3) 비밀번호 매니저가 어떤 라벨이 어떤 input에 해당하는지 더 정확히 매칭합니다. 단, <label>텍스트<input></label>처럼 감싸는 형태도 같은 효과를 냅니다.
a 태그를 onclick과 함께 쓰면 안 되는 이유?
a는 "주소가 바뀐다"는 약속이라 사용자는 (1) 우클릭으로 새 탭, (2) 미들 클릭으로 새 탭, (3) 마우스오버로 URL 미리 보기를 기대합니다. href="#" + onclick 패턴은 이 모든 기대를 깨뜨립니다. 주소가 바뀌면 <a href="...">, 동작만 한다면 <button>이 정답이며, 두 의도를 섞지 마세요.
접근성을 위해 ARIA를 많이 붙이는 게 좋은가요?
그 반대입니다. "잘못된 ARIA가 ARIA 없는 것보다 나쁘다"가 W3C의 1번 규칙입니다. ARIA는 시맨틱 HTML로 표현 안 되는 경우(예: 커스텀 콤보박스) 보강 도구이지 1순위가 아닙니다. button을 만들 거면 div role="button"이 아니라 그냥 <button>. ARIA는 정확히 알고 쓸 때만 가치가 있고, 잘못된 role/state는 보조기기를 혼란시킵니다.
img의 alt에 무엇을 적어야 하나요?
그 이미지가 콘텐츠를 전달하면 이미지가 표현하는 의미를 적습니다("로그인 버튼이 있는 헤더 사진"이 아니라 "갈색 고양이가 창틀에 앉아 있다"). 의미 없이 장식만이라면 빈 문자열(alt="")을 명시 — 스크린리더가 건너뜁니다. alt 속성을 아예 빼면 스크린리더가 파일명을 읽어 버려 끔찍해집니다.
SPA에서 SEO를 어떻게 챙기나요?
클라이언트 사이드 렌더링만 하는 SPA는 첫 HTML이 거의 비어 있어 검색 봇이 콘텐츠를 못 봅니다. 대응 옵션은 (1) SSR(Next.js·Nuxt) — 서버에서 첫 HTML을 만들어 응답, (2) SSG — 빌드 타임에 정적 HTML 생성, (3) Prerender — 봇 요청만 헤드리스 브라우저로 미리 그려 응답. 그리고 모든 페이지에 고유한 title·description·canonical을 두는 것이 기본입니다.
canonical 태그는 왜 필요한가요?
같은 콘텐츠가 여러 URL로 접근 가능할 때(예: ?utm=..., 페이지네이션, https vs http) 검색엔진이 중복으로 판단해 SEO 점수가 분산되거나 페널티가 생길 수 있습니다. <link rel="canonical" href="...">로 "이 콘텐츠의 정본은 여기"라고 명시하면 검색엔진이 색인 결과를 한쪽으로 모읍니다.
