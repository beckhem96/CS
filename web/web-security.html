웹 보안 기본 — XSS·CSRF·CSP·SameSite

웹 보안은 브라우저가 신뢰할 수 없는 입력과 외부 사이트를 다루는 환경에서 사용자와 서비스를 보호하는 기술입니다. 핵심 위협은 XSS, CSRF, 잘못된 CORS, 민감한 토큰 저장이며, 방어는 출력 인코딩·CSP·SameSite/HttpOnly/Secure 쿠키·CSRF 토큰으로 시작합니다.

구체적인 개념 설명

웹은 기본적으로 열린 플랫폼입니다. 사용자는 임의의 사이트를 방문하고, 브라우저는 여러 Origin의 리소스를 동시에 로드하며, JS는 강력한 DOM/네트워크 API를 가집니다. 이 유연성이 공격 표면이 됩니다.

웹 보안의 핵심은 "누가 이 코드를 실행하게 했는가", "이 요청은 사용자의 의도인가", "이 응답을 JS가 읽어도 되는가", "이 토큰은 탈취되어도 안전한가"를 계속 묻는 것입니다.

비유 — 출입증과 대리 서명

XSS는 공격자가 내 사무실 안에 몰래 들어와 내 컴퓨터로 명령을 실행하는 것이고, CSRF는 내가 로그인된 상태를 악용해 공격자가 서류를 대신 제출하게 만드는 것입니다. 둘은 모두 사용자 권한을 악용하지만 경로가 다릅니다.

왜 필요한가?

  • 사용자 보호 — 세션 탈취, 개인정보 유출, 악성 요청 방지.
  • 서비스 보호 — 계정 takeover, 결제/설정 변경, 데이터 변조 방지.
  • 브라우저 정책 이해 — SOP/CORS/쿠키 속성을 잘못 이해하면 취약점이 생긴다.
  • 기본값 강화 — 보안은 나중에 붙이는 기능이 아니라 초기 설계다.

어떻게 동작하는가?

1) XSS(Cross-Site Scripting)

XSS는 공격자가 삽입한 스크립트가 피해자의 브라우저에서 서비스의 Origin 권한으로 실행되는 취약점입니다. DOM을 읽고, 사용자를 대신해 요청을 보내고, localStorage의 토큰을 훔칠 수 있습니다.

XSS는 입력 검증만으로 부족하고, 출력 위치에 맞는 인코딩이 핵심이다.
// 위험: 사용자 입력을 HTML로 해석
container.innerHTML = userComment;

// 안전: 텍스트로 삽입
container.textContent = userComment;
  • Stored XSS — DB에 저장된 악성 스크립트가 여러 사용자에게 실행.
  • Reflected XSS — URL/요청 값이 즉시 응답에 반사되어 실행.
  • DOM XSS — 서버 없이 클라이언트 JS가 location/hash 값을 위험하게 DOM에 삽입.

2) CSRF(Cross-Site Request Forgery)

CSRF는 사용자가 로그인된 사이트에 대해, 공격 사이트가 사용자의 브라우저를 시켜 원치 않는 요청을 보내게 하는 공격입니다. 쿠키는 요청에 자동으로 붙기 때문에 문제가 됩니다.

CSRF는 JS가 응답을 읽는 문제가 아니라, 요청이 사용자 인증 상태로 보내지는 문제다.

4) CSP(Content Security Policy)

CSP는 브라우저에게 "어떤 스크립트/스타일/이미지 출처를 허용할지" 알려주는 정책입니다. XSS가 발생해도 임의 inline script 실행을 막는 보조 방어선이 됩니다.

Content-Security-Policy:
  default-src 'self';
  script-src 'self' 'nonce-random123';
  object-src 'none';
  base-uri 'self';

CSP는 XSS를 고치는 대신이 아니라 피해를 줄이는 안전망입니다. 기본 방어는 여전히 출력 인코딩과 위험한 DOM API 회피입니다.

5) CORS 오해

CORS는 서버를 보호하는 인증/인가 기능이 아닙니다. 브라우저가 cross-origin 응답을 JS에게 보여줄지 정하는 정책입니다. 서버 API는 CORS와 별개로 인증과 권한 검사를 반드시 해야 합니다.

CORS는 브라우저 보안 정책이지 API 접근 제어의 대체물이 아니다.

6) 토큰 저장 위치

  • localStorage — 구현은 쉽지만 XSS에 탈취됩니다. 민감 토큰 저장은 피하는 것이 좋습니다.
  • HttpOnly Cookie — JS가 읽을 수 없어 XSS 탈취에 강하지만 CSRF를 고려해야 합니다.
  • 메모리 — 새로고침 시 사라지지만 노출면이 작습니다. SPA에서 access token을 짧게 유지할 때 사용.

실무에서 어디에 쓰이는가?

  • 로그인 세션 — HttpOnly + Secure + SameSite 쿠키.
  • 사용자 입력 출력 — HTML escape, sanitize, markdown renderer 안전 설정.
  • 관리자 페이지 — CSRF 토큰과 권한 검사 필수.
  • 프론트 배포 — CSP, HSTS, X-Content-Type-Options 같은 보안 헤더.
  • OAuth/OIDC — state/nonce/PKCE로 CSRF와 code injection 방어.

면접에 나올 질문과 답변

XSS와 CSRF의 차이는?
XSS는 공격자의 스크립트가 피해자 브라우저에서 서비스 Origin 권한으로 실행되는 공격입니다. CSRF는 공격 사이트가 피해자 브라우저의 로그인 쿠키를 이용해 사용자가 의도하지 않은 요청을 보내게 하는 공격입니다. XSS는 코드 실행, CSRF는 요청 위조가 핵심입니다.
XSS 방어의 핵심은?
출력 위치에 맞는 인코딩과 위험한 DOM API 회피입니다. 사용자 입력을 innerHTML로 넣지 말고 텍스트는 textContent로 넣어야 합니다. CSP와 HttpOnly 쿠키는 피해를 줄이는 보조 방어입니다.
CSRF는 왜 쿠키 인증에서 문제가 되나요?
브라우저는 요청 대상 도메인의 쿠키를 자동으로 붙입니다. 사용자가 bank.com에 로그인된 상태에서 evil.com을 방문하면 evil.com이 bank.com으로 form 요청을 보내도 쿠키가 자동 첨부될 수 있습니다. 그래서 SameSite와 CSRF 토큰이 필요합니다.
HttpOnly 쿠키는 XSS를 완전히 막나요?
아닙니다. HttpOnly는 JS가 쿠키 값을 읽지 못하게 해 토큰 탈취를 막지만, XSS 스크립트가 사용자를 대신해 같은 Origin으로 요청을 보내는 것까지 막지는 못합니다. XSS 자체는 출력 인코딩과 CSP로 줄여야 합니다.
CORS를 막으면 API가 안전한가요?
아닙니다. CORS는 브라우저 JS가 cross-origin 응답을 읽을 수 있는지 제한하는 정책입니다. curl, 서버-서버 요청, 모바일 앱에는 적용되지 않습니다. API 보안은 인증/인가/입력 검증을 서버에서 해야 합니다.
localStorage에 JWT를 저장하면 왜 위험한가요?
localStorage는 JS에서 접근 가능하므로 XSS가 발생하면 토큰을 그대로 읽어 외부로 보낼 수 있습니다. 인증 토큰은 가능하면 짧은 수명의 access token + HttpOnly refresh cookie 또는 서버 세션 방식으로 관리하는 것이 안전합니다.
