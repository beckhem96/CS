브라우저 렌더링 파이프라인

브라우저 렌더링 파이프라인은 HTML/CSS/JS를 실제 픽셀로 바꾸는 과정입니다. 큰 흐름은 HTML 파싱 → DOM/CSSOM → Render Tree → Layout → Paint → Composite이며, 성능 최적화의 핵심은 이 단계 중 무엇을 다시 하게 만드는지 이해하는 것입니다.

구체적인 개념 설명

브라우저는 HTML 파일을 받자마자 위에서 아래로 파싱하며 DOM을 만들고, CSS를 파싱해 CSSOM을 만듭니다. DOM과 CSSOM을 합쳐 실제 화면에 그릴 노드만 모은 Render Tree를 만들고, 각 박스의 위치와 크기를 계산한 뒤(Layout), 픽셀로 칠하고(Paint), 레이어를 GPU가 합성합니다(Composite).

웹 성능 문제는 대개 이 파이프라인을 너무 자주, 너무 크게 다시 실행시키는 데서 옵니다. 예를 들어 width를 바꾸면 layout부터 다시 필요하지만, transform은 합성 단계에서 처리될 수 있어 더 저렴합니다.

비유 — 신문 제작

기사를 모으고(DOM), 스타일 가이드를 적용하고(CSSOM), 실제 지면에 들어갈 요소만 고르고(Render Tree), 지면 배치를 계산하고(Layout), 잉크를 칠하고(Paint), 여러 판을 합치는 것(Composite)이 렌더링 파이프라인입니다.

왜 필요한가?

  • 성능 튜닝 — reflow/repaint 원인을 알아야 빠른 UI를 만든다.
  • 렌더링 버그 해결 — display/visibility/position/z-index 문제를 파이프라인 관점으로 해석.
  • JS 최적화 — DOM 읽기/쓰기를 섞으면 layout thrashing이 생긴다.
  • Core Web Vitals 이해 — LCP/INP/CLS는 렌더링 흐름과 직접 연결된다.

어떻게 동작하는가?

1) 전체 파이프라인

렌더링 최적화는 어느 단계를 다시 실행시키는지 줄이는 일이다.

2) DOM과 CSSOM

DOM은 문서 구조, CSSOM은 스타일 규칙 트리입니다. CSS는 render-blocking 자원입니다. CSSOM이 완성되어야 정확한 스타일을 계산할 수 있으므로, 큰 CSS는 첫 렌더링을 늦출 수 있습니다.

<link rel="stylesheet" href="style.css">
<script src="app.js" defer></script>

일반 script는 파서를 막을 수 있으므로, 초기 렌더링에 필요 없는 JS는 defer나 type="module"을 고려합니다.

3) Render Tree

Render Tree는 DOM과 CSSOM을 결합해 실제로 그릴 노드만 포함합니다. display: none 요소는 Render Tree에 들어가지 않지만, visibility: hidden은 공간은 차지하므로 들어갑니다.

4) Layout(Reflow)

Layout은 각 박스의 위치와 크기를 계산합니다. viewport 크기, 폰트, 콘텐츠, display/position/width/height 변화가 영향을 줍니다. Layout은 트리의 다른 부분까지 영향을 줄 수 있어 비쌉니다.

모든 CSS 변경 비용이 같지 않다. 레이아웃을 바꾸는 속성이 가장 비싸다.

5) Paint와 Composite

Paint는 텍스트, 배경, border, shadow 등을 픽셀로 그리는 단계입니다. Composite는 여러 레이어를 합쳐 최종 화면을 만듭니다. GPU가 처리할 수 있는 레이어는 합성이 빠르지만, 레이어를 너무 많이 만들면 메모리 비용이 커집니다.

6) Layout Thrashing

레이아웃 읽기와 쓰기를 반복하면 브라우저가 최신 값을 계산하기 위해 layout을 동기적으로 강제 실행합니다.

// 나쁜 예: 반복마다 layout 강제 가능
for (const box of boxes) {
  box.style.width = "200px";
  console.log(box.offsetWidth);
}

// 좋은 예: 읽기와 쓰기 분리
const widths = boxes.map(box => box.offsetWidth);
boxes.forEach(box => { box.style.width = "200px"; });

7) requestAnimationFrame

requestAnimationFrame은 브라우저의 다음 렌더링 프레임 직전에 콜백을 실행합니다. 애니메이션과 DOM 쓰기를 프레임에 맞추면 불필요한 중간 렌더링을 줄일 수 있습니다.

function move() {
  box.style.transform = `translateX(${x}px)`;
  x += 2;
  requestAnimationFrame(move);
}
requestAnimationFrame(move);

실무에서 어디에 쓰이는가?

  • 애니메이션 최적화 — top/left 대신 transform 사용.
  • 스크롤 성능 — passive listener, DOM 조작 최소화.
  • 대형 목록 — virtual scrolling으로 DOM 노드 수 제한.
  • 초기 로딩 — critical CSS, defer script, lazy loading.

면접에 나올 질문과 답변

브라우저 렌더링 파이프라인을 설명해 주세요.
HTML을 파싱해 DOM을 만들고, CSS를 파싱해 CSSOM을 만든 뒤 둘을 결합해 Render Tree를 만듭니다. 이후 각 요소의 위치와 크기를 계산하는 Layout, 픽셀을 그리는 Paint, 레이어를 합치는 Composite 단계를 거쳐 화면에 표시됩니다.
Reflow와 Repaint의 차이는?
Reflow는 layout 재계산으로 요소의 위치와 크기를 다시 계산하는 과정이고, Repaint는 위치/크기는 그대로인데 색·배경·그림자 같은 픽셀을 다시 칠하는 과정입니다. Reflow는 보통 Repaint를 동반하고 비용이 더 큽니다.
display:none과 visibility:hidden의 렌더링 차이는?
display:none은 Render Tree에서 제외되어 공간을 차지하지 않습니다. visibility:hidden은 보이지 않을 뿐 layout 공간은 차지하므로 Render Tree에 남습니다.
왜 transform 애니메이션이 top/left보다 좋나요?
top/left는 layout을 다시 계산하게 만들 수 있지만, transform은 별도 레이어에서 composite 단계만으로 처리될 수 있습니다. 그래서 프레임 드랍이 적고 GPU 가속을 활용하기 쉽습니다.
Layout thrashing이란?
DOM 쓰기와 레이아웃 읽기를 반복해 브라우저가 최신 레이아웃 값을 얻기 위해 layout을 동기적으로 여러 번 강제 실행하는 현상입니다. 읽기와 쓰기를 분리하고 batch 처리하면 줄일 수 있습니다.
CSS가 render-blocking인 이유는?
브라우저가 정확한 스타일을 알아야 Render Tree와 layout을 만들 수 있기 때문입니다. CSSOM이 완성되지 않으면 잘못된 화면을 그릴 수 있어 CSS 로딩은 초기 렌더링을 막을 수 있습니다.
