DOM과 이벤트

DOM(Document Object Model)은 HTML 문서를 JavaScript가 조작할 수 있는 트리 형태의 객체 모델로 바꾼 것입니다. 이벤트는 클릭·입력·스크롤 같은 사용자/브라우저 행동을 객체로 전달하는 메커니즘이며, 캡처링 → 타깃 → 버블링 순서로 흐릅니다.

구체적인 개념 설명

브라우저는 HTML 문자열을 그대로 화면에 그리지 않습니다. 먼저 HTML을 파싱해 노드 트리인 DOM을 만들고, JavaScript는 이 DOM API를 통해 요소를 찾고, 속성을 바꾸고, 이벤트를 등록합니다. 즉 DOM은 HTML과 JS 사이의 다리입니다.

이벤트는 특정 요소에서만 끝나지 않고 트리를 따라 이동합니다. 이 특성 덕분에 부모 요소 하나에 이벤트 핸들러를 걸어 자식들의 이벤트를 한 번에 처리하는 이벤트 위임(event delegation)이 가능합니다.

비유 — 건물 안내도와 안내 방송

DOM은 건물의 층·방·문을 나타내는 안내도입니다. 이벤트는 어느 방에서 일어난 안내 방송이고, 방송은 복도와 층을 지나 관리실까지 전파될 수 있습니다.

왜 필요한가?

  • 동적 UI — JS가 DOM을 조작해 화면을 바꾼다.
  • 사용자 상호작용 — 클릭, 입력, 키보드, 스크롤을 처리한다.
  • 성능 최적화 — DOM 조작은 layout/paint 비용을 유발할 수 있다.
  • 프레임워크 이해 — React/Vue도 결국 DOM 업데이트를 효율화하는 도구다.

어떻게 동작하는가?

1) HTML에서 DOM 트리로

DOM 노드는 부모/자식/형제 관계를 가진 객체이며 JS가 이 객체를 탐색·수정한다.
const button = document.querySelector("button");
button.textContent = "저장";
button.classList.add("primary");

2) 이벤트 흐름 — capture → target → bubble

이벤트는 위에서 내려와 타깃에 도달한 뒤 다시 위로 올라간다.
button.addEventListener("click", e => {
  console.log("bubble phase");
});

document.addEventListener("click", e => {
  console.log("capture phase");
}, { capture: true });

3) 이벤트 위임(Event Delegation)

리스트 항목이 1,000개라면 각 버튼에 핸들러를 붙이는 대신 부모 ul에 하나만 붙이고, event.target으로 실제 클릭 대상을 확인합니다.

document.querySelector("ul").addEventListener("click", (event) => {
  const button = event.target.closest("button[data-id]");
  if (!button) return;
  deleteItem(button.dataset.id);
});
  • 동적으로 추가되는 자식에도 자동 적용됩니다.
  • 핸들러 수가 줄어 메모리와 등록 비용이 줄어듭니다.
  • 단, 이벤트가 bubble되지 않는 종류도 있어 확인이 필요합니다.

4) 기본 동작과 전파 제어

form.addEventListener("submit", (event) => {
  event.preventDefault();   // 브라우저 기본 submit/navigation 막기
  saveForm();
});

button.addEventListener("click", (event) => {
  event.stopPropagation();  // 부모로 bubble되는 것 막기
});

preventDefault()는 기본 행동을 막고, stopPropagation()은 이벤트 전파를 막습니다. 둘은 역할이 다릅니다.

5) DOM 조작 비용

DOM을 바꾸면 브라우저가 style/layout/paint를 다시 해야 할 수 있습니다. 특히 레이아웃 정보를 읽고 쓰는 작업을 반복하면 forced synchronous layout이 생깁니다.

// 나쁜 예: 읽기와 쓰기가 섞여 layout이 반복될 수 있음
for (const item of items) {
  const h = item.offsetHeight;
  item.style.height = `${h + 10}px`;
}

// 좋은 예: 읽기 먼저, 쓰기 나중
const heights = items.map(item => item.offsetHeight);
items.forEach((item, i) => item.style.height = `${heights[i] + 10}px`);

시간/공간 복잡도 또는 trade-off

기법장점주의점왜 그런가?
직접 핸들러단순요소 많으면 비용 증가노드마다 listener 등록
이벤트 위임동적 요소와 대량 목록에 강함target 판별 필요bubble 흐름 활용
DOM batch렌더링 비용 감소코드 구조 고려 필요layout/paint 횟수 감소
stopPropagation충돌 방지남용 시 디버깅 어려움상위 핸들러가 이벤트를 못 봄

실무에서 어디에 쓰이는가?

  • 대량 목록 UI — 이벤트 위임으로 버튼·메뉴 처리.
  • 폼 처리 — submit 기본 동작을 막고 fetch로 저장.
  • 모달/드롭다운 — document click으로 바깥 클릭 감지.
  • 성능 튜닝 — DOM 읽기/쓰기 분리, fragment 사용, 가상 DOM/렌더링 최적화.

면접에 나올 질문과 답변

DOM이란 무엇인가요?
HTML 문서를 브라우저가 파싱해 만든 트리 형태의 객체 모델입니다. JavaScript는 DOM API를 통해 요소를 탐색하고, 속성·텍스트·스타일을 바꾸며, 이벤트를 등록합니다.
이벤트 버블링과 캡처링의 차이는?
캡처링은 이벤트가 document에서 타깃으로 내려가는 단계이고, 버블링은 타깃에서 document로 올라가는 단계입니다. 기본 addEventListener는 버블링 단계에서 실행되고, {capture: true} 옵션으로 캡처 단계에서 실행할 수 있습니다.
이벤트 위임이란?
자식 각각에 핸들러를 붙이지 않고 부모에 하나만 붙여 이벤트 버블링을 이용해 처리하는 패턴입니다. 동적으로 추가되는 요소에도 적용되고, 대량 요소에서 listener 수를 줄일 수 있습니다.
preventDefault와 stopPropagation의 차이는?
preventDefault는 링크 이동이나 폼 제출 같은 브라우저 기본 동작을 막고, stopPropagation은 이벤트가 부모로 전파되는 것을 막습니다. 기본 동작과 전파는 별개의 개념입니다.
DOM 조작이 왜 느릴 수 있나요?
DOM 변경은 style 계산, layout, paint, composite 같은 렌더링 작업을 유발할 수 있습니다. 특히 레이아웃 값을 읽고 바로 스타일을 쓰는 작업을 반복하면 브라우저가 layout을 동기적으로 여러 번 수행합니다.
동적으로 추가된 버튼에 이벤트를 걸려면 어떻게 하나요?
부모 요소에 이벤트 위임을 사용합니다. 부모에 클릭 핸들러를 등록하고 event.target.closest()로 실제 버튼을 찾으면 나중에 추가된 자식도 처리할 수 있습니다.
