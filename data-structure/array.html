배열(Array)

배열이란, 같은 타입의 데이터를 메모리상에 연속(contiguous)으로 늘어놓고 인덱스(index)로 한 번에 꺼낼 수 있게 만든 자료구조입니다.

구체적인 개념 설명

배열을 가장 쉽게 떠올리는 비유는 학교 사물함입니다. 0번부터 N-1번까지 번호가 붙은 칸이 한 줄로 쭉 이어져 있고, 각 칸의 크기는 모두 똑같습니다. "37번 사물함 열어!" 하면 굳이 1번부터 세지 않아도 한 번에 갈 수 있죠. 배열도 마찬가지입니다.

배열의 핵심 성질은 두 가지입니다.

  • 연속된 메모리 — 첫 번째 원소의 주소를 알면, "주소 + (인덱스 × 한 칸 크기)"로 모든 원소의 위치를 즉시 계산할 수 있습니다.
  • 고정된 원소 크기 — 모든 칸이 같은 크기여야 위 계산이 성립합니다. 그래서 같은 타입(int면 int, double이면 double)으로 통일됩니다.

이 성질 덕분에 인덱스로 접근하는 비용이 O(1)이라는 강력한 장점이 생기지만, 반대로 중간에 끼워넣기/빼내기는 비싸다는 단점도 같이 따라옵니다 (뒤 원소들을 한 칸씩 밀어야 하니까).

정적 배열 vs 동적 배열

C의 int arr[10]처럼 크기를 고정해두는 게 정적 배열이고, Java의 ArrayList나 Python의 list처럼 자동으로 늘어나는 게 동적 배열입니다. 동적 배열은 내부적으로 정적 배열을 잡아두고, 다 차면 더 큰 배열로 옮겨담는(보통 2배) 방식으로 동작해요.

왜 필요한가?

"데이터 여러 개를 한 변수에 묶고 싶다"는 게 출발점입니다. 변수를 100개 만들어 score1, score2, ..., score100으로 관리할 수는 없잖아요. 거기에 더해 "i번째 데이터를 빠르게 꺼내고 싶다"는 요구가 생기면 배열이 답이 됩니다.

예를 들어 학생 100만 명의 성적 평균을 구한다고 해봅시다. 배열이라면 arr[i]로 i번째 점수에 즉시 접근하면서 한 바퀴 돌면 끝이지만, 만약 임의 접근이 안 되는 자료구조였다면 매번 처음부터 i번까지 따라가야 했을 거예요.

어떻게 동작하는가?

1) 메모리 배치

아래는 길이 5짜리 정수 배열이 메모리에 어떻게 놓이는지 보여줍니다. 한 칸이 4바이트라고 가정합니다.

시작 주소 + (인덱스 × 원소 크기) 한 번이면 어떤 위치든 바로 갈 수 있다.

2) 중간에 삽입할 때 일어나는 일

인덱스 1에 값 15를 끼워넣는다고 해보세요. 뒤 원소들을 모두 한 칸씩 밀어야 합니다.

앞쪽에 끼워넣을수록 밀어야 할 원소가 많아져 비용이 커진다 — 최악 O(n).

3) Java 코드로 보기

// 정적 배열
int[] arr = new int[5];
arr[0] = 10;
arr[1] = 20;
int x = arr[1];     // O(1) 접근

// 동적 배열 (ArrayList) — 내부적으로 배열을 더 큰 것으로 옮김
import java.util.ArrayList;

ArrayList<Integer> list = new ArrayList<>();
list.add(10);          // 끝에 추가, 평균 O(1)
list.add(20);
list.add(1, 15);       // 인덱스 1에 삽입 → 뒤 원소 밀기 발생, O(n)
int y = list.get(1);   // O(1)

시간/공간 복잡도

인덱스 접근 arr[i]O(1)O(1)주소 계산 한 번이면 끝
끝에 추가 (동적 배열)O(1)O(n)가끔 일어나는 재할당 때만 O(n), 평균은 상수
중간 삽입/삭제O(n)O(n)뒷 원소들을 한 칸씩 이동
탐색 (값으로 찾기)O(n)O(n)처음부터 훑어야 함 (정렬되어 있으면 이진 탐색으로 O(log n))

공간 복잡도는 원소 n개에 대해 O(n)으로, 다른 자료구조 대비 메모리 오버헤드(포인터 등)가 거의 없는 편입니다.

실무에서 어디에 쓰이는가?

  • 거의 모든 컬렉션의 기반 — Java ArrayList, Python list, JS Array, Go slice 모두 동적 배열.
  • 이미지/영상 처리 — 픽셀을 1차원 또는 2차원 배열에 담아 다룸.
  • 버퍼/캐시 — 네트워크 패킷 버퍼, 오디오 버퍼처럼 빠른 순차 접근이 중요한 곳.
  • 해시 테이블의 내부 저장소 — 해시 인덱스를 배열 인덱스로 매핑하면서 사용.
  • CPU 캐시 친화적 — 메모리가 연속이라 캐시 히트율이 높아, 같은 작업도 연결 리스트보다 훨씬 빠른 경우가 많음.

면접에 나올 질문과 답변

배열과 연결 리스트의 차이를 설명해 주세요.
배열은 메모리에 연속으로 저장돼서 인덱스 접근이 O(1)이지만, 중간 삽입/삭제가 O(n)으로 비쌉니다. 연결 리스트는 노드들이 포인터로 이어져 있어 임의 접근은 O(n)이지만, 위치만 알고 있다면 삽입/삭제가 O(1)입니다. 또 배열은 캐시 친화적이라 순회 성능이 보통 더 좋습니다.
동적 배열에 끝에 추가하는 게 평균 O(1)인 이유는요?
보통 용량이 가득 차면 두 배 크기의 새 배열을 만들어 옮깁니다. 이 재할당은 O(n)이지만, n번을 추가하는 동안 재할당이 일어나는 총 비용을 평균하면 한 번 추가당 상수 시간이 됩니다. 이걸 분할상환 분석(Amortized Analysis)이라고 하고, 결과적으로 평균 O(1)이 됩니다.
2차원 배열은 메모리에 어떻게 놓이나요?
대부분의 언어(C, Java 등)는 행 우선(row-major)으로 저장합니다. arr[0][0], arr[0][1], ..., arr[0][n-1], arr[1][0], ... 순으로 한 줄씩 이어 붙입니다. 그래서 같은 데이터를 다뤄도 행 방향으로 순회하는 게 열 방향보다 캐시 친화적이라 보통 더 빠릅니다.
배열이 캐시 친화적이라는 건 무슨 뜻인가요?
CPU는 메모리에서 한 번에 한 줄(보통 64바이트)을 캐시로 가져오는데, 배열은 연속된 데이터라 한 번 가져오면 근처 원소들도 같이 캐시에 올라옵니다. 다음 원소를 읽을 때 이미 캐시에 있으니 빠르죠. 반면 연결 리스트는 노드들이 메모리 곳곳에 흩어져 있어서 매번 캐시 미스가 날 가능성이 높습니다.
