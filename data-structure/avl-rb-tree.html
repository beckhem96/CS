AVL / 레드-블랙 트리

AVL 트리와 레드-블랙(RB) 트리는, 삽입·삭제 후 트리가 한쪽으로 치우치지 않도록 스스로 균형을 맞추는 BST입니다. 입력이 어떻게 들어오든 항상 높이 O(log n)을 보장해 탐색·삽입·삭제 모두 O(log n) 최악 시간을 보장합니다.

구체적인 개념 설명

일반 BST는 입력 순서가 정렬되어 들어오면 한쪽으로만 자라 사실상 연결 리스트가 됩니다. 그러면 모든 연산이 O(n)이 되죠. 이걸 막기 위해, 삽입·삭제가 일어날 때마다 약간의 추가 작업을 통해 균형을 다시 맞추는 BST가 자가 균형 BST(self-balancing BST)입니다. AVL과 RB는 그중 가장 많이 쓰이는 두 가지예요.

AVL 트리

AVL 트리는 가장 엄격한 균형을 유지합니다. 모든 노드에서 왼쪽 서브트리 높이와 오른쪽 서브트리 높이의 차이가 1 이하여야 합니다. 이 차이를 균형 인수(Balance Factor)라고 부릅니다 (BF = h(left) − h(right)). 삽입/삭제 후 BF가 ±2가 되면 회전(rotation)으로 균형을 복원합니다.

레드-블랙 트리

레드-블랙 트리는 모든 노드에 빨강 / 검정 색을 부여하고, 다음 5가지 속성을 항상 유지합니다.

  1. 모든 노드는 빨강 또는 검정.
  2. 루트는 검정.
  3. 모든 잎(NIL, 비어 있는 자식)은 검정.
  4. 빨강 노드의 자식은 모두 검정 (빨강이 연속으로 나오면 안 됨).
  5. 임의의 노드에서 그 후손 NIL까지의 모든 경로에 들어 있는 검정 노드 수가 같다 (블랙 높이).

이 속성들 덕분에 RB 트리는 가장 긴 경로가 가장 짧은 경로의 최대 2배를 넘지 않습니다. AVL만큼 빡빡한 균형은 아니지만, 삽입·삭제 시 회전 횟수가 더 적어 평균적으로 더 빠른 갱신 성능을 보입니다.

AVL vs RB — 한눈에

  • AVL은 더 균형이 잡혀 있어 탐색 성능이 약간 더 빠름. 읽기가 많은 워크로드에 유리.
  • RB는 균형이 덜 빡빡해 삽입·삭제 시 재조정 비용이 적음. 쓰기가 잦은 워크로드와 범용 라이브러리에 유리.
  • 그래서 Java TreeMap·TreeSet, C++ std::map·std::set, Linux 커널의 일부(스케줄러·timer)는 모두 RB 트리를 씁니다.

왜 필요한가?

BST의 평균 O(log n)은 입력이 무작위라는 가정 하에서만 성립합니다. 실세계 데이터는 정렬되어 있거나 거의 정렬되어 있는 경우가 흔하고, 그러면 BST가 편향 트리로 망가져 O(n)이 됩니다. 라이브러리나 데이터베이스 같은 곳에서 최악 보장을 깔고 가야 한다면 자가 균형 BST가 필수입니다.

또한 정렬 순서·범위 검색이 필요한 상황에서는 해시 테이블이 답이 안 되니, 자가 균형 BST가 사실상 유일한 선택지입니다. 그래서 표준 라이브러리의 정렬 맵/셋은 거의 모두 RB 트리로 구현되어 있습니다.

어떻게 동작하는가?

1) AVL — LL 회전 예시

30, 20, 10 순으로 삽입하면 왼쪽으로 치우친 트리가 됩니다. 30의 균형 인수가 2가 되어 불균형 → 오른쪽 회전(LL)으로 복원합니다.

왼쪽 자식이 더 무거운 경우(LL) → 부모를 오른쪽으로 회전. 4가지 케이스(LL/RR/LR/RL)가 있다.

2) AVL의 4가지 회전 케이스

케이스상황해결
LL왼쪽 자식의 왼쪽 서브트리에 삽입오른쪽 단일 회전
RR오른쪽 자식의 오른쪽 서브트리에 삽입왼쪽 단일 회전
LR왼쪽 자식의 오른쪽 서브트리에 삽입왼쪽 회전 → 오른쪽 회전 (이중)
RL오른쪽 자식의 왼쪽 서브트리에 삽입오른쪽 회전 → 왼쪽 회전 (이중)

3) 레드-블랙 트리의 모양

색깔 속성을 시각화한 RB 트리 예시입니다 (NIL 잎은 생략).

색깔 속성 + 회전 + 색 재칠하기로 균형을 유지한다.

4) Java 코드로 보기

표준 라이브러리에 자가 균형 BST가 들어 있어 직접 구현할 일은 거의 없습니다. 보통은 이렇게 씁니다.

import java.util.TreeMap;
import java.util.NavigableSet;
import java.util.TreeSet;

// 정렬된 맵 — 내부적으로 레드-블랙 트리
TreeMap<Integer, String> map = new TreeMap<>();
map.put(10, "a");
map.put(5,  "b");
map.put(20, "c");

map.firstKey();         // 5  — 가장 작은 키
map.lastKey();          // 20
map.floorKey(15);       // 10 — 15 이하의 가장 큰 키
map.ceilingKey(15);     // 20 — 15 이상의 가장 작은 키
map.subMap(5, 20);      // [5, 20) 범위 검색

// 정렬된 셋
TreeSet<Integer> set = new TreeSet<>();

// 회전(개념 코드) — AVL의 오른쪽 회전
Node rotateRight(Node y) {
    Node x = y.left;
    Node t = x.right;

    x.right = y;
    y.left  = t;

    y.height = 1 + Math.max(height(y.left), height(y.right));
    x.height = 1 + Math.max(height(x.left), height(x.right));
    return x;   // 새 루트
}

시간/공간 복잡도

연산AVLRB일반 BST(편향)
탐색O(log n)O(log n)O(n)
삽입O(log n)O(log n)O(n)
삭제O(log n)O(log n)O(n)
회전 횟수 (삽입 시)최대 2회최대 2회—
회전 횟수 (삭제 시)O(log n)최대 3회—

AVL은 더 엄격한 균형이라 탐색이 약간 빠르고, RB는 삭제 시 회전이 상수 번이라 갱신이 빠릅니다. 공간은 둘 다 O(n)이고 노드당 추가 정보(AVL=높이, RB=색상 1비트)를 들고 다닙니다.

실무에서 어디에 쓰이는가?

  • 표준 라이브러리의 정렬된 맵/셋 — Java TreeMap/TreeSet, C++ std::map/std::set 모두 RB 트리.
  • 리눅스 커널 — CFS 스케줄러의 가상 런타임 정렬, epoll의 등록 fd 관리 등에 RB 트리 사용.
  • 인메모리 데이터베이스의 정렬 인덱스 — 범위 질의가 필요한 곳.
  • 구간 탐색·일정 관리 — 가장 가까운 시각의 작업, 다음 만료 이벤트 등.
  • 참고로 디스크 기반 인덱스는 RB가 아니라 B-Tree/B+Tree를 씁니다 (디스크 I/O 특성 때문).

면접에 나올 질문과 답변

AVL과 레드-블랙 트리 중 어느 걸 더 자주 쓰나요? 왜죠?
실무에서 표준 라이브러리에는 거의 모두 레드-블랙 트리가 쓰입니다. AVL이 균형은 더 빡빡해 탐색이 조금 빠르지만, 삽입·삭제 시 회전이 더 자주/더 깊게 일어납니다. RB는 색깔 속성으로 적당한 균형만 유지해서 갱신 비용이 평균적으로 낮고, 범용 라이브러리에선 갱신과 탐색이 섞이기 때문에 균형 잡힌 성능이 중요해 RB가 선택됩니다.
레드-블랙 트리의 색 속성은 무엇을 보장하나요?
다섯 가지 속성이 함께 작용해서 "가장 긴 경로가 가장 짧은 경로의 2배를 넘지 않는다"를 보장합니다. 특히 "빨강 노드의 자식은 검정"이라는 규칙 때문에 빨강이 연속으로 나오지 못하고, "어떤 경로든 검정 노드 수가 같다"는 규칙 때문에 양쪽 깊이 차이가 무한히 벌어지지 못합니다. 이 두 가지가 함께 트리 높이를 O(log n)으로 묶어줍니다.
자가 균형 BST 대신 해시 테이블을 쓰면 안 되나요?
단순한 키-값 매핑이라면 해시 테이블이 더 빠릅니다(평균 O(1)). 하지만 자가 균형 BST는 정렬된 순서로 데이터를 다룰 수 있고, "X 이상의 첫 키" 같은 범위·근접 질의에 강합니다. 해시 테이블은 그런 질의가 안 되기 때문에, 정렬·범위가 필요하면 자가 균형 BST가 정답입니다.
왜 디스크 기반 인덱스에는 RB 트리가 아니라 B-Tree를 쓸까요?
디스크 I/O는 메모리 접근보다 수십만 배 비싸서, 트리의 높이를 최대한 낮추는 것이 중요합니다. RB 트리는 한 노드가 키 1개를 가져 트리가 깊어지지만, B-Tree는 한 노드가 수백~수천 개의 키를 담을 수 있어 같은 데이터 양에서도 높이가 훨씬 낮습니다. 그래서 디스크 접근 횟수가 줄어 인덱스에 적합합니다. 자세한 건 B-Tree 페이지 참고.
