이진 탐색 트리(Binary Search Tree)

이진 탐색 트리(BST)란, 모든 노드에서 왼쪽 서브트리의 값 < 자기 값 < 오른쪽 서브트리의 값이라는 규칙이 지켜지는 이진 트리입니다. 이 규칙 덕분에 평균 O(log n)에 탐색·삽입·삭제가 가능합니다.

구체적인 개념 설명

비유로 가장 잘 맞는 건 책 페이지 찾기입니다. 500쪽짜리 책에서 247쪽을 찾을 때, 먼저 책을 절반 펴서 그 자리가 247보다 앞이면 앞쪽 절반에서 다시 절반을 펴고… 이런 식으로 줄여가죠. BST도 똑같습니다. 한 노드를 보고 크면 오른쪽, 작으면 왼쪽으로 한 단계씩 내려가면서 탐색 범위를 절반씩 줄입니다.

그래서 BST의 핵심은 그저 "이진 트리 + 정렬 규칙"입니다. 이 규칙은 모든 노드에서 성립해야 하고, 덕분에 중위 순회를 하면 정렬된 순서로 값이 나옵니다 — 이게 BST의 시그니처 성질입니다.

BST의 한계 — 균형 문제

BST의 성능은 트리의 높이에 의존합니다. 입력 값이 정렬된 순서(예: 1, 2, 3, 4, 5)로 들어오면 트리가 한쪽으로만 자라나 사실상 연결 리스트가 되어 모든 연산이 O(n)으로 떨어집니다. 이걸 막기 위해 AVL 트리, 레드-블랙 트리 같은 자가 균형 BST를 씁니다.

왜 필요한가?

"정렬된 데이터 위에서 빠르게 검색·삽입·삭제하고 싶다"는 게 출발점입니다.

  • 정렬된 배열은 이진 탐색으로 검색은 O(log n)이지만, 삽입/삭제는 O(n) (뒤 원소를 다 밀어야 해서).
  • 해시 테이블은 평균 O(1) 검색이 가능하지만, 정렬된 순서나 범위 검색은 못합니다.
  • BST는 검색·삽입·삭제 모두 평균 O(log n)이고, 게다가 정렬 순회와 범위 검색이 자연스럽게 됩니다.

이런 이유로 데이터베이스의 정렬된 인덱스, "어떤 값보다 큰 첫 번째 키"를 찾는 자료구조, Java의 TreeMap·TreeSet 같은 곳에서 BST 계열이 활약합니다.

어떻게 동작하는가?

1) BST의 모양

다음 트리는 BST 규칙을 만족합니다. 어느 노드를 보더라도 왼쪽은 더 작고 오른쪽은 더 큽니다.

중위 순회: 1 3 6 8 10 12 14 — 정렬된 결과가 자연스럽게 나온다.

2) 탐색 — 6을 찾는 과정

루트부터 시작해, 찾는 값과 비교하며 한쪽으로만 내려갑니다.

탐색 비용 = 트리 높이. 균형 잡혀 있으면 O(log n).

3) 삽입 — 5를 넣는 과정

탐색과 똑같이 내려가다가 빈 자리를 만나면 거기에 잎으로 추가합니다.

새 노드는 항상 잎(leaf) 위치에 추가된다.

4) 삭제 — 가장 까다로운 연산

삭제하려는 노드의 자식 수에 따라 처리가 달라집니다.

  • 자식 0개(잎) — 그냥 떼어낸다.
  • 자식 1개 — 그 자식이 자리를 대체한다.
  • 자식 2개 — 중위 후속자(왼쪽 서브트리의 최댓값 또는 오른쪽 서브트리의 최솟값)로 값을 덮어쓰고, 그 후속자 노드를 다시 삭제한다(후속자는 자식이 0~1개라 위 두 케이스로 환원됨).

5) Java 코드로 보기

class Node {
    int value;
    Node left, right;
    Node(int v) { this.value = v; }
}

class BST {
    Node root;

    // 탐색 — O(h)
    Node search(Node node, int key) {
        if (node == null || node.value == key) return node;
        return key < node.value ? search(node.left, key)
                                 : search(node.right, key);
    }

    // 삽입 — O(h)
    Node insert(Node node, int key) {
        if (node == null) return new Node(key);
        if (key < node.value)      node.left  = insert(node.left, key);
        else if (key > node.value) node.right = insert(node.right, key);
        return node;
    }

    // 삭제 — O(h)
    Node delete(Node node, int key) {
        if (node == null) return null;
        if (key < node.value)      node.left  = delete(node.left, key);
        else if (key > node.value) node.right = delete(node.right, key);
        else {
            // 자식이 0~1개
            if (node.left == null)  return node.right;
            if (node.right == null) return node.left;
            // 자식 2개: 오른쪽 서브트리 최솟값을 후속자로
            Node succ = node.right;
            while (succ.left != null) succ = succ.left;
            node.value = succ.value;
            node.right = delete(node.right, succ.value);
        }
        return node;
    }
}

시간/공간 복잡도

연산평균(균형)최악(편향)설명
탐색O(log n)O(n)높이 h만큼 내려감
삽입O(log n)O(n)탐색 후 잎에 추가
삭제O(log n)O(n)후속자 찾기 포함해도 높이 비례
정렬 순회O(n)O(n)중위 순회로 정렬된 결과 획득

공간은 노드 수만큼 O(n), 재귀 호출 스택은 O(h)가 추가됩니다.

실무에서 어디에 쓰이는가?

  • Java TreeMap, TreeSet — 내부적으로 레드-블랙 트리(자가 균형 BST).
  • "어떤 값 이상의 첫 키" 같은 범위 질의 — floorKey, ceilingKey처럼 정렬 순서 활용.
  • 구간 합/구간 최솟값 같은 트리 기반 자료구조의 기반 — 세그먼트 트리, 펜윅 트리도 이진 트리 변형.
  • 스케줄러, 타이머 큐 — 가장 가까운 시각의 작업을 빨리 찾고 추가/삭제도 빠른 게 필요할 때.
  • 인메모리 정렬된 인덱스 — 일부 NoSQL/임베디드 DB가 사용.

면접에 나올 질문과 답변

BST와 해시 테이블의 차이를 설명해 주세요.
해시 테이블은 평균 O(1) 검색이지만 정렬 순서가 없고 범위 검색이 안 됩니다. BST는 평균 O(log n)으로 조금 느리지만 정렬 순서와 범위 검색을 자연스럽게 지원합니다. 그래서 단순한 키-값 매핑은 해시 테이블, 정렬·범위 질의가 필요하면 BST를 쓰는 게 일반적입니다.
BST가 O(n)으로 떨어지는 경우는 언제인가요?
삽입 순서가 정렬되어 들어올 때입니다. 예를 들어 1, 2, 3, 4, 5를 차례로 삽입하면 모든 노드가 오른쪽으로만 자라 편향 트리가 되고 사실상 연결 리스트가 됩니다. 이걸 막으려면 입력을 무작위로 섞거나, AVL/레드-블랙 트리 같은 자가 균형 BST를 씁니다.
BST에서 자식이 2개인 노드를 삭제하는 방법은요?
중위 후속자(오른쪽 서브트리의 최솟값) 또는 중위 선행자(왼쪽 서브트리의 최댓값)의 값을 삭제할 노드 자리에 복사한 뒤, 그 후속자/선행자 노드를 다시 삭제합니다. 그 노드는 자식이 0~1개로 보장되므로 간단한 케이스로 환원됩니다. 이렇게 해야 BST 규칙이 깨지지 않습니다.
왜 중위 순회의 결과가 정렬된 순서가 되나요?
BST 규칙상 어떤 노드든 왼쪽 서브트리의 모든 값이 자기보다 작고 오른쪽이 더 큽니다. 중위 순회는 "왼쪽 → 자신 → 오른쪽" 순으로 방문하기 때문에, 작은 값부터 큰 값으로 자연히 이어집니다. 그래서 BST를 한 번 중위 순회하면 그게 곧 오름차순 정렬 결과입니다.
