해시 테이블(Hash Table)

해시 테이블이란, 키(key)를 해시 함수에 넣어 인덱스를 얻은 뒤 그 위치의 배열 칸에 값(value)을 저장하는 자료구조입니다. 평균 O(1)에 삽입·검색·삭제가 가능한 강력한 무기입니다.

구체적인 개념 설명

해시 테이블의 비유는 옷가게 보관함입니다. 손님 이름을 적어내면 직원이 이름을 일정한 규칙으로 바꿔서 "37번 보관함"이라고 알려줍니다. 다음에 다시 와서 이름을 말하면 같은 규칙으로 같은 번호가 나오니까 바로 그 보관함에서 옷을 꺼내줄 수 있죠. 매번 모든 보관함을 뒤지지 않아도 됩니다.

핵심 구성 요소는 세 가지입니다.

  • 해시 함수(hash function) — 임의 키를 정수 인덱스로 변환하는 함수.
  • 버킷 배열(buckets) — 인덱스로 바로 접근 가능한 저장소.
  • 충돌 처리(collision resolution) — 서로 다른 키가 같은 인덱스를 받았을 때의 처리 방법.

이상적인 경우 인덱스 한 번 계산이면 끝나서 O(1)이지만, 충돌이 자주 일어나면 그만큼 비용이 커집니다. 그래서 좋은 해시 함수와 적절한 부하율 관리가 성능을 좌우합니다.

해시 함수의 조건

  • 결정적 — 같은 입력에는 항상 같은 출력.
  • 균등 분포 — 결과가 인덱스 공간에 골고루 퍼져야 충돌이 줄어듦.
  • 빠름 — 자주 호출되므로 가벼워야 함.
  • (보안용일 경우) 일방향 — 출력에서 입력을 역산하기 어려워야 함. 일반 해시 테이블에는 불필요.

왜 필요한가?

"키로 값을 빠르게 찾고 싶다"가 출발점입니다. 배열은 인덱스로만 빠른 접근이 가능하고, 연결 리스트는 처음부터 훑어야 하고, 정렬된 트리도 O(log n)이 걸립니다. 만약 키-값 쌍이 수백만 개라면 차이가 어마어마합니다.

해시 테이블은 키를 직접 인덱스로 바꿔버리기 때문에, 데이터가 아무리 많아져도 평균 검색 시간이 거의 그대로입니다. 덕분에 캐시, 데이터베이스 인덱스(특히 동등 비교용), 사전(dictionary), 집합(set) 등 거의 모든 곳에 깔려 있어요.

어떻게 동작하는가?

1) 해시 함수로 인덱스 계산

크기 7짜리 버킷 배열에 "apple", "banana"를 넣는다고 해봅시다.

해시 함수가 키를 정수로 바꾸고, 그 정수를 버킷 배열의 인덱스로 사용한다.

2) 충돌 — 체이닝(Chaining)

서로 다른 키가 같은 인덱스를 받으면 어떡할까요? 가장 단순한 방법은 그 자리에 연결 리스트를 매달아두는 것입니다. 새 항목이 오면 그 리스트 끝에 추가하고, 검색은 같은 버킷 안에서 키를 비교하며 찾습니다.

버킷에 충돌이 일어나면 같은 자리에서 리스트로 이어 붙인다. 검색은 그 리스트 안에서 키 비교.

3) 충돌 — 오픈 어드레싱(Open Addressing)

또 다른 방법은 충돌이 나면 다른 빈 칸을 찾아가는 것입니다. 가장 단순한 형태인 선형 탐사(linear probing)는 충돌 시 다음 칸, 그 다음 칸… 식으로 한 칸씩 이동하면서 비어 있는 자리를 찾습니다.

검색 시에도 같은 규칙으로 따라가면서 키를 비교한다. 삭제는 tombstone 처리가 필요해 까다롭다.

4) 부하율(Load Factor)과 리해싱(Rehashing)

부하율 = 저장된 항목 수 / 버킷 수. 이 값이 높아질수록 충돌이 많아져 성능이 떨어집니다. 그래서 보통 임계값(예: 0.75)을 넘으면 더 큰 배열을 만들어 모든 항목을 다시 해싱해서 옮깁니다. 이 작업을 리해싱이라고 부르고, 한 번에 비싸지만(O(n)) 분할상환으로 보면 평균 O(1)이 됩니다.

5) Java 코드로 보기

import java.util.HashMap;
import java.util.Map;

Map<String, Integer> map = new HashMap<>();

map.put("apple", 1);     // 평균 O(1)
map.put("banana", 2);

int v = map.get("apple"); // O(1)
boolean has = map.containsKey("banana");
map.remove("apple");

// 자주 묻는 패턴: 빈도수 세기
String[] words = {"a", "b", "a", "c", "b", "a"};
Map<String, Integer> cnt = new HashMap<>();
for (String w : words) {
    cnt.merge(w, 1, Integer::sum);
}
// {a=3, b=2, c=1}

// 사용자 정의 키를 쓸 때는 equals/hashCode를 반드시 같이 오버라이드
class Point {
    final int x, y;
    Point(int x, int y) { this.x = x; this.y = y; }

    @Override public boolean equals(Object o) {
        if (!(o instanceof Point p)) return false;
        return x == p.x && y == p.y;
    }
    @Override public int hashCode() {
        return 31 * x + y;
    }
}

시간/공간 복잡도

연산평균최악설명
삽입 (put)O(1)O(n)충돌이 심하면 같은 버킷의 모든 항목 비교
검색 (get)O(1)O(n)좋은 해시 함수에선 거의 일어나지 않음
삭제 (remove)O(1)O(n)최악은 검색과 동일

Java 8부터는 한 버킷의 체인이 일정 길이를 넘으면 레드-블랙 트리로 변환해 최악을 O(log n)으로 낮춥니다. 공간은 O(n)이지만 부하율을 낮게 유지하기 때문에 실제로는 항목 수의 1.3~2배 메모리를 씁니다.

실무에서 어디에 쓰이는가?

  • 언어의 기본 사전/맵 자료구조 — Java HashMap, Python dict, JS Object/Map.
  • 캐시 — 키로 결과를 빠르게 찾는 모든 종류의 캐시(Memcached, Redis 등).
  • 데이터베이스 인덱스 — 동등 비교 위주의 해시 인덱스. 범위 검색은 약함.
  • 중복 제거 / 집합 연산 — HashSet으로 O(1) 멤버십 검사.
  • 파일/메시지 무결성 검사 — SHA-256 같은 암호학적 해시는 변조 감지에 사용.
  • 로드 밸런서 — 키의 해시값으로 서버를 고르는 일관 해싱(Consistent Hashing).

면접에 나올 질문과 답변

해시 충돌이란 무엇이고 어떻게 해결하나요?
서로 다른 키가 같은 인덱스를 받는 현상입니다. 두 가지 대표적인 해결책이 있습니다. 체이닝은 같은 인덱스에 연결 리스트나 트리를 매달아 모든 충돌 항목을 그 안에 저장하고, 오픈 어드레싱은 충돌 시 다른 빈 칸(선형/이차/이중 해싱)을 찾아 저장합니다. Java HashMap은 체이닝 + (체인이 길어지면) 트리화 방식을 씁니다.
해시 테이블의 평균은 O(1)인데 최악은 왜 O(n)인가요?
모든 키가 같은 버킷에 몰리는 극단적 상황을 가정하면, 한 버킷 안의 리스트가 곧 전체 항목과 같아져 검색이 O(n)이 됩니다. 실제로는 일어나기 어렵지만, 악의적인 키를 넣어 의도적으로 충돌을 유도하는 해시 충돌 공격(HashDoS)이 가능해서 Java 8은 트리화로 최악을 O(log n)으로 완화했습니다.
equals와 hashCode는 왜 함께 오버라이드해야 하나요?
해시 테이블은 먼저 hashCode로 버킷을 찾고, 그 안에서 equals로 같은 키인지 확인합니다. 둘이 일관되지 않으면 — 예를 들어 같은 객체인데 hashCode가 다르면 — 다른 버킷에 들어가 영영 못 찾고, 반대로 hashCode는 같은데 equals가 false면 충돌은 났는데 같다고 인정 안 되는 일이 생깁니다. 그래서 "equals가 같으면 hashCode도 같아야 한다"는 계약이 항상 깔립니다.
체이닝과 오픈 어드레싱 중 어느 것이 더 좋나요?
상황에 따라 다릅니다. 체이닝은 부하율이 1을 넘어도 동작 가능하고 삭제가 단순하지만, 노드 할당과 포인터 추적 비용이 있습니다. 오픈 어드레싱은 메모리 연속성이 좋아 캐시 효율이 높지만, 부하율이 0.7~0.8을 넘으면 성능이 급격히 나빠지고 삭제가 까다롭습니다(빈 칸과 삭제된 칸을 구분해야 함). 라이브러리는 보통 체이닝, 고성능 인메모리 자료구조는 오픈 어드레싱을 선호합니다.
