B-Tree / B+Tree

B-Tree는 한 노드에 여러 개의 키와 자식을 담아 트리의 높이를 낮춘 자가 균형 트리입니다. 디스크 I/O를 최소화하기 위해 만들어진 구조로, 거의 모든 데이터베이스 인덱스와 파일 시스템의 표준입니다. B+Tree는 그 변형으로, 데이터를 잎(leaf)에만 저장하고 잎끼리 연결 리스트로 묶어 범위 검색에 강합니다.

구체적인 개념 설명

비유로는 도서관의 분류 안내판이 가깝습니다. 입구의 큰 안내판 하나만 봐도 "100~199번대는 1층 동쪽, 200~299번대는 1층 서쪽…"처럼 많은 정보가 한꺼번에 보이죠. 그래서 단 몇 번의 안내판 읽기로 책에 도달합니다. B-Tree도 마찬가지로 한 노드가 키를 여러 개 들고 있어, 트리를 몇 단계만 내려가면 원하는 데이터에 도달합니다.

M차(M-way) B-Tree의 규칙은 다음과 같습니다.

  • 각 노드는 최대 M-1개의 키와 M개의 자식을 가진다.
  • 루트를 제외한 모든 노드는 최소 ⌈M/2⌉-1개의 키를 가진다 (반쯤은 차 있어야 함).
  • 한 노드 안의 키들은 정렬되어 있고, 자식들은 키 사이의 범위를 담당한다.
  • 모든 잎의 깊이가 같다 (완전한 균형).

B-Tree vs B+Tree

두 변종의 핵심 차이만 정리하면 이렇습니다.

  • B-Tree — 데이터(value 또는 그 포인터)가 모든 노드(내부+잎)에 저장됨. 잎끼리 연결 안 됨.
  • B+Tree — 데이터는 잎에만 저장되고, 내부 노드는 키만 들고 라우팅 역할. 잎끼리 연결 리스트로 묶여 있어 범위 스캔이 매우 효율적.

왜 RB 트리 대신 B-Tree?

메모리 안에서는 RB 트리도 충분히 빠르지만, 디스크 I/O는 메모리 접근보다 수십만 배 비쌉니다. 디스크는 한 번 접근할 때마다 보통 4KB~16KB의 페이지(블록)를 통째로 읽습니다. 그러면 한 노드에 키 하나만 담는 RB 트리는 페이지의 대부분을 낭비하게 되죠. B-Tree는 한 노드를 디스크 페이지 하나에 맞춰 키 수백~수천 개를 채우기 때문에 같은 데이터 양에서 트리 높이가 훨씬 낮고, 탐색에 필요한 디스크 I/O 수가 압도적으로 적습니다.

왜 필요한가?

"메모리에 다 못 올리는 거대한 데이터를 디스크에 두고 빠르게 조회하고 싶다"가 출발점입니다. MySQL, PostgreSQL 같은 DB의 인덱스에는 수억 행짜리 테이블도 들어 있고, 파일 시스템도 디렉토리 안에 수백만 개 파일이 있을 수 있습니다. 이 정도 규모에선 트리의 높이가 1만 늘어도 디스크 접근이 한 번 더 일어나서 성능에 큰 차이를 냅니다.

B-Tree는 한 노드가 디스크 페이지(예: 16KB) 하나를 통째로 사용하도록 설계되어, 한 번의 디스크 읽기로 수백 개의 키를 비교할 수 있습니다. 그래서 1억 건의 데이터에서도 보통 3~4번의 디스크 접근이면 어떤 키든 찾아냅니다.

어떻게 동작하는가?

1) B-Tree(차수 4) 예시

한 노드에 최대 3개의 키, 최대 4개의 자식을 가진 B-Tree입니다. 모든 잎의 깊이가 같다는 점에 주목하세요.

한 노드에 여러 키 → 트리 높이가 낮음 → 디스크 접근 적음.

2) 탐색 — 15를 찾기

루트에서 키와 비교해 어느 자식으로 내려갈지 결정합니다. 한 노드 안에서는 보통 이진 탐색으로 키를 찾습니다.

  1. 루트 [10, 20, 30] → 15는 10과 20 사이 → 두 번째 자식으로.
  2. 잎 [12, 15, 17] → 15 발견.

3) B+Tree — 잎끼리 연결

B+Tree는 데이터를 잎에만 두고, 잎끼리 양방향 연결 리스트로 묶습니다. 그래서 "20 이상의 모든 키"같은 범위 질의는 잎 하나를 찾은 뒤 옆으로 쭉 따라가면 됩니다.

B+Tree의 핵심 — 데이터는 잎에만, 잎끼리 연결 리스트.

4) 분할(split) — 노드가 가득 찼을 때

한 노드에 더 이상 키를 넣을 수 없을 때는, 노드를 두 개로 쪼개고 가운데 키를 부모로 올려 보냅니다. 부모도 가득 찼다면 같은 과정이 위로 전파되고, 루트까지 차면 트리 높이가 1 늘어납니다. 덕분에 트리는 항상 잎 깊이가 같은 균형 상태를 유지합니다.

5) Java 코드로 보기

직접 구현할 일은 거의 없고, DB나 라이브러리에 깔려 있는 구현을 사용합니다. Java의 TreeMap은 인메모리용이라 RB 트리지만, 디스크 친화적인 인메모리 자료구조가 필요하다면 jdbm·MapDB 같은 임베디드 라이브러리가 B-Tree를 제공합니다. 

// 실무에서는 보통 DB가 알아서 B+Tree 인덱스를 만든다 — SQL 한 줄이면 됨
//
//   CREATE INDEX idx_users_email ON users(email);
//
// 이 인덱스를 통한 조회는 내부적으로 B+Tree 탐색을 수행한다.

// 범위 검색이 B+Tree에서 효율적인 이유:
//
//   SELECT * FROM users WHERE id BETWEEN 1000 AND 2000;
//
// → 1) 트리 탐색으로 id=1000을 가진 잎에 도달 (몇 번의 디스크 I/O)
//   2) 잎 연결 리스트를 따라 id=2000까지 시퀀셜 읽기
// → 인덱스를 따라 점프하지 않아도 되어 매우 빠름.

시간/공간 복잡도

연산시간설명
탐색O(log n)로그의 밑이 M(차수)이라 RB 트리보다 트리 높이가 훨씬 낮음
삽입 / 삭제O(log n)분할(split)·병합(merge)이 위로 전파될 수 있음
범위 검색 (B+Tree)O(log n + k)k는 결과 개수. 잎 연결 따라가기만 하면 됨

"log의 밑이 M"이 핵심입니다. M이 100이라면 1억 개 데이터의 트리 높이가 약 4 정도입니다. 공간은 O(n)이지만, 노드의 빈 공간을 일부 허용하기 때문에 실제 사용량은 1.5n 정도가 보통입니다.

실무에서 어디에 쓰이는가?

  • 관계형 데이터베이스 인덱스 — MySQL/InnoDB, PostgreSQL의 기본 인덱스가 모두 B+Tree.
  • NoSQL 일부 — MongoDB의 인덱스도 B-Tree 계열.
  • 파일 시스템 — NTFS, HFS+, ext4의 디렉토리 인덱스, ReiserFS, Btrfs(Btr는 B-Tree에서 따옴) 등.
  • 임베디드 DB — SQLite, BerkeleyDB도 B+Tree 페이지 기반.
  • 전공정 추적/시계열 — InfluxDB, TimeScaleDB의 인덱스에도 B+Tree 변형이 등장.

면접에 나올 질문과 답변

B-Tree와 B+Tree의 차이를 설명해 주세요.
B-Tree는 데이터(또는 데이터 포인터)가 모든 노드에 분산 저장되고 잎끼리 연결되어 있지 않습니다. B+Tree는 데이터를 잎에만 저장하고 내부 노드는 키만 가지며, 잎끼리 연결 리스트로 묶여 있습니다. 그래서 B+Tree는 한 노드(특히 내부 노드)가 더 많은 키를 담을 수 있어 트리 높이가 더 낮고, 잎 연결 덕분에 범위 검색이 매우 빠릅니다. 오늘날 DB 인덱스는 거의 모두 B+Tree를 씁니다.
왜 디스크 기반 인덱스에는 B-Tree를, 메모리 기반에는 RB 트리를 쓰나요?
디스크 I/O는 한 번에 4KB~16KB의 페이지를 통째로 읽기 때문에, 한 노드에 가능한 한 많은 키를 채우는 게 유리합니다. B-Tree는 한 노드를 페이지 하나에 맞춰 수백~수천 개의 키를 담아 트리 높이를 극단적으로 낮춥니다. 반대로 메모리에서는 한 워드 단위로 빠르게 접근하기 때문에 노드를 크게 만들 필요가 없고, 오히려 회전 같은 단순한 재조정이 효율적이라 RB 트리가 더 적합합니다.
B+Tree에서 범위 검색이 효율적인 이유는요?
잎끼리 연결 리스트로 묶여 있어서, 범위의 시작 키를 가진 잎을 찾은 뒤 잎 포인터를 따라 끝 키까지 읽기만 하면 됩니다. 매번 트리를 위에서 다시 탐색할 필요가 없어, 디스크에서는 시퀀셜 읽기가 되어 랜덤 액세스보다 훨씬 빠릅니다. SQL의 BETWEEN, ORDER BY ... LIMIT 같은 쿼리가 이 성질 덕분에 빠릅니다.
B-Tree는 분할(split)이 일어나면 무슨 일이 벌어지나요?
가득 찬 노드에 새 키가 들어오면, 그 노드를 가운데 키 기준으로 둘로 쪼개고 가운데 키를 부모 노드로 올려 보냅니다. 부모가 또 가득 차 있으면 같은 과정이 위로 전파되고, 결국 루트까지 차면 새 루트가 생기면서 트리 높이가 1 증가합니다. 이 방식 때문에 B-Tree는 항상 모든 잎이 같은 깊이에 있는 완전 균형 상태를 유지합니다.
