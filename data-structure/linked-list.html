연결 리스트(Linked List)

연결 리스트란, 데이터와 "다음 데이터의 위치"를 함께 담은 노드(node)들이 포인터로 줄줄이 이어져 있는 자료구조입니다. 메모리상에서 떨어져 있어도 포인터로 따라가면 순서대로 만날 수 있습니다.

구체적인 개념 설명

연결 리스트는 보물찾기를 떠올리면 쉽습니다. 첫 번째 쪽지에 "다음은 화단 옆"이라고 적혀 있고, 거기엔 또 "다음은 미끄럼틀 밑"이라고 적혀 있죠. 쪽지(데이터)와 다음 위치(포인터)를 한 묶음으로 들고 다니는 게 바로 노드입니다.

각 노드의 구조는 보통 이렇게 생겼습니다.

class Node {
    int value;
    Node next;   // 다음 노드를 가리키는 포인터(참조)
}

그리고 head라는 변수가 첫 번째 노드를 가리키고 있어서, 이걸 시작점으로 쭉 따라갑니다. 마지막 노드의 next는 보통 null이고, 그러면 "끝"이라는 신호가 됩니다.

변종이 여러 개 있다

  • 단일 연결 리스트(Singly) — 다음 노드만 가리킴. 가장 기본 형태.
  • 이중 연결 리스트(Doubly) — 다음과 이전을 모두 가리킴. 양방향 이동 가능.
  • 원형 연결 리스트(Circular) — 마지막 노드의 next가 head를 가리킴. 라운드 로빈 같은 곳에 활용.

왜 필요한가?

배열의 가장 큰 약점은 중간 삽입/삭제가 비싸다는 점입니다. 끼워넣을 때마다 뒤 원소를 다 밀어야 하니까요. "삽입과 삭제가 잦은 상황"에서 이 비용을 줄이고 싶다는 게 연결 리스트의 출발점입니다.

연결 리스트는 노드끼리 포인터로 이어져 있을 뿐, 메모리상으로는 떨어져 있어도 됩니다. 그래서 중간에 노드 하나를 끼워넣을 때 주변 두 노드의 포인터만 고쳐주면 끝입니다. 대신 i번째 데이터를 꺼내려면 head에서 i번 따라가야 해서 임의 접근은 느려집니다. 트레이드오프가 명확한 자료구조죠.

어떻게 동작하는가?

1) 기본 구조

head가 첫 노드를 가리키고, 노드들은 next로 이어지며, 마지막은 null로 끝납니다.

각 노드는 [값 | next 포인터]의 두 칸으로 이루어진다.

2) 노드 사이에 새 노드 삽입

10과 20 사이에 15를 끼워넣어 봅시다. 포인터 두 개만 고쳐주면 끝납니다.

노드 자체를 옮기지 않고 포인터 2개만 다시 묶어주면 삽입 완료.

3) Java 코드로 보기

class Node {
    int value;
    Node next;
    Node(int v) { this.value = v; }
}

class LinkedList {
    Node head;

    // 맨 앞에 추가 — O(1)
    void addFirst(int v) {
        Node node = new Node(v);
        node.next = head;
        head = node;
    }

    // prev 노드 뒤에 삽입 — O(1) (단, prev를 미리 알고 있을 때)
    void insertAfter(Node prev, int v) {
        Node node = new Node(v);
        node.next = prev.next;
        prev.next = node;
    }

    // i번째 노드 가져오기 — O(n)
    Node get(int i) {
        Node cur = head;
        for (int k = 0; k < i && cur != null; k++) {
            cur = cur.next;
        }
        return cur;
    }
}

시간/공간 복잡도

연산단일 연결 리스트설명
맨 앞 삽입/삭제O(1)head만 바꿔주면 됨
맨 뒤 삽입/삭제O(n)꼬리까지 따라가야 함 (tail 포인터를 따로 두면 삽입은 O(1))
특정 노드 뒤에 삽입O(1)이미 그 노드를 알고 있다는 전제
인덱스 i 접근O(n)head부터 i번 따라가야 함
값으로 탐색O(n)처음부터 비교해야 함

공간은 데이터 1개당 포인터 비용이 추가로 듭니다(이중이라면 2배). 그래서 같은 데이터라도 배열보다 메모리를 더 먹는 게 보통입니다.

실무에서 어디에 쓰이는가?

  • LRU 캐시 구현 — 이중 연결 리스트 + 해시 맵 조합으로 O(1) 캐시를 만듦.
  • 운영체제의 프로세스 큐 — 스케줄링 대기열, 메모리 페이지 리스트 등.
  • 탐색기·에디터의 Undo/Redo — 이전/다음 상태를 양방향으로 따라가야 할 때.
  • JVM의 일부 컬렉션 — Java LinkedList는 이중 연결 리스트, LinkedHashMap은 해시 + 연결 리스트 조합.
  • 해시 테이블의 충돌 처리(체이닝) — 같은 버킷에 들어온 데이터를 연결 리스트로 매닮.

면접에 나올 질문과 답변

언제 배열 대신 연결 리스트를 써야 하나요?
삽입과 삭제가 빈번한데 그 위치를 미리 알고 있는 경우에 적합합니다. 반대로 인덱스 접근이 잦거나, 메모리 사용량을 줄여야 하거나, 캐시 효율이 중요한 상황에선 배열이 더 좋습니다. 실무에선 의외로 연결 리스트보다 동적 배열이 선택되는 경우가 많아요 — 캐시 친화성 때문입니다.
단일 연결 리스트에서 삭제가 O(1)이 될 수 있나요?
삭제할 노드의 이전 노드를 이미 알고 있다면 O(1)입니다. 단일 연결 리스트는 next만 알기 때문에 보통은 그 이전 노드를 찾기 위해 head부터 따라가야 해서 O(n)이 됩니다. 그래서 양방향 이동이 필요하면 이중 연결 리스트를 씁니다.
연결 리스트에 사이클이 있는지 어떻게 알 수 있나요?
플로이드의 토끼와 거북이 알고리즘이 유명합니다. 한 칸씩 가는 느린 포인터와 두 칸씩 가는 빠른 포인터를 동시에 출발시켜서, 둘이 만나면 사이클이 있고 빠른 포인터가 null에 도달하면 사이클이 없는 겁니다. 시간 O(n), 공간 O(1)에 검사할 수 있어요.
Java의 LinkedList와 ArrayList 중 어느 걸 써야 하나요?
대부분의 경우 ArrayList가 답입니다. 인덱스 접근이 O(1)이고, 메모리 연속성으로 캐시도 잘 받습니다. LinkedList는 양 끝에서 자주 추가/제거하면서 인덱스 접근은 거의 안 하는 큐/덱 용도일 때 의미가 있는데, 그것마저도 보통 ArrayDeque가 더 빠릅니다.
