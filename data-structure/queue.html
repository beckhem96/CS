큐(Queue)

큐란, 한쪽 끝(rear)에서 데이터를 넣고 반대쪽 끝(front)에서 꺼내는 FIFO(First-In-First-Out, 선입선출) 구조입니다. 먼저 들어온 데이터가 먼저 나옵니다.

구체적인 개념 설명

비유로 가장 직관적인 건 매표소 줄서기입니다. 줄 뒤로 가서 서고, 앞쪽부터 차례로 표를 받고 빠져나갑니다. 새치기 없이 들어온 순서대로 처리되는 게 큐의 핵심이에요.

큐의 기본 연산은 두 가지입니다.

  • enqueue(x) — rear 쪽에 새 데이터를 넣는다.
  • dequeue() — front 쪽 데이터를 꺼내서 반환한다.

여기에 peek()(front 들여다보기), isEmpty(), size()가 따라옵니다. 스택과 달리 들어오는 쪽과 나가는 쪽이 반대편이라는 점이 핵심입니다.

큐의 변종

  • 선형 큐(Linear Queue) — 배열 한 줄에 front/rear 인덱스를 두는 가장 단순한 형태. 메모리 낭비 문제가 있음.
  • 원형 큐(Circular Queue) — 배열의 끝과 처음을 이어 붙여 메모리를 재활용하는 형태.
  • 우선순위 큐(Priority Queue) — 들어온 순서가 아니라 우선순위에 따라 꺼내짐. 보통 힙으로 구현.
  • 덱(Deque) — 양 끝에서 모두 넣고 뺄 수 있는 일반화된 큐. 덱 페이지 참고.

왜 필요한가?

"들어온 순서대로 공정하게 처리해야 한다"는 상황에서 큐는 정답입니다. OS 스케줄러가 작업을 받는 순서대로 처리하거나, 프린터에 전송한 순서대로 인쇄되거나, 네트워크 패킷이 도착한 순서대로 라우팅되어야 할 때 모두 큐가 깔려 있죠.

또 하나, BFS(너비 우선 탐색)도 큐 없이는 구현할 수 없습니다. 현재 위치에서 갈 수 있는 곳을 큐에 넣어두고 가까운 곳부터 차례로 꺼내 탐색해야 "최단 거리"를 보장할 수 있기 때문입니다.

어떻게 동작하는가?

1) enqueue / dequeue 단계별 시각화

빈 큐에 10, 20, 30을 enqueue한 뒤 한 번 dequeue하는 과정입니다.

front은 가장 오래된 원소, rear는 가장 최근에 들어온 원소를 가리킨다.

2) 선형 큐의 문제 — 그리고 원형 큐

배열로 큐를 만들면 dequeue를 반복할 때마다 front가 오른쪽으로 이동해서, 앞쪽에 빈 자리가 생기지만 재활용되지 않습니다. 결국 배열이 안 찼는데도 rear가 끝에 도달해서 더 이상 enqueue가 안 되는 일이 생깁니다.

원형 큐는 인덱스를 모듈러 연산( i % capacity )으로 다뤄 메모리를 끝까지 활용한다.

3) Java 코드로 보기

import java.util.ArrayDeque;
import java.util.Queue;

// Queue 인터페이스의 표준 구현으로 ArrayDeque 권장
Queue<Integer> queue = new ArrayDeque<>();

queue.offer(10);   // enqueue
queue.offer(20);
queue.offer(30);

int x = queue.poll();      // dequeue → 10
int peek = queue.peek();   // 20 (꺼내지는 않음)
boolean empty = queue.isEmpty();

// 응용: 그래프 BFS
import java.util.*;

void bfs(Map<Integer, List<Integer>> graph, int start) {
    Queue<Integer> q = new ArrayDeque<>();
    Set<Integer> visited = new HashSet<>();

    q.offer(start);
    visited.add(start);

    while (!q.isEmpty()) {
        int cur = q.poll();
        // cur 처리

        for (int next : graph.getOrDefault(cur, List.of())) {
            if (!visited.contains(next)) {
                visited.add(next);
                q.offer(next);
            }
        }
    }
}

시간/공간 복잡도

연산시간설명
enqueueO(1)rear에 추가
dequeueO(1)front에서 제거
peekO(1)front 들여다보기
탐색 (특정 값)O(n)큐의 본 목적이 아님

공간 복잡도는 O(n). 단, 선형 큐를 단순 배열로 만들면 dequeue 후 shift를 한다거나 하면 dequeue가 O(n)이 될 수 있어, 실무에선 원형 큐 또는 연결 리스트 기반 구현을 씁니다.

실무에서 어디에 쓰이는가?

  • OS 스케줄링 — 프로세스/스레드 대기열, 라운드 로빈 스케줄링.
  • BFS(너비 우선 탐색) — 미로 최단 경로, 소셜 네트워크의 "친구의 친구" 탐색.
  • 메시지 큐(Message Queue) — Kafka, RabbitMQ, SQS 등 비동기 시스템의 핵심.
  • 프린터 작업 대기열, 콜센터 통화 대기열 — 공정한 순서 처리가 필요한 모든 곳.
  • 버퍼/캐싱 — 동영상 스트리밍의 패킷 버퍼, 키보드 입력 버퍼.
  • 자바 ExecutorService — 작업 큐(BlockingQueue)에 작업을 넣고 워커 스레드가 꺼내 처리.

면접에 나올 질문과 답변

스택과 큐의 차이를 한 문장으로 설명하면?
스택은 LIFO(가장 최근 것이 먼저), 큐는 FIFO(가장 오래된 것이 먼저)입니다. 스택은 한쪽 끝에서만 넣고 빼고, 큐는 한쪽 끝에서 넣고 반대쪽 끝에서 뺍니다.
선형 큐의 단점과 원형 큐가 어떻게 그것을 해결하는지 설명해 주세요.
선형 큐는 dequeue가 반복되면 front가 계속 오른쪽으로 이동해 앞쪽 메모리를 다시 못 씁니다. 결국 배열이 가득 차지 않았는데도 rear가 끝에 닿아 enqueue가 막히는 "false-full" 문제가 생기죠. 원형 큐는 인덱스를 i % capacity로 다뤄서, rear가 끝에 닿으면 0번으로 돌아가 비어 있는 앞쪽 메모리를 채우게 합니다.
우선순위 큐는 어떻게 동작하나요?
들어온 순서가 아니라 우선순위가 높은 원소가 먼저 나옵니다. 보통 힙(Heap)으로 구현해서 삽입과 최우선 원소 추출이 모두 O(log n)에 가능합니다. 다익스트라 알고리즘, 이벤트 시뮬레이션, 작업 스케줄러처럼 "가장 중요한 것을 먼저"가 필요할 때 씁니다.
왜 BFS는 큐를, DFS는 스택을 쓰나요?
BFS는 가까운 곳부터 차례로 탐색해야 하기 때문에 "먼저 발견한 노드를 먼저 처리"하는 큐가 필요합니다. DFS는 한 방향으로 끝까지 들어갔다가 막히면 되돌아오는 방식이라, "가장 최근에 발견한 노드를 먼저"인 스택이 자연스럽습니다. DFS는 재귀로도 구현되는데 그 재귀가 결국 호출 스택을 쓰는 것과 같은 원리예요.
