트라이(Trie)

트라이란, 문자열을 한 글자씩 트리의 간선에 매핑해 저장하는 자료구조입니다. 공통 접두사를 공유해서 메모리를 절약하고, 길이 L짜리 문자열의 검색·삽입을 O(L)에 처리합니다. 이름은 Retrieval(검색)에서 따왔어요.

구체적인 개념 설명

트라이의 가장 좋은 비유는 전화번호부 이름표 트리입니다. 공통된 첫 글자 'A'로 시작하는 이름들은 한 가지 아래에 모이고, 두 번째 글자가 'P'인 이름들은 또 그 아래로 묶입니다. "Apple"과 "App", "Apply"는 모두 A → P → P까지 같은 길을 공유하고, 그 다음에 갈라지죠. 이게 트라이의 핵심 아이디어입니다.

각 노드는 한 글자가 아니라, "여기까지 입력했을 때의 상태"를 나타냅니다. 간선(edge)이 한 글자에 해당하고, 노드는 자식들을 자식 글자별로 가리키는 포인터를 가집니다. 그리고 어떤 노드에 도달했을 때 "이 위치에서 단어가 끝난다"는 표시(end-of-word 플래그)를 두면 그 노드까지 이어진 글자들이 하나의 단어가 됩니다.

자식 포인터를 어떻게 저장할까?

  • 고정 크기 배열 (예: 영문 소문자 26칸) — 간단하고 빠르지만, 자식이 적은 노드에서 메모리가 낭비됨.
  • 해시 맵 (Map<Character, Node>) — 자식이 적을 때 메모리 효율적. 약간 느리지만 알파벳 외 문자도 자유롭게 처리.
  • 압축 트라이(Radix Tree / Patricia Trie) — 자식이 하나뿐인 연속된 노드들을 합쳐 압축. 메모리 더욱 절약.

왜 필요한가?

문자열 검색을 해시 테이블이나 BST로도 할 수 있지만, 두 가지 한계가 있습니다.

  • 해시 테이블은 접두사 검색이 안 됩니다. "ap로 시작하는 모든 단어"를 빠르게 가져올 수 없어요.
  • BST는 비교가 일어날 때마다 문자열 전체를 비교해야 해서 검색이 O(L log n)이 됩니다.

트라이는 검색·삽입이 모두 O(L)(L = 문자열 길이)로 단어 개수와 무관하고, 접두사 검색이 자연스럽게 됩니다. 게다가 공통 접두사를 한 번만 저장하기 때문에 메모리도 효율적이죠. 자동완성·맞춤법 검사·IP 라우팅 같은 "접두사가 핵심인 문제"에서 빛을 발합니다.

어떻게 동작하는가?

1) 트라이의 모양

"app", "apple", "ape", "bat"을 저장한 트라이입니다. 노란 노드가 단어가 끝나는 자리(end-of-word).

"app", "apple"이 a-p-p까지 같은 길을 공유한다. 메모리 절약의 핵심.

2) 검색 — "apple" 찾기

루트에서 시작해 글자 하나씩 따라 내려가며, 모든 글자를 따라가고 마지막 노드에 end-of-word 플래그가 켜져 있으면 단어가 존재합니다.

  1. 루트 → 'a' 자식 (있음)
  2. 'a' → 'p' 자식 (있음)
  3. 'p' → 'p' 자식 (있음)
  4. 'p' → 'l' 자식 (있음)
  5. 'l' → 'e' 자식 (있음, 그리고 ✓) → 존재

중간에 자식이 없으면 그 시점에서 "단어 없음"으로 종료합니다.

3) 접두사로 시작하는 모든 단어 찾기

"ap"로 시작하는 모든 단어를 찾고 싶다면, 먼저 a → p까지 따라가서 그 노드 아래의 서브트리를 DFS로 순회하며 ✓가 켜진 모든 노드를 모으면 됩니다. 자동완성의 동작 원리가 정확히 이거예요.

4) Java 코드로 보기

class TrieNode {
    Map<Character, TrieNode> children = new HashMap<>();
    boolean isEnd = false;
}

class Trie {
    TrieNode root = new TrieNode();

    // 삽입 — O(L)
    void insert(String word) {
        TrieNode cur = root;
        for (char c : word.toCharArray()) {
            cur = cur.children.computeIfAbsent(c, k -> new TrieNode());
        }
        cur.isEnd = true;
    }

    // 정확히 그 단어가 있는지 — O(L)
    boolean contains(String word) {
        TrieNode node = walk(word);
        return node != null && node.isEnd;
    }

    // 접두사 검색 — O(L)
    boolean startsWith(String prefix) {
        return walk(prefix) != null;
    }

    private TrieNode walk(String s) {
        TrieNode cur = root;
        for (char c : s.toCharArray()) {
            cur = cur.children.get(c);
            if (cur == null) return null;
        }
        return cur;
    }

    // 접두사로 시작하는 모든 단어 — 자동완성용
    List<String> autocomplete(String prefix) {
        List<String> result = new ArrayList<>();
        TrieNode node = walk(prefix);
        if (node != null) collect(node, new StringBuilder(prefix), result);
        return result;
    }

    private void collect(TrieNode node, StringBuilder path, List<String> out) {
        if (node.isEnd) out.add(path.toString());
        for (var entry : node.children.entrySet()) {
            path.append(entry.getKey());
            collect(entry.getValue(), path, out);
            path.deleteCharAt(path.length() - 1);
        }
    }
}

시간/공간 복잡도

연산시간설명
삽입O(L)L = 문자열 길이. 단어 개수와 무관
탐색O(L)중간에 자식이 없으면 즉시 종료
접두사 검색O(L)접두사 끝까지 따라가기만 하면 됨
접두사로 시작하는 단어 모두 모으기O(L + k·M)k = 결과 개수, M = 단어 평균 길이

공간이 트라이의 약점입니다. 노드마다 자식 포인터 배열/맵을 들고 있어, 메모리 사용량은 총 글자 수 × 알파벳 크기가 최악입니다. 대안으로 해시 맵 자식 또는 Radix Tree(Patricia)로 압축하면 크게 줄일 수 있습니다.

실무에서 어디에 쓰이는가?

  • 자동완성 — 검색창, IDE의 변수명/메서드 추천, 휴대폰 한글 입력기.
  • 맞춤법 검사 / 사전 검색 — 후보 단어를 빠르게 찾기.
  • IP 라우팅 테이블 — Longest Prefix Match를 위한 Radix Tree (트라이의 압축 변형).
  • 문자열 다중 매칭 — Aho-Corasick 알고리즘은 트라이 + 실패 링크로 여러 패턴을 한 번에 검색.
  • 금칙어/태그 필터링 — 미리 사전을 트라이로 만들어 두고 입력을 한 번 훑어 검사.
  • 유전체 서열 분석 — 짧은 문자열(서열) 검색에 트라이 변형 사용.

면접에 나올 질문과 답변

트라이와 해시 테이블 중 문자열 저장에 어느 게 좋은가요?
단순한 존재 여부만 빠르게 확인하면 해시 테이블이 평균 O(1)로 더 빠릅니다. 하지만 접두사 검색(자동완성, "apple로 시작하는 모든 단어" 등)이 필요하다면 트라이가 압도적으로 유리합니다. 해시 테이블은 키 전체가 같아야 매칭되지만, 트라이는 접두사 노드까지만 따라가도 그 아래 모든 단어를 얻을 수 있어요.
트라이의 메모리 단점을 줄이는 방법은요?
고정 크기 자식 배열 대신 해시 맵으로 자식을 저장해 빈 슬롯을 없애는 게 가장 단순합니다. 더 줄이려면 Radix Tree(Patricia Trie)처럼 자식이 하나뿐인 연속 노드들을 하나의 간선으로 합쳐 압축할 수 있습니다. 영어 사전이나 IP 라우팅 테이블처럼 분기가 적은 데이터에서는 압축 효과가 큽니다.
트라이로 자동완성을 어떻게 구현하나요?
사용자가 입력한 접두사를 따라 트라이를 내려가서 해당 노드를 찾고, 그 서브트리를 DFS로 순회하며 end-of-word가 켜진 단어들을 모읍니다. 실무에선 후보가 너무 많을 수 있어 빈도수나 점수를 노드에 저장해 두고 우선순위 큐로 상위 N개만 뽑거나, 각 노드에 미리 인기 단어 K개를 캐싱해 두는 식의 최적화를 합니다.
Aho-Corasick 알고리즘은 트라이를 어떻게 활용하나요?
여러 개의 검색 패턴을 트라이로 만든 뒤, 각 노드에 "현재 매칭이 실패했을 때 어디로 다시 시작할지"를 가리키는 실패 링크(failure link)를 추가합니다. 이렇게 하면 입력 문자열을 한 번 훑는 것만으로 모든 패턴의 모든 등장 위치를 찾을 수 있어, 다중 패턴 매칭이 O(텍스트 길이 + 패턴 총 길이 + 발견 횟수)에 끝납니다. 스팸/금칙어 필터, IDS 패킷 검사 같은 곳에서 사용해요.
