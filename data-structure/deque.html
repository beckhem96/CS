덱(Deque)

덱(Double-Ended Queue)이란, 양쪽 끝에서 모두 삽입과 삭제가 가능한 큐입니다. 스택과 큐의 동작을 모두 흡수한 더 일반적인 자료구조라고 보면 됩니다.

구체적인 개념 설명

덱은 발음 그대로 "Double-Ended Queue"의 줄임말입니다. 큐는 한쪽 끝에서만 넣고 반대쪽에서만 빼지만, 덱은 front와 rear 양쪽에서 모두 push/pop이 가능합니다. 그래서 다음 4가지 동작을 모두 지원합니다.

  • addFirst(x) / removeFirst() — front 쪽 삽입/삭제
  • addLast(x) / removeLast() — rear 쪽 삽입/삭제

이 4가지를 어떻게 조합하느냐에 따라 덱은 스택처럼도, 큐처럼도, 또는 그 둘을 섞은 더 유연한 형태로도 동작합니다. 덱을 알면 스택과 큐를 따로 구현할 필요가 사실상 없어진다는 의미예요.

덱 = 스택 + 큐

  • 한쪽 끝에서만 addFirst / removeFirst 만 쓰면 → 스택
  • addLast로 넣고 removeFirst로 빼면 → 큐
  • 네 동작을 모두 쓰면 → 양방향 슬라이딩 윈도우, 회문 검사 등 더 복잡한 패턴 처리 가능

왜 필요한가?

"한쪽 끝뿐만 아니라 반대쪽 끝도 가끔 다뤄야 한다"는 상황에서 덱이 필요해집니다. 대표적인 예가 슬라이딩 윈도우 최댓값 같은 알고리즘 문제입니다. 윈도우가 오른쪽으로 이동할 때 새 원소를 rear에 추가하고, 윈도우 밖으로 나간 원소는 front에서 제거하는 작업을 모두 O(1)에 해야 하기 때문에 덱이 딱 맞습니다.

또 Java 같은 언어에서는 스택과 큐의 표준 구현으로 ArrayDeque를 권장합니다. 덱이 더 일반적이라 한 자료구조로 양쪽 용도를 다 커버할 수 있고, 동기화 오버헤드도 없어 빠르기 때문입니다.

어떻게 동작하는가?

1) 4가지 연산 시각화

각 연산이 front/rear 어느 쪽에 작용하는지 한눈에 봅시다.

덱은 양 끝에서 4가지 연산이 모두 O(1).

2) 응용 — 슬라이딩 윈도우 최댓값

배열 [1, 3, -1, -3, 5, 3, 6, 7]에서 윈도우 크기 3의 최댓값을 매번 구하라는 문제를 봅시다. 매번 윈도우 안을 다 훑으면 O(n·k)인데, 덱을 쓰면 O(n)에 끝낼 수 있습니다.

핵심 아이디어: 덱 안에는 "앞으로 답이 될 가능성이 있는 인덱스"만 인덱스 순서대로 유지합니다. 새 원소를 넣을 때 그보다 작은 값이 rear에 있다면 모두 제거하고(어차피 더 큰 새 값이 들어왔으니 영영 답이 아님), 윈도우 밖으로 나간 인덱스는 front에서 제거합니다. 그러면 항상 front에 답이 있습니다.

덱에 단조 감소 인덱스만 유지하기 때문에 한 원소가 들어오고 나가는 횟수는 각각 1번 → 전체 O(n).

3) Java 코드로 보기

import java.util.ArrayDeque;
import java.util.Deque;

Deque<Integer> dq = new ArrayDeque<>();

dq.addFirst(10);    // [10]
dq.addLast(20);     // [10, 20]
dq.addFirst(5);     // [5, 10, 20]

int f = dq.peekFirst();   // 5
int l = dq.peekLast();    // 20

dq.removeFirst();   // → 5,  덱: [10, 20]
dq.removeLast();    // → 20, 덱: [10]

// 슬라이딩 윈도우 최댓값
int[] maxSlidingWindow(int[] nums, int k) {
    int[] ans = new int[nums.length - k + 1];
    Deque<Integer> dq = new ArrayDeque<>(); // 인덱스를 저장

    for (int i = 0; i < nums.length; i++) {
        // 윈도우 밖으로 나간 인덱스 제거
        if (!dq.isEmpty() && dq.peekFirst() <= i - k) {
            dq.pollFirst();
        }
        // 새 원소보다 작은 값들은 답이 될 수 없으니 제거
        while (!dq.isEmpty() && nums[dq.peekLast()] < nums[i]) {
            dq.pollLast();
        }
        dq.offerLast(i);

        if (i >= k - 1) ans[i - k + 1] = nums[dq.peekFirst()];
    }
    return ans;
}

시간/공간 복잡도

연산시간설명
addFirst / addLastO(1)양 끝 삽입 (동적 배열 기반은 분할상환 O(1))
removeFirst / removeLastO(1)양 끝 삭제
peekFirst / peekLastO(1)들여다보기
중간 접근/삽입O(n)덱의 본 목적이 아님

구현은 보통 원형 배열 기반(Java ArrayDeque)이거나 이중 연결 리스트 기반(Java LinkedList)입니다. 대부분의 경우 캐시 효율이 좋은 원형 배열 기반이 빠릅니다.

실무에서 어디에 쓰이는가?

  • 스택/큐의 표준 구현 — Java에서는 ArrayDeque가 사실상 둘의 기본 자리.
  • 슬라이딩 윈도우 알고리즘 — 윈도우 최댓값/최솟값, 일정 시간 동안의 통계 등.
  • 회문 검사 — 양 끝에서 동시에 비교하면서 좁혀 들어가는 패턴.
  • 작업 스케줄러의 양방향 큐 — 우선 처리 / 일반 처리를 양 끝에 나눠 관리.
  • 실행 취소(Undo)와 최근 기록(Recent History) — 한쪽 끝에 추가하다가 일정 개수가 넘으면 반대쪽 끝에서 오래된 것을 제거.

면접에 나올 질문과 답변

덱과 큐, 스택의 관계를 설명해 주세요.
덱은 양 끝에서 모두 삽입/삭제가 가능한 자료구조라, 한쪽 끝만 사용하면 스택, 한쪽으로 넣고 반대쪽에서 빼면 큐로 동작합니다. 그래서 덱은 스택과 큐의 일반화된 형태이고, 하나만 있으면 두 가지 다 만들 수 있습니다.
왜 Java에서 Stack이나 LinkedList 대신 ArrayDeque를 권장하나요?
Stack은 Vector 기반이라 동기화 오버헤드가 있고 설계가 낡았습니다. LinkedList는 노드별 메모리 할당과 포인터 추적 때문에 캐시 효율이 떨어집니다. ArrayDeque는 원형 배열 기반이라 캐시 친화적이고, 동기화도 없으며, 스택과 큐 양쪽 용도를 모두 커버합니다.
슬라이딩 윈도우 문제에서 덱이 왜 효율적인가요?
덱에 "앞으로 답이 될 가능성이 있는 후보들"만 단조롭게 유지하면, 각 원소가 덱에 한 번 들어가고 한 번 나옵니다. 그래서 전체 작업이 O(n)에 끝납니다. 매번 윈도우 안을 다 훑는 단순한 방법이 O(n·k)인 것에 비해 큰 이득이죠. 이때 양 끝에서 빠르게 제거할 수 있는 덱의 성질이 핵심입니다.
덱은 어떻게 구현하는 게 좋은가요?
대부분의 상황에서 원형 배열 기반 구현이 좋습니다(Java의 ArrayDeque가 그렇습니다). front/rear 인덱스를 모듈러 연산으로 관리하면 양 끝 연산이 O(1)이고, 메모리 연속성 덕분에 캐시 효율이 좋습니다. 다만 동시성이 필요하다면 ConcurrentLinkedDeque처럼 락프리 연결 리스트 기반 구현을 고려합니다.
