HTTP / HTTPS

HTTP(HyperText Transfer Protocol)는 클라이언트와 서버가 자원을 주고받기 위한 요청–응답형 텍스트 기반 프로토콜입니다. HTTPS는 그 위에 TLS(Transport Layer Security)로 암호화·무결성·서버 인증을 더한 것이고, 이름 그대로 HTTP + Secure입니다. 두 프로토콜 모두 응용 계층(L7)에서 동작하며, 무상태성(stateless)을 기본 원칙으로 삼습니다.

구체적인 개념 설명

HTTP는 우리가 매일 쓰는 웹의 대화 규칙입니다. 브라우저가 "이 자원을 줘"라고 요청하면 서버가 "여기 있어"라고 응답합니다. 이 단순한 골격 위에 메서드, 상태 코드, 헤더, 캐시 같은 약속들이 쌓여 있습니다. 텍스트 기반이라 사람이 직접 읽을 수 있고, 덕분에 디버깅이 쉽다는 게 큰 장점입니다(curl -v 한 번이면 끝).

무상태성은 HTTP의 가장 중요한 설계 원칙입니다 — 서버는 이전 요청을 "기억"하지 않습니다. 매 요청은 그 자체로 완결되어야 하며, 서로 연관된 요청들의 맥락은 쿠키·세션·토큰 같은 별도 메커니즘으로 만들어 줘야 합니다. 이 단순함이 거대한 분산 시스템의 확장성을 가능하게 했습니다 — 서버를 마구 늘려도 요청을 어느 서버가 받든 상관없으니까요.

HTTPS는 그 위에 TLS를 한 겹 깐 것입니다. TLS가 보장하는 것은 세 가지: 데이터를 도청할 수 없는 기밀성, 중간에 변조되면 알아챌 수 있는 무결성, 내가 접속한 서버가 진짜 그 서버인지 확인하는 인증입니다. HTTP 자체의 문법은 그대로지만 그 메시지를 암호화된 통로 안으로 흘려보낸다고 보면 됩니다.

HTTPS는 "암호화"만이 아니다

TLS의 진짜 가치는 셋 모두입니다. 암호화만 있다면 공격자가 서버인 척 위장(MITM)해 가짜 응답을 보내도 알 수 없습니다. 인증서로 서버 신원을 확인하고, MAC/AEAD로 변조를 막고, 대칭 키로 데이터를 암호화 — 이 세 가지가 모두 한 번에 작동해야 합니다.

왜 필요한가?

HTTP는 1991년 단순한 문서 공유용으로 시작했지만, 지금은 거의 모든 응용 통신의 사실상 표준이 되었습니다. 이유는 단순함과 유연함입니다 — 텍스트 헤더로 무엇이든 표현할 수 있고, 어떤 방화벽도 80/443 포트는 열어두기 때문에 다른 프로토콜이 끼어들 틈이 없습니다. gRPC, WebSocket, SSE 같은 후계자들도 결국 HTTP의 핸드셰이크와 헤더를 빌려 쓰는 이유입니다.

HTTPS가 필요한 이유는 두 가지입니다. 첫째, 평문 HTTP는 같은 Wi-Fi에 있는 누구나 들여다볼 수 있고 변조할 수 있습니다(공항·카페 와이파이 광고 삽입 사례). 둘째, 인증서가 없으면 사용자는 자기가 진짜 은행 사이트에 접속한 건지 가짜 사이트에 접속한 건지 알 수 없습니다. 그래서 현대 브라우저는 HTTP만으로는 "Not Secure" 경고를 띄우고, 일부 API(Geolocation, Service Worker 등)는 HTTPS에서만 동작합니다.

어떻게 동작하는가?

1) HTTP 요청·응답 메시지 구조

HTTP 메시지는 세 부분으로 나뉩니다 — start line, headers, body. 헤더와 본문은 빈 줄(\r\n\r\n)로 구분합니다.

요청·응답 모두 첫 줄(start line) → 헤더들 → 빈 줄 → 본문의 동일한 골격을 가진다.

2) HTTP 메서드 — 안전성과 멱등성

메서드는 자원에 대해 무엇을 할 것인지 알리는 동사입니다. 단순히 GET/POST 두 개가 아니라 안전성(safety)과 멱등성(idempotency) 두 축으로 분류해 외워두면 좋습니다.

메서드용도안전한가?멱등한가?본문 있나?
GET자원 조회OO(원칙적으로) X
HEAD응답 헤더만 조회OOX
OPTIONS지원 메서드 확인 / preflightOOX
POST자원 생성, 또는 일반적인 처리XXO
PUT자원 전체 교체(또는 생성)XOO
PATCH자원 부분 수정XX (보장 안 함)O
DELETE자원 삭제XO(선택)
  • 안전(safe) = 서버 상태를 변경하지 않음. GET·HEAD·OPTIONS만 안전.
  • 멱등(idempotent) = 같은 요청을 여러 번 보내도 결과가 같음. PUT/DELETE는 멱등이지만 POST/PATCH는 보장 안 함.
  • 이 분류는 단순한 학문적 구분이 아닙니다. 멱등 메서드는 네트워크 오류 시 안전하게 재시도할 수 있어 클라이언트·프록시·로드밸런서가 자동 재시도해도 됩니다.

3) 상태 코드 5범주

범주의미대표 코드
1xx정보(처리 중)100 Continue, 101 Switching Protocols(WebSocket 업그레이드)
2xx성공200 OK, 201 Created, 204 No Content, 206 Partial Content
3xx리다이렉션301 영구 이동, 302/303/307/308 이동, 304 Not Modified(캐시)
4xx클라이언트 오류400 Bad Request, 401 인증 안 됨, 403 권한 없음, 404 없음, 409 충돌, 429 너무 많은 요청
5xx서버 오류500 Internal Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout

자주 헷갈리는 코드 정리:

  • 301 vs 308: 둘 다 영구 이동이지만, 301은 일부 클라이언트가 메서드를 GET으로 바꿔버리고, 308은 메서드와 본문을 그대로 유지합니다.
  • 302 vs 307: 302는 같은 이유로 메서드가 바뀔 수 있고, 307은 메서드 보존을 명확히 합니다. 폼 POST 후 리다이렉트엔 303을 권장.
  • 401 vs 403: 401은 "당신이 누군지 모르겠다(인증 필요)", 403은 "누군지는 알지만 권한 없다".
  • 502 vs 504: 둘 다 게이트웨이 문제. 502는 백엔드가 잘못된 응답을 줬을 때, 504는 백엔드가 시간 안에 응답 못 했을 때.

4) 헤더 — 카테고리로 묶어 이해

헤더는 종류가 매우 많지만 대부분 다음 4범주 중 하나입니다:

  • 일반(General) — 메시지 자체에 관한 정보. Date, Connection, Cache-Control.
  • 요청(Request) — 요청 클라이언트나 조건. Host, User-Agent, Accept, Authorization, Cookie, If-Modified-Since.
  • 응답(Response) — 응답 서버나 조건. Server, Set-Cookie, WWW-Authenticate, Location, ETag.
  • 엔티티(Entity) — 본문에 관한 메타. Content-Type, Content-Length, Content-Encoding(gzip/br).

5) 캐시 — Conditional GET

HTTP는 같은 자원을 여러 번 받지 않도록 강력한 캐시 메커니즘을 가집니다. 핵심은 조건부 GET입니다. 서버는 응답에 ETag(콘텐츠의 지문)나 Last-Modified를 같이 줍니다. 다음에 클라이언트는 If-None-Match나 If-Modified-Since로 "내가 이 버전 갖고 있는데 바뀌었어?"라고 묻고, 서버는 안 바뀌었으면 304 Not Modified(본문 없음)로 응답합니다.

캐시 정책은 Cache-Control로 표현합니다: public/private(누가 캐싱 가능), max-age=N(N초간 신선), no-cache(캐시 사용 전 매번 검증), no-store(아예 저장 금지), immutable(절대 변하지 않음).

6) 무상태성과 상태 유지

앞서 말했듯 HTTP 자체는 무상태입니다. 그러나 실제 웹은 "로그인", "장바구니" 같은 상태가 필요하니 별도 메커니즘으로 만들어줍니다:

  • 쿠키 — 서버가 Set-Cookie로 발급, 클라이언트가 다음 요청부터 Cookie 헤더로 자동 전송.
  • 세션 — 서버 메모리/DB에 상태를 두고 식별자만 쿠키에 담음.
  • 토큰(JWT) — 서명된 토큰을 클라이언트가 보관, 서버는 토큰을 검증만 하면 되어 stateless 유지.

각 방식의 장단점은 별도 페이지(쿠키 / 세션 / JWT — 작성 예정)에서 깊이 다룹니다.

7) HTTPS — HTTP + TLS

HTTPS의 동작은 단순합니다. TCP 연결이 수립되면 곧장 TLS 핸드셰이크를 해서 안전한 통로를 만들고, 그 통로 안으로 평소의 HTTP 메시지를 흘려보냅니다. 클라이언트와 서버는 핸드셰이크에서:

  1. 지원 가능한 암호 스위트(cipher suite) 협상
  2. 서버 인증서로 서버 신원 검증 (CA 체인 → 루트 CA)
  3. 키 교환(보통 ECDHE)으로 양쪽이 동일한 대칭 키 도출
  4. 이후 그 대칭 키로 모든 메시지를 AEAD(예: AES-GCM, ChaCha20-Poly1305)로 암호화·인증
TLS 1.3는 핸드셰이크가 1 RTT(0-RTT 옵션도 가능). TLS 1.2는 2 RTT가 필요했다. 상세는 별도 페이지에서.

TLS 1.2는 핸드셰이크에 2 RTT가 필요했지만, TLS 1.3은 1 RTT로 줄였고 0-RTT 재연결도 지원합니다. 자세한 메시지 단계와 0-RTT의 보안 trade-off는 "TLS 핸드셰이크 상세"(작성 예정) 페이지에서 다룹니다.

8) Java 코드

// (1) HttpClient — JDK 11+ 표준
HttpClient client = HttpClient.newBuilder()
    .version(HttpClient.Version.HTTP_2)         // HTTPS면 자동으로 TLS 사용
    .connectTimeout(Duration.ofSeconds(3))
    .build();

HttpRequest req = HttpRequest.newBuilder()
    .uri(URI.create("https://api.example.com/users/42"))
    .header("Accept", "application/json")
    .header("Authorization", "Bearer " + token)
    .GET()
    .build();

HttpResponse<String> res = client.send(req, HttpResponse.BodyHandlers.ofString());
System.out.println(res.statusCode());           // 200
System.out.println(res.headers().firstValue("content-type").orElse(""));

// (2) 캐시 검증 — If-None-Match
String etag = previousResponse.headers().firstValue("ETag").orElse(null);
HttpRequest req = HttpRequest.newBuilder(uri)
    .header("If-None-Match", etag)
    .GET().build();
// 응답이 304 Not Modified면 본문 없음 → 기존 캐시 사용

// (3) POST + JSON
HttpRequest post = HttpRequest.newBuilder(URI.create("https://api.example.com/users"))
    .header("Content-Type", "application/json")
    .POST(HttpRequest.BodyPublishers.ofString("{\"name\":\"Bob\"}"))
    .build();
// 응답: 201 Created + Location: /users/43

실무에서 어디에 쓰이는가?

  • 웹·모바일 API — 거의 모든 백엔드 API가 HTTP(S) 기반. REST·GraphQL·gRPC 모두 HTTP를 빌려 씀.
  • 서비스 간 통신 — 마이크로서비스의 내부 호출도 대부분 HTTP. 사이드카(mTLS)로 mutual TLS 자동화.
  • CDN과 캐시 계층 — Cache-Control·ETag로 엣지 캐시 정책 결정. 적절한 캐시는 곧 비용 절감.
  • 인증·보안 — Authorization 헤더, OAuth 2.0, mTLS, HSTS 헤더로 강제 HTTPS.
  • 장애 분석 — 5xx 비율, P95/P99 응답 시간, 4xx 패턴이 알람·SLO의 기본 지표.
  • 관측·디버깅 — curl -v, 브라우저 DevTools, mitmproxy로 헤더와 본문을 들여다봄.

면접에 나올 질문과 답변

HTTP의 무상태성(stateless)이란 무엇이고 왜 중요한가요?
서버가 이전 요청의 맥락을 기억하지 않고, 매 요청을 그 자체로 완결되도록 처리한다는 원칙입니다. 덕분에 서버를 자유롭게 추가·교체할 수 있어 수평 확장이 쉬워지고, 캐시·로드밸런서가 단순해집니다. 대신 인증 같은 상태는 쿠키, 세션, 토큰으로 별도 표현합니다.
HTTP 메서드의 안전성과 멱등성은 어떻게 다른가요?
안전(safe)은 서버 상태를 변경하지 않는 메서드(GET·HEAD·OPTIONS)를 말하고, 멱등(idempotent)은 같은 요청을 여러 번 보내도 같은 결과가 되는 메서드(GET·PUT·DELETE 등)입니다. 멱등 메서드는 네트워크 오류 시 안전하게 자동 재시도할 수 있어, 클라이언트·프록시 설계에 큰 영향을 줍니다.
PUT과 PATCH의 차이는 무엇인가요?
PUT은 자원 전체를 보낸 본문으로 교체합니다. 같은 요청을 여러 번 보내도 결과는 동일해 멱등입니다. PATCH는 자원의 일부만 수정합니다. 표준상 멱등성이 보장되지 않으며, 서버 구현에 따라 다릅니다(JSON Patch는 비멱등, JSON Merge Patch는 멱등 가능).
401과 403의 차이는 무엇인가요?
401은 "인증되지 않음(unauthenticated)" — 누구인지 모르겠으니 자격을 증명하라는 의미로, 응답에 WWW-Authenticate 헤더를 같이 보냅니다. 403은 "인증은 됐지만 권한 없음(forbidden)" — 누구인지 알지만 이 자원에 접근할 자격이 없다는 의미입니다.
HTTP 캐시 검증은 어떻게 동작하나요?
서버가 응답에 ETag(콘텐츠 지문) 또는 Last-Modified를 줍니다. 클라이언트는 다음에 If-None-Match나 If-Modified-Since로 보유한 버전을 알리고, 서버는 변경이 없으면 304 Not Modified로 본문 없이 응답합니다. Cache-Control과 함께 쓰면 강력한 신선도 + 검증 캐시가 됩니다.
HTTPS는 무엇을 보장하나요?
세 가지를 동시에 보장합니다 — 기밀성(도청 방지), 무결성(변조 감지), 서버 인증(접속한 곳이 진짜 그 서버임 확인). 대칭 암호로 데이터를, AEAD로 무결성을, 인증서 체인으로 신원을 처리합니다. 셋 중 하나라도 빠지면 안전하지 않습니다.
TLS 핸드셰이크는 왜 필요하고 무엇을 합니까?
평문 채널 위에 안전한 채널을 만드는 협상 절차입니다. 양쪽이 지원하는 암호 스위트를 합의하고, 서버 인증서로 신원을 확인하고, ECDHE 같은 키 교환으로 양쪽이 동일한 대칭 키를 도출합니다. 이후 모든 메시지는 그 대칭 키로 암호화·인증됩니다. TLS 1.3은 1 RTT, 0-RTT까지 가능합니다.
CONNECT 메서드는 언제 쓰이나요?
주로 HTTPS 프록시 통과에 쓰입니다. 클라이언트가 프록시에 CONNECT host:443을 보내면 프록시는 그 호스트에 TCP 터널을 만들고 그 안으로 TLS를 그대로 흘려보냅니다. 프록시는 본문을 못 봅니다(end-to-end 암호화 보존).
