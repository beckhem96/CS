CORS(Cross-Origin Resource Sharing)

CORS는 브라우저가 다른 출처(Origin)의 서버 자원에 접근할 수 있도록 허용 여부를 검사하는 보안 메커니즘입니다. 기본 원칙은 Same-Origin Policy(동일 출처 정책)로 막고, 서버가 Access-Control-* 헤더로 명시적으로 허용하면 브라우저가 응답을 JavaScript에 넘겨줍니다.

구체적인 개념 설명

CORS는 서버끼리 통신할 때 생기는 문제가 아니라 브라우저가 끼어 있을 때만 생기는 문제입니다. 서버 A가 서버 B를 호출하는 건 마음대로 가능하지만, 브라우저 안의 JavaScript가 다른 출처의 API를 호출하면 브라우저가 보안 검사를 합니다.

여기서 출처(Origin)는 프로토콜 + 호스트 + 포트 세 가지가 모두 같은 경우에만 동일합니다. https://example.com과 http://example.com은 프로토콜이 다르고, https://example.com과 https://api.example.com은 호스트가 다르므로 다른 출처입니다.

CORS 오류는 서버가 응답을 안 한 게 아니다

서버가 200 OK를 보내도, 브라우저가 CORS 헤더를 보고 "이 응답은 JS에 보여주면 안 돼"라고 판단하면 콘솔에 CORS 오류를 띄웁니다. Network 탭에는 응답이 보이는데 코드에서는 실패하는 이유가 바로 이겁니다.

왜 필요한가?

Same-Origin Policy가 없으면 악성 사이트가 사용자의 브라우저 권한을 빌려 은행·메일·사내 시스템에 요청을 보내고 응답을 읽을 수 있습니다. 사용자는 이미 해당 서비스에 로그인되어 있으므로 쿠키가 자동으로 붙고, 악성 JS는 민감한 데이터를 훔칠 수 있습니다.

다만 현대 웹은 프론트엔드와 API 서버의 도메인이 다른 경우가 많습니다. 그래서 완전히 막는 대신, 서버가 허용한 출처만 예외적으로 열어주는 방식이 CORS입니다.

어떻게 동작하는가?

1) 동일 출처와 교차 출처

서브도메인만 달라도 다른 출처다. 쿠키 Domain 설정과 CORS Origin은 별개의 개념이다.

2) Simple Request

브라우저가 바로 본 요청을 보내는 경우입니다. 조건은 꽤 제한적입니다:

  • 메서드가 GET, HEAD, POST 중 하나
  • 직접 설정한 헤더가 CORS-safelisted 헤더뿐 (Accept, Content-Language 등)
  • Content-Type이 application/x-www-form-urlencoded, multipart/form-data, text/plain 중 하나

서버가 응답에 Access-Control-Allow-Origin: https://app.example.com을 주면 성공하고, 없으면 브라우저가 응답을 JS에 넘기지 않습니다.

3) Preflight Request

PUT, DELETE, Authorization 헤더, application/json 같은 요청은 실제 요청 전에 브라우저가 OPTIONS preflight를 먼저 보냅니다. "이런 요청을 보내도 돼?"라고 서버에 사전 허가를 받는 단계입니다.

preflight는 브라우저가 자동으로 보내며, 응용 코드에서 직접 OPTIONS를 작성하는 것이 아니다.

5) Credentials 모드의 함정

쿠키나 HTTP 인증 정보를 교차 출처 요청에 포함하려면 프론트와 서버가 둘 다 명시해야 합니다. 

// 브라우저 fetch
fetch("https://api.example.com/me", {
  credentials: "include"
});

HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Credentials: true

이때 Access-Control-Allow-Origin: *와 credentials는 같이 쓸 수 없습니다. 쿠키를 포함한다는 건 사용자 권한을 싣는다는 뜻이라, 모든 출처(*)에 열면 위험하기 때문입니다.

6) Java/Spring 예시

// Spring MVC 예시 — 특정 출처만 허용
@Configuration
public class CorsConfig implements WebMvcConfigurer {
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/api/**")
            .allowedOrigins("https://app.example.com")
            .allowedMethods("GET", "POST", "PUT", "DELETE")
            .allowedHeaders("Authorization", "Content-Type")
            .allowCredentials(true)
            .maxAge(3600);
    }
}

// 위험한 설정 — 인증이 필요한 API에 쓰면 안 됨
registry.addMapping("/**")
    .allowedOriginPatterns("*")
    .allowedMethods("*")
    .allowCredentials(true); // 동적 echo와 결합하면 거의 모든 사이트에 쿠키 API를 열 수 있음

실무에서 어디에 쓰이는가?

  • SPA + API 서버 분리 — app.example.com에서 api.example.com 호출.
  • 로컬 개발 — localhost:3000 프론트가 localhost:8080 백엔드 호출. 포트가 달라 다른 Origin.
  • CDN 정적 자원 — 폰트·이미지·스크립트는 CORS 정책이 필요할 수 있음.
  • 사내 API 보안 — allowlist를 정확히 두고, 인증 API엔 * 금지.

면접에 나올 질문과 답변

CORS는 왜 브라우저에서만 문제가 되나요?
CORS는 브라우저의 보안 정책입니다. 서버끼리의 HTTP 호출에는 Same-Origin Policy가 적용되지 않습니다. 브라우저가 악성 JS가 사용자의 인증 쿠키를 악용해 다른 출처의 응답을 읽는 것을 막기 위해 검사합니다.
Origin은 무엇으로 구성되나요?
프로토콜, 호스트, 포트 세 가지입니다. 셋 중 하나라도 다르면 다른 Origin입니다. 경로(path)는 Origin에 포함되지 않습니다.
preflight는 언제 발생하나요?
simple request 조건을 벗어날 때 발생합니다. 예를 들어 PUT/DELETE, Authorization 헤더, application/json Content-Type을 쓰면 브라우저가 실제 요청 전에 OPTIONS로 허가를 확인합니다.
Access-Control-Allow-Origin: *와 쿠키를 같이 쓸 수 있나요?
불가능합니다. credentials 요청은 서버가 특정 Origin을 명시해야 하고 Access-Control-Allow-Credentials: true도 필요합니다. 모든 출처에 인증 정보를 노출하면 위험하기 때문입니다.
CORS와 CSRF는 어떤 관계인가요?
CORS는 교차 출처 응답을 JS가 읽는 것을 통제하고, CSRF는 사용자의 쿠키를 이용해 원치 않는 요청을 보내는 공격입니다. CORS가 CSRF를 완전히 막아주지 않습니다. SameSite 쿠키와 CSRF token이 별도로 필요합니다.
왜 Postman에서는 되는데 브라우저에서는 CORS 오류가 나나요?
Postman은 브라우저가 아니므로 Same-Origin Policy를 적용하지 않습니다. 서버 응답 자체는 정상이어도 브라우저가 CORS 헤더 부족으로 JS 접근을 막을 수 있습니다.
