3-way / 4-way Handshake

3-way handshake는 TCP 연결을 여는 세 번의 메시지 교환(SYN → SYN+ACK → ACK)이고, 4-way handshake는 연결을 닫는 네 번의 메시지 교환(FIN → ACK → FIN → ACK)입니다. 열 때는 양쪽 시퀀스 번호를 맞추고, 닫을 때는 양쪽이 각자 보낼 게 끝났다는 신호를 따로 주고받습니다.

구체적인 개념 설명

TCP는 신뢰성 있는 연결형 프로토콜입니다. "신뢰성"을 보장하려면 그 전에 양쪽이 통신할 준비가 됐다는 사실, 그리고 어떤 번호부터 데이터를 셀지를 합의해야 합니다. 이 합의가 3-way handshake입니다.

연결을 끊을 때도 그냥 "안녕"하고 사라지면 안 됩니다. 한쪽이 보낼 게 끝났다고 해서 상대도 보낼 게 끝났다는 뜻은 아니기 때문입니다. 그래서 양쪽이 각자 "나는 이제 안 보낼게"를 따로 선언하고 확인받습니다 — 이게 4-way handshake입니다.

주요 플래그 5개를 외워두면 그림이 잘 보입니다: SYN(연결 시작), ACK(확인), FIN(종료), RST(즉시 끊기), PSH(즉시 전달).

왜 "악수"라 부르는가?

실제 사람들이 만나 인사할 때도 한쪽이 손을 내밀고, 상대가 잡고, 서로 흔드는 짧은 의식이 있죠. TCP의 handshake도 한쪽이 "연결할까?", 상대가 "그래, 너도 받을 수 있어?", 마지막으로 "응 받아져"의 순서로 진행됩니다.

왜 필요한가?

그냥 데이터를 보내면 되지 굳이 인사부터 할 필요가 있을까 싶지만, 핸드셰이크는 다음 일을 동시에 합니다:

  • 상대 존재 확인 — 상대가 살아 있고 이 포트에서 듣고 있는가?
  • 초기 시퀀스 번호(ISN) 합의 — 양쪽이 각자 임의의 번호에서 시작해, 이후 ACK로 추적할 기반을 만든다.
  • 옵션 협상 — MSS(최대 세그먼트 크기), window scale, SACK 허용 여부 같은 파라미터를 맞춘다.
  • 오래된 패킷 차단 — 이전 연결의 늦은 패킷이 새 연결에 섞이는 것을 막는다(이래서 ISN을 무작위로 잡음).

어떻게 동작하는가?

1) 3-way handshake — 연결 수립

두 번째 패킷에서 SYN과 ACK가 합쳐져 3번이 된다 — 이론적으로는 4번을 효율화한 형태.
  1. Client → Server: SYN, seq=x — "내 시작 번호는 x야, 연결할래?"
  2. Server → Client: SYN+ACK, seq=y, ack=x+1 — "x 잘 받았어. 내 시작 번호는 y야."
  3. Client → Server: ACK, ack=y+1 — "y도 잘 받았어. 시작하자."

여기서 ack=x+1은 "x번까지 받았으니 다음으로 x+1을 기대한다"는 의미입니다. 세 번째 ACK 패킷은 데이터를 함께 실어 보낼 수도 있어서(piggyback) 실제론 첫 데이터 전송과 합쳐지는 경우가 많습니다.

2) 4-way handshake — 연결 종료

4번이 되는 이유는 양쪽이 각자 FIN을 보내고 각자 ACK를 받기 때문이다.
  1. Client → Server: FIN, seq=u — "나는 더 보낼 게 없어."
  2. Server → Client: ACK, ack=u+1 — "알았어. 근데 나는 마저 보내야 할 게 있어."
  3. Server → Client: FIN, seq=v — "나도 다 보냈어."
  4. Client → Server: ACK, ack=v+1 — "확인."

3) 왜 "3-way"는 3번이고 "4-way"는 4번인가?

열 때는 서버가 "받았다(ACK)"와 "나도 시작할게(SYN)"를 한 번에 묶어 보낼 수 있어 4번이 3번으로 줄어듭니다. 반면 닫을 때는 한쪽이 FIN을 보내도 상대는 보낼 데이터가 더 남아 있을 수 있어 ACK와 FIN을 따로 보내야 합니다. 그래서 4번이 됩니다(드물게 합칠 수 있는 경우 3번이 되기도 합니다).

4) TIME_WAIT — 왜 2×MSL을 기다리나?

종료를 시작한 쪽은 마지막 ACK를 보낸 뒤 곧장 닫지 않고 2×MSL(Maximum Segment Lifetime)만큼 대기합니다(보통 60초). 이유는 두 가지입니다:

  • 마지막 ACK 분실에 대비 — 상대가 못 받으면 FIN을 다시 보내올 텐데, 너무 빨리 닫으면 RST가 발생해 깔끔하지 않게 끝납니다.
  • 늦게 도착한 옛 패킷 차단 — 같은 (IP, 포트) 쌍으로 새 연결을 만들었을 때 옛 연결의 늦은 패킷이 섞이지 않도록.

서버에 TIME_WAIT 소켓이 너무 많이 쌓이면 새 연결을 못 받는 문제가 생기기도 합니다. 해결로는 keep-alive로 연결 재사용, SO_REUSEADDR/tw_reuse 같은 커널 튜닝이 흔히 쓰입니다.

5) Java 코드

// 클라이언트 — connect()가 내부적으로 3-way handshake를 수행한다
try (Socket s = new Socket()) {
    s.connect(new InetSocketAddress("example.com", 80), 3000);
    // 여기 도달하면 ESTABLISHED 상태
    var out = s.getOutputStream();
    out.write("GET / HTTP/1.1\r\nHost: example.com\r\n\r\n".getBytes());
    var in = s.getInputStream();
    in.readAllBytes();
}                                  // try-with-resources 종료 → close() → 4-way handshake

// 서버 — accept()가 핸드셰이크 완료된 소켓을 반환
try (ServerSocket ss = new ServerSocket(8080)) {
    while (true) {
        Socket client = ss.accept();   // 3번째 ACK까지 도착해야 반환
        // ...
        client.close();                // FIN 송신 시작
    }
}

시간/공간 복잡도 또는 trade-off

  • 1 RTT의 비용 — 3-way handshake는 최소 1 RTT가 더 들어 짧은 요청에선 무시할 수 없습니다. 그래서 HTTP/2의 keep-alive, TLS 1.3의 0-RTT, QUIC가 등장했습니다.
  • SYN flood — 공격자가 SYN만 잔뜩 보내고 ACK를 안 보내면 서버의 half-open 큐가 꽉 차 정상 연결이 막힙니다. SYN cookies로 큐를 쓰지 않고 응답하는 방어가 일반적입니다.
  • 지나친 연결 생성 — TIME_WAIT 누적, 포트 고갈 같은 부작용이 있어 연결 풀 / keep-alive를 적극 활용해야 합니다.

실무에서 어디에 쓰이는가?

  • HTTP keep-alive / 커넥션 풀 — 핸드셰이크 비용을 아끼기 위해 한 번 연 연결을 여러 요청에 재사용합니다.
  • TLS 1.3 / QUIC — 암호화 핸드셰이크를 TCP handshake와 합치거나 0-RTT로 단축합니다.
  • 로드밸런서 헬스체크 — TCP handshake가 성공하는지로 백엔드 생존 여부를 판단(L4 healthcheck).
  • 네트워크 디버깅 — tcpdump 'tcp[tcpflags] & (tcp-syn|tcp-fin) != 0'으로 핸드셰이크만 골라서 캡처할 수 있습니다.

면접에 나올 질문과 답변

3-way handshake 과정을 설명해 주세요.
클라이언트가 SYN(seq=x)을 보내면, 서버가 SYN+ACK(seq=y, ack=x+1)로 응답하고, 클라이언트가 ACK(ack=y+1)을 보내 연결을 확립합니다. 이 과정에서 양쪽의 초기 시퀀스 번호와 옵션을 합의하고, 상대의 도달성을 확인합니다.
왜 종료는 4-way인가요? 3-way로 줄일 수 없나요?
한쪽이 FIN을 보내도 상대는 보낼 데이터가 남아 있을 수 있기 때문에, ACK는 즉시 보내지만 FIN은 그 데이터를 다 보낸 뒤에 보냅니다. 보낼 데이터가 없는 경우 ACK와 FIN을 합쳐 3-way가 되기도 합니다.
TIME_WAIT 상태는 무엇이고 왜 필요한가요?
능동 종료 측이 마지막 ACK를 보낸 뒤 2×MSL 동안 머무는 상태입니다. 마지막 ACK가 분실됐을 때 재전송된 FIN에 다시 ACK할 수 있도록 하고, 옛 연결의 늦은 패킷이 새 연결에 섞이지 않도록 막아줍니다. 보통 60초 정도이며, 트래픽이 많은 서버에서는 keep-alive·tw_reuse로 영향을 줄입니다.
SYN flood 공격은 무엇이고 어떻게 막나요?
공격자가 SYN만 보내고 마지막 ACK를 보내지 않아 서버의 half-open 연결 큐를 꽉 채워 정상 사용자의 연결을 막는 공격입니다. 대표 방어는 SYN cookies로, 큐에 상태를 저장하는 대신 SYN+ACK의 시퀀스 번호에 인코딩한 값으로 검증해 자원을 거의 안 쓰고도 정상성을 확인합니다.
왜 초기 시퀀스 번호(ISN)는 무작위로 정하나요?
예측 가능한 ISN은 공격자가 연결을 가로채거나 위조하는 데 악용될 수 있고, 또 같은 (IP, 포트) 쌍으로 다시 연결됐을 때 이전 연결의 늦은 패킷이 새 연결에 섞이는 것을 막기 위해서입니다. 그래서 OS는 시간 기반의 무작위성을 ISN에 더합니다.
