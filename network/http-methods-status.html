HTTP 메서드와 상태 코드

HTTP 메서드는 클라이언트가 서버에 어떤 동작을 원하는지 표현하는 동사이고, 상태 코드는 서버가 그 결과를 한 줄로 답하는 표준 응답입니다. 이 두 가지가 정확하게 약속되어 있기에 브라우저·프록시·캐시·CDN이 같은 규칙으로 동작할 수 있습니다.

구체적인 개념 설명

HTTP 요청은 항상 메서드 + 경로 + 헤더 + (옵션) 본문으로 시작하고, 응답은 항상 상태 코드 + 헤더 + 본문으로 시작합니다. GET /users/42 HTTP/1.1의 GET이 메서드, HTTP/1.1 200 OK의 200이 상태 코드입니다. 메서드와 상태 코드는 단순한 라벨이 아니라 중간 장치(프록시·캐시·LB·브라우저)에게 동작을 지시하는 신호입니다.

GET이라고 했는데 서버가 데이터를 변경하면, 프록시는 그 응답을 캐시하고 봇은 그 URL을 그대로 다시 긁어 사고를 만듭니다. 404로 응답해야 할 것을 200으로 응답하면, 모니터링은 정상으로 보고 사용자만 깨진 화면을 봅니다. 메서드와 상태 코드를 정확하게 쓰는 일은 사실 인프라 전체와의 약속을 지키는 일입니다.

두 핵심 속성 — 안전성(safe)과 멱등성(idempotent)

  • 안전(safe) — 서버 상태를 바꾸지 않는 메서드. GET/HEAD/OPTIONS.
  • 멱등(idempotent) — 같은 요청을 여러 번 보내도 결과가 같은 메서드. GET/HEAD/PUT/DELETE/OPTIONS.
  • POST는 둘 다 아님 — 두 번 보내면 두 번 생성될 수 있고, 서버 상태도 바뀐다.

왜 필요한가?

  • 중간 장치의 동작 결정 — 캐시는 GET/HEAD만 캐시하고, 프록시는 안전 메서드를 자동 재시도합니다. 메서드 의미를 지키지 않으면 이 동작이 깨집니다.
  • 재시도 안전성 — 네트워크가 일시적으로 끊겼을 때, 멱등 메서드는 다시 보낼 수 있습니다. 비멱등이라면 사용자에게 확인을 받아야 합니다.
  • 운영 가시성 — 상태 코드가 정확하면 4xx/5xx만 봐도 클라이언트 잘못인지 서버 잘못인지 즉시 분간됩니다. 200으로 모든 걸 답하면 모니터링이 무용지물이 됩니다.
  • API 설계의 기반 — REST·OpenAPI는 메서드/상태 코드 의미를 전제로 동작합니다.

2) 안전성·멱등성 — 같은 요청을 다시 보내도 되는가

DELETE/PUT은 여러 번 보내도 최종 상태가 같다. POST는 그 자체로는 안전한 재시도가 불가능하다.

3) 상태 코드 5분류

상태 코드의 백의 자리는 "성공/실패의 큰 분류", 뒤 두 자리는 "그 안의 구체적 사유".

5) 자주 헷갈리는 페어

  • 301 vs 302 — 301은 영구(검색엔진/브라우저가 URL을 갱신·캐시), 302는 임시. 잘못 301을 쓰면 옛 URL로 다시는 안 돌아옵니다.
  • 401 vs 403 — 401은 "누구인지 모름(인증 필요)", 403은 "누군지는 알지만 권한 없음". 둘을 섞어 쓰면 클라이언트가 로그인 페이지를 잘못 띄웁니다.
  • 404 vs 410 — 404는 "지금 없음(앞으로 생길 수도)", 410은 "영구 사라짐". SEO에서 의미가 다릅니다.
  • 502 vs 504 — 502는 업스트림이 깨진 응답을 줌(예: 빈 응답·잘못된 헤더), 504는 응답 자체를 시간 안에 못 받음(타임아웃).
  • 422 vs 400 — 400은 형식이 잘못됨(JSON 깨짐), 422는 형식은 맞지만 의미가 안 맞음(이메일 형식 위반). 더 정밀한 메시지를 줄 수 있습니다.

6) RESTful 설계에서의 메서드·상태 코드 매핑

GET    /users          → 200 OK + 목록
POST   /users          → 201 Created + Location: /users/42
GET    /users/42       → 200 OK + 단일 리소스
PUT    /users/42       → 200 OK 또는 204 No Content (전체 교체)
PATCH  /users/42       → 200 OK 또는 204 (부분 수정)
DELETE /users/42       → 204 No Content
GET    /users/9999     → 404 Not Found
POST   /users (중복)   → 409 Conflict
PATCH  /users/42 (검증 실패) → 422 Unprocessable Entity

trade-off

설계 결정장점비용
표준 코드를 정확히 사용모니터링·캐시·도구가 자동으로 잘 동작코드 의미를 팀이 학습해야 함
모든 응답을 200으로 (래핑된 에러 객체)프런트 처리가 균일HTTP 레벨 모니터링·재시도 로직이 무용
POST 위주 RPC식 API도구·CSRF 정책이 단순캐시·재시도·LB가 못 도와줌, 의미 표현 약함
PATCH로 부분 수정네트워크·DB 비용 절감JSON Patch·Merge Patch 등 표준 선택 필요
존재 노출 회피로 404 통일리소스 존재 정보 누설 방지디버깅 어려움 — 권한 문제도 404처럼 보임

실무에서 어디에 쓰이는가?

  • 모니터링·알람 — 5xx 비율, 4xx 비율을 SLI로 잡아 SLO/알람의 기반으로 씁니다.
  • CDN·캐시 — GET/HEAD만 캐시되고, Cache-Control·304 협상이 동작합니다.
  • 리트라이 정책 — 멱등 메서드(GET/PUT/DELETE)는 자동 재시도, POST는 Idempotency-Key로 보호.
  • 로드밸런서/프록시 — 502/503/504는 백엔드 헬스체크와 직결됩니다.
  • OpenAPI 명세 — 메서드·상태 코드별 예시·스키마를 적어 클라이언트 SDK를 자동 생성합니다.
  • SEO — 301/302/410/404 차이가 검색 색인에 직접 영향을 줍니다.

면접에 나올 질문과 답변

PUT과 PATCH의 차이는 무엇인가요?
PUT은 리소스를 요청 본문의 내용으로 통째로 교체합니다. 일부 필드를 빼고 보내면 그 필드는 빈 값이 되거나 사라질 수 있습니다. PATCH는 부분 수정으로, 보낸 필드만 바뀌고 나머지는 그대로 유지됩니다. PUT은 멱등이 보장되지만, PATCH는 적용 방식(JSON Patch / JSON Merge Patch)에 따라 멱등이 아닐 수도 있습니다.
멱등성과 안전성은 무엇이 다른가요?
안전성은 "서버 상태를 바꾸지 않는가"이고, 멱등성은 "같은 요청을 여러 번 보내도 최종 상태가 같은가"입니다. GET은 둘 다 만족하고, PUT/DELETE는 멱등이지만 안전하진 않습니다(서버 상태를 바꿈). POST는 둘 다 만족하지 않아 두 번 보내면 두 개가 생성될 수 있고, 그래서 결제·주문은 Idempotency-Key 같은 보호장치를 별도로 둡니다.
401과 403의 차이는 무엇인가요?
401 Unauthorized는 인증 자체가 안 된 상태로, "누구인지 모르겠으니 로그인하라"는 의미입니다. 403 Forbidden은 인증은 됐지만 그 사용자에게 해당 자원에 대한 권한이 없을 때 사용합니다. 클라이언트가 둘을 구분해야 로그인 페이지로 보낼지, "권한 없음" 메시지를 보일지 결정할 수 있습니다.
301과 302의 차이는 무엇이고, 잘못 쓰면 어떤 문제가 생기나요?
301은 영구 이동으로, 브라우저와 검색엔진이 옛 URL을 새 URL로 갱신하고 응답을 적극 캐시합니다. 302는 임시 이동으로, 옛 URL이 여전히 유효하다는 가정 아래 다음에 다시 옛 URL을 시도합니다. 실수로 임시 상황에 301을 쓰면 옛 URL이 영구 캐시되어 되돌릴 수 없게 되고, SEO에도 영향을 줍니다.
502와 504의 차이는 무엇인가요?
502 Bad Gateway는 리버스 프록시·로드밸런서가 업스트림(백엔드)에서 받은 응답이 잘못됐을 때(빈 응답, 깨진 헤더 등) 발생합니다. 504 Gateway Timeout은 업스트림에 연결은 됐지만 응답이 정해진 시간 안에 오지 않을 때 발생합니다. 둘 다 백엔드 쪽 문제지만 502는 "응답이 망가진" 사고, 504는 "응답이 늦은" 사고로 디버깅 방향이 다릅니다.
POST를 안전하게 재시도하려면 어떻게 해야 하나요?
POST는 본질적으로 멱등이 아니라 단순 재시도하면 같은 결과가 두 번 만들어질 수 있습니다. 실무에선 클라이언트가 요청마다 고유한 Idempotency-Key 헤더를 붙이고, 서버는 같은 키의 요청이 들어오면 처음 결과를 그대로 반환하도록 구현합니다. Stripe 같은 결제 API가 이 방식을 표준으로 채택해, 네트워크 재시도와 중복 결제를 동시에 막습니다.
