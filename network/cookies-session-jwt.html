쿠키 / 세션 / JWT

쿠키는 브라우저가 서버별로 저장하고 매 요청에 자동으로 붙이는 작은 데이터이고, 세션은 서버가 상태를 저장하고 쿠키에는 세션 ID만 담는 방식입니다. JWT(JSON Web Token)는 서명된 토큰 안에 사용자 정보와 만료 시간을 담아 서버가 저장소 조회 없이 검증할 수 있게 하는 방식입니다.

구체적인 개념 설명

HTTP는 무상태(stateless)입니다. 서버는 방금 전 요청을 보낸 사용자가 누구였는지 기본적으로 기억하지 않습니다. 그런데 실제 서비스에는 로그인, 장바구니, 권한 확인처럼 상태가 필요합니다. 그래서 브라우저와 서버 사이에 "너는 방금 로그인한 사용자야"를 표현하는 장치가 필요합니다.

가장 기본 도구가 쿠키입니다. 서버가 응답에 Set-Cookie를 보내면 브라우저가 저장하고, 이후 같은 조건에 맞는 요청마다 Cookie 헤더로 자동 전송합니다. 세션은 이 쿠키에 실제 사용자 정보를 넣지 않고, 무작위 세션 ID만 넣은 뒤 서버 저장소에서 상태를 찾는 방식입니다. JWT는 반대로 상태 일부를 토큰 자체에 담고 서명으로 위조를 막습니다.

쿠키는 저장 위치, 세션/JWT는 인증 설계 방식

쿠키 vs JWT처럼 비교하면 조금 헷갈립니다. JWT도 쿠키에 저장할 수 있고, 세션 ID도 쿠키에 저장합니다. 정확히는 "브라우저가 무엇을 저장하느냐"와 "서버가 상태를 어디에 두느냐"를 나눠 봐야 합니다.

왜 필요한가?

로그인 후 모든 요청에 아이디와 비밀번호를 다시 보내는 건 위험하고 불편합니다. 대신 로그인 성공 시 서버가 "이 사용자는 인증됐다"는 증표를 발급하고, 이후 요청은 그 증표만 보내게 합니다. 이 증표가 세션 ID일 수도 있고 JWT일 수도 있습니다.

좋은 인증 방식은 보안과 운영 편의의 균형입니다. 세션은 서버가 즉시 무효화할 수 있어 안전하지만 저장소가 필요합니다. JWT는 여러 서버가 쉽게 검증할 수 있어 확장성이 좋지만, 발급된 토큰을 중간에 회수하기 어렵습니다.

어떻게 동작하는가?

1) 쿠키의 기본 흐름

쿠키는 브라우저가 조건에 맞는 요청에 자동 첨부한다. JS가 직접 붙이는 Authorization 헤더와 다르다.

2) 쿠키 속성

속성의미보안 영향
Domain쿠키를 보낼 도메인 범위너무 넓게 잡으면 서브도메인 탈취 영향 증가
Path쿠키를 보낼 경로 범위보안 경계로 믿으면 안 됨
Expires / Max-Age만료 시각 / 상대 시간짧을수록 탈취 피해 감소
HttpOnlyJavaScript에서 읽기 금지XSS로 토큰 탈취 방지에 중요
SecureHTTPS에서만 전송평문 네트워크 탈취 방지
SameSite교차 사이트 요청에 쿠키 전송 제한CSRF 방어 핵심

3) 세션 방식

세션 방식은 서버 저장소에 사용자 상태를 두고, 브라우저에는 랜덤한 세션 ID만 둡니다. 요청이 오면 서버가 세션 ID로 Redis/DB/메모리에서 사용자를 찾습니다.

4) JWT 방식

JWT는 세 부분으로 이루어집니다: Header.Payload.Signature. Header는 알고리즘, Payload는 claim(사용자 ID, 권한, 만료 등), Signature는 위조 방지 서명입니다.

5) Access Token + Refresh Token

JWT는 한 번 발급되면 만료 전까지 회수하기 어렵습니다. 그래서 실무에서는 짧은 수명의 Access Token과 긴 수명의 Refresh Token을 나눕니다.

  • Access Token: API 호출에 사용, 수명 5~30분.
  • Refresh Token: 새 Access Token 발급용, 수명 며칠~몇 주. 서버 DB에 저장해 회수 가능하게 운영.
  • Refresh Token Rotation: 갱신할 때마다 refresh token도 새로 발급하고 이전 것은 폐기해 탈취 재사용을 감지.

6) Java 예시

// Spring에서 안전한 세션 쿠키 설정 예시
ResponseCookie cookie = ResponseCookie.from("sid", sessionId)
    .httpOnly(true)
    .secure(true)
    .sameSite("Lax")
    .path("/")
    .maxAge(Duration.ofHours(2))
    .build();
response.addHeader(HttpHeaders.SET_COOKIE, cookie.toString());

// JWT 검증의 핵심 흐름(개념 코드)
String token = authorizationHeader.substring("Bearer ".length());
Jws<Claims> parsed = Jwts.parserBuilder()
    .setSigningKey(publicKeyOrSecret)
    .build()
    .parseClaimsJws(token);       // 서명 검증 + exp 검증
String userId = parsed.getBody().getSubject();

시간/공간 복잡도 또는 trade-off

방식장점단점적합한 경우

JWT서버 저장소 조회 없이 검증, 서비스 간 전달 쉬움회수 어려움, 토큰 커짐, payload 노출분산 API, SSO, 짧은 토큰

localStorage 저장CSRF 영향 적음, JS 제어 쉬움XSS 시 탈취 쉬움권장 낮음, 네이티브 앱과 구분 필요

실무에서 어디에 쓰이는가?

  • 전통 서버 렌더링 웹 — 세션 + HttpOnly/Secure/SameSite 쿠키.
  • SPA + API — 쿠키 세션 또는 JWT. CORS credentials와 SameSite 설정을 함께 고려.
  • 모바일 앱 — Authorization Bearer 토큰 저장, OS 보안 저장소 사용.
  • SSO — OAuth 2.0/OIDC에서 ID Token(JWT)과 Access Token을 사용.
  • 마이크로서비스 — 게이트웨이에서 토큰 검증 후 내부 claim 전달.

면접에 나올 질문과 답변

세션과 JWT의 핵심 차이는?
세션은 상태를 서버 저장소에 두고 클라이언트에는 ID만 둡니다. JWT는 토큰 자체에 claim을 담고 서명으로 위조를 막습니다. 세션은 회수가 쉽고, JWT는 분산 검증이 쉽습니다.
JWT payload에 비밀번호나 개인정보를 넣어도 되나요?
안 됩니다. JWT의 payload는 Base64Url 인코딩일 뿐 암호화가 아닙니다. 누구나 디코딩해서 볼 수 있으므로 민감정보는 넣지 말고 식별자와 최소 권한 정보만 넣어야 합니다.
HttpOnly와 Secure의 의미는?
HttpOnly는 JavaScript에서 쿠키를 읽지 못하게 해 XSS로 토큰을 훔치는 것을 어렵게 합니다. Secure는 HTTPS 연결에서만 쿠키를 보내 평문 네트워크 탈취를 막습니다.
SameSite는 무엇을 막나요?
교차 사이트 요청에 쿠키를 자동으로 보낼지 제한해 CSRF를 줄입니다. Lax는 일반 링크 이동에는 보내지만 위험한 cross-site POST에는 제한하고, Strict는 더 엄격하며, None은 Secure와 함께 써야 합니다.
JWT를 로그아웃시키기 어려운 이유는?
서버가 상태를 저장하지 않고 서명만 검증하므로, 이미 발급된 토큰은 만료 전까지 유효합니다. 해결하려면 짧은 access token, refresh token rotation, blacklist 또는 token version을 사용해야 합니다.
쿠키 기반 인증은 왜 CSRF에 취약한가요?
브라우저가 해당 사이트 쿠키를 자동으로 붙이기 때문입니다. 악성 사이트가 폼 제출이나 이미지 요청을 유도하면 사용자의 쿠키가 함께 전송될 수 있습니다. SameSite와 CSRF token으로 방어합니다.
