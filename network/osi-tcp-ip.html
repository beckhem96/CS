OSI 7계층 / TCP/IP 4계층

OSI 7계층은 네트워크 통신을 7개의 책임으로 나눈 이론적 참조 모델이고, TCP/IP 4계층은 실제 인터넷이 사용하는 구현 모델입니다. 각 계층은 자기 책임만 다하고 위·아래 계층의 내부 사정은 모르기 때문에, 한 계층의 기술을 바꿔도 다른 계층이 영향받지 않습니다.

구체적인 개념 설명

네트워크 통신을 한 덩어리로 짜면 너무 복잡합니다. 케이블 신호 처리, 경로 찾기, 신뢰성 보장, 데이터 표현, 화면 표시 같은 성격이 다른 일들을 한꺼번에 해야 하니까요. 그래서 책임을 분리해 층층이 쌓는 방식으로 설계합니다. 이게 계층 모델입니다.

층을 나누면 좋은 점은 한 층만 바꿔도 된다는 것입니다. 예를 들어 케이블이 광섬유로 바뀌어도(Layer 1) 그 위에서 동작하는 TCP·HTTP는 그대로입니다. 반대로 HTTP를 HTTP/2로 올려도 IP·이더넷은 변하지 않습니다.

OSI 모델은 1984년에 표준화된 참조용 그림이고, 우리가 실제로 쓰는 인터넷은 그것보다 일찍 나온 TCP/IP 모델입니다. 그래서 둘은 비슷하지만 일치하지 않습니다 — OSI는 7계층, TCP/IP는 4계층(또는 5계층으로 세분)입니다.

왜 굳이 둘 다 외우나?

실무에선 TCP/IP만으로도 충분하지만, 면접에선 "L4 스위치", "L7 로드밸런서" 같은 표현이 OSI 번호를 그대로 쓰기 때문에 7계층의 이름과 위치를 알면 의사소통이 편합니다. L4 = 전송, L7 = 응용이라고만 외워둬도 절반은 끝납니다.

왜 필요한가?

계층 모델은 단순한 분류가 아니라 설계 원칙입니다. 각 계층은 위 계층에 일정한 서비스를 제공하고, 아래 계층은 블랙박스로 취급합니다. 덕분에:

  • 개별 진화가 가능합니다. IPv4 → IPv6, HTTP/1.1 → HTTP/2 → HTTP/3 모두 다른 계층에 영향이 적었습니다.
  • 장비 분업이 가능합니다. 라우터(L3), 스위치(L2), 로드밸런서(L4/L7), 방화벽 등 각자 자기 계층만 봅니다.
  • 문제 격리가 쉽습니다. 인터넷이 안 된다 → ping(L3) 되나? → DNS(L7) 되나? → HTTP(L7) 응답이 오나?

어떻게 동작하는가?

1) 두 모델의 매핑

OSI는 책임을 7개로 잘게 나눴고, TCP/IP는 응용 3개와 하위 2개를 묶어 4개로 단순화했다.

2) 각 계층의 책임

계층주요 단위주소대표적인 일
응용(L7)메시지URL, 도메인HTTP 요청, 이메일, DNS 질의
표현(L6)데이터—UTF-8 인코딩, TLS 암호화, 압축
세션(L5)——대화 시작·종료, 동기화 지점
전송(L4)세그먼트(TCP) / 데이터그램(UDP)포트 번호신뢰성, 흐름 제어, 혼잡 제어
네트워크(L3)패킷IP 주소라우팅, 단편화
데이터링크(L2)프레임MAC 주소같은 링크 내 전달, 오류 검출
물리(L1)비트—전기/빛 신호로 변환해 매체에 흘림

3) 캡슐화와 역캡슐화 — 데이터가 옷을 갈아입는 과정

위에서 내려갈 때마다 각 계층은 자기 헤더를 앞에 붙입니다(캡슐화). 받는 쪽은 반대로 한 겹씩 헤더를 벗기며 위로 올립니다(역캡슐화). 이 양파 같은 구조 덕분에 각 계층은 자기 헤더만 보고 자기 일에 집중할 수 있습니다.

각 계층은 자기 헤더만 붙이고 위 계층 내용은 페이로드로 본다.

4) 실생활 예: 웹 페이지 요청

  1. L7 응용 — 브라우저가 GET /index.html 메시지 작성
  2. L6 표현 — TLS로 암호화(HTTPS면)
  3. L4 전송 — TCP가 출발/도착 포트 붙이고(예: 서버 443) 신뢰성 보장
  4. L3 네트워크 — IP가 출발/목적지 IP 붙이고 경로 결정
  5. L2 데이터링크 — 다음 라우터 MAC 주소 붙이기
  6. L1 물리 — 비트를 전기 신호로 케이블에 흘려보냄

시간/공간 복잡도 또는 trade-off

  • 오버헤드 — 각 계층 헤더가 더해져 페이로드 비율이 줄어듭니다. 1500바이트 이더넷 프레임에서 헤더 합이 보통 약 50~80바이트.
  • 지연(latency) — 캡슐화/역캡슐화·복사·체크섬 계산이 모두 비용입니다. 그래서 고성능 NIC는 일부 작업을 하드웨어로 오프로드합니다.
  • 유연성 vs 단순함 — OSI는 책임이 명확하지만 실제 구현은 어색하고, TCP/IP는 단순해 빠르게 보급됐지만 응용 안에 표현·세션이 섞여 있습니다.

실무에서 어디에 쓰이는가?

  • L4 vs L7 로드밸런서 — L4는 IP/포트만 보고 빠르게 분배, L7은 HTTP 헤더·쿠키·URL을 보고 라우팅합니다. AWS의 NLB/ALB가 정확히 이 차이.
  • L7 방화벽 / WAF — HTTP 메시지를 해석해 SQL 인젝션 같은 공격을 막습니다.
  • 네트워크 디버깅 — tcpdump, wireshark로 각 계층 헤더를 펼쳐 보며 문제 위치를 좁힙니다.
  • 장애 진단 — "ping은 되는데 HTTP만 안 된다"면 L3는 되고 L7이 문제, 같은 식으로 계층 단위로 잘라서 봅니다.

면접에 나올 질문과 답변

OSI 7계층을 위에서부터 말해보세요.
응용 - 표현 - 세션 - 전송 - 네트워크 - 데이터링크 - 물리입니다. 각 계층은 자기 책임만 다하고 인접 계층에만 의존하므로, 한 계층의 기술을 바꿔도 다른 계층이 영향받지 않습니다.
OSI 모델과 TCP/IP 모델의 차이는 무엇인가요?
OSI는 7계층의 이론적 참조 모델이고, TCP/IP는 응용 3계층(5+6+7)을 하나로 합치고 하위 2계층(1+2)을 묶어 4계층으로 단순화한 실제 구현 모델입니다. 실제 인터넷은 TCP/IP 모델로 동작합니다.
L4 스위치와 L7 로드밸런서는 어떻게 다른가요?
L4는 IP와 포트만 보고 빠르게 트래픽을 분산합니다. 콘텐츠를 보지 않으니 가볍습니다. L7은 HTTP 메서드·헤더·쿠키·URL 같은 응용 계층 정보를 해석해 더 똑똑하게 라우팅합니다(예: /api/*는 A 그룹, /img/*는 B 그룹). 대신 비용이 큽니다.
캡슐화(encapsulation)란 무엇이고 왜 하나요?
각 계층이 위에서 받은 데이터에 자기 헤더를 붙여 아래 계층으로 전달하는 과정입니다. 덕분에 각 계층은 자기 헤더만 다루면 되고, 한 계층의 변경이 다른 계층에 영향을 주지 않아 모듈성과 진화 가능성이 보장됩니다.
"인터넷이 안 된다"는 사용자 신고를 받았다면 어떻게 좁혀 가나요?
계층별로 잘라 봅니다. 케이블/Wi-Fi 신호(L1·L2) → ip addr로 IP 받았는지(L3) → ping 8.8.8.8(L3 도달성) → nslookup으로 DNS(L7) → curl로 HTTP(L7). 어디서 끊기는지 찾으면 원인 계층이 좁혀집니다.
