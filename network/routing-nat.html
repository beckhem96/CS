라우팅(Routing) / NAT

라우팅은 목적지 IP 주소를 보고 패킷을 다음 어디로 보낼지 결정하는 과정입니다. NAT(Network Address Translation)는 사설 IP와 공인 IP 사이에서 주소와 포트를 바꿔 여러 내부 장치가 하나의 공인 IP를 공유하게 하는 기술입니다.

구체적인 개념 설명

인터넷은 하나의 거대한 도로망입니다. 내 컴퓨터가 어떤 서버로 패킷을 보내면, 그 패킷은 여러 라우터를 지나 목적지까지 갑니다. 각 라우터는 전체 경로를 다 아는 게 아니라, 자기 라우팅 테이블에서 가장 구체적으로 맞는 경로(Longest Prefix Match)를 찾아 다음 홉(next hop)으로 넘깁니다.

NAT는 주소 부족 때문에 널리 쓰이게 됐습니다. 집 안의 노트북, 휴대폰, TV는 모두 사설 IP를 쓰지만 인터넷에는 공유기 하나의 공인 IP로 나갑니다. 공유기는 내부 주소와 포트를 외부 주소와 포트로 바꾸고, 되돌아오는 응답을 다시 원래 내부 장치로 돌려보냅니다.

라우팅은 "길 찾기", NAT는 "주소 바꾸기"

라우팅은 패킷을 어느 방향으로 보낼지 결정하는 일이고, NAT는 패킷 안의 출발지/목적지 주소·포트를 수정하는 일입니다. 공유기는 보통 둘을 동시에 합니다.

왜 필요한가?

라우팅이 없으면 인터넷은 동작할 수 없습니다. 모든 장치가 서로 직접 연결되어 있지 않기 때문에, 중간 라우터들이 목적지 네트워크까지 패킷을 이어 전달해야 합니다. CIDR 덕분에 라우터는 개별 IP가 아니라 네트워크 묶음 단위로 경로를 관리합니다.

NAT는 IPv4 주소 부족을 완화했고, 내부망을 외부에서 직접 보이지 않게 만드는 부수적 보호 효과도 있습니다. 하지만 NAT는 end-to-end 원칙을 깨기 때문에 P2P, VoIP, 게임, WebRTC 같은 양방향 통신을 어렵게 만듭니다.

어떻게 동작하는가?

1) 라우팅 테이블과 Longest Prefix Match

라우터는 목적지 IP와 라우팅 테이블의 prefix를 비교합니다. 여러 경로가 매칭되면 가장 긴 prefix, 즉 가장 구체적인 경로를 선택합니다.

DestinationNext HopInterface
0.0.0.0/0203.0.113.1eth0
10.0.0.0/810.0.0.1eth1
10.1.0.0/1610.1.0.1eth2
10.1.2.0/2410.1.2.1eth3

목적지가 10.1.2.42라면 네 경로 중 10.1.2.0/24가 가장 길게 매칭되므로 eth3로 나갑니다.

3) 정적 라우팅과 동적 라우팅

  • 정적 라우팅: 운영자가 직접 경로를 설정. 단순하고 예측 가능하지만 장애 대응이 수동.
  • 동적 라우팅: 라우터끼리 프로토콜로 경로 정보를 교환. 장애 시 자동 우회 가능.
프로토콜범위방식특징
RIP내부Distance Vector단순, hop count 기준, 현재는 거의 안 씀
OSPF내부Link State조직 내부망에서 빠른 수렴
BGP인터넷Path VectorAS 간 라우팅, 정책 기반

4) NAT의 기본 — SNAT/PAT

가정 공유기의 일반 NAT는 대부분 PAT(Port Address Translation)입니다. 여러 내부 장치가 하나의 공인 IP를 공유하므로, 포트까지 바꿔 매핑 테이블을 만듭니다.

외부에서 보면 둘 다 203.0.113.5지만, 포트가 달라 NAT가 내부 장치를 구분한다.

5) NAT 종류

종류방향의미예시
SNAT나가는 트래픽출발지 주소 변경사설 IP → 공인 IP
DNAT들어오는 트래픽목적지 주소 변경공인 IP:80 → 내부 서버:8080
PAT/NAPT양방향주소 + 포트 변경가정 공유기
1:1 NAT고정 매핑공인 IP 하나 ↔ 사설 IP 하나서버 공개

6) 포트포워딩과 헤어핀 NAT

포트포워딩은 외부에서 들어온 특정 포트를 내부 서버로 보내는 DNAT 규칙입니다. 예를 들어 공유기의 203.0.113.5:443을 내부 192.168.0.20:8443으로 전달합니다.

헤어핀 NAT은 내부 클라이언트가 공유기의 공인 IP로 내부 서버에 접속할 때, 공유기가 다시 내부로 돌려보내는 기능입니다. 지원하지 않으면 내부에서는 도메인 접속이 안 되고, 외부에서는 되는 이상한 현상이 생깁니다.

7) NAT traversal — STUN / TURN / ICE

NAT는 외부에서 내부로 먼저 들어오는 연결을 어렵게 만듭니다. WebRTC, VoIP, P2P 게임은 이를 우회해야 합니다.

  • STUN: "내가 외부에서 어떤 IP:포트로 보이니?"를 알려주는 서버.
  • TURN: 직접 연결이 실패하면 중계 서버가 트래픽을 대신 전달.
  • ICE: 여러 후보 경로(host, srflx, relay)를 모아 가장 좋은 연결을 선택하는 절차.

STUN은 싸고 빠르지만 NAT 종류에 따라 실패할 수 있고, TURN은 거의 항상 되지만 서버 비용과 지연이 큽니다.

8) 리눅스 명령어

# 라우팅 테이블 확인
ip route
# default via 192.168.0.1 dev wlan0
# 10.0.0.0/8 via 10.0.0.1 dev eth1

# 특정 목적지로 어떤 경로를 쓰는지
ip route get 8.8.8.8

# 경로 추적
traceroute example.com
mtr example.com

# NAT 규칙 확인(nftables 권장, iptables도 흔함)
nft list ruleset
iptables -t nat -L -n -v

시간/공간 복잡도 또는 trade-off

  • 라우팅 테이블 집약은 메모리와 조회 비용을 줄이지만, 너무 큰 prefix는 세밀한 제어가 어렵습니다.
  • 동적 라우팅은 장애 복구가 빠르지만 설정 복잡도와 잘못된 광고 위험이 있습니다.
  • NAT는 IPv4 주소를 절약하지만, 연결 상태 테이블을 유지해야 하므로 장비 자원을 씁니다.
  • NAT traversal은 P2P 연결 성공률을 높이지만 TURN 중계는 비용·지연이 큽니다.

실무에서 어디에 쓰이는가?

  • 클라우드 VPC — public subnet은 인터넷 게이트웨이, private subnet은 NAT Gateway를 통해 외부로 나감.
  • 사내망 — OSPF/BGP로 내부 라우팅, 방화벽에서 NAT 정책.
  • 쿠버네티스 — Service NAT, kube-proxy, CNI 라우팅.
  • WebRTC — STUN/TURN/ICE가 NAT 뒤 사용자끼리 연결을 만듦.
  • 장애 진단 — traceroute, mtr, ip route get으로 어디서 막히는지 확인.

면접에 나올 질문과 답변

라우팅에서 Longest Prefix Match란?
여러 경로가 목적지 IP와 매칭될 때 prefix 길이가 가장 긴, 즉 가장 구체적인 경로를 선택하는 규칙입니다. 10.1.2.42는 10.0.0.0/8보다 10.1.2.0/24가 더 구체적이므로 /24 경로를 씁니다.
기본 게이트웨이는 무엇인가요?
로컬 라우팅 테이블에 더 구체적인 경로가 없을 때 보내는 다음 홉입니다. 보통 0.0.0.0/0 경로로 표현하며, 집에서는 공유기가 기본 게이트웨이입니다.
SNAT와 DNAT의 차이는?
SNAT는 나가는 패킷의 출발지 주소를 바꿉니다. 사설 IP가 인터넷에 나갈 때 공인 IP로 바꾸는 게 대표적입니다. DNAT는 들어오는 패킷의 목적지 주소를 바꿉니다. 포트포워딩이 대표 예입니다.
NAT가 P2P를 어렵게 만드는 이유는?
NAT 장비는 내부에서 먼저 만든 매핑이 있어야 외부 응답을 돌려보낼 수 있습니다. 외부 피어가 먼저 내부 장치로 연결하려 하면 매핑이 없어 차단됩니다. 그래서 STUN/TURN/ICE 같은 NAT traversal이 필요합니다.
traceroute는 어떻게 경로를 알아내나요?
TTL을 1부터 증가시키며 패킷을 보냅니다. 각 라우터는 TTL이 0이 되면 ICMP Time Exceeded를 돌려주고, traceroute는 그 응답의 출발지 IP를 기록해 경로를 추정합니다.
NAT는 보안 장비인가요?
NAT는 원래 주소 변환 기술이지 방화벽이 아닙니다. 내부 주소를 숨기고 외부에서 직접 들어오는 연결을 어렵게 만드는 부수 효과가 있지만, 접근 제어는 방화벽 정책으로 별도 관리해야 합니다.
