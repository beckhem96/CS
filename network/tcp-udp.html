TCP vs UDP

TCP(Transmission Control Protocol)는 연결을 설정한 뒤 데이터가 순서대로·빠짐없이 도착하도록 보장하는 신뢰성 있는 전송 프로토콜이고, UDP(User Datagram Protocol)는 연결도 보장도 없이 그냥 보내는 가볍고 빠른 전송 프로토콜입니다. 둘 다 전송 계층(L4)에서 동작하지만 책임의 무게가 정반대입니다.

구체적인 개념 설명

두 사람이 정보를 주고받는 방식에 비유해 보면, TCP는 등기 우편에 가깝습니다. "받았다"는 확인이 와야 안심하고, 빠진 게 있으면 다시 보내고, 순서가 뒤집히면 정렬해 줍니다. UDP는 그냥 던지는 종이비행기에 가깝습니다. 도착했는지 모르고, 도착하더라도 순서가 뒤죽박죽일 수 있고, 잃어버려도 신경 쓰지 않습니다.

신기한 점은 UDP가 나쁜 게 아니라는 것입니다. 등기 우편의 "확인하고 다시 보내는" 절차는 시간을 잡아먹습니다. 음성 통화나 게임에선 0.3초 늦은 정확한 데이터보다 약간 빠진 실시간 데이터가 더 가치 있습니다. 그래서 도구로 골라 쓰는 것입니다.

"신뢰성"의 정확한 뜻

TCP의 신뢰성은 ① 보낸 데이터가 도착함을 보장 ② 도착 순서를 보장 ③ 중복을 제거함, 의 세 가지를 의미합니다. "내용이 변조되지 않았다"의 보장은 TCP 체크섬 정도이고, 진짜 무결성·기밀성은 그 위에 TLS를 올려야 얻을 수 있습니다.

왜 필요한가?

IP 자체는 베스트 에포트(best-effort)입니다 — 패킷이 사라져도, 순서가 뒤집혀도 IP는 신경 쓰지 않습니다. 이걸 그대로 쓰면 파일이 깨지거나 페이지가 사라지는 일이 비일비재합니다. 그래서 IP 위에 전송 계층을 둡니다.

누구는 신뢰성이 절대적으로 필요하고(파일 다운로드, 결제, 웹 페이지), 누구는 속도가 절대적으로 중요합니다(영상 통화, 게임). 두 요구를 한 프로토콜로 만족시키긴 어려워서 TCP와 UDP가 나란히 존재합니다. 위 응용은 자기 성격에 맞는 쪽을 골라 씁니다.

어떻게 동작하는가?

1) 헤더 비교

헤더 크기 차이가 두 프로토콜의 책임 차이를 그대로 보여준다.

2) 통신 흐름 비교

TCP는 연결 수립과 ACK·재전송이 핵심, UDP는 "쏘고 잊기".

4) TCP의 신뢰성 메커니즘 — 한 장 정리

  • Sequence Number — 보낸 바이트마다 번호. 수신측은 이 번호로 순서 정렬·중복 제거.
  • ACK — "n번까지 잘 받았다"는 확인. 일정 시간 안에 ACK가 안 오면 재전송.
  • 흐름 제어(window) — 수신측이 처리 가능한 양만큼만 보내라고 알림.
  • 혼잡 제어(slow start, AIMD) — 네트워크 막힘을 감지하면 전송 속도를 줄임.
  • 체크섬 — 헤더+페이로드 손상 검사(약한 보장).

5) Java 코드 비교

// TCP 서버 — Socket / ServerSocket
try (ServerSocket ss = new ServerSocket(9000)) {
    Socket s = ss.accept();           // 연결 수립까지 대기
    var in  = s.getInputStream();
    var out = s.getOutputStream();
    // 스트림이라 아래 read 한 번에 메시지 경계를 지키지 않음
    byte[] buf = new byte[1024];
    int n = in.read(buf);
    out.write(("got " + n).getBytes());
}

// UDP 서버 — DatagramSocket
try (DatagramSocket ds = new DatagramSocket(9001)) {
    byte[] buf = new byte[1024];
    DatagramPacket pkt = new DatagramPacket(buf, buf.length);
    ds.receive(pkt);                  // 메시지 경계가 그대로 보존됨
    DatagramPacket reply = new DatagramPacket(
        "ok".getBytes(), 2, pkt.getAddress(), pkt.getPort());
    ds.send(reply);                   // 도착 보장 없음
}

코드만 봐도 TCP는 "스트림 + 연결", UDP는 "메시지 + 무연결"의 사고방식 차이가 드러납니다. 특히 TCP는 메시지 경계가 사라진다는 점이 초보자가 가장 자주 놓치는 부분입니다 — 한 번 send해도 두 번에 걸쳐 read될 수 있고 그 반대도 가능합니다.

시간/공간 복잡도 또는 trade-off

  • 지연 vs 신뢰성 — TCP의 안정성은 핸드셰이크·ACK·재전송 비용을 동반합니다. 짧고 자주 일어나는 통신이라면 누적 비용이 큽니다.
  • 처리량 — TCP는 한 연결 안에서 손실 한 번이면 그 흐름의 뒷부분이 모두 막히는 head-of-line blocking이 있습니다. HTTP/3가 UDP 위에서 도는 이유입니다.
  • NAT 친화도 — TCP는 연결 상태가 명확해 NAT 통과가 비교적 쉽고, UDP는 stateless라 P2P에서 NAT traversal에 추가 작업이 필요합니다.

실무에서 어디에 쓰이는가?

  • 웹 — HTTP/1·2는 TCP, HTTP/3는 UDP 위의 QUIC. "더 빠른 웹"의 답이 UDP 위로 이동하는 중입니다.
  • DNS — 기본 UDP(빠름), 응답이 크면 TCP로 폴백.
  • 실시간 — VoIP, 화상회의, 게임은 UDP. 약간의 손실은 무시하고 지연을 최소화.
  • 스트리밍 — HLS/DASH는 TCP 위 HTTP, WebRTC 미디어는 UDP 기반 RTP/SRTP.
  • 관측·디버깅 — ss -tunap으로 현재 TCP/UDP 소켓 확인, tcpdump 'tcp port 80'으로 캡처.

면접에 나올 질문과 답변

TCP와 UDP의 가장 큰 차이는 무엇인가요?
TCP는 연결을 맺고 데이터의 도착·순서·중복 제거를 보장하는 신뢰성 프로토콜이고, UDP는 연결 없이 그냥 던지는 비신뢰 프로토콜입니다. TCP는 안전한 대신 핸드셰이크·ACK 비용이 있고, UDP는 빠른 대신 안전성을 응용이 직접 챙겨야 합니다.
UDP는 신뢰성이 없는데 왜 쓰나요?
실시간성이 더 중요한 통신이 있기 때문입니다. 음성 통화에서 0.3초 지연된 정확한 음성보다 약간 끊긴 실시간 음성이 낫고, DNS 같은 짧은 질의응답은 핸드셰이크 비용을 들이느니 다시 묻는 게 효율적입니다. 필요하면 응용 계층에서 직접 신뢰성을 구현할 수도 있습니다(QUIC).
TCP는 "스트림 기반"이라는 게 무슨 뜻인가요?
TCP는 메시지 단위가 아니라 바이트의 흐름으로 데이터를 다룹니다. send를 두 번 했어도 수신측에서 한 번에 읽힐 수 있고 반대도 가능합니다. 그래서 응용 프로토콜(HTTP, 자체 프로토콜)은 메시지 경계를 표시하는 헤더(길이 필드, 줄바꿈 등)를 직접 정해야 합니다.
TCP의 흐름 제어와 혼잡 제어는 어떻게 다른가요?
흐름 제어는 수신측이 처리할 수 있는 양에 맞추는 것(rwnd, 수신 윈도우)이고, 혼잡 제어는 네트워크 전체가 막히는 것을 막기 위해 송신측이 속도를 조절하는 것(cwnd, 슬로 스타트·AIMD)입니다. 둘 다 송신량을 줄이지만 보고 있는 대상이 다릅니다.
HTTP/3가 UDP 위에서 동작하는 이유는?
TCP는 한 연결 안에서 패킷 손실 하나가 그 흐름 전체를 막는 HOL blocking이 있습니다. HTTP/3는 UDP 위의 QUIC에서 멀티 스트림을 직접 구현해, 한 스트림의 손실이 다른 스트림에 영향을 주지 않게 했습니다. 또 0-RTT 재연결, 연결 마이그레이션 같은 이점도 함께 가집니다.
