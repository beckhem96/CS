이더넷 / MAC / ARP — 같은 LAN 안에서 패킷이 전달되는 방식

이더넷(Ethernet)은 같은 LAN 안에서 전기·광 신호로 데이터를 주고받는 가장 흔한 데이터링크 계층 기술이고, 그 안에서 각 NIC를 식별하는 48비트 주소가 MAC 주소입니다. ARP(Address Resolution Protocol)는 IP 주소만 아는 상태에서 같은 LAN 안의 MAC 주소를 찾아내는 프로토콜로, IP 패킷이 실제 전선 위에서 전달되기 위해 반드시 거치는 단계입니다.

구체적인 개념 설명

"192.168.1.10에 있는 동료 컴퓨터에 핑을 보내라"고 명령하면, 우리는 IP 주소만 떠올립니다. 하지만 실제 LAN 케이블 위로 흐르는 신호는 IP 주소를 모릅니다 — 이더넷은 MAC 주소로만 목적지를 인식합니다. 그래서 IP 패킷을 전송하기 전에 "이 IP를 가진 NIC의 MAC 주소가 무엇인가?"를 알아내야 하는데, 그 일을 ARP가 합니다.

이 흐름을 한 번 알면 핑이 안 갈 때, 같은 서브넷 안에서 통신이 안 될 때, 가상 머신·컨테이너 네트워크가 이상할 때 어디를 봐야 할지가 명확해집니다. 네트워크 디버깅의 가장 깊은 층이 바로 이 L2/ARP 영역입니다.

핵심 한 줄 — IP는 "어디로", MAC은 "지금 누구에게"

IP 주소는 출발지에서 목적지까지의 전체 여정을 알려주고, MAC 주소는 한 LAN 안에서 다음 한 발짝을 알려줍니다. 그래서 IP는 패킷이 라우터를 지나는 동안 그대로 유지되지만, MAC 주소는 라우터를 지날 때마다 바뀝니다.

왜 필요한가?

  • 물리 매체와 논리 주소의 분리 — IP를 NIC에 박아두면 NIC를 교체할 때마다 IP가 바뀌어 운영이 어렵습니다. MAC은 NIC가 갖고 IP는 OS가 부여하는 분리가 자연스럽습니다.
  • 같은 LAN 안의 직접 전달 — 라우터를 거치지 않고 같은 스위치에 연결된 호스트끼리는 직접 프레임을 주고받아야 합니다. 이때 필요한 건 IP가 아니라 MAC입니다.
  • 주소 변환의 자동화 — IP만 알고 있는 응용/OS가 ARP 덕분에 MAC을 자동으로 알아냅니다. 사람이 직접 ARP 테이블을 적어둘 필요가 없습니다.
  • 네트워크 디버깅의 기반 — "IP는 보이는데 통신이 안 된다"는 문제는 거의 항상 ARP/L2 단계에서 발생합니다.

어떻게 동작하는가?

1) 이더넷 프레임 구조

출발지·목적지 MAC, 어떤 상위 프로토콜인지(EtherType), 그리고 페이로드와 검증 코드.

2) MAC 주소 — 48비트의 의미

MAC 주소는 48비트(6바이트)로 보통 aa:bb:cc:dd:ee:ff처럼 표기합니다. 앞 24비트(OUI, Organizationally Unique Identifier)는 IEEE가 제조사에 발급하고, 뒤 24비트는 제조사가 NIC마다 부여합니다. 그래서 MAC만 보고도 어떤 회사 NIC인지 추정 가능합니다(예: 00:50:56 = VMware).

  • 유니캐스트 — 첫 바이트의 LSB가 0. 한 NIC를 정확히 가리킴.
  • 멀티캐스트 — 첫 바이트 LSB가 1. 여러 NIC가 함께 받기.
  • 브로드캐스트 — ff:ff:ff:ff:ff:ff. LAN 안의 모든 NIC가 받기. ARP가 사용.
  • 로컬 / 글로벌 — 두 번째 비트(U/L)가 1이면 로컬 관리(가상 머신·컨테이너가 자주 만드는 MAC).

3) ARP — IP를 MAC으로 바꾸는 4단계

ARP는 LAN 전체에 "누구야?"를 묻는 브로드캐스트로 시작해, 당사자만 답하는 유니캐스트로 끝난다.

4) ARP 캐시 — 자주 묻지 않으려는 장치

매 패킷마다 ARP를 하면 LAN 트래픽이 폭발하므로, OS는 한 번 알아낸 IP↔MAC 매핑을 잠시 캐시합니다(보통 수 초 ~ 수 분). arp -n(리눅스), arp -a(맥/윈도우)로 현재 캐시를 볼 수 있습니다. 

$ arp -n
Address              HWtype  HWaddress           Iface
192.168.1.1          ether   00:11:22:33:44:55   wlan0
192.168.1.10         ether   bb:bb:bb:bb:bb:bb   wlan0

캐시가 만료되거나 호스트가 사라지면 다음 통신 직전 ARP가 다시 발생합니다. Gratuitous ARP는 자기가 새 IP를 갖게 됐을 때 LAN 전체에 미리 알리는 ARP로, 페일오버나 IP 이전 직후 캐시 갱신에 쓰입니다.

5) 같은 LAN vs 다른 LAN — 패킷 전달의 결정적 차이

목적지 IP가 같은 서브넷이면 그 호스트의 MAC으로 직접 보냅니다. 다른 서브넷이면 그 IP의 MAC을 알 수 없고, 대신 "기본 게이트웨이의 MAC"으로 보냅니다. 라우터가 받아서 IP는 그대로 둔 채 자기가 다음 홉에 적합한 MAC으로 다시 감싸 전달합니다 — IP는 끝까지 유지, MAC은 매 홉마다 교체.

목적지 IP가 같은 서브넷인지 아닌지에 따라 ARP 대상이 달라진다 — 호스트의 MAC vs 게이트웨이의 MAC.

6) ARP 보안 — Spoofing과 그 방어

ARP는 응답 진위를 검증하지 않습니다. 누가 "192.168.1.1은 내 MAC이야"라고 답해도 그대로 캐시에 들어갑니다. 이를 악용해 공격자가 자기를 게이트웨이로 위장해 트래픽을 중간에서 가로채는 공격이 ARP Spoofing(또는 ARP Poisoning)입니다. 방어는 보통 다음 조합으로 합니다.

  • 스위치의 Dynamic ARP Inspection(DAI) — DHCP 스누핑 정보로 가짜 ARP를 차단.
  • 정적 ARP 항목 — 게이트웨이 등 핵심 호스트는 수동으로 MAC을 못 박기.
  • 네트워크 분리 — VLAN·private VLAN으로 브로드캐스트 도메인을 작게 유지.
  • L2 모니터링 — arpwatch 같은 도구로 매핑 변화를 감시.

실무에서 어디에 쓰이는가?

  • "같은 서브넷인데 통신이 안 됨" 디버깅 — arp -n으로 매핑이 비어 있거나 잘못됐는지 확인하는 게 1순위입니다.
  • VRRP / Keepalived 페일오버 — 가상 IP가 다른 노드로 옮겨갈 때 Gratuitous ARP를 보내 LAN의 캐시를 갱신합니다.
  • 컨테이너 / VM 네트워크 — 가상 NIC마다 자체 MAC이 부여되고, 브리지·OVS가 ARP를 처리합니다. 가상 MAC 관리가 모든 컨테이너 네트워크의 기반입니다.
  • 네트워크 보안 모니터링 — ARP 변화 감지로 spoofing·rogue 디바이스를 식별합니다.
  • 패킷 분석 — Wireshark에서 EtherType·MAC 헤더가 가장 먼저 보이고, 디버깅의 출발점이 됩니다.
  • 스위치 학습 테이블 — 스위치는 프레임의 출발지 MAC을 보고 어느 포트가 누구인지 학습합니다(MAC learning).

면접에 나올 질문과 답변

IP 주소와 MAC 주소의 차이는 무엇인가요?
IP 주소는 네트워크 계층에서 사용하는 논리 주소로, 인터넷 전체에서 출발지부터 목적지까지의 여정을 식별합니다. MAC 주소는 데이터링크 계층에서 사용하는 물리 주소로, 같은 LAN 안에서 다음 홉을 식별합니다. 그래서 IP는 패킷이 라우터를 지나도 그대로지만, MAC은 한 홉마다 출발지·목적지가 바뀝니다.
ARP는 무엇이고 왜 필요한가요?
ARP는 같은 LAN 안에서 IP 주소만 알고 있을 때 그에 해당하는 MAC 주소를 찾기 위한 프로토콜입니다. 이더넷은 MAC 주소로만 프레임을 전달하므로, IP 패킷을 보내기 전에 반드시 MAC을 알아내야 합니다. 호스트는 LAN에 "이 IP의 MAC이 뭐야?"를 브로드캐스트로 묻고, 해당 IP를 가진 호스트만 자기 MAC을 유니캐스트로 회신합니다.
같은 서브넷과 다른 서브넷의 통신은 ARP 측면에서 어떻게 다른가요?
같은 서브넷이면 그 호스트의 MAC을 ARP로 알아내 직접 프레임을 보냅니다. 다른 서브넷이면 목적지 호스트의 MAC을 알 수 없고, 대신 기본 게이트웨이(라우터)의 MAC을 ARP로 알아낸 뒤 그 게이트웨이에게 프레임을 보냅니다. 라우터가 IP를 보고 다음 홉을 결정해 다시 그쪽 LAN의 MAC으로 감싸 전달합니다 — IP는 끝까지 유지, MAC은 홉마다 바뀝니다.
ARP 캐시는 왜 사용하고, 보안적으로 위험한 점은 무엇인가요?
매 패킷마다 ARP를 하면 LAN에 브로드캐스트가 폭주하므로, 한 번 알아낸 IP↔MAC 매핑을 짧게 캐시합니다. 하지만 ARP는 응답을 검증하지 않아 누구든 "이 IP는 내 MAC이야"라고 답할 수 있고, 이를 악용한 ARP Spoofing으로 트래픽을 가로챌 수 있습니다. 방어로 스위치의 Dynamic ARP Inspection, 정적 ARP, VLAN 분리, arpwatch 같은 모니터링을 함께 사용합니다.
Gratuitous ARP는 무엇이고 언제 쓰이나요?
Gratuitous ARP는 호스트가 누가 묻지 않았는데도 "내가 이 IP를 갖고 있고 내 MAC은 X다"를 LAN에 알리는 ARP입니다. 가상 IP가 다른 노드로 페일오버될 때, IP가 새로 할당될 때, MAC이 바뀐 뒤 LAN의 ARP 캐시를 갱신하기 위해 사용됩니다. VRRP·Keepalived 같은 HA 도구가 페일오버 직후 자동으로 보냅니다.
이더넷 프레임의 EtherType은 어떤 역할인가요?
EtherType은 프레임이 담고 있는 페이로드가 어떤 상위 프로토콜에 속하는지 알려 줍니다. 0x0800은 IPv4, 0x86DD는 IPv6, 0x0806은 ARP를 의미합니다. 수신 측은 EtherType을 보고 페이로드를 IP 스택으로 보낼지 ARP 처리기로 보낼지 결정합니다. 이 한 필드가 같은 케이블 위에서 여러 프로토콜을 공존시키는 토대가 됩니다.
