TCP 혼잡 제어(Congestion Control)

혼잡 제어는 네트워크 자체가 막히는 것을 막기 위해 송신측이 자신의 전송 속도(cwnd)를 조절하는 메커니즘입니다. 네트워크에서 직접 "혼잡해!"라고 알려주는 일은 거의 없으므로, TCP는 패킷 손실·중복 ACK·RTT 증가 같은 간접 신호로 혼잡을 추정합니다. 대표 알고리즘은 슬로 스타트, 혼잡 회피(AIMD), Fast Retransmit/Recovery이고, 변형으로 Tahoe·Reno·NewReno·Cubic·BBR이 있습니다.

구체적인 개념 설명

혼잡 제어는 비유하면 여러 차가 한 도로에 진입할 때의 자율 협상입니다. 각 차가 자기 속도만 챙기다 보면 도로 전체가 정체되고 결국 모두가 손해를 봅니다. 네트워크에선 라우터의 큐가 넘치면 패킷이 버려지고, 모두가 재전송하면 더 막히는 congestion collapse가 일어날 수 있습니다. 1980년대 인터넷이 실제로 이것 때문에 마비된 사건이 있었고, 이후 Van Jacobson의 알고리즘이 표준이 됐습니다.

혼잡 제어가 흐름 제어와 헷갈리기 쉬운데, 둘은 보고 있는 대상이 다릅니다. 흐름 제어는 수신자의 버퍼(rwnd)를 보고, 혼잡 제어는 네트워크 전체의 신호(cwnd)를 봅니다. 실제 송신 가능량은 min(rwnd, cwnd)로 결정됩니다 — 둘 중 빡빡한 쪽이 곧 그 순간의 한계.

핵심 변수 두 개를 외우면 나머지가 쉽습니다:

  • cwnd (Congestion Window) — 송신측이 한 번에 in-flight 상태로 둘 수 있는 데이터 양의 추정치(MSS 단위로 셈).
  • ssthresh (Slow Start Threshold) — 슬로 스타트와 혼잡 회피를 가르는 경계선. 손실이 감지되면 보통 cwnd의 절반으로 떨어진다.

혼잡의 신호는 거의 항상 "손실"

라우터는 큐가 가득 차면 그냥 패킷을 버립니다(Tail Drop). 송신측은 이 손실을 감지해 "네트워크가 막혔구나"라고 추정합니다. 그래서 전통적인 TCP의 혼잡 제어는 loss-based입니다. BBR처럼 RTT 증가와 대역폭 추정을 함께 쓰는 model-based 알고리즘은 비교적 최근에야 등장했습니다.

왜 필요한가?

IP 자체는 best-effort라 혼잡이 생기면 그냥 패킷을 버립니다. TCP는 이 "버려짐"을 신뢰성으로 메우려 재전송하는데, 혼잡 신호 없이 무작정 재전송하면 혼잡이 더 심해지는 악순환이 됩니다. 1986년 인터넷이 실제로 이것 때문에 처리량이 1/1000로 떨어졌습니다.

또 한 가지 이유는 공정성(fairness)입니다. 한 송신자가 회선을 독차지하면 다른 사용자가 굶주리게 됩니다. AIMD(Additive Increase, Multiplicative Decrease)는 여러 흐름이 함께 사용할 때 자연스럽게 비슷한 점유율로 수렴하는 멋진 성질을 가집니다.

어떻게 동작하는가?

1) 핵심 변수와 송신 가능량

송신 가능량은 흐름 제어의 rwnd와 혼잡 제어의 cwnd 중 작은 쪽입니다: 

SendableWindow = min(rwnd, cwnd)
in-flight ≤ SendableWindow

cwnd는 보통 1~10 MSS로 시작해(IW, Initial Window), 네트워크 상황에 따라 늘었다 줄었다 합니다. ssthresh는 혼잡이 감지되면 보통 cwnd/2로 떨어지고, 슬로 스타트는 이 임계값을 넘기 전까지 동작합니다.

2) Slow Start — 지수적으로 살피기

처음에는 네트워크 사정을 모르니 천천히 시작합니다. 단, "천천히"라고 부르지만 실제로는 매 RTT마다 두 배로 늘리는 지수 증가입니다. "느린 출발"이라는 이름은 그 이전의 무모한 즉시-최대 송신과 비교해서 붙은 이름일 뿐입니다.

구체적으로는 ACK를 받을 때마다 cwnd를 1 MSS씩 증가시킵니다. 한 RTT 동안 cwnd만큼의 ACK가 도착하므로 결과적으로 RTT마다 두 배. cwnd가 ssthresh에 닿으면 혼잡 회피로 전환합니다.

3) Congestion Avoidance(AIMD) — 선형으로 늘리고, 곱으로 줄이기

ssthresh 위로 올라오면 송신측은 한 RTT마다 cwnd를 1 MSS씩만 더 늘립니다(Additive Increase). 그러다 손실이 감지되면 cwnd를 절반으로 줄이고 ssthresh도 절반으로 떨어뜨립니다(Multiplicative Decrease). 이 비대칭(천천히 증가, 갑자기 감소)이 네트워크 안정성과 공정성의 비밀입니다.

Tahoe 변형의 전형적 모습 — 손실 시 cwnd를 1로 떨어뜨리고 슬로 스타트부터 다시.

4) Fast Retransmit — 손실을 더 빨리 알아채기

RTO(재전송 타임아웃)를 기다려서 손실을 감지하면 시간이 너무 많이 듭니다(보통 200ms 이상). 그래서 같은 ACK 번호가 3번 연속 오면(3 dup ACK) 손실로 간주하고 즉시 재전송합니다. 수신측이 순서가 안 맞는 패킷을 받았을 때 "내가 다음으로 기대하는 번호"를 ACK로 다시 보내기 때문에 중복 ACK가 손실의 단서가 됩니다.

5) Fast Recovery — 손실 후 재출발 비용 줄이기

Tahoe는 손실 시 cwnd를 1로 떨어뜨리고 슬로 스타트를 다시 합니다 — 안전하지만 회복이 느립니다. Reno는 다음 규칙으로 회복 비용을 줄입니다:

  • 3 dup ACK로 손실 감지 → ssthresh = cwnd/2, cwnd = ssthresh + 3 (3은 dup ACK가 의미하는 in-flight 데이터)
  • 이후 dup ACK가 올 때마다 cwnd += 1 (네트워크에 이미 도달한 패킷이 있으니 윈도우를 잠깐 키워줘도 OK)
  • 새 ACK 도달 → cwnd = ssthresh, 혼잡 회피로 즉시 전환
Reno는 손실 후 cwnd가 1까지 떨어지지 않아 회복이 훨씬 빠르다.

6) NewReno와 SACK

Reno는 한 윈도우 안에 손실이 여러 개면 약합니다. 첫 손실은 빠르게 회복해도 두 번째는 다시 RTO를 기다려야 할 수 있습니다. NewReno는 부분 ACK(partial ACK)를 보고 "아직 더 잃은 게 있구나"를 알아내 같은 회복 라운드 안에서 연속 재전송합니다.

SACK(Selective ACK)는 한 단계 더 나아가, 수신측이 "1~10번 받았고 12~15번 받았다"처럼 받은 구간을 명시합니다. 송신측이 정확히 어떤 패킷이 빠졌는지 알 수 있어 한 RTT 안에 여러 손실을 모두 재전송할 수 있게 됩니다. 현대 TCP에선 거의 표준.

7) 알고리즘 변형 비교

알고리즘등장핵심장점한계

Reno1990Fast Retransmit + Fast Recovery회복 빠름한 윈도우 다중 손실에 약함

현대 리눅스 기본은 Cubic이고, 구글 서버·YouTube 등 일부 환경은 BBR을 씁니다. 윈도우는 Cubic 변형(CTCP, Compound TCP)을 쓰다가 최근 BBR도 추가했습니다. BBR은 분량이 충분히 커서 별도 페이지로 분리할 가치가 있는 주제입니다.

8) ECN — 손실 없이 혼잡 알리기

손실로 혼잡을 추정하는 건 사실 비효율적입니다. 손실이 일어나기 전에 알 수 있으면 좋겠죠. ECN(Explicit Congestion Notification)은 라우터가 패킷을 버리지 않고 IP 헤더의 ECN 비트를 1로 표시해 "곧 막힐 것 같아"를 알립니다. 수신측은 이걸 보고 ACK에 ECE 플래그를 켜서 송신측에 전달, 송신측은 cwnd를 줄입니다. 이론적으로는 우아하지만, 중간 경로의 모든 장비가 ECN을 지원해야 하고 실제 활성화는 제한적입니다.

9) RTO 백오프와 타이머

RTO(Retransmission Timeout)가 두 번 연속 발생하면 RTO 자체를 두 배로 늘립니다(지수 백오프). 네트워크가 실제로 심하게 막힌 상황에서 송신측이 더 자주 시도하지 않도록 막는 안전장치입니다. RTO 계산은 RTT를 평활화한 SRTT와 분산 RTTVAR로 결정합니다(Jacobson 알고리즘): 

SRTT   ← (1 - α) · SRTT   + α · RTT_sample        // α = 1/8
RTTVAR ← (1 - β) · RTTVAR + β · |SRTT - RTT_sample| // β = 1/4
RTO    ← SRTT + max(G, K · RTTVAR)                  // K = 4

혼잡 제어의 본질적 어려움은 "네트워크 상태를 직접 볼 수 없다"는 점입니다. 손실·RTT·ECN 같은 간접 신호로 모델을 만들어야 하니 가정이 깨지면 부작용이 큽니다. 예를 들어 무선 환경의 손실은 혼잡과 상관없이 자주 일어나는데, loss-based 알고리즘은 이걸 혼잡으로 오인해 cwnd를 깎습니다.

실무에서 어디에 쓰이는가?

  • OS 기본값 선택 — 리눅스 서버는 Cubic이 기본. 대용량 트래픽 서비스는 BBR로 바꿔 처리량 향상을 보고하기도 합니다.
  • CDN과 백본 — 구글, Cloudflare 등은 BBR을 적극 도입. 사용자 응답 시간(P95, P99)이 줄어드는 효과가 보고됨.
  • 회선 진단 — 처리량이 안 나오면 cwnd가 작은지(혼잡), rwnd가 작은지(흐름)를 ss -tin으로 분리해 봅니다.
  • 무선/셀룰러 — 손실 기반 알고리즘이 자주 cwnd를 깎아 처리량이 들쭉날쭉해질 수 있음. BBR이나 LTE 친화 알고리즘 선택 검토.
  • 혼잡 제어 튜닝 — net.ipv4.tcp_slow_start_after_idle=0으로 idle 후 슬로 스타트 재시작을 끄는 최적화가 keep-alive 트래픽에서 흔히 사용됩니다.

면접에 나올 질문과 답변

슬로 스타트의 원리와 "느리다"라는 이름의 의미는?
ACK가 올 때마다 cwnd를 1 MSS 늘리므로 RTT마다 두 배가 됩니다. 즉 지수 증가입니다. "Slow"라는 이름은 그 이전의 무모하게 즉시 최대 송신과 비교해 "이건 천천히 시작한다"는 의미일 뿐, 실제로는 매우 빠르게 늘어납니다. cwnd가 ssthresh에 도달하면 혼잡 회피로 전환합니다.
AIMD의 의미와 왜 이런 비대칭을 쓰는지 설명해 주세요.
Additive Increase, Multiplicative Decrease — 늘릴 땐 RTT마다 +1처럼 천천히, 줄일 땐 절반처럼 큰 폭으로. 이 비대칭은 두 가지 효과를 줍니다. 첫째, 손실 시 즉시 큰 폭으로 양보해 네트워크 안정을 회복하고, 둘째, 여러 흐름이 같은 회선을 쓸 때 비슷한 점유율로 자연스럽게 수렴해 공정성이 보장됩니다.
Fast Retransmit과 Fast Recovery는 무엇이 다른가요?
Fast Retransmit은 RTO를 기다리지 않고 3 dup ACK가 오면 즉시 재전송하는 손실 감지 방법입니다. Fast Recovery는 그 후의 회복 동작으로, cwnd를 1로 떨어뜨리지 않고 ssthresh로 절반만 떨어뜨려 즉시 혼잡 회피 모드로 진입합니다. Tahoe는 Fast Retransmit만, Reno는 둘 다 가집니다.
왜 손실로만 혼잡을 판단하는 게 한계가 있나요?
무선·모바일 환경처럼 혼잡과 무관한 손실이 자주 일어나는 곳에서 cwnd를 잘못 깎아 처리량이 떨어집니다. 또 손실은 이미 큐가 가득 찼다는 뜻이라 반응이 늦습니다. BBR은 이를 보완하기 위해 RTT 증가와 대역폭 추정을 직접 사용해 손실 전에 적정 cwnd를 찾으려 합니다.
Cubic이 왜 현대 리눅스의 기본인가요?
Reno류는 RTT가 길어질수록 회복이 느려지는 약점이 있습니다. Cubic은 cwnd를 시간의 3차 함수로 모델링해 손실 직전 값 부근에선 천천히, 멀어지면 빠르게 늘립니다. 그 결과 BDP가 큰 환경에서 처리량이 좋고, RTT가 다른 흐름들 사이에서도 비교적 공정합니다.
cwnd, rwnd, ssthresh의 관계를 설명해 주세요.
실제 송신 가능량은 min(rwnd, cwnd)입니다. rwnd는 수신자 버퍼 보호용(흐름 제어), cwnd는 네트워크 보호용(혼잡 제어)입니다. ssthresh는 슬로 스타트와 혼잡 회피의 경계로, 손실 감지 시 보통 cwnd/2로 설정되어 다음 동작 모드를 결정합니다.
ECN은 무엇이고 실무에서 자주 쓰이나요?
라우터가 패킷을 버리는 대신 IP 헤더의 ECN 비트로 혼잡을 알리는 메커니즘입니다. 손실 없이 송신측이 cwnd를 줄일 수 있어 효율적입니다. 그러나 경로 전체의 모든 장비와 양 끝단이 모두 지원해야 하고, 일부 장비는 ECN 비트를 잘못 처리해 활성화에 신중해야 합니다. AWS·일부 데이터센터 내부에선 적극적으로 쓰지만 일반 인터넷에서는 제한적입니다.
RTO는 어떻게 계산되며 왜 백오프하나요?
Jacobson 알고리즘으로 RTT 측정값을 평활화한 SRTT와 분산 RTTVAR을 유지하고, RTO = SRTT + 4·RTTVAR로 계산합니다. 타임아웃이 연속 발생하면 RTO 자체를 두 배로 늘려(지수 백오프) 이미 막힌 네트워크에 부담을 더 주지 않습니다. Karn 알고리즘은 재전송된 세그먼트의 RTT는 측정에 쓰지 않아 모호성을 피합니다.
