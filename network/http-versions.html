HTTP/1.1 vs HTTP/2 vs HTTP/3

세 버전은 모두 같은 HTTP 의미론(메서드·상태 코드·헤더)을 공유하지만 전송 방식이 완전히 다릅니다. HTTP/1.1은 텍스트 기반·연결당 하나의 요청, HTTP/2는 이진 프레임 기반·한 TCP 연결에서 멀티플렉싱, HTTP/3는 TCP를 버리고 QUIC(UDP 위 신뢰성 프로토콜)로 옮겨 TCP의 HOL blocking을 근본적으로 해결합니다.

구체적인 개념 설명

세 버전을 한 문장으로 묶으면 "같은 의미를 더 빠르게 전달하기 위한 진화"입니다. 메서드 GET, 상태 코드 200, Content-Type 헤더 같은 의미는 그대로지만, 그 메시지를 어떻게 인코딩하고 어떻게 흘려보내는지가 매 버전마다 바뀝니다.

HTTP/1.1(1997)은 한 TCP 연결에서 요청을 순서대로 하나씩 처리합니다. 페이지 하나에 수십 개 자원을 받아야 하는 현대 웹에는 너무 느려, 브라우저는 도메인당 6개 정도의 연결을 동시에 열어 우회했습니다. 비효율과 헤더 오버헤드가 누적됐죠.

HTTP/2(2015)는 이진 프레임으로 인코딩하고, 한 TCP 연결 안에 여러 개의 스트림을 두어 동시에 여러 요청을 주고받습니다(멀티플렉싱). 그러나 TCP 자체의 패킷 손실 한 번이 한 연결의 모든 스트림을 막는 TCP 레벨 HOL(Head-Of-Line) blocking이 남았습니다.

HTTP/3(2022)는 이 마지막 걸림돌을 없애기 위해 UDP 위에 QUIC이라는 새 프로토콜을 만들고, 그 안에서 멀티플렉싱·암호화·재전송을 모두 직접 처리합니다. 한 스트림의 손실이 다른 스트림에 영향을 주지 않습니다.

의미는 그대로, 와이어 포맷이 다르다

앱에서 GET /index.html을 보내는 방식은 셋 다 똑같습니다. 다만 그게 케이블 위로 흐를 때 1.1은 텍스트 줄, 2는 이진 프레임, 3은 QUIC 패킷으로 변할 뿐입니다. 그래서 응용 코드는 거의 그대로 두고도 버전을 올리는 효과를 볼 수 있습니다.

왜 필요한가?

현대 웹 페이지 한 개를 열려면 평균적으로 50~100개의 HTTP 요청이 필요합니다(HTML, CSS, JS, 폰트, 이미지, API). RTT가 50~100ms인 일반 환경에서 1.1처럼 직렬로 처리하면 페이지 로드가 수십 초가 됩니다. 병렬화·멀티플렉싱·압축·QUIC은 모두 "이 RTT 비용을 어떻게 줄일까"라는 한 질문에 대한 답입니다.

또 모바일·무선 환경에서는 패킷 손실이 잦은데, TCP의 신뢰성 보장이 오히려 발목을 잡는 경우가 많습니다. HTTP/3는 이런 환경에서 특히 큰 차이를 보입니다 — 손실이 한 스트림에만 머무르고, 셀룰러↔Wi-Fi 전환 같은 IP 변경에도 연결을 유지할 수 있습니다.

어떻게 동작하는가?

1) HTTP/1.1 — 직렬 처리와 keep-alive

1.0에서는 요청 하나마다 TCP 연결을 새로 만들었습니다. 1.1은 keep-alive로 한 연결을 여러 요청에 재사용하는 게 기본이 됐고, 파이프라이닝(요청을 여러 개 미리 보내기)도 정의됐지만 응답 순서 보장 문제로 거의 안 쓰였습니다. 결국 실무에서는 요청 → 응답 → 다음 요청의 직렬 모델이 됐습니다.

병렬화는 도메인당 여러 TCP 연결로 우회했고, 그래도 연결 비용·헤더 중복이 컸다.

2) HTTP/2 — 이진 프레임과 멀티플렉싱

HTTP/2의 가장 큰 변화는 두 가지입니다:

  • 이진 프레이밍 — 메시지를 텍스트가 아닌 작은 이진 프레임으로 쪼개 보냅니다. 파싱이 빠르고 정확합니다.
  • 스트림 멀티플렉싱 — 한 TCP 연결 안에 가상의 "스트림" 여러 개를 두고, 프레임에 스트림 ID를 붙여 인터리빙합니다. 한 연결에서 동시에 여러 요청·응답이 흐르며, 응답 순서가 요청 순서와 달라도 됩니다.
스트림 ID가 있는 이진 프레임 덕분에 응답이 뒤섞여 도착해도 재조립할 수 있다.

추가로 HPACK이라는 헤더 압축이 도입됐습니다. 같은 헤더를 매번 보내지 않고 인덱스 테이블과 허프만 인코딩으로 크기를 크게 줄입니다 — 보통 80~90% 감소. 또한 서버 푸시(요청 안 한 자원도 미리 보내기)가 정의됐지만 캐시 효율이 낮아 결국 HTTP/2에서 폐기됐습니다.

3) TCP HOL blocking — HTTP/2의 마지막 약점

여기서 한 가지 함정이 있습니다. HTTP/2의 멀티플렉싱은 응용 계층의 마법일 뿐, 그 아래 TCP는 여전히 단일 바이트 스트림입니다. 한 패킷이 손실되면 TCP는 그 자리에서 멈추고 재전송을 기다립니다. 결과적으로 스트림 1의 손실이 스트림 3, 5까지 모두 멈추게 만듭니다.

QUIC은 스트림별로 독립된 신뢰성 채널을 가져 한 스트림의 손실이 다른 스트림을 막지 않는다.

4) HTTP/3 — UDP 위 QUIC

HTTP/3의 핵심 결정은 "TCP를 버리자"입니다. 하지만 인터넷의 모든 미들박스(NAT, 방화벽)가 TCP/UDP만 이해하므로, 완전히 새 프로토콜은 만들 수 없습니다. 그래서 UDP 위에 QUIC이라는 사용자 공간 프로토콜을 올렸습니다. 미들박스는 평범한 UDP로 보지만, 양 끝단은 QUIC으로 신뢰성·암호화·멀티플렉싱을 직접 합니다.

QUIC의 주요 이점:

  • 스트림별 독립 신뢰성 — 한 스트림의 손실이 다른 스트림을 막지 않음.
  • 핸드셰이크 통합 — TCP 핸드셰이크와 TLS 핸드셰이크를 한 번에. 1 RTT, 또는 0-RTT 재연결.
  • 연결 마이그레이션 — IP가 바뀌어도(셀룰러↔Wi-Fi 전환) Connection ID로 연결을 유지.
  • 암호화 기본 — QUIC 자체에 TLS 1.3이 내장. 평문 QUIC은 존재하지 않음.

헤더 압축은 QPACK으로 바뀌었습니다 — HPACK은 스트림 순서에 의존했는데, 멀티스트림 환경에선 그게 다시 HOL blocking을 만들 수 있어 QPACK은 동적 테이블을 별도 스트림으로 분리해 이 문제를 풀었습니다.

6) Java 코드

// HttpClient — JDK 11+. 자동 협상으로 가장 좋은 버전을 선택.
HttpClient client = HttpClient.newBuilder()
    .version(HttpClient.Version.HTTP_2)         // HTTP/2 우선, 협상 실패 시 1.1
    .build();

HttpRequest req = HttpRequest.newBuilder()
    .uri(URI.create("https://example.com/api"))
    .GET().build();

// 같은 클라이언트로 여러 요청을 동시에 보내면 HTTP/2면 한 연결 멀티플렉싱
List<CompletableFuture<HttpResponse<String>>> futures = List.of(
    client.sendAsync(req, HttpResponse.BodyHandlers.ofString()),
    client.sendAsync(req, HttpResponse.BodyHandlers.ofString()),
    client.sendAsync(req, HttpResponse.BodyHandlers.ofString())
);

// 응답 후 어떤 버전이 협상됐는지 확인
HttpResponse<String> res = futures.get(0).join();
System.out.println(res.version());              // HTTP_1_1 또는 HTTP_2

// HTTP/3는 JDK 표준 미지원(2026년 현재).
// jdk.httpclient에서 HTTP/3 옵션이 incubator로 진행 중이며,
// 운영 환경에서는 보통 Netty/Jetty 또는 Reactor Netty + JNI 라이브러리(quiche, msquic)로 처리.

// 협상 메커니즘
//   HTTP/1.1 ↔ HTTP/2: TLS 핸드셰이크의 ALPN 확장으로 협상 ("h2" 또는 "http/1.1")
//   HTTP/3:           응답 헤더의 Alt-Svc로 "다음엔 h3로 와줘"를 알림
// 즉 첫 접속은 HTTP/2(TCP), 이후 같은 사이트는 HTTP/3로 자동 전환되는 게 일반적.

시간/공간 복잡도 또는 trade-off

  • 1.1의 단순함 — 디버깅·도구 호환성은 1.1이 가장 쉽습니다. telnet으로도 직접 칠 수 있죠. 하지만 처리량과 응답 시간에서 큰 손해.
  • 2의 멀티플렉싱 vs HOL — 한 연결로 효율을 높였지만 TCP HOL이 잠복. 손실률이 낮고 RTT가 짧은 환경에선 1.1과 큰 차이가 없을 때도 있습니다.
  • 3의 QUIC 비용 — 사용자 공간에서 신뢰성·암호화를 처리하므로 CPU 사용량이 TCP보다 높습니다. 하드웨어 오프로드가 약해 큰 트래픽에선 부담.
  • 방화벽 호환성 — 일부 기업·국가 환경에서 UDP 443이 차단되거나 비대칭으로 처리되어 HTTP/3가 동작하지 않을 수 있습니다. 그래서 fallback이 필수.
  • 0-RTT의 보안 trade-off — 빠르지만 재전송 공격(replay)에 취약합니다. 멱등 GET에만 허용하는 식의 정책이 필요합니다.

실무에서 어디에 쓰이는가?

  • 대형 사이트의 기본값 — Cloudflare, Google, Facebook 등은 HTTP/3를 1순위, HTTP/2를 fallback으로 운영합니다. Alt-Svc: h3=":443" 헤더 사용.
  • API 게이트웨이 — 백엔드 간 통신은 여전히 HTTP/2가 일반적(gRPC도 HTTP/2 위에 동작). 외부 노출은 HTTP/3 + 1.1 fallback.
  • 모바일 앱 — 셀룰러↔Wi-Fi 전환이 잦은 환경에서 HTTP/3의 연결 마이그레이션 효과가 큽니다.
  • 로드밸런서 — AWS ALB, GCP HTTPS LB 모두 HTTP/2를 지원, HTTP/3는 Cloudflare·Akamai 등 CDN 위주로 보급 중.
  • 관측 — 브라우저 DevTools의 "Protocol" 컬럼으로 어느 버전으로 응답됐는지 확인 가능 (h2, h3-29 등).

면접에 나올 질문과 답변

HTTP/1.1과 HTTP/2의 가장 큰 차이는 무엇인가요?
HTTP/1.1은 텍스트 기반에 한 연결당 한 요청을 직렬 처리하는 반면, HTTP/2는 이진 프레임으로 인코딩하고 한 TCP 연결 안에 여러 스트림을 멀티플렉싱합니다. 덕분에 헤드 오브 라인 문제와 연결 비용이 응용 계층에서는 사라졌습니다. 다만 TCP 자체의 HOL blocking은 그대로 남아 있어 이게 HTTP/3의 동기가 됐습니다.
HTTP/2의 멀티플렉싱이 있는데 왜 HTTP/3가 필요한가요?
HTTP/2의 멀티플렉싱은 응용 계층에서만 작동합니다. 그 아래 TCP는 단일 바이트 스트림이라 패킷 한 개 손실이 모든 스트림을 멈추게 만듭니다(TCP HOL blocking). HTTP/3는 UDP 위 QUIC에서 스트림별로 독립적인 신뢰성 채널을 만들어 이 문제를 근본적으로 해결합니다.
QUIC은 무엇인가요? 그냥 UDP를 쓰면 되지 않나요?
QUIC은 UDP 위에서 신뢰성, 암호화(TLS 1.3 내장), 흐름 제어, 멀티플렉싱을 모두 사용자 공간에서 구현한 프로토콜입니다. UDP를 그대로 쓰면 신뢰성·암호화·혼잡 제어가 없어 응용이 직접 만들어야 하는데, QUIC이 그것을 표준화했습니다. UDP는 미들박스 통과를 위한 "껍데기" 역할입니다.
HPACK과 QPACK의 차이는 무엇인가요?
둘 다 헤더 압축인데, HPACK(HTTP/2)은 단일 TCP 스트림 위에서 동작하므로 헤더 인덱스 갱신 순서가 보장돼 단순합니다. QPACK(HTTP/3)은 멀티스트림 환경에서 인덱스 동기화 자체가 HOL blocking을 만들 수 있어, 동적 테이블 갱신을 별도 스트림으로 분리해 이 문제를 회피합니다.
HTTP/3의 0-RTT 재연결의 장점과 위험은?
장점은 응답 시간 — 이전 연결 정보를 바탕으로 핸드셰이크 없이 곧장 데이터 전송이 가능합니다. 위험은 재전송 공격(replay) — 공격자가 0-RTT 데이터를 캡처해 다시 보내면 서버가 같은 작업을 두 번 수행할 수 있습니다. 그래서 0-RTT는 멱등 GET 같은 안전한 요청에만 허용하는 게 원칙입니다.
연결 마이그레이션이란 무엇이고 왜 중요한가요?
QUIC 연결은 IP/포트가 아닌 Connection ID로 식별됩니다. 그래서 클라이언트의 IP가 바뀌어도(셀룰러↔Wi-Fi, NAT 재바인딩) 연결을 끊지 않고 이어갈 수 있습니다. TCP는 IP가 바뀌면 즉시 끊기므로, 모바일 환경에서 큰 차이를 만듭니다.
HTTP/2 서버 푸시가 사라진 이유는?
서버가 클라이언트가 곧 요청할 자원을 미리 보내는 기능이었지만, 클라이언트의 캐시 상태나 우선순위를 모른 채 보내다 보니 오히려 대역폭을 낭비하는 경우가 많았습니다. 측정 결과 효과가 작거나 음의 효과가 있어 크롬 등이 지원을 중단했고, 사실상 deprecated 상태입니다. 대안으로 Link: rel=preload 헤더와 103 Early Hints가 사용됩니다.
한 사이트가 어떤 HTTP 버전을 쓰고 있는지 어떻게 확인하나요?
브라우저 DevTools의 Network 탭에서 "Protocol" 컬럼을 보면 됩니다(h2, h3, http/1.1). curl --http3 https://example.com -v로 강제 시도해 볼 수 있고, 응답 헤더의 Alt-Svc: h3=":443"로 다음 접속에 HTTP/3를 쓰겠다고 알릴 수 있습니다.
