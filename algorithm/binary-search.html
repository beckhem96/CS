이진 탐색(Binary Search)

이진 탐색이란, 정렬된 배열에서 가운데 값을 보고 찾는 값과 비교해 탐색 범위를 절반씩 줄여가며 O(log n)에 원소를 찾는 알고리즘입니다. "정답을 매개변수로 두고 조건을 만족하는지 검사"하는 매개변수 탐색으로 일반화되어 광범위하게 쓰입니다.

구체적인 개념 설명

비유는 단순합니다. 500쪽짜리 책에서 247쪽 찾기. 책 절반을 펴서 250쪽이 나오면 "247은 앞쪽이네" 하고 앞쪽 절반에서 다시 절반을 펴고… 이런 식으로 한 번에 후보를 절반씩 버리면 금방 찾는 페이지에 도달하죠. 이진 탐색이 정확히 이거예요.

알고리즘은 두 개의 포인터 lo, hi로 탐색 범위를 표현하고, mid = (lo + hi) / 2의 값을 찾는 값과 비교해 다음 중 하나로 분기합니다.

  • arr[mid] == target — 찾았다, 종료.
  • arr[mid] < target — 답은 오른쪽 절반에 있다 → lo = mid + 1
  • arr[mid] > target — 답은 왼쪽 절반에 있다 → hi = mid - 1

이진 탐색의 전제 — "단조성(monotonicity)"

이진 탐색이 동작하려면 데이터에 순서가 있어서, 한쪽에 답이 있다는 걸 한 번의 비교로 결정할 수 있어야 합니다. 정렬된 배열이 가장 흔한 예지만, "조건 X를 만족하는 가장 작은 값"처럼 결정 함수가 단조롭기만 하면 배열이 아닌 문제(매개변수 탐색)에도 적용됩니다.

왜 필요한가?

선형 탐색은 평균적으로 n/2번, 최악으로 n번 비교가 필요합니다. n이 1억이라면 1억 번이죠. 반면 이진 탐색은 한 번 비교할 때마다 후보가 절반씩 줄어 log₂ 1억 ≈ 27번이면 끝납니다. 데이터가 수백만 개를 넘어가면 차이가 어마어마합니다.

더 큰 가치는 응용 가능성에 있어요. "조건을 만족하는 최소/최대 값"을 찾는 문제, 회전된 정렬 배열에서 값 찾기, 함수의 영점 찾기 같은 문제도 모두 이진 탐색의 변형입니다. 코딩테스트의 단골 도구이기도 합니다.

어떻게 동작하는가?

1) 단계별 시각화 — 정렬된 배열에서 23 찾기

9개 원소를 단 3번의 비교로 끝낸다 — log₂ 9 ≈ 3.17.

2) 자주 헷갈리는 변형 — lower_bound / upper_bound

"정확히 일치하는 인덱스"가 아니라 경계를 찾고 싶을 때가 더 많습니다. 두 가지 표준 변형을 외워두면 응용이 쉬워요.

  • lower_bound(target) — target 이상의 첫 번째 인덱스. 즉 "target을 삽입할 수 있는 가장 왼쪽 자리".
  • upper_bound(target) — target 초과의 첫 번째 인덱스. 즉 "target을 삽입할 수 있는 가장 오른쪽 자리".

두 값을 빼면 target의 등장 횟수(= upper - lower)가 나옵니다. 정렬된 배열에서 등장 횟수, 범위 카운트 같은 문제가 깔끔하게 풀려요.

3) Java 코드로 보기

// 표준 이진 탐색 — 값 인덱스 반환, 없으면 -1
int binarySearch(int[] arr, int target) {
    int lo = 0, hi = arr.length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;     // 오버플로 방지
        if (arr[mid] == target) return mid;
        if (arr[mid] < target) lo = mid + 1;
        else                   hi = mid - 1;
    }
    return -1;
}

// lower_bound — target 이상의 첫 인덱스 (없으면 arr.length)
int lowerBound(int[] arr, int target) {
    int lo = 0, hi = arr.length;          // hi는 length로!
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (arr[mid] < target) lo = mid + 1;
        else                   hi = mid;
    }
    return lo;
}

// 매개변수 탐색 — "조건 ok(x)가 단조롭게 false→true로 바뀐다"고 가정
int parametricSearch(int lo, int hi) {
    // ok(x) = (x가 충분히 큰가)
    while (lo < hi) {
        int mid = lo + (hi - lo) / 2;
        if (ok(mid)) hi = mid;            // 조건 만족하는 가장 작은 mid를 찾음
        else         lo = mid + 1;
    }
    return lo;
}

// 표준 라이브러리 활용
import java.util.Arrays;
int idx = Arrays.binarySearch(arr, target);  // 음수 반환은 "삽입 자리의 ~값"

시간/공간 복잡도

버전시간공간
반복문 구현O(log n)O(1)
재귀 구현O(log n)O(log n) — 호출 스택

"한 번 비교에 후보가 절반"이 핵심입니다. n번에 도달하려면 n / 2 / 2 / ... / 2 = 1이 되는 횟수, 즉 log₂ n 번. 재귀로 짜면 코드가 깔끔하지만 호출 스택이 쌓이므로, 보통은 반복문으로 짭니다.

실무에서 어디에 쓰이는가?

  • 정렬된 배열에서 키 검색 — Java Arrays.binarySearch, C++ std::lower_bound 등 표준 라이브러리.
  • 데이터베이스 인덱스 — B+Tree 노드 안에서 키를 찾을 때 이진 탐색 사용.
  • 매개변수 탐색 — "최소 시간 안에 끝낼 수 있는 가장 큰 작업 크기" 같은 최적화 문제.
  • Git의 git bisect — 어떤 커밋에서 버그가 들어왔는지 이진 탐색으로 찾음.
  • 회전된 정렬 배열에서 검색, 버전 관리에서 첫 결함 버전 찾기 같은 변형 문제.
  • 코딩테스트의 단골 — "정답을 이분 탐색"하는 문제는 아주 흔한 패턴.

면접에 나올 질문과 답변

이진 탐색에서 mid = (lo + hi) / 2 대신 lo + (hi - lo) / 2를 쓰는 이유는요?
정수 오버플로 방지 때문입니다. lo + hi가 int 범위를 넘으면 음수가 되어 잘못된 mid를 계산할 수 있습니다. 실제로 자바 표준 라이브러리도 한때 이 버그가 있었어요. lo + (hi - lo) / 2는 차이를 먼저 구해 더하기 때문에 오버플로가 일어나지 않습니다.
중복 값이 있는 배열에서 첫 번째 위치는 어떻게 찾나요?
lower_bound를 쓰면 됩니다. arr[mid] < target일 때만 lo = mid + 1로 옮기고, arr[mid] >= target이면 hi = mid로 두는 식이죠. 이렇게 하면 같은 값이 여러 개 있어도 그중 가장 왼쪽 인덱스를 보장하게 됩니다. upper_bound도 같은 패턴인데 부등호만 살짝 다릅니다.
이진 탐색을 쓸 수 있는 조건은 무엇인가요?
한 번의 비교로 답이 어느 쪽에 있는지 결정할 수 있어야 합니다 — 즉 단조성이 있어야 합니다. 가장 흔한 경우가 정렬된 배열이지만, "어떤 조건 ok(x)가 false → true로 한 번만 바뀌는 함수"라면 배열이 아니어도 적용 가능합니다(매개변수 탐색). 단조성이 깨지면 이진 탐색은 잘못된 답을 줄 수 있어요.
매개변수 탐색은 일반 이진 탐색과 어떻게 다른가요?
구조는 같습니다. 차이는 탐색 대상이 배열의 인덱스가 아니라 정답 후보의 값이라는 점이에요. 예: "출판할 수 있는 책의 최소 두께"를 구하라 → 두께를 mid로 두고, 그 두께로 출판이 가능한지 ok(mid) 함수로 확인합니다. ok가 단조롭게 변한다는 보장이 있으면 이진 탐색으로 답을 좁혀갈 수 있습니다.
