유니온-파인드(Union-Find / DSU)

유니온-파인드(또는 서로소 집합 자료구조, Disjoint Set Union)는, 원소들을 여러 개의 집합으로 나눠 두고 두 가지 연산만 빠르게 처리하는 자료구조입니다. find(x)는 x가 속한 집합의 대표(루트)를 반환하고, union(a, b)는 a와 b가 속한 두 집합을 하나로 합칩니다. 경로 압축 + 랭크/크기 기반 합치기를 같이 쓰면 거의 O(1)(엄밀히는 O(α(n)))에 동작합니다.

구체적인 개념 설명

비유는 친구 그룹입니다. 처음에 모두가 따로 떨어져 있다가, "A와 B가 친해졌다" → 두 그룹을 합치고, "C와 D가 같은 그룹인가?" 라고 물으면 두 사람의 그룹 대표가 같은지만 확인하면 됩니다. 이걸 자료구조로 표현한 게 유니온-파인드예요.

구현은 의외로 단순합니다. 각 원소가 자기 부모를 가리키는 parent[] 배열 하나면 충분해요. 처음엔 모두 자기 자신이 부모(즉 자기가 속한 집합의 루트)이고, union을 하면 한 집합의 루트가 다른 집합의 루트의 자식이 되도록 부모를 바꿉니다. find는 부모를 따라가다 자기 자신이 부모인 노드(루트)에 도달하면 그게 그 집합의 대표입니다.

두 가지 최적화로 거의 상수 시간이 된다

  • 경로 압축(Path Compression) — find 도중 거쳐간 노드들의 부모를 모두 루트로 직접 연결해서, 다음 find가 O(1)에 가까워지게 함.
  • 랭크/크기 기반 합치기(Union by Rank / Size) — 작은 트리를 큰 트리 밑에 붙여서 트리 높이가 천천히 자라게 함.

둘을 함께 쓰면 모든 연산이 평균 O(α(n)) — α는 역애커만 함수로, 실세계 n에서는 사실상 상수(4 이하)입니다.

왜 필요한가?

"어떤 두 원소가 같은 그룹에 속하는가?"라는 질문이 빈번하게 나오는 문제들이 있습니다. 네트워크의 연결 관계, 그래프의 사이클 검출, 최소 신장 트리 알고리즘(크루스칼), 이미지 처리에서의 영역 분할, 게임에서 같은 길드 멤버 확인 등이 대표적이에요.

매번 BFS/DFS로 확인하면 한 번 질의에 O(V+E)가 걸리는데, 유니온-파인드는 미리 동치 관계를 합쳐두기 때문에 질의 한 번이 거의 상수 시간입니다. "동적으로 합쳐지는 그룹들"을 다루는 거의 모든 문제에서 표준 도구입니다.

어떻게 동작하는가?

1) 기본 표현 — 부모 배열

원소 6개로 시작합시다. 처음엔 모두 자기 자신이 부모(= 자기가 속한 집합의 루트).

각 원소가 자기 자신을 가리킨다 = 모두 별개 집합의 루트.

2) union 연산 — 두 집합 합치기

union(0, 1) → union(2, 3) → union(0, 2)를 순서대로 적용한 결과입니다.

0 트리는 의미만 그룹을 나타낼 뿐, 모양은 union 순서에 따라 다양해진다.

3) 경로 압축 — find 한 번 호출로 트리가 평탄해짐

깊은 트리에서 find가 한 번 호출되면, 그 경로 위의 모든 노드를 루트의 직접 자식으로 다시 매다는 게 경로 압축입니다. 그러면 다음번에 같은 노드에 대해 find를 호출하면 거의 O(1)에 끝납니다.

한 번 깊이 들어간 비용을 다음번에는 안 치르게 만든다 — amortization의 좋은 예시.

4) Java 코드로 보기

class UnionFind {
    int[] parent, rank;

    UnionFind(int n) {
        parent = new int[n];
        rank   = new int[n];
        for (int i = 0; i < n; i++) parent[i] = i;
    }

    // find — 경로 압축
    int find(int x) {
        if (parent[x] != x) {
            parent[x] = find(parent[x]);    // 재귀하며 루트로 직접 연결
        }
        return parent[x];
    }

    // union — 랭크 기반 합치기, true면 실제 합쳐짐
    boolean union(int a, int b) {
        int ra = find(a), rb = find(b);
        if (ra == rb) return false;          // 이미 같은 집합

        if (rank[ra] < rank[rb]) {
            parent[ra] = rb;
        } else if (rank[ra] > rank[rb]) {
            parent[rb] = ra;
        } else {
            parent[rb] = ra;
            rank[ra]++;
        }
        return true;
    }

    boolean connected(int a, int b) {
        return find(a) == find(b);
    }
}

// 응용: 그래프의 연결 요소 개수
int countComponents(int n, int[][] edges) {
    UnionFind uf = new UnionFind(n);
    int comps = n;
    for (int[] e : edges) {
        if (uf.union(e[0], e[1])) comps--;   // 새로 합쳐질 때마다 1 감소
    }
    return comps;
}

시간/공간 복잡도

구현find / union 평균설명
최적화 없음O(n)최악으로 한쪽으로 늘어진 트리
경로 압축만O(log n)분할상환 분석
랭크 기반 합치기만O(log n)트리 높이를 log n 이하로
경로 압축 + 랭크 기반O(α(n)) ≈ O(1)역애커만 함수. 실세계 n에서 4 이하

공간은 부모 배열과 랭크 배열로 O(n).

실무에서 어디에 쓰이는가?

  • 크루스칼 최소 신장 트리 — 간선을 가중치 순으로 정렬한 뒤, 사이클을 만들지 않으면 합치기 — 사이클 판단을 union-find로.
  • 그래프 연결성 / 사이클 검출 — 동적으로 간선이 추가될 때 두 정점이 이미 연결되어 있는지 즉시 확인.
  • 이미지 처리의 연결 요소 라벨링(CCL) — 같은 색 픽셀들을 한 영역으로 묶기.
  • 네트워크 연결성 — "이 두 컴퓨터 사이에 경로가 있는가?" 질의의 빠른 처리.
  • 오프라인 쿼리 처리 — 시간 역순으로 간선을 추가하며 답하는 트릭에서 자주 등장.
  • 친구 관계 / 길드 / 같은 컴포넌트 검사 — 게임·SNS 같은 도메인.

면접에 나올 질문과 답변

유니온-파인드는 트리 구조로 표현하는데, 왜 트리 클래스 없이 배열만으로 만드나요?
실제로는 트리 자식 정보를 알 필요가 없고, "내 부모가 누구인가"만 알면 find와 union이 모두 가능하기 때문입니다. 그래서 parent[] 배열 하나에 인덱스를 부모로 매핑해두는 게 메모리·캐시 측면에서 가장 효율적입니다. 클래스로 노드를 만들면 포인터 추적과 GC 비용이 늘어 오히려 느려지죠.
경로 압축과 랭크 기반 합치기 중 하나만 쓰면 안 되나요?
하나만 써도 평균 O(log n)으로 충분히 빠르고 실용적입니다. 다만 두 가지를 함께 쓰면 평균이 거의 상수 — α(n) — 가 되어 대규모 입력에서 차이가 명확해집니다. 실무에서 코드 양이 거의 늘지 않으므로 보통 둘을 함께 적용합니다.
유니온-파인드를 BFS/DFS로 대체할 수 있을까요?
한 번 그래프가 고정되어 있다면 BFS/DFS 한 번으로 연결 요소를 모두 라벨링할 수 있어 대안이 됩니다. 하지만 간선이 동적으로 추가/제거되거나 "두 정점이 같은 그룹인가?"를 수많은 질의로 해야 한다면 매번 BFS를 다시 돌리는 건 비효율적입니다. 유니온-파인드는 합치기와 질의가 모두 거의 O(1)이라 이런 상황에 적합합니다. 반대로 동적 삭제가 필요하면 일반 유니온-파인드로는 어렵고, 더 무거운 자료구조(Link-Cut Tree 등)가 필요해집니다.
α(n)이 사실상 상수라는 말이 무슨 뜻인가요?
α는 역애커만 함수(inverse Ackermann)로, 입력 n이 커져도 매우 매우 천천히 증가합니다. n이 우주의 원자 수만큼 커져도 α(n) ≤ 5 정도로, 알고리즘 분석에서 사실상 상수로 취급됩니다. 이론적으로는 O(1)이 아니지만, 실무 분석에서는 "사실상 O(1)"이라고 말해도 무방합니다.
