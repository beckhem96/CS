다익스트라(Dijkstra)

다익스트라 알고리즘이란, 음수 가중치가 없는 가중 그래프에서 한 출발 정점으로부터 모든 정점까지의 최단 경로를 구하는 알고리즘입니다. "현재까지 거리가 가장 짧은 정점을 골라 확정"하는 그리디 + 우선순위 큐 조합으로 O((V+E) log V)에 동작합니다.

구체적인 개념 설명

비유로 가장 잘 어울리는 건 출발지에서 모든 도시까지의 최단 거리 계산입니다. 일단 모든 도시의 거리를 ∞로 두고 출발지만 0으로 시작합니다. 그리고 매 라운드마다 "지금까지 알려진 거리가 가장 짧은 도시"를 하나 골라 확정하고, 그 도시를 거쳐가는 다른 도시들의 거리를 갱신해 갑니다(relaxation).

한 번 확정된 정점의 거리는 다시 바뀌지 않습니다. 이게 가능한 이유가 "음수 가중치가 없다"는 가정 덕분이에요. 모든 간선이 0 이상이라면, 지금까지 가장 짧은 거리로 도달한 정점은 나중에 어디를 들르더라도 그보다 더 짧아질 수 없기 때문입니다.

왜 음수 가중치가 안 되나?

다익스트라는 "지금 가장 가까운 정점은 더 짧아질 수 없다"는 그리디 가정을 사용합니다. 음수 간선이 있으면 그 정점을 한 번 더 들렀을 때 거리가 줄어들 수 있어 가정이 깨집니다. 그래서 음수가 있는 그래프에서는 벨만-포드나 SPFA 같은 다른 알고리즘을 써야 합니다. 다만 음수만 있을 뿐 음수 사이클이 없다면 한 번 변환을 거쳐 다익스트라를 적용하는 기법(존슨 알고리즘)도 있습니다.

왜 필요한가?

"지도에서 가장 빠른 길 찾기"는 일상에서 가장 자주 쓰이는 그래프 문제입니다. BFS는 간선의 가중치를 모두 1로 가정하기 때문에 "도로 길이가 다른" 실제 도로망에는 맞지 않아요. 다익스트라는 가중치를 정확히 반영해 최단 경로를 보장하면서도, 우선순위 큐 덕분에 빠릅니다.

네트워크 라우팅(OSPF), 게임의 경로 찾기, 음성 인식의 최적 경로, 화학 반응 경로 등 "비용이 다른 간선들 위에서 최단/최저비용 경로"가 필요한 곳에는 거의 항상 다익스트라(또는 그 변형 A*)가 등장합니다.

어떻게 동작하는가?

1) 단계별 시각화

다음 그래프에서 정점 A에서 출발해 모든 정점까지의 최단 거리를 구해봅시다.

가중치가 있는 무방향 그래프 (예: 도시와 도로 거리).

거리 표가 라운드마다 어떻게 갱신되는지 봅시다. ✓는 확정, 굵은 숫자는 그 라운드에 갱신된 값.

라운드꺼낸 정점ABCDEF
초기—0∞∞∞∞∞
1A ✓0 ✓42∞∞∞
2C ✓ (거리 2)0 ✓32 ✓∞5∞
3B ✓ (거리 3)0 ✓3 ✓2 ✓85∞
4E ✓ (거리 5)0 ✓3 ✓2 ✓75 ✓∞
5D ✓ (거리 7)0 ✓3 ✓2 ✓7 ✓5 ✓8
6F ✓ (거리 8)0 ✓3 ✓2 ✓7 ✓5 ✓8 ✓

라운드 2에서 C(거리 2)가 B(거리 4)보다 먼저 확정되는 게 핵심입니다. 그 후 C에서 B로 가는 길(2+1=3)이 발견되어 B의 거리가 4 → 3으로 줄어듭니다. 이런 식으로 "이미 짧은 정점을 거쳐 더 짧은 길을 찾는" 갱신을 반복합니다.

2) 알고리즘 골격

  1. 모든 거리 dist[v]를 ∞로, 출발 정점만 0으로 초기화. 우선순위 큐에 (0, 출발) 삽입.
  2. 큐에서 거리가 가장 작은 정점 u를 꺼냄. 이미 처리한 정점이면 건너뜀.
  3. u의 모든 이웃 v에 대해 dist[u] + w(u, v) < dist[v]이면 갱신하고 큐에 (새 거리, v) 추가.
  4. 큐가 빌 때까지 반복.

3) Java 코드로 보기

import java.util.*;

class Edge {
    int to, weight;
    Edge(int t, int w) { to = t; weight = w; }
}

int[] dijkstra(List<List<Edge>> adj, int n, int start) {
    int[] dist = new int[n];
    Arrays.fill(dist, Integer.MAX_VALUE);
    dist[start] = 0;

    // (거리, 정점) — 거리 오름차순으로 꺼냄
    PriorityQueue<int[]> pq = new PriorityQueue<>(Comparator.comparingInt(a -> a[0]));
    pq.offer(new int[]{0, start});

    while (!pq.isEmpty()) {
        int[] cur = pq.poll();
        int d = cur[0], u = cur[1];

        // 큐에 들어 있던 옛 거리는 무시 (lazy deletion)
        if (d > dist[u]) continue;

        for (Edge e : adj.get(u)) {
            int nd = d + e.weight;
            if (nd < dist[e.to]) {
                dist[e.to] = nd;
                pq.offer(new int[]{nd, e.to});
            }
        }
    }
    return dist;
}

// 경로까지 복원하려면 prev[v] = u 를 추가로 기록한 뒤 v에서 거꾸로 따라가면 됨

시간/공간 복잡도

구현시간비고
이진 힙(우선순위 큐)O((V + E) log V)가장 일반적인 구현. 자바 PriorityQueue 등.
피보나치 힙O(E + V log V)이론적 최선. 상수 인자가 커서 실무에선 거의 안 씀.
인접 행렬 + 단순 배열O(V²)V가 작거나 매우 밀집한 그래프에 유리.

공간은 인접 리스트와 거리 배열, 우선순위 큐를 합쳐 O(V + E).

실무에서 어디에 쓰이는가?

  • 지도 / 내비게이션 — 거리·시간을 가중치로 한 최단 경로. 보통 A*로 발전시켜 사용.
  • 네트워크 라우팅 — OSPF — 라우터가 LSDB를 다익스트라로 풀어 최단 경로 트리 계산.
  • 게임 AI 길 찾기 — 타일 그리드 위에서 NPC가 플레이어에게 접근하는 경로.
  • 물류·통신비 최적화 — 비용을 가중치로 두고 최저 비용 경로 산출.
  • 음성 인식 / 자연어 처리 — 비터비(Viterbi)와 유사한 최적 경로 탐색.
  • 부분 문제로 활용 — Johnson's 알고리즘처럼 더 큰 알고리즘의 빌딩 블록.

면접에 나올 질문과 답변

다익스트라가 음수 가중치를 처리할 수 없는 이유는요?
다익스트라는 "현재 거리가 가장 짧은 정점은 더 짧아질 수 없다"는 그리디 가정 위에서 동작합니다. 모든 간선이 0 이상이어야 이 가정이 성립하죠. 음수 간선이 있으면 어떤 정점을 한 번 거쳐 다시 그 정점에 가는 길이 더 짧아질 수 있어, 이미 확정한 정점의 거리가 잘못된 채로 남게 됩니다. 음수 간선이 필요한 경우엔 벨만-포드(O(V·E))나 SPFA를 씁니다.
우선순위 큐에서 옛 거리 값을 어떻게 처리하나요?
보통 lazy deletion을 씁니다. 거리가 갱신되어도 큐에 있는 옛 항목을 직접 제거하지 않고, 나중에 큐에서 꺼냈을 때 d > dist[u]면 무시하고 건너뛰는 식입니다. 이렇게 하면 큐의 크기가 O(E)까지 늘 수 있지만 구현이 단순해지고 시간 복잡도는 그대로 유지됩니다. 엄밀하게 갱신하려면 인덱스드 우선순위 큐가 필요한데, 그만큼 복잡도가 올라갑니다.
BFS와 다익스트라의 관계는요?
BFS는 모든 간선의 가중치가 1이라고 가정하는 다익스트라의 특수 케이스로 볼 수 있습니다. 가중치가 모두 같으면 큐만 있어도 가까운 순으로 정점이 처리되어 우선순위 큐가 필요 없죠. 간선 가중치가 0과 1만 있는 그래프에선 0-1 BFS(덱 활용)로 다익스트라보다 간단하고 빠르게 풀 수 있습니다.
A* 알고리즘과 다익스트라의 차이는?
다익스트라는 "출발지에서 거리"만 보고 후보를 선택하는데, A*는 거기에 "이 정점에서 목표까지의 추정 거리(휴리스틱 h)"를 더한 값으로 우선순위를 매깁니다. 목표 방향으로 가는 정점을 우선 탐색하기 때문에 보통 훨씬 빠릅니다. h가 항상 실제 거리보다 작거나 같다면(admissible) A*도 최단 경로를 보장합니다. 지도 위 길찾기처럼 출발-목표가 명확할 땐 다익스트라보다 A*가 자연스러운 선택이에요.
