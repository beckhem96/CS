그리디(Greedy)

그리디 알고리즘이란, 매 순간 가능한 선택 중 지금 가장 좋아 보이는 선택을 하고, 그 선택을 되돌리지 않는 알고리즘 설계 기법입니다. 단순하고 빠르지만, 항상 최적해를 보장하지는 않습니다. 그리디가 통하려면 그리디 선택 속성과 최적 부분 구조가 필요합니다.

구체적인 개념 설명

그리디는 말 그대로 "욕심쟁이" 방식입니다. 멀리 있는 전체 그림을 완벽히 계산하지 않고, 지금 눈앞에서 가장 이득인 선택을 합니다. 예를 들어 거스름돈을 줄 때 큰 동전부터 최대한 많이 쓰는 방식이 전형적인 그리디입니다. 500원, 100원, 50원, 10원 동전 체계에서는 이 방식이 항상 최소 동전 수를 줍니다.

하지만 아무 문제나 그리디로 풀면 틀립니다. 지금 좋아 보이는 선택이 나중에 큰 손해를 만들 수 있기 때문입니다. 그래서 그리디 문제의 핵심은 "어떤 기준으로 고를 것인가"보다 먼저, 그 기준으로 골라도 항상 최적해가 되는 이유를 설명할 수 있는가입니다.

그리디는 증명이 중요하다

코딩테스트에서 그리디는 코드보다 판단이 더 어렵습니다. 정렬하고 앞에서부터 고르는 코드가 맞아 보이더라도, "왜 이 선택을 해도 최적해를 잃지 않는가"를 교환 논증(exchange argument)이나 귀납으로 설명할 수 있어야 합니다.

왜 필요한가?

DP나 완전 탐색은 모든 상태나 경우를 고려하기 때문에 정확하지만 느릴 수 있습니다. 반면 그리디는 선택을 되돌리지 않으므로 매우 빠르고, 보통 정렬 O(n log n) 또는 한 번 순회 O(n)으로 끝납니다. 대규모 스케줄링, 네트워크 설계, 압축, 최단 경로 일부 알고리즘처럼 빠른 의사결정이 필요한 곳에서 강력합니다.

대표적인 그리디 알고리즘으로는 활동 선택(Activity Selection), 허프만 코딩, 크루스칼/프림 MST, 다익스트라, 구간 스케줄링, 일부 문자열/배열 최적화 문제가 있습니다.

어떻게 동작하는가?

1) 활동 선택 문제 — 가장 빨리 끝나는 회의부터 고르기

회의실 하나에 최대한 많은 회의를 배정하려고 합니다. 각 회의는 시작 시간과 종료 시간이 있고, 겹치면 안 됩니다. 그리디 전략은 종료 시간이 가장 빠른 회의부터 선택하는 것입니다.

빨리 끝나는 회의를 고르면 남은 시간이 가장 많이 확보된다.

2) 왜 "가장 빨리 끝나는 회의"가 맞을까?

최적해에서 첫 번째로 선택된 회의를 X라고 합시다. 우리가 고른 회의 G는 모든 회의 중 가장 빨리 끝납니다. 따라서 G의 종료 시간은 X의 종료 시간보다 같거나 빠릅니다. 최적해의 X를 G로 바꿔도 이후 회의들과 겹치지 않고, 회의 개수도 줄지 않습니다. 즉 항상 G로 시작하는 최적해가 존재합니다. 이런 식의 증명을 교환 논증이라고 합니다.

3) 그리디가 실패하는 예 — 임의 동전 체계

동전이 1, 3, 4원이고 6원을 만들어야 한다고 해봅시다. 큰 동전부터 고르면 4 + 1 + 1 = 3개지만, 최적해는 3 + 3 = 2개입니다. 즉 "큰 것부터"라는 그리디가 항상 맞는 건 아닙니다. 한국 동전처럼 특수한 체계에서만 맞습니다.

그리디는 빠르지만, 문제 구조가 맞지 않으면 틀린 답을 낸다.

4) Java 코드로 보기

import java.util.*;

// 활동 선택 문제: 종료 시간이 빠른 순으로 회의 선택
class Meeting {
    int start, end;
    Meeting(int s, int e) { start = s; end = e; }
}

int maxMeetings(List<Meeting> meetings) {
    meetings.sort(Comparator.comparingInt((Meeting m) -> m.end)
                            .thenComparingInt(m -> m.start));

    int count = 0;
    int lastEnd = Integer.MIN_VALUE;

    for (Meeting m : meetings) {
        if (m.start >= lastEnd) {
            count++;
            lastEnd = m.end;
        }
    }
    return count;
}

// 거스름돈 — 한국 동전 체계에서는 그리디가 최적
int coinChangeGreedy(int amount) {
    int[] coins = {500, 100, 50, 10};
    int count = 0;
    for (int coin : coins) {
        count += amount / coin;
        amount %= coin;
    }
    return count;
}

실무에서 어디에 쓰이는가?

  • 스케줄링 — 회의실 배정, 작업 마감시간 관리, 구간 선택.
  • 데이터 압축 — 허프만 코딩은 빈도가 낮은 문자부터 합치는 그리디.
  • 네트워크 설계 — 크루스칼/프림 최소 신장 트리.
  • 최단 경로 — 다익스트라는 "현재 최단 거리 정점 확정"이라는 그리디.
  • 캐시 정책 — 일부 온라인 정책은 지금 당장 가장 나쁜 후보를 버리는 그리디 사고를 활용.

면접에 나올 질문과 답변

그리디와 DP의 차이는 무엇인가요?
그리디는 매 순간 하나의 선택을 하고 되돌리지 않습니다. DP는 여러 선택의 결과를 상태로 저장해 비교합니다. 그래서 그리디는 빠르지만 적용 조건이 까다롭고, DP는 더 일반적이지만 상태 수만큼 비용이 큽니다. 문제에서 "지금의 최선 선택이 전체 최선으로 이어진다"는 걸 증명할 수 있으면 그리디, 아니면 DP를 의심하는 게 좋습니다.
그리디가 맞다는 걸 어떻게 증명하나요?
가장 흔한 방법은 교환 논증입니다. 어떤 최적해가 있다고 가정하고, 그 최적해의 첫 선택을 내가 고른 그리디 선택으로 바꿔도 최적성이 깨지지 않음을 보입니다. 그 다음 남은 문제에도 같은 논리를 적용하면 전체 그리디 선택이 최적해라는 결론이 나옵니다.
거스름돈 문제는 항상 그리디로 풀리나요?
아닙니다. 동전 체계에 따라 다릅니다. 한국 동전처럼 큰 동전이 작은 동전들의 조합을 잘 지배하는 체계에서는 큰 것부터 고르는 그리디가 맞지만, 동전이 1, 3, 4이고 목표가 6이면 그리디는 4+1+1(3개)을 고르고 최적해 3+3(2개)을 놓칩니다. 일반 동전 체계에서는 DP로 풀어야 안전합니다.
활동 선택 문제에서 왜 종료 시간이 빠른 순으로 고르나요?
빨리 끝나는 회의를 고르면 이후에 남는 시간이 가장 넓어집니다. 더 엄밀히는, 어떤 최적해의 첫 회의가 X라고 할 때 종료 시간이 가장 빠른 회의 G로 X를 바꿔도 이후 회의들과 겹치지 않고 회의 수가 줄지 않습니다. 따라서 G로 시작하는 최적해가 항상 존재하므로 매번 가장 빨리 끝나는 회의를 고르는 그리디가 맞습니다.
