동적 계획법(Dynamic Programming)

동적 계획법(DP)이란, 큰 문제를 작은 부분 문제로 나누되, 같은 부분 문제가 반복해서 등장할 때 그 답을 저장해 두고 재사용하는 알고리즘 설계 기법입니다. 핵심은 "한 번 푼 문제는 다시 풀지 않는다"입니다.

구체적인 개념 설명

DP는 이름이 어렵지만 아이디어는 단순합니다. 같은 계산을 계속 반복하지 않도록 메모장에 적어두는 방식이에요. 피보나치 수열을 재귀로 구하면 fib(5) 안에서 fib(3)을 여러 번 다시 계산합니다. 그런데 fib(3)=2라는 결과를 저장해두면 다음부터는 바로 꺼내 쓰면 됩니다.

DP를 적용하려면 보통 두 조건이 필요합니다.

  • 중복 부분 문제(Overlapping Subproblems) — 같은 작은 문제가 반복해서 등장한다.
  • 최적 부분 구조(Optimal Substructure) — 큰 문제의 최적해가 작은 문제의 최적해로 구성된다.

DP의 두 구현 방식

  • 메모이제이션(Memoization, Top-down) — 재귀로 내려가며 필요할 때 계산하고 저장.
  • 타뷸레이션(Tabulation, Bottom-up) — 작은 문제부터 표를 채워 큰 문제까지 올라감.

둘은 같은 DP를 다른 방향에서 구현한 것입니다. 재귀 관계가 직관적이면 Top-down, 반복문으로 깔끔하면 Bottom-up이 좋습니다.

왜 필요한가?

완전 탐색은 가능한 모든 경우를 다 시도하기 때문에 금방 폭발합니다. 하지만 그 경우들 안에 같은 부분 문제가 반복된다면, DP는 중복 계산을 제거해 지수 시간을 다항 시간으로 줄일 수 있습니다.

예를 들어 피보나치 재귀는 O(2ⁿ)에 가깝지만, DP로 바꾸면 O(n)입니다. 0/1 배낭, 최장 증가 부분 수열(LIS), 최장 공통 부분 수열(LCS), 편집 거리, 최단 경로의 일부 변형처럼 "선택의 누적 결과"를 다루는 문제에서 DP가 자주 등장합니다.

어떻게 동작하는가?

1) 피보나치 — 중복 계산 제거

fib(5)를 순수 재귀로 풀면 같은 노드가 여러 번 등장합니다. DP는 이 반복을 저장으로 제거합니다.

DP는 재귀 트리의 반복 노드를 테이블의 한 칸으로 접어 넣는 느낌이다.

2) DP 문제 풀이 절차

  1. 상태 정의 — dp[i] 또는 dp[i][j]가 무엇을 의미하는지 한 문장으로 정한다.
  2. 점화식 — 큰 상태를 작은 상태로 어떻게 계산할지 식을 세운다.
  3. 초기값 — 가장 작은 문제의 답을 채운다.
  4. 계산 순서 — 필요한 작은 상태가 먼저 계산되도록 반복 방향을 정한다.
  5. 정답 위치 — 최종 답이 dp[n]인지, 테이블 전체의 최댓값인지 확인한다.

3) 0/1 배낭 문제 맛보기

배낭 용량이 W이고, 각 물건은 한 번만 넣을 수 있다고 해봅시다. dp[i][w]를 "앞 i개 물건만 고려하고 용량이 w일 때 얻을 수 있는 최대 가치"로 정의하면: 

dp[i][w] = max(
    dp[i-1][w],                         // i번째 물건을 안 넣음
    dp[i-1][w - weight[i]] + value[i]   // i번째 물건을 넣음
)

4) Java 코드로 보기

// Top-down: 메모이제이션 피보나치
int[] memo;

int fibMemo(int n) {
    if (n <= 1) return n;
    if (memo[n] != -1) return memo[n];
    return memo[n] = fibMemo(n - 1) + fibMemo(n - 2);
}

// Bottom-up: 타뷸레이션 피보나치
int fibTab(int n) {
    if (n <= 1) return n;
    int[] dp = new int[n + 1];
    dp[0] = 0;
    dp[1] = 1;
    for (int i = 2; i <= n; i++) {
        dp[i] = dp[i - 1] + dp[i - 2];
    }
    return dp[n];
}

// 공간 최적화: 직전 두 값만 필요
int fibOptimized(int n) {
    if (n <= 1) return n;
    int prev2 = 0, prev1 = 1;
    for (int i = 2; i <= n; i++) {
        int cur = prev1 + prev2;
        prev2 = prev1;
        prev1 = cur;
    }
    return prev1;
}

// 0/1 배낭 — 2차원 DP
int knapsack(int[] weight, int[] value, int W) {
    int n = weight.length;
    int[][] dp = new int[n + 1][W + 1];

    for (int i = 1; i <= n; i++) {
        for (int w = 0; w <= W; w++) {
            dp[i][w] = dp[i - 1][w];
            if (w >= weight[i - 1]) {
                dp[i][w] = Math.max(dp[i][w],
                    dp[i - 1][w - weight[i - 1]] + value[i - 1]);
            }
        }
    }
    return dp[n][W];
}

실무에서 어디에 쓰이는가?

  • 문자열 비교 — 편집 거리(Edit Distance), LCS, diff 도구.
  • 추천/최적화 — 제한된 예산에서 최대 가치 조합 찾기(배낭류).
  • 경로 계획 — 격자 최단 경로, 장애물 회피의 일부 변형.
  • 컴파일러/파서 — CYK 파싱 같은 문법 분석 알고리즘.
  • 캐싱 사고방식 — 실제 서비스에서도 중복 계산 제거를 위해 메모이제이션/캐시를 적극 활용.

면접에 나올 질문과 답변

DP와 분할 정복의 차이는 무엇인가요?
둘 다 큰 문제를 작은 문제로 나누지만, 분할 정복은 부분 문제가 서로 독립인 경우가 많고, DP는 같은 부분 문제가 반복해서 등장합니다. 병합 정렬은 왼쪽/오른쪽이 독립이라 분할 정복이고, 피보나치는 fib(3) 같은 부분 문제가 여러 경로에서 반복되어 DP가 적합합니다.
Top-down과 Bottom-up 중 어느 쪽이 좋나요?
Top-down은 재귀 관계를 그대로 코드로 옮겨 직관적이고, 실제로 필요한 상태만 계산합니다. Bottom-up은 호출 스택이 없고 반복문이라 빠르고 안정적이며, 공간 최적화가 쉬운 경우가 많습니다. 면접에서는 먼저 상태와 점화식을 명확히 설명하고, 구현은 문제에 맞춰 선택하면 됩니다.
DP에서 가장 어려운 부분은 무엇인가요?
대부분 상태 정의입니다. dp[i][j]가 정확히 무엇을 의미하는지 한 문장으로 못 쓰면 점화식도 흔들립니다. 좋은 습관은 "dp[i] = ...의 최댓값/최솟값/개수"처럼 의미를 먼저 고정하고, 마지막 선택이 무엇인지 생각해 점화식을 세우는 것입니다.
DP 공간 최적화는 언제 가능한가요?
현재 상태를 계산할 때 필요한 이전 상태가 제한적이면 가능합니다. 피보나치는 직전 두 값만 필요해서 O(1)로 줄일 수 있고, 배낭 문제는 이전 행만 필요하므로 1차원 배열 O(W)로 줄일 수 있습니다. 단, 1차원으로 줄일 때는 같은 물건을 중복 사용하지 않도록 반복 방향(보통 용량을 큰 쪽에서 작은 쪽으로)을 조심해야 합니다.
