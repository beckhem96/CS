가상 메모리 / MMU

가상 메모리(Virtual Memory)는 각 프로세스에게 자신만의 연속된 메모리 공간이 있는 것처럼 보이게 하는 추상화입니다. MMU(Memory Management Unit)는 CPU가 만든 가상 주소를 실제 RAM의 물리 주소로 변환하는 하드웨어입니다.

구체적인 개념 설명

프로그램은 보통 "내가 0번 주소부터 큰 메모리를 혼자 쓰고 있다"고 생각합니다. 하지만 실제 RAM은 여러 프로세스가 함께 나눠 씁니다. 운영체제와 MMU는 각 프로세스에게 독립된 가상 주소 공간을 제공하고, 실제 물리 메모리의 어디에 매핑되는지는 숨깁니다.

이 변환은 보통 페이지(page) 단위로 이루어집니다. 가상 메모리를 일정 크기(예: 4KB)의 페이지로 나누고, 물리 메모리도 같은 크기의 프레임(frame)으로 나눕니다. 페이지 테이블은 "가상 페이지 번호 → 물리 프레임 번호" 매핑을 저장합니다.

가상 메모리가 주는 세 가지 가치

  • 격리 — 프로세스가 다른 프로세스의 메모리를 함부로 읽거나 쓰지 못하게 합니다.
  • 편의 — 각 프로세스가 연속된 큰 주소 공간을 가진 것처럼 프로그래밍할 수 있습니다.
  • 확장 — 실제 RAM보다 큰 주소 공간을 디스크와 함께 사용하는 demand paging이 가능합니다.

왜 필요한가?

가상 메모리가 없다면 프로그램은 실제 RAM 주소를 직접 다뤄야 하고, 여러 프로그램이 서로의 메모리를 덮어쓰는 사고가 쉽게 발생합니다. 또한 큰 프로그램을 실행하려면 반드시 그 전체가 RAM에 올라와 있어야 합니다.

가상 메모리는 이런 문제를 해결합니다. 프로세스마다 독립된 주소 공간을 제공하고, 필요한 페이지만 실제 메모리에 올리며, 접근 권한을 페이지 단위로 관리합니다. 그래서 OS 보안, 프로세스 격리, 메모리 보호, 동적 로딩, 파일 매핑이 가능해집니다.

어떻게 동작하는가?

1) 주소 변환 흐름

오프셋은 페이지 안 위치이므로 그대로 유지되고, 페이지 번호만 프레임 번호로 바뀐다.

2) 페이지 테이블 예시

가상 페이지 번호(VPN)물리 프레임 번호(PFN)권한상태
05읽기/실행메모리에 있음
19읽기/쓰기메모리에 있음
2—읽기/쓰기디스크에 있음(page fault)
32읽기 전용메모리에 있음

3) TLB — 주소 변환용 캐시

매 메모리 접근마다 페이지 테이블을 RAM에서 읽으면 너무 느립니다. 그래서 MMU 안에는 최근 주소 변환 결과를 저장하는 작은 캐시인 TLB(Translation Lookaside Buffer)가 있습니다. TLB hit이면 페이지 테이블을 거치지 않고 바로 물리 주소를 얻습니다.

TLB는 주소 변환에도 캐시가 필요하다는 사실을 보여준다.

4) Java 코드와 가상 메모리 관점

// Java 객체 생성
byte[] data = new byte[1024 * 1024];

// 개발자는 연속된 배열처럼 사용하지만,
// OS/MMU 관점에서는 여러 가상 페이지가 여러 물리 프레임에 매핑된다.
for (int i = 0; i < data.length; i += 4096) {
    data[i] = 1; // 새 페이지를 처음 건드리면 page fault가 발생할 수 있음
}

// 대용량 메모리 매핑 파일은 가상 메모리 기능을 적극 활용한다.

실무에서 어디에 쓰이는가?

  • 프로세스 격리 — 각 프로세스가 독립 주소 공간을 가져 안정성과 보안이 좋아집니다.
  • 메모리 매핑 파일 — 파일을 메모리처럼 접근하는 mmap, DB의 페이지 캐시.
  • 컨테이너/가상화 — 주소 공간 격리와 2단계 주소 변환.
  • 성능 튜닝 — TLB miss, page fault, huge page 설정이 서버 성능에 영향.
  • JVM — 힙 메모리 예약/커밋, GC와 페이지 접근 패턴 이해에 도움.

면접에 나올 질문과 답변

가상 메모리는 왜 필요한가요?
프로세스마다 독립된 주소 공간을 제공해 서로의 메모리를 침범하지 못하게 하고, 실제 물리 메모리 배치를 숨겨 프로그래밍을 단순하게 만듭니다. 또한 필요한 페이지만 메모리에 올리는 demand paging으로 RAM보다 큰 주소 공간을 다룰 수 있습니다.
MMU는 어떤 역할을 하나요?
CPU가 생성한 가상 주소를 물리 주소로 변환합니다. 이 과정에서 페이지 테이블과 TLB를 사용하고, 페이지 권한을 검사해 읽기/쓰기/실행이 허용되는지도 확인합니다. 권한 위반이나 페이지 부재가 있으면 예외 또는 page fault가 발생합니다.
Page fault는 항상 오류인가요?
아닙니다. 필요한 페이지가 아직 RAM에 없어서 발생하는 정상적인 page fault도 있습니다. OS는 디스크에서 페이지를 읽어 물리 메모리에 올린 뒤 실행을 재개합니다. 다만 접근 권한이 없거나 유효하지 않은 주소라면 segmentation fault 같은 오류로 이어집니다.
TLB가 왜 중요한가요?
모든 메모리 접근마다 페이지 테이블을 조회하면 메모리 접근이 두 번 이상 필요해집니다. TLB는 최근 가상 페이지 → 물리 프레임 변환 결과를 캐싱해 이 비용을 크게 줄입니다. TLB miss가 많으면 캐시 hit가 좋아도 주소 변환 비용 때문에 성능이 떨어질 수 있습니다.
