인터럽트(Interrupt)

인터럽트란 CPU가 현재 실행 중인 일을 잠시 멈추고, 더 급하거나 중요한 이벤트를 처리한 뒤 다시 원래 작업으로 돌아오게 하는 메커니즘입니다. 키보드 입력, 디스크 I/O 완료, 타이머, 네트워크 패킷, 예외 상황 등이 인터럽트를 통해 처리됩니다.

구체적인 개념 설명

인터럽트는 업무 중 걸려오는 긴급 전화와 비슷합니다. 문서를 작성하던 중 화재 경보가 울리면 하던 일을 잠시 멈추고 경보를 처리해야 합니다. 중요한 건 다시 돌아와서 문서 작성을 이어갈 수 있도록, 지금 어디까지 했는지를 저장해둬야 한다는 점입니다.

CPU도 마찬가지입니다. 인터럽트가 들어오면 현재 PC, 상태 레지스터, 필요한 레지스터 값을 저장하고, 인터럽트 벡터 테이블을 통해 알맞은 인터럽트 서비스 루틴(ISR, Interrupt Service Routine)으로 점프합니다. ISR이 끝나면 저장했던 상태를 복원하고 원래 명령어 흐름으로 돌아옵니다.

인터럽트 vs 폴링

폴링은 CPU가 장치 상태를 계속 확인하는 방식이고, 인터럽트는 장치가 필요할 때 CPU에게 알리는 방식입니다. 폴링은 단순하지만 CPU를 낭비할 수 있고, 인터럽트는 효율적이지만 문맥 저장/복원과 우선순위 관리가 필요합니다.

왜 필요한가?

CPU가 모든 장치 상태를 계속 확인하면 대부분의 시간이 기다림에 낭비됩니다. 예를 들어 디스크 읽기는 CPU 연산보다 훨씬 느리기 때문에, CPU가 디스크 완료를 계속 기다리면 전체 시스템 처리량이 크게 떨어집니다.

인터럽트를 사용하면 CPU는 다른 일을 하다가, I/O가 끝나거나 타이머가 울리거나 네트워크 패킷이 도착했을 때만 잠시 처리하면 됩니다. 운영체제의 선점형 스케줄링, 비동기 I/O, 장치 드라이버, 시스템 콜 처리도 모두 인터럽트/트랩 개념과 연결됩니다.

어떻게 동작하는가?

1) 인터럽트 처리 흐름

인터럽트는 실행 흐름을 잠시 다른 코드(ISR)로 옮겼다가 다시 되돌리는 제어 메커니즘이다.

2) 인터럽트 종류

종류발생 원인예시
하드웨어 인터럽트외부 장치가 CPU에 알림키보드 입력, 네트워크 패킷, 디스크 I/O 완료
타이머 인터럽트주기적으로 발생OS 스케줄러가 CPU를 다른 프로세스에 넘김
소프트웨어 인터럽트 / 트랩명령어로 의도적으로 발생시스템 콜
예외(Exception)명령어 실행 중 오류/특수 상황0으로 나누기, page fault, 권한 위반

3) 인터럽트 벡터 테이블

인터럽트가 발생하면 CPU는 인터럽트 번호를 확인하고, 해당 번호에 대응하는 ISR 주소를 인터럽트 벡터 테이블에서 찾습니다. 덕분에 키보드 인터럽트는 키보드 처리 루틴으로, 타이머 인터럽트는 스케줄러 관련 루틴으로 이동할 수 있습니다.

인터럽트 번호가 곧 어떤 처리 루틴으로 점프할지 결정하는 인덱스가 된다.

4) Java 코드와 인터럽트 관점

// Java에서 파일 읽기를 요청하면, 실제로는 OS와 장치 드라이버가 비동기적으로 동작한다.
byte[] data = java.nio.file.Files.readAllBytes(path);

// 단순화한 내부 흐름
// 1. 사용자 프로그램이 시스템 콜로 파일 읽기 요청
// 2. OS가 디스크 컨트롤러에 I/O 명령 전달
// 3. CPU는 다른 작업 실행 가능
// 4. 디스크 I/O 완료 시 하드웨어 인터럽트 발생
// 5. OS의 ISR/드라이버가 완료 상태를 처리
// 6. 대기 중이던 스레드를 다시 실행 가능 상태로 전환

시간/공간 복잡도 또는 trade-off

방식장점단점
폴링구현 단순, 예측 가능CPU가 계속 확인하느라 낭비 가능
인터럽트이벤트가 있을 때만 처리해 효율적문맥 저장/복원, 우선순위, 중첩 처리 복잡
DMA + 인터럽트대량 I/O를 CPU 개입 없이 전송캐시 일관성/버퍼 관리 필요

인터럽트는 효율적이지만 너무 자주 발생하면 오히려 시스템이 인터럽트 처리에 대부분의 시간을 쓰는 interrupt storm 문제가 생길 수 있습니다. 고성능 네트워크에서는 인터럽트와 폴링을 섞은 NAPI 같은 방식도 사용합니다.

실무에서 어디에 쓰이는가?

  • 운영체제 스케줄링 — 타이머 인터럽트로 현재 프로세스를 멈추고 다른 프로세스로 전환합니다.
  • 비동기 I/O — 디스크/네트워크 작업 완료 알림.
  • 장치 드라이버 — 키보드, 마우스, NIC, SSD 같은 장치 이벤트 처리.
  • 시스템 콜 — 사용자 모드에서 커널 모드로 들어가는 제어된 진입점.
  • 페이지 폴트 처리 — 가상 메모리에서 필요한 페이지를 로드하거나 접근 위반을 감지.

면접에 나올 질문과 답변

인터럽트가 발생하면 CPU는 무엇을 하나요?
현재 실행 상태(PC, 상태 레지스터, 필요한 레지스터)를 저장하고, 인터럽트 번호에 맞는 ISR 주소를 찾아 점프합니다. ISR이 이벤트를 처리한 뒤 저장한 상태를 복원하고 원래 프로그램 실행을 이어갑니다.
인터럽트와 예외의 차이는 무엇인가요?
인터럽트는 보통 외부 장치나 타이머처럼 CPU 외부 이벤트로 비동기적으로 발생합니다. 예외는 현재 실행 중인 명령어 때문에 동기적으로 발생합니다. 0으로 나누기, page fault, 권한 위반 등이 예외입니다. 둘 다 CPU 흐름을 ISR/핸들러로 옮긴다는 점에서는 비슷합니다.
폴링 대신 인터럽트를 쓰는 이유는요?
폴링은 CPU가 계속 장치 상태를 확인해야 해서 이벤트가 드문 경우 낭비가 큽니다. 인터럽트는 장치가 필요할 때만 CPU에게 알려주므로 CPU가 다른 일을 할 수 있습니다. 다만 이벤트가 너무 빈번하면 인터럽트 처리 오버헤드가 커져 폴링이나 하이브리드 방식이 더 나을 수 있습니다.
타이머 인터럽트는 왜 중요한가요?
선점형 멀티태스킹의 기반입니다. 타이머가 주기적으로 인터럽트를 발생시키면 OS는 현재 프로세스가 CPU를 너무 오래 독점하지 못하게 하고, 스케줄러를 실행해 다른 프로세스나 스레드로 전환할 수 있습니다.
