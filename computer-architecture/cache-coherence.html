캐시 일관성(Cache Coherence)과 MESI

캐시 일관성은 멀티코어 CPU에서 여러 코어의 캐시가 같은 메모리 값을 서로 다르게 알고 있지 않도록 맞추는 규칙입니다. MESI는 캐시 라인을 Modified, Exclusive, Shared, Invalid 네 상태로 관리하는 대표적인 일관성 프로토콜입니다.

구체적인 개념 설명

싱글코어에서는 CPU 하나가 캐시를 쓰므로 같은 데이터의 복사본이 많아도 비교적 단순합니다. 하지만 멀티코어에서는 Core 1과 Core 2가 같은 메모리 주소를 각자 캐시에 올려둘 수 있습니다. 이때 Core 1이 값을 바꿨는데 Core 2가 예전 값을 계속 읽으면 프로그램은 이상하게 동작합니다.

캐시 일관성은 이런 문제를 막기 위해 한 코어가 캐시 라인을 쓰면 다른 코어의 복사본을 무효화하거나 최신 값을 보게 만드는 약속입니다. 여기서 캐시 라인(cache line)은 캐시가 데이터를 주고받는 기본 단위입니다. 보통 int 하나가 아니라 주변 바이트 묶음이 함께 이동합니다.

일관성(coherence)과 일관된 순서(consistency)는 다르다

캐시 일관성은 같은 주소에 대한 여러 캐시 복사본이 서로 모순되지 않게 하는 문제입니다. 메모리 일관성 모델(memory consistency model)은 여러 주소에 대한 읽기/쓰기 순서가 스레드 사이에서 어떻게 보이는지를 정하는 더 넓은 규칙입니다.

왜 필요한가?

멀티코어 성능은 각 코어가 가까운 캐시를 적극적으로 사용할 때 좋아집니다. 하지만 캐시를 쓰면 같은 메모리 값의 복사본이 여러 곳에 생깁니다. 공유 변수, 락, 큐, 카운터처럼 여러 스레드가 만지는 데이터는 항상 최신 값이 보장되어야 합니다.

캐시 일관성이 없다면 volatile, synchronized, 원자 연산 같은 동시성 도구도 하드웨어 위에서 신뢰할 수 없습니다. 그래서 CPU는 캐시 일관성 프로토콜과 메모리 배리어를 조합해 멀티스레드 프로그램의 기반을 제공합니다.

어떻게 동작하는가?

1) 같은 값의 복사본이 여러 캐시에 생기는 상황

읽기만 할 때는 여러 코어가 같은 값을 공유해도 문제가 없다. 문제는 누군가 쓰기 시작할 때 생긴다.

2) MESI 네 가지 상태

상태의미직관
Modified이 코어만 최신 값을 가지고 있고 메모리는 아직 예전 값내가 고친 원본 초안
Exclusive이 코어만 가지고 있지만 메모리와 값은 같음나만 가진 깨끗한 복사본
Shared여러 코어가 같은 값을 읽기용으로 공유여러 명이 보는 읽기 전용 문서
Invalid더 이상 믿으면 안 되는 캐시 라인폐기된 복사본

3) 쓰기가 일어나면 다른 복사본을 무효화한다

쓰기 권한은 사실상 독점권이다. 다른 코어가 가진 오래된 복사본은 Invalid가 된다.

4) 상태 전이의 큰 흐름

MESI의 핵심은 읽기는 공유하고, 쓰기는 독점권을 얻은 뒤 진행하는 것이다.

5) Java 코드와 false sharing

// 서로 다른 변수처럼 보여도 같은 캐시 라인에 있으면 문제가 될 수 있다.
class Counters {
    volatile long a = 0;
    volatile long b = 0;
}

Counters counters = new Counters();

// Thread 1: counters.a++ 반복
// Thread 2: counters.b++ 반복

// a와 b가 같은 캐시 라인에 있으면,
// 두 스레드는 서로 다른 값을 고치는데도 같은 캐시 라인 소유권을 계속 뺏고 빼앗긴다.
// 이런 현상을 false sharing이라고 한다.

시간/공간 복잡도 또는 trade-off

항목장점trade-off
캐시 일관성멀티코어에서도 공유 데이터의 최신성 보장코어 간 통신, 무효화 메시지 비용
쓰기 무효화쓰기 독점권을 명확히 관리다른 코어가 다시 읽으면 캐시 미스 발생
공유 읽기읽기 많은 데이터는 여러 코어가 빠르게 접근쓰기 발생 시 모든 공유 복사본 관리 필요
false sharing 회피불필요한 캐시 라인 경쟁 감소패딩으로 메모리 사용량 증가 가능

실무에서 어디에 쓰이는가?

  • 멀티스레드 성능 튜닝 — 공유 카운터, 큐, 락 경합에서 캐시 라인 이동 비용을 이해할 수 있습니다.
  • false sharing 분석 — 서로 다른 필드가 같은 캐시 라인에 있어 성능이 떨어지는 문제를 찾습니다.
  • 락과 원자 연산 — CAS, synchronized, volatile이 왜 비싸질 수 있는지 설명하는 기반입니다.
  • NUMA 시스템 — 코어/소켓 사이 데이터 이동이 더 비싸지는 서버 환경을 이해하는 데 필요합니다.
  • 고성능 자료구조 — lock-free queue, ring buffer, work stealing deque 설계에서 중요합니다.

면접에 나올 질문과 답변

캐시 일관성이 왜 필요한가요?
멀티코어에서는 같은 메모리 주소의 값이 여러 코어 캐시에 복사될 수 있습니다. 한 코어가 값을 수정했는데 다른 코어가 오래된 복사본을 읽으면 프로그램 결과가 틀릴 수 있으므로, 하드웨어가 캐시 복사본의 상태를 맞춰줘야 합니다.
MESI의 네 상태는 무엇인가요?
Modified는 이 캐시만 최신 값을 가진 더러운 상태, Exclusive는 이 캐시만 가진 깨끗한 상태, Shared는 여러 캐시가 같은 값을 공유하는 상태, Invalid는 더 이상 사용할 수 없는 상태입니다.
한 코어가 Shared 상태의 캐시 라인에 쓰려면 어떻게 되나요?
먼저 해당 캐시 라인의 독점 소유권을 얻어야 합니다. 이 과정에서 다른 코어가 가진 같은 라인의 복사본은 Invalid가 되고, 쓰는 코어의 라인은 보통 Modified 상태가 됩니다.
false sharing이란 무엇인가요?
논리적으로는 서로 다른 변수를 각 스레드가 수정하지만, 두 변수가 같은 캐시 라인에 있어서 캐시 일관성 트래픽이 계속 발생하는 현상입니다. 실제 공유 데이터가 아닌데도 캐시 라인을 공유하기 때문에 성능이 크게 떨어질 수 있습니다.
