명령어 사이클(Instruction Cycle)

명령어 사이클이란 CPU가 하나의 명령어를 처리하기 위해 반복하는 기본 흐름입니다. 보통 Fetch(가져오기) → Decode(해석) → Execute(실행) → Memory Access(메모리 접근) → Writeback(결과 저장) 단계로 설명합니다.

구체적인 개념 설명

프로그램은 결국 명령어들의 줄입니다. CPU는 그 줄을 한 줄씩 읽고, 무슨 뜻인지 해석하고, 계산하거나 메모리를 읽고, 결과를 저장합니다. 이 반복이 명령어 사이클입니다.

가장 중요한 레지스터는 PC(Program Counter)입니다. PC는 다음에 가져올 명령어의 주소를 들고 있습니다. CPU는 PC가 가리키는 명령어를 가져오고, 보통 PC를 다음 명령어 주소로 증가시킵니다. 단, 분기(branch)나 점프(jump)가 실행되면 PC는 다른 주소로 바뀔 수 있습니다.

실제 CPU는 더 복잡하다

현대 CPU는 명령어를 단순히 하나씩 끝까지 처리하지 않습니다. 파이프라인으로 여러 명령어 단계를 겹치고, 분기 예측, out-of-order execution, speculative execution 같은 기법으로 처리량을 높입니다. 그래도 기본 개념은 Fetch → Decode → Execute 흐름에서 출발합니다.

왜 필요한가?

명령어 사이클을 이해하면 고급 언어 코드가 어떻게 실제 실행으로 이어지는지 감을 잡을 수 있습니다. 예를 들어 반복문 안에서 메모리를 자주 읽는 코드가 느린 이유, 분기문이 많은 코드가 예측 실패로 느려질 수 있는 이유, 함수 호출이 PC와 스택을 바꾸는 작업이라는 사실을 설명할 수 있습니다.

또한 운영체제의 인터럽트, 컨텍스트 스위칭, 파이프라이닝, 캐시 미스 같은 개념은 모두 명령어 실행 흐름 위에서 이해해야 자연스럽습니다.

어떻게 동작하는가?

1) 명령어 사이클 전체 흐름

한 명령어가 끝나면 PC가 가리키는 다음 명령어로 같은 흐름이 반복된다.

2) 예시: LOAD R1, [100]

메모리 주소 100의 값을 레지스터 R1에 가져오는 명령어라고 해봅시다.

단계일어나는 일관련 부품
FetchPC가 가리키는 명령어를 메모리에서 읽어 IR에 저장PC, MAR, MDR, IR
Decode명령어가 LOAD이고 대상이 R1, 주소가 100임을 해석제어 장치
Execute유효 주소 100 계산ALU 또는 주소 계산 유닛
Memory메모리 주소 100에서 값을 읽음캐시/메모리, MDR
Writeback읽은 값을 R1에 저장레지스터 파일

3) 분기 명령어는 PC를 바꾼다

일반 명령어는 끝나면 PC가 다음 주소로 증가합니다. 하지만 if, while, 함수 호출 같은 제어 흐름은 기계어 수준에서 분기/점프 명령어가 되어 PC를 다른 주소로 바꿉니다.

제어 흐름은 결국 PC를 어느 주소로 바꿀지의 문제다.

4) Java 코드와 명령어 사이클

// Java 코드
int sum = 0;
for (int i = 0; i < arr.length; i++) {
    sum += arr[i];
}

// CPU 관점에서 반복되는 일
// 1. i와 arr.length 비교 (CMP)
// 2. 조건 분기: 반복 종료인지 판단 (BRANCH)
// 3. arr[i] 주소 계산 (주소 계산)
// 4. 메모리/캐시에서 arr[i] 읽기 (LOAD)
// 5. sum에 더하기 (ADD)
// 6. i 증가 (ADD)
// 7. 반복 시작 주소로 점프 (JUMP)

시간/공간 복잡도 또는 trade-off

명령어 사이클은 알고리즘 복잡도보다는 CPI(Cycles Per Instruction)와 IPC(Instructions Per Cycle)로 성능을 봅니다.

지표의미성능 해석
CPI명령어 하나당 필요한 평균 클록 사이클낮을수록 좋음
IPC한 사이클에 처리한 평균 명령어 수높을수록 좋음
클록초당 사이클 수높아도 CPI가 크면 느릴 수 있음
캐시 미스메모리 대기 때문에 사이클 낭비CPI를 크게 올림
분기 예측 실패잘못 가져온 명령어를 버림파이프라인 flush 비용 발생

실무에서 어디에 쓰이는가?

  • 성능 튜닝 — 반복문의 분기, 메모리 접근, 캐시 미스가 CPU 사이클을 어떻게 쓰는지 이해.
  • 컴파일러/JIT — 고급 언어 코드를 어떤 명령어열로 바꿀지 최적화.
  • 디버깅 — 어셈블리, stack trace, instruction pointer를 해석할 때 필요.
  • 보안 — speculative execution, side-channel 공격 이해의 출발점.
  • 운영체제 — 인터럽트, 컨텍스트 스위칭, 시스템 콜이 명령어 흐름을 바꾸는 방식 이해.

면접에 나올 질문과 답변

명령어 사이클의 단계를 설명해 주세요.
보통 Fetch, Decode, Execute, Memory Access, Writeback으로 설명합니다. Fetch는 PC가 가리키는 명령어를 가져오는 단계, Decode는 명령어를 해석하는 단계, Execute는 ALU 연산이나 주소 계산을 하는 단계, Memory Access는 필요하면 메모리를 읽고 쓰는 단계, Writeback은 결과를 레지스터에 저장하는 단계입니다.
PC와 IR의 차이는 무엇인가요?
PC는 다음에 가져올 명령어의 주소를 저장합니다. IR은 방금 가져와 현재 실행 중인 명령어 자체를 저장합니다. 즉 PC는 "다음에 어디를 읽을지"를 가리키고, IR은 "지금 읽은 명령어가 무엇인지"를 들고 있습니다.
분기 명령어가 성능에 영향을 주는 이유는요?
현대 CPU는 파이프라인으로 다음 명령어를 미리 가져오며 실행합니다. 분기가 있으면 다음 주소가 어디인지 확정되기 전까지 추측해야 하고, 예측이 틀리면 미리 가져온 명령어들을 버리고 다시 가져와야 합니다. 이 파이프라인 flush 비용 때문에 분기 예측 실패가 성능에 영향을 줍니다.
CPI와 클록 속도 중 무엇이 더 중요한가요?
둘 다 중요합니다. 실행 시간은 대략 명령어 수 × CPI × 클록 주기입니다. 클록이 높아도 캐시 미스나 분기 실패 때문에 CPI가 크면 느릴 수 있고, 클록이 조금 낮아도 IPC가 높고 캐시 효율이 좋으면 더 빠를 수 있습니다.
