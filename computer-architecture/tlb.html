TLB(Translation Lookaside Buffer)

TLB는 최근 사용한 가상 페이지 번호(VPN) → 물리 프레임 번호(PFN) 변환 결과를 저장하는 MMU 내부의 작은 고속 캐시입니다. 주소 변환용 캐시라고 보면 됩니다.

구체적인 개념 설명

가상 메모리에서는 CPU가 만든 가상 주소를 물리 주소로 바꿔야 합니다. 이때 페이지 테이블을 매번 메모리에서 조회하면, 데이터 하나를 읽기 위해 주소 변환용 메모리 접근까지 추가되어 너무 느려집니다. TLB는 이 변환 결과를 캐싱해 "이 가상 페이지는 저 물리 프레임"이라는 답을 빠르게 돌려줍니다.

비유하면, 페이지 테이블은 거대한 주소록이고 TLB는 책상 위에 적어둔 최근 연락처 메모입니다. 방금 전화한 사람에게 다시 전화할 때마다 두꺼운 주소록을 뒤질 필요 없이 메모에서 바로 찾는 셈입니다.

TLB hit / miss

  • TLB hit — 변환 결과가 TLB에 있어 바로 물리 주소를 만든다.
  • TLB miss — 페이지 테이블을 조회해야 한다. 이 과정이 느리고, 필요하면 page walk가 발생한다.

캐시 hit rate가 좋아도 TLB miss가 많으면 메모리 접근이 느려질 수 있습니다.

왜 필요한가?

가상 메모리는 프로세스 격리와 편의성을 제공하지만, 주소 변환 비용이라는 대가가 있습니다. CPU는 거의 모든 load/store마다 주소 변환을 해야 하므로, 이 비용이 크면 성능이 크게 떨어집니다. TLB는 주소 변환 경로에서 가장 자주 쓰는 결과를 가까운 곳에 저장해 이 오버헤드를 줄입니다.

특히 큰 배열을 넓게 건너뛰며 접근하거나, 여러 큰 자료구조를 동시에 순회하면 한정된 TLB 항목으로 커버하지 못하는 페이지가 많아져 TLB miss가 증가합니다. 그래서 고성능 시스템에서는 캐시뿐 아니라 TLB 지역성도 중요합니다.

3) TLB reach

TLB reach는 TLB가 한 번에 커버할 수 있는 메모리 범위입니다. 예를 들어 TLB 엔트리가 64개이고 페이지 크기가 4KB라면, 대략 256KB 범위를 빠르게 변환할 수 있습니다. 작업 집합이 이 범위를 크게 넘으면 TLB miss가 늘어날 수 있습니다.

Huge page는 페이지 크기를 키워 TLB reach를 크게 늘리는 대표적인 방법이다.

4) Java 코드와 TLB 관점

// 큰 배열을 페이지마다 한 번씩 건드리는 패턴
// 4096바이트 간격 접근은 캐시 라인뿐 아니라 페이지/TLB 관점에서도 중요하다.
byte[] data = new byte[1024 * 1024 * 1024];
long sum = 0;

for (int i = 0; i < data.length; i += 4096) {
    sum += data[i];  // 매번 다른 페이지를 건드려 TLB pressure 증가 가능
}

// 연속적으로 촘촘히 접근하면 한 페이지 안에서 여러 캐시 라인을 활용하므로 TLB 효율이 좋다.
for (int i = 0; i < data.length; i++) {
    sum += data[i];
}

실무에서 어디에 쓰이는가?

  • DB 버퍼풀 / 인메모리 캐시 — 큰 메모리 영역을 다룰 때 TLB miss가 지연의 원인이 될 수 있습니다.
  • Huge Page 설정 — JVM, DB, Redis 같은 대용량 메모리 워크로드에서 TLB 효율을 높입니다.
  • 가상화 — 게스트 가상 주소 → 게스트 물리 주소 → 호스트 물리 주소의 2단계 변환에서 TLB가 중요합니다.
  • 컨텍스트 스위칭 — 프로세스 전환 시 TLB flush 비용이 성능에 영향을 줍니다.
  • NUMA/고성능 서버 — 메모리 지역성, 페이지 배치, TLB reach를 함께 고려합니다.

면접에 나올 질문과 답변

TLB는 왜 필요한가요?
가상 주소를 물리 주소로 바꾸려면 페이지 테이블을 조회해야 하는데, 이를 매 메모리 접근마다 수행하면 너무 느립니다. TLB는 최근 주소 변환 결과를 캐싱해 페이지 테이블 조회를 생략하게 해줍니다.
TLB miss가 발생하면 어떻게 되나요?
하드웨어 또는 OS가 페이지 테이블을 조회해 변환 정보를 찾습니다. 페이지가 유효하면 그 결과를 TLB에 넣고 명령을 계속합니다. 페이지 자체가 메모리에 없거나 권한 문제가 있으면 page fault나 예외가 발생합니다.
TLB와 CPU cache는 어떻게 다른가요?
CPU cache는 실제 데이터나 명령어를 캐싱합니다. TLB는 데이터가 아니라 주소 변환 결과(VPN→PFN)를 캐싱합니다. 둘 다 지역성을 활용하지만, 캐싱 대상이 다릅니다.
Huge page가 TLB 성능에 도움이 되는 이유는요?
한 TLB 엔트리가 커버하는 메모리 범위가 페이지 크기만큼 커지기 때문입니다. 4KB 페이지보다 2MB huge page를 쓰면 같은 엔트리 수로 훨씬 넓은 메모리를 변환할 수 있어 TLB miss가 줄어듭니다. 대신 내부 단편화와 운영 복잡도가 생길 수 있습니다.
