CPU 분기 예측(Branch Prediction)

분기 예측은 CPU가 if, while, jump 같은 분기 명령의 결과를 알기 전에 어느 쪽으로 실행될지 미리 추측하는 기법입니다. 파이프라인을 비우지 않고 계속 명령어를 가져오기 위해 필요합니다.

구체적인 개념 설명

파이프라인 CPU는 다음 명령어를 미리 가져와야 처리량이 높습니다. 그런데 분기 명령을 만나면 다음 주소가 두 갈래로 나뉩니다. 조건이 참이면 A로, 거짓이면 B로 가야 하죠. 조건 계산이 끝날 때까지 기다리면 파이프라인이 멈춥니다. 그래서 CPU는 과거 패턴을 보고 "이번엔 아마 이쪽"이라고 예측하고 먼저 실행을 시작합니다.

예측이 맞으면 파이프라인은 계속 흐릅니다. 예측이 틀리면 잘못 가져온 명령어들을 버리고(flush), 올바른 주소에서 다시 가져와야 합니다. 이 비용을 branch misprediction penalty라고 합니다.

예측은 성능 최적화지만, 보안 이슈와도 연결된다

CPU는 예측한 경로를 실제로 먼저 실행해볼 수 있습니다(speculative execution). 나중에 틀렸다고 결과를 버려도, 캐시 상태 같은 미세한 흔적은 남을 수 있습니다. Spectre 계열 공격은 이런 특성을 악용한 대표적인 side-channel 공격입니다.

왜 필요한가?

분기는 프로그램에 매우 자주 등장합니다. 조건문, 반복문, 함수 호출/반환, switch 모두 분기입니다. 파이프라인이 깊은 CPU에서 분기 결과를 기다리면 매번 여러 사이클이 낭비됩니다. 분기 예측은 이 대기 시간을 숨겨 높은 IPC를 유지하는 핵심 장치입니다.

특히 서버 코드처럼 조건 분기가 많은 프로그램에서도 예측 가능한 패턴이면 성능이 좋고, 데이터가 랜덤해 분기 결과가 들쭉날쭉하면 같은 코드라도 느려질 수 있습니다.

어떻게 동작하는가?

1) 예측 성공과 실패

맞으면 이득, 틀리면 flush 비용. 예측 정확도가 성능을 좌우한다.

2) 2-bit saturating counter

가장 유명한 기본 예측기는 각 분기마다 2비트 상태를 두고, 최근 결과에 따라 상태를 조금씩 이동합니다. 한 번 틀렸다고 바로 방향을 바꾸지 않아 루프 끝의 한 번 다른 결과에 덜 흔들립니다.

루프는 대부분 taken이다가 마지막 한 번 not taken이므로 2-bit 예측기가 잘 맞는다.

3) Java 코드와 분기 예측

// 예측 쉬움: 정렬된 데이터는 조건 결과가 한쪽으로 몰림
int count = 0;
for (int x : sortedArray) {
    if (x > 100) count++;
}

// 예측 어려움: 랜덤 데이터는 참/거짓이 불규칙
for (int x : randomArray) {
    if (x > 100) count++;
}

// 실제 성능은 JVM JIT, 벡터화, CPU 모델에 따라 달라지지만,
// 불규칙한 분기는 파이프라인 flush 비용을 만들 수 있다.

시간/공간 복잡도 또는 trade-off

기법장점trade-off
정적 예측단순, 하드웨어 비용 낮음패턴 적응 불가
동적 예측실행 이력으로 정확도 향상예측 테이블/전력/복잡도 증가
BTB분기 대상 주소를 빠르게 예측테이블 충돌 가능
Speculative Execution대기 시간 숨김보안 side-channel 위험

실무에서 어디에 쓰이는가?

  • 성능 최적화 — hot loop의 분기 패턴이 성능에 영향을 줍니다.
  • 컴파일러/JIT — branch layout, profile-guided optimization으로 예측 성공률을 높입니다.
  • 보안 — Spectre, branch target injection 같은 공격 이해에 필요합니다.
  • 데이터 처리 — branchless programming, SIMD가 분기 비용을 줄이는 데 사용됩니다.
  • 시스템 튜닝 — 예측 실패는 perf 같은 프로파일러에서 branch-misses 지표로 관찰할 수 있습니다.

면접에 나올 질문과 답변

분기 예측은 왜 필요한가요?
파이프라인 CPU가 다음 명령어를 계속 가져오려면 분기 결과를 기다릴 수 없습니다. 분기 예측은 다음 경로를 미리 추측해 파이프라인을 채워두고, 맞으면 처리량을 유지합니다. 틀리면 잘못 가져온 명령어를 버리고 다시 시작해야 하므로 비용이 발생합니다.
분기 예측 실패가 왜 비싼가요?
CPU가 예측한 경로의 명령어들을 이미 파이프라인에 넣어둔 상태이기 때문입니다. 예측이 틀리면 이 명령어들을 flush하고 올바른 주소에서 다시 fetch해야 합니다. 파이프라인이 깊을수록 버려야 할 작업이 많아 비용이 커집니다.
2-bit predictor가 1-bit predictor보다 나은 이유는요?
1-bit는 한 번 결과가 바뀌면 바로 예측 방향을 바꿉니다. 루프처럼 대부분 taken이다가 마지막 한 번 not taken인 패턴에서는 루프 종료와 다음 루프 시작에서 연속으로 틀릴 수 있습니다. 2-bit는 강한/약한 상태를 둬 한 번 틀렸다고 바로 방향을 바꾸지 않아 더 안정적입니다.
branchless code는 항상 좋은가요?
항상 그렇지는 않습니다. 분기가 예측 가능하면 일반 분기가 더 빠를 수 있고, branchless 코드는 불필요한 연산을 더 할 수 있습니다. 하지만 분기 결과가 랜덤하고 misprediction이 많다면 조건부 이동, 비트 연산, SIMD 등 branchless 기법이 유리할 수 있습니다.
