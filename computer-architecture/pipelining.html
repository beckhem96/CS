파이프라이닝(Pipelining)

파이프라이닝이란 CPU 명령어 처리 단계를 여러 구간으로 나누고, 여러 명령어가 서로 다른 단계를 동시에 겹쳐 실행하도록 만드는 기법입니다. 한 명령어의 지연 시간(latency)을 줄이기보다는, 단위 시간당 처리량(throughput)을 높이는 구조입니다.

구체적인 개념 설명

파이프라이닝은 자동차 조립 라인과 같습니다. 한 사람이 차 한 대를 처음부터 끝까지 다 만들면 느리지만, 차체 조립, 도색, 엔진 장착, 검사처럼 단계를 나누면 여러 자동차가 동시에 서로 다른 공정을 지나갈 수 있습니다. CPU도 명령어를 Fetch, Decode, Execute, Memory, Writeback 같은 단계로 나누고, 각 단계가 서로 다른 명령어를 처리하게 합니다.

중요한 점은 파이프라인이 각 명령어를 마법처럼 더 빨리 끝내는 게 아니라는 겁니다. 첫 명령어는 여전히 모든 단계를 지나야 하므로 지연 시간은 비슷합니다. 하지만 파이프라인이 가득 차면 매 클록마다 명령어 하나씩 완료될 수 있어 처리량이 크게 올라갑니다.

Latency vs Throughput

Latency는 한 명령어가 시작해서 끝날 때까지 걸리는 시간이고, Throughput은 일정 시간 동안 완료되는 명령어 수입니다. 파이프라이닝은 주로 throughput을 높입니다. 첫 결과는 늦게 나오지만, 이후부터는 공장 라인처럼 계속 결과가 나옵니다.

왜 필요한가?

CPU 내부 부품을 놀리지 않기 위해서입니다. 파이프라인이 없다면 Fetch 단계가 끝난 뒤 Decode가 시작되고, 그동안 Fetch 장치는 놀게 됩니다. 반대로 파이프라인을 쓰면 첫 번째 명령어가 Decode로 넘어간 순간 Fetch 장치는 다음 명령어를 가져올 수 있습니다.

현대 CPU 성능의 핵심은 클록 속도만이 아니라 한 사이클에 얼마나 많은 일을 겹쳐 처리하느냐입니다. 파이프라이닝은 슈퍼스칼라, out-of-order execution, 분기 예측 같은 고급 기법의 기반이 됩니다.

어떻게 동작하는가?

1) 5단계 파이프라인

명령어들이 계단처럼 겹쳐 실행되어 처리량이 높아진다.

2) 파이프라인 해저드(Hazard)

파이프라인은 항상 매끄럽게 흐르지 않습니다. 다음 명령어가 앞 명령어 결과를 기다리거나, 분기 결과를 몰라 다음 명령어를 잘못 가져오는 일이 생깁니다. 이런 장애물을 해저드라고 합니다.

종류의미예시대응
구조적 해저드하드웨어 자원 충돌명령어 fetch와 data load가 같은 메모리 포트 요구자원 분리, stall
데이터 해저드앞 명령어 결과가 아직 준비 안 됨ADD R1,... 다음에 SUB R3,R1,...forwarding, stall
제어 해저드분기 결과를 알기 전 다음 PC를 모름if, while, jump분기 예측, flush

3) 데이터 해저드 예시

데이터 의존성이 있으면 파이프라인이 기다리거나 우회 전달해야 한다.

4) Java 코드와 파이프라인 관점

// 분기가 예측하기 어려운 코드: 데이터에 따라 if 결과가 들쭉날쭉
int sum = 0;
for (int x : arr) {
    if (x % 2 == 0) {     // 예측 실패가 많으면 파이프라인 flush 가능
        sum += x;
    }
}

// 의존성이 긴 코드: 이전 결과를 바로 다음 연산이 사용
int a = 1;
a = a * 3;
a = a + 7;
a = a ^ 11;

// 실제 CPU/JIT는 instruction scheduling, branch prediction 등으로 이를 완화한다.

시간/공간 복잡도 또는 trade-off

이상적인 k단계 파이프라인은 처리량을 최대 k배까지 높일 수 있지만, 실제로는 해저드와 단계 불균형 때문에 그보다 낮습니다.

분기 예측제어 해저드 완화예측 실패 시 flush 비용
Out-of-order 실행독립 명령어를 먼저 실행해 대기 숨김하드웨어 설계 복잡도와 전력 증가

실무에서 어디에 쓰이는가?

  • 성능 튜닝 — 분기 예측, 데이터 의존성, 메모리 대기가 CPU 처리량에 주는 영향 이해.
  • 컴파일러 최적화 — instruction scheduling으로 해저드를 줄입니다.
  • JIT 최적화 — JVM이 루프를 재배치하거나 분기를 줄이는 이유를 이해할 수 있습니다.
  • 보안 — speculative execution 기반 side-channel 공격 이해의 배경입니다.
  • 하드웨어 선택 — GHz만이 아니라 IPC, branch predictor, pipeline depth가 성능에 영향.

면접에 나올 질문과 답변

파이프라이닝은 명령어 실행 시간을 줄이나요?
한 명령어의 latency를 크게 줄이는 기법은 아닙니다. 오히려 한 명령어는 여전히 모든 단계를 거쳐야 합니다. 대신 여러 명령어를 겹쳐 실행해 처리량을 높입니다. 공장 라인처럼 첫 결과는 늦게 나오지만 이후에는 계속 결과가 나오는 구조입니다.
파이프라인 해저드의 종류를 설명해 주세요.
구조적 해저드는 하드웨어 자원 충돌, 데이터 해저드는 앞 명령어 결과를 뒤 명령어가 아직 사용할 수 없는 상황, 제어 해저드는 분기/점프 때문에 다음 PC를 알기 어려운 상황입니다. 각각 자원 추가, forwarding/stall, 분기 예측/flush 등으로 완화합니다.
Stall과 flush의 차이는 무엇인가요?
Stall은 필요한 데이터나 자원이 준비될 때까지 파이프라인 일부를 잠시 멈추는 것입니다. Flush는 잘못 가져온 명령어들을 파이프라인에서 버리는 것입니다. 보통 분기 예측 실패가 나면 잘못된 경로의 명령어를 flush합니다.
분기 예측이 왜 중요한가요?
파이프라인은 다음 명령어를 미리 가져와야 처리량이 유지됩니다. 분기 결과를 기다리면 파이프라인이 비고, 잘못 예측하면 이미 가져온 명령어를 버려야 합니다. 파이프라인이 깊을수록 예측 실패 비용이 커져 분기 예측 정확도가 성능에 큰 영향을 줍니다.
