Q1. 직무 지식·기술 습득 경험 — Java/SQL/Python

🎯 한 줄 핵심 메시지

"Java/SQL은 실행 계획 기반의 성능 튜닝 경험으로, Python은 ETL부터 추천 API 서빙까지 end-to-end 구현으로 깊이를 키웠고, 이 두 축을 진료비 심사 ETL 성능 개선과 운영 자동화에 그대로 옮겨 쓸 수 있다."

📄 자소서 원문 요약

공공 교육 플랫폼에서 통계 대시보드 1분+ 응답 지연을 실행 계획 + DB 로그 분석으로 진단, PreparedStatement 바인드 변수가 옵티마이저의 인덱스 선택을 막아 Full Scan이 발생한 것을 확인. 파라미터 참조 방식을 개선해 1초 내로 단축. Python으로는 Pandas/NumPy로 공공 도서관 데이터 ETL, FastAPI 기반 추천 API를 구현. 입사 후 진료비 심사 데이터 ETL 성능 개선과 운영 자동화·데이터 분석에 활용 예정.

🔥 경험 deep dive 질문

통계 대시보드의 응답 지연을 어떻게 진단했나요? 어떤 도구·로그를 봤는지 구체적으로 설명해 주세요.
먼저 WAS 로그에서 어떤 쿼리가 1분 이상 걸리는지 식별한 뒤, 해당 쿼리를 DB에 직접 붙여 EXPLAIN(또는 Oracle EXPLAIN PLAN)으로 실행 계획을 확인했습니다. 예상은 인덱스를 탄 Index Range Scan이었지만 실제로는 TABLE FULL SCAN이 잡혔고, COST와 ROWS 추정치가 비현실적으로 컸습니다. DB 측의 슬로우 쿼리 로그와 세션 통계로 "이 쿼리가 PreparedStatement로 바인딩되어 들어왔다"는 사실을 확인하고, 바인드 변수의 데이터 분포(특히 NULL 비율)에 따라 옵티마이저가 인덱스를 포기한다는 점을 가설로 세웠습니다.
"파라미터 참조 방식을 개선했다"고 했는데, 정확히 무엇을 어떻게 바꿨는지 설명해 주세요.
문제가 된 쿼리는 검색 조건이 동적이었고, 사용하지 않는 파라미터에는 NULL을 바인딩한 뒤 WHERE col = ? OR ? IS NULL 패턴을 쓰고 있었습니다. 이 형태에서는 옵티마이저가 인덱스 선택성을 판단하기 어려워 Full Scan으로 떨어졌습니다. 개선은 두 단계로 진행했습니다 — ① 조건 분기 SQL을 동적으로 조립해 실제 사용되는 컬럼만 WHERE에 들어가도록 하고, ② 자주 사용되는 키 컬럼은 리터럴 또는 명시적 캐스팅으로 옵티마이저가 선택성을 정확히 추정할 수 있게 했습니다. 그 결과 동일 데이터에서 Index Range Scan으로 전환되며 1분+ → 1초 이내로 단축되었습니다.
ETL 프로세스를 어떻게 설계했고, Pandas와 NumPy를 각각 어디에 활용했나요?
Extract 단계에서 공공 도서관 OpenAPI를 호출해 JSON을 받아오고, Transform에서 Pandas로 결측치 처리·중복 제거·도서/모임 키 정규화를 수행했습니다. 장르 키워드 벡터화나 추천 점수 계산같이 행 단위 연산이 많은 부분은 NumPy 벡터 연산으로 바꿔 for 루프 대비 수십 배 빠르게 만들었습니다. Load는 정제된 데이터프레임을 청크 단위로 DB에 적재했고, 멱등성을 위해 자연키 기준 upsert로 처리했습니다. 실패 시 재실행이 안전하도록 단계별로 idempotent하게 분리한 것이 가장 신경 쓴 부분입니다.
FastAPI를 선택한 이유는 무엇인가요? Flask·Django와 비교해 주세요.
세 가지 이유였습니다. 첫째, 비동기(async/await) 지원이 1급 시민이라 추천 API 호출 시 외부 OpenAPI를 동시에 부르는 데 유리했고, 둘째, Pydantic 기반 자동 스키마 검증과 OpenAPI 문서 자동 생성이 작은 팀에 큰 생산성을 줬습니다. 셋째, Django는 ORM·관리자·인증까지 묶인 풀스택이라 추천 API용으로는 무거웠고, Flask는 가볍지만 타입·문서화를 직접 갖춰야 했습니다. FastAPI는 그 중간에서 "API에만 집중한 가벼운 풀세트"라서 적합했습니다.
진료비 심사 데이터의 대용량 ETL 성능 개선은 구체적으로 어떻게 접근하시겠어요?
먼저 병목 측정이 우선입니다 — 추출·변환·적재 중 어디가 병목인지를 측정해야 잘못된 최적화를 피할 수 있습니다. 그 위에 일반적으로는 ① 행 단위 처리 → 배치 단위(executeBatch·COPY·BULK INSERT)로 전환, ② 인덱스를 일시적으로 비활성화한 뒤 적재 후 재생성, ③ 파티셔닝과 병렬 처리, ④ 변환 단계는 가능한 한 DB 안(SET 연산)에서 처리해 네트워크 왕복을 줄이는 순서로 적용합니다. 장애·재실행을 고려해 모든 단계는 멱등하게 설계하고, 실패 시 체크포인트부터 재개할 수 있게 만듭니다.

🧠 연관 CS 개념 질문

PreparedStatement의 장점이 무엇이고, 왜 SQL Injection을 막을 수 있나요?
쿼리 텍스트와 데이터가 분리되어 DB에 전달되므로, 데이터에 어떤 문자열이 들어와도 SQL의 일부로 해석되지 않습니다. 그래서 '; DROP TABLE users; -- 같은 페이로드도 그저 한 문자열 값일 뿐 실행되지 않습니다. 부수적으로 동일 쿼리의 실행 계획을 캐싱해 재사용할 수 있어 반복 호출 성능에도 이점이 있고, 타입 안전성도 강해집니다.
Full Scan과 Index Scan은 각각 언제 더 유리한가요?
Full Scan은 테이블의 모든 블록을 순차 읽기로 가져오므로, 전체 행의 상당 비율(보통 10~20% 이상)을 읽어야 할 때 오히려 빠릅니다. Index Scan은 인덱스를 거쳐 필요한 행만 랜덤 액세스로 읽으므로, 선택성이 높을 때(전체의 일부만 필요) 유리합니다. 옵티마이저는 통계 정보(행 수, 카디널리티, 데이터 분포)를 보고 둘 중 더 싸 보이는 쪽을 고르므로, 통계가 오래되거나 바인드 변수 분포를 잘못 추정하면 Full Scan으로 떨어지는 사고가 생깁니다.
실행 계획에서 어떤 항목을 가장 먼저 확인하시나요?
① 접근 방식(Seq/Index Scan, 어떤 인덱스를 탔는지), ② 예상 ROWS와 실제 ROWS 차이(통계 신뢰도), ③ JOIN 알고리즘(Nested Loop / Hash / Sort-Merge)과 그 순서, ④ Sort·Hash 같이 메모리·디스크를 쓰는 연산이 있는지, ⑤ 마지막으로 COST의 절댓값이 아니라 단계별 분포를 봅니다. 예상과 실제 ROWS가 크게 어긋나면 통계 갱신 또는 hint·SQL 재작성을 고려합니다.
ETL과 ELT의 차이는 무엇인가요? 심사 데이터에는 어떤 게 더 적합할까요?
ETL은 추출 후 변환을 별도 워커에서 처리한 뒤 적재하는 방식이고, ELT는 일단 적재해 둔 뒤 DB·DW의 SQL 엔진으로 변환하는 방식입니다. 대용량·반복적인 변환 로직과 강력한 DB(예: 컬럼나 DW)가 있다면 ELT가 변환을 병렬·SET 단위로 처리해 유리합니다. 진료비 심사처럼 정합성과 검증 규칙이 복잡하고 규제 추적성이 필요하다면, 적재 전에 검증을 끝내는 ETL이 안전한 경우가 많습니다. 실제로는 두 가지를 섞어 쓰는 하이브리드 형태가 흔합니다.
대용량 처리에서 데이터베이스 인덱스를 어떻게 다루시겠어요?
대량 적재 시에는 인덱스 갱신 비용이 매번 누적되므로, ① 적재 전 인덱스를 비활성화·삭제 → ② 데이터 적재 → ③ 인덱스 재생성 순서가 일반적입니다. 다만 운영 중인 테이블이라면 가용성을 고려해 온라인 인덱스 빌드 옵션을 사용하거나, 파티션 단위로 적재·인덱스 작업을 끊어 진행합니다. 조회 성능을 위해서는 자주 사용되는 WHERE·JOIN 키에 복합 인덱스를 잡고, 카디널리티 순서를 고려해 컬럼 순서를 결정합니다.
추천 알고리즘은 어떤 방식으로 구현했나요?
프로젝트에서는 사용자가 입력한 관심 키워드와 도서 메타데이터(장르·저자·키워드)를 벡터화한 뒤 코사인 유사도로 점수를 매기는 콘텐츠 기반 필터링(Content-based)을 적용했습니다. Cold start 문제(신규 사용자에게 데이터가 없음)를 완화하기 위해 인기 도서·신간을 백오프로 끼워 넣었고, 장기적으로는 사용자 행동 로그가 쌓이면 협업 필터링(Collaborative Filtering)이나 두 가지를 결합한 하이브리드로 발전시킬 계획이었습니다.
Python의 GIL이 있는데 ETL 같은 데이터 처리에서 병렬성을 어떻게 확보할 수 있나요?
GIL은 같은 프로세스 안의 파이썬 바이트코드 실행을 한 번에 한 스레드만 허용하므로 CPU-bound 작업에서 멀티스레딩 효과가 제한됩니다. 대신 multiprocessing이나 concurrent.futures.ProcessPoolExecutor로 별도 프로세스를 띄워 진짜 병렬을 얻을 수 있고, I/O bound 작업(API 호출, DB 적재)은 asyncio로 한 스레드 안에서도 동시성을 충분히 얻습니다. Pandas/NumPy 연산은 내부 C 구현이 GIL을 풀기 때문에 사실상 단일 프로세스로도 빠르게 동작하고, 더 큰 규모는 Dask/PySpark로 확장합니다.
