Q2. 관행 개선 — CLOVA Speech API PoC와 실시간 자막 파이프라인

🎯 한 줄 핵심 메시지

"협력사 STT 모델만 기다리던 관행을 깨고 Naver CLOVA Speech API PoC로 자막 서비스를 재개시켰고, gRPC 양방향 스트리밍 + @Synchronized 임계영역 제어 + lockCanvas 재시도 방어 로직으로 시스템 앱 권한 환경에서 안정적인 실시간 자막 파이프라인을 완성했다."

📄 자소서 원문 요약

ATV(안드로이드 TV 미들웨어 셋톱박스) 자막 서비스 개발 중 협력사 STT 모델 납기가 무기한 지연 → 자막 개발 전면 중단. "협력사 모델만 사용" 전제가 수단에 고착됐다고 판단, 외부 STT API PoC를 팀장께 제안해 진행. 이후 Naver CLOVA Speech API와 gRPC 양방향 스트리밍 기반 실시간 텍스트 변환 파이프라인을 구축하고 멀티스레드 환경의 Race Condition을 @Synchronized 기반 임계영역 제어로 해결, 자막 미지원 서드파티 OTT 앱·방송 채널 위에 시스템 오버레이로 자막을 제공.

🔥 경험 deep dive 질문

팀장께 외부 API 도입을 어떻게 설득하셨나요? 어떤 근거로 PoC 승인을 받았는지 구체적으로 설명해 주세요.
제안서 한 장이 아니라 End-to-End가 동작하는 작은 PoC 결과로 설득했습니다. TV 오디오 출력 → 캡처 → CLOVA Speech API → 화면 자막 렌더링까지 전 구간을 일주일 안에 한 번 돌려 보여 주었습니다. "협력사 모델은 결국 들어올 것이지만 그동안 자막 기능 개발 전체가 멈춰 있는 비용이 더 크다", "API와 협력사 모델 모두를 추상화 계층 뒤에 두면 추후 교체가 가능하다"는 두 논거로 진행 결정을 받았습니다.
CLOVA Speech API와 통신할 때 왜 gRPC 양방향 스트리밍을 선택했나요? 직접 proto 파일을 설계하셨다고 했는데 어떤 점을 신경 썼나요?
실시간 자막은 오디오 청크(512byte PCM 16bit)를 끊임없이 보내고 partial 텍스트를 동시에 받아야 하는 워크로드라 REST의 요청-응답 모델로는 매번 연결 오버헤드가 누적됐습니다. gRPC 양방향 스트리밍은 한 HTTP/2 스트림 위에서 양방향을 동시에 처리해 자연스러웠습니다. proto 설계에서는 oneof를 활용해 첫 메시지는 인식 설정(샘플링레이트·언어), 그 이후는 오디오 데이터로 분기되도록 만들어 설정과 오디오 데이터를 단일 파이프라인에서 처리하도록 통신 구조를 단순화했고, async callback으로 partial 결과를 char-by-char로 자막 렌더링에 흘려보냈습니다.
Race Condition은 정확히 어떤 상황에서 발생했고, @Synchronized 외에 어떤 방어 로직을 추가하셨나요?
ATV는 일반 앱이 아닌 시스템 앱(sharedUserId=systemapp) 권한으로 동작해 SurfaceFlinger에 직접 접근하는 구조였습니다. 사용자가 채널을 변경하는 등 하드웨어 인터럽트가 발생하면 렌더링 스레드와 UI 스레드가 같은 Surface에 동시에 접근해 크래시 위험이 있었습니다. 해결은 두 단계였습니다 — ① @Synchronized로 Surface 접근 임계영역을 엄격히 제어, ② lockCanvas 실패 시 즉시 앱이 종료되지 않도록 최대 20회까지 재시도 후 안전하게 락을 포기하는 방어 로직을 추가했습니다. 이 두 가지로 채널 변경 시점의 크래시 발생률을 0으로 떨어뜨렸습니다.
자막 렌더링에서 메모리 효율은 어떻게 챙겼나요?
매 자막 갱신마다 새 View를 생성하면 GC 압박이 누적되어 셋톱박스의 제한된 자원에서 버퍼링·끊김의 원인이 됩니다. 그래서 3행 × 23열의 그리드 버퍼 구조를 직접 구현해 한 번 만든 View를 재활용하고 글자만 갱신하는 방식으로 메모리 할당을 최소화했습니다. 매번 view inflate를 피하면 GC 횟수가 줄고 첫 자막 표시 latency도 짧아져, 실시간성과 안정성을 동시에 확보했습니다.
"협력사 모델만 써야 한다"는 전제를 깬 결과로 협력사 관계나 보안 이슈는 없었나요?
세 가지를 미리 검토했습니다 — ① 외부 데이터 전송에 대한 법무·정책 확인(고객사 컨펌 필요), ② API 의존이 늘면 비용·SLA가 외부에 종속되므로 인터페이스 추상화로 교체 가능성 확보, ③ 협력사 관계 측면에서 "PoC는 임시 우회"라는 점을 명시. 결과적으로 "교체 가능한 임시 구현 + 추상화" 형태로 만들었기 때문에, 협력사 모델 입고 후에도 어댑터만 바꾸면 되도록 설계했습니다.

🧠 연관 CS 개념 질문

gRPC와 REST의 차이는 무엇이고, 왜 자막 파이프라인에 gRPC가 더 적합했나요?
REST는 텍스트 JSON과 HTTP/1.1을 기반으로 한 요청-응답 모델이라 양방향 스트림에 적합하지 않습니다. gRPC는 HTTP/2 위의 binary protobuf 프로토콜로 multiplexing·스트리밍·strong typing을 1급 시민으로 지원합니다. 오디오 청크를 끊임없이 보내며 partial 텍스트를 동시에 받아야 하는 자막 파이프라인에는 gRPC 양방향 스트리밍이 가장 자연스러웠고, protobuf의 IDL로 클라이언트·서버 코드가 자동 생성되어 인터페이스 어긋남 사고도 줄어듭니다.
gRPC의 4가지 통신 모드를 설명해 주세요.
① Unary — 1 요청 / 1 응답, ② Server streaming — 1 요청 / N 응답(대용량 결과를 청크로), ③ Client streaming — N 요청 / 1 응답(파일 업로드처럼 청크로 보내기), ④ Bidirectional streaming — N 요청 / N 응답(채팅·실시간 STT). ATV에서는 양방향 스트리밍을 사용했고, 첫 메시지에 설정·이후 메시지에 오디오 데이터를 보내는 흐름이었습니다.
protobuf의 oneof가 무엇이고 왜 자막 파이프라인 설계에 활용했나요?
oneof는 한 메시지 안에 여러 필드 후보 중 한 번에 하나만 값이 설정되도록 정의하는 protobuf 기능입니다. ATV에서는 같은 stream으로 첫 메시지에 인식 설정(언어·샘플레이트), 그 이후는 오디오 chunk가 흘러야 했는데, 두 종류 메시지를 별도 메서드로 나누면 stream을 두 개 열어야 해 복잡해졌을 겁니다. oneof로 한 메시지 안에 두 케이스를 두면 단일 파이프라인으로 통신 구조를 단순화할 수 있어 적합했습니다.
Race Condition을 막는 도구로 mutex, semaphore, atomic 변수가 있는데 각각 어떤 차이가 있나요?
뮤텍스(mutex)는 임계영역에 한 번에 한 스레드만 들어가게 하는 잠금이고, 락을 잡은 스레드가 반드시 해제해야 합니다. 세마포어(semaphore)는 동시 진입 가능한 수를 N개로 제한하는 카운팅 잠금으로, 자원 풀 제한이나 producer-consumer에서 유용합니다. atomic 변수는 CAS 같은 명령으로 락 없이 짧은 갱신을 보장하는 도구로, 카운터·플래그처럼 단순 연산에는 atomic, 여러 변수를 함께 보호해야 하면 뮤텍스가 적합합니다. Kotlin의 @Synchronized는 메서드/블록 단위 모니터 락으로, 본질적으로 뮤텍스에 해당합니다.
"임계영역(critical section)"이 무엇이고, 잘못 설정하면 어떤 문제가 생기나요?
공유 자원에 동시 접근하면 안 되는 코드 구간이 임계영역입니다. 잘못 설정하면 ① 영역을 너무 좁게 잡으면 race condition을 못 막고, ② 너무 넓게 잡으면 동시성을 잃어 처리량이 떨어지고 최악의 경우 데드락이 생깁니다. 그래서 "정확성을 보장하는 가장 좁은 범위"를 잡는 게 원칙이고, ATV에서도 @Synchronized 범위를 Surface 접근 구간만으로 좁혀 자막 갱신 throughput을 유지했습니다.
HTTP/2가 HTTP/1.1과 비교해 gRPC에 적합한 이유는 무엇인가요?
HTTP/1.1은 한 연결에서 동시에 하나의 요청만 진행할 수 있어(헤더 블로킹) 양방향 스트리밍이 사실상 불가능합니다. HTTP/2는 한 TCP 연결 위에 여러 스트림을 다중화(multiplexing)할 수 있고, 헤더 압축(HPACK)과 binary framing으로 오버헤드가 작습니다. gRPC는 이 위에서 동작하므로 한 연결로 양방향 스트림을 자연스럽게 구성하고, 셋톱박스처럼 자원이 제한된 환경에서도 효율적입니다.
시스템 앱 권한과 일반 앱 권한의 차이는 무엇이고, 왜 ATV는 시스템 앱이어야 했나요?
일반 앱은 자기 프로세스 안에서만 화면을 그릴 수 있고 다른 앱 위에 그릴 수 없습니다. ATV의 자막 서비스는 자막을 자체 지원하지 않는 서드파티 OTT 앱·방송 채널 위에 오버레이로 자막을 띄워야 했기에, SYSTEM_ALERT_WINDOW·SurfaceFlinger 같은 시스템 권한이 필요했고, sharedUserId=systemapp으로 시스템 앱으로 빌드했습니다. 대신 시스템 권한이 있다는 건 한 번의 크래시가 다른 앱·채널에까지 영향을 미친다는 뜻이라, 안정성에 더 큰 책임이 따랐습니다.
