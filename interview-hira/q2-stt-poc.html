Q2. 관행 개선 — STT API PoC와 실시간 자막 파이프라인

🎯 한 줄 핵심 메시지

"협력사 모델만 사용한다는 관행을 '고객사 일정 준수'라는 더 상위 원칙으로 다시 보고, 외부 STT API PoC → gRPC 스트리밍 파이프라인 → Race Condition 임계영역 제어까지 한 사이클을 직접 끝내 자막 서비스를 완성했다."

📄 자소서 원문 요약

안드로이드 TV 자막 서비스에서 협력사 STT 모델 납기가 무기한 지연 → 자막 개발 전면 중단. "협력사 모델만 사용" 전제가 수단에 고착됐다고 판단, 외부 STT API PoC를 팀장께 제안해 진행. 이후 gRPC 스트리밍 기반 실시간 텍스트 변환 파이프라인을 구축하고 멀티스레드 환경의 Race Condition을 임계영역 제어로 해결해 안정적인 자막 서비스를 완성.

🔥 경험 deep dive 질문

팀장께 외부 API를 제안할 때 어떻게 설득하셨나요? 거절될 가능성도 있었을 텐데요.
제안서가 아니라 2~3일짜리 작은 PoC 결과로 설득했습니다. 외부 API의 인식 정확도, 평균 지연(스트리밍 partial latency), 비용, 그리고 협력사 모델로 갈아탈 때의 인터페이스 변경 비용을 한 페이지로 정리했습니다. "협력사 모델은 결국 들어올 것이지만, 그동안 중단되는 비용이 더 크다", "API와 협력사 모델 모두를 추상화 계층 뒤에 두면 나중에 교체가 가능하다"는 두 가지 논거로 진행 결정을 받았습니다.
PoC를 어떤 기준으로 진행하셨나요? "PoC 성공"의 기준이 무엇이었습니까?
세 가지 정량 기준을 정했습니다 — ① 한국어 음성에 대한 단어 오류율(WER) 일정 수준 이하, ② 첫 partial 결과까지의 지연 1초 이내, ③ 30분 연속 스트리밍에서의 끊김·재연결 횟수 0. 여기에 비기능 기준으로 비용 단가와 정책(데이터 외부 전송 가능 여부)을 함께 검토했습니다. PoC는 "성공·실패"를 미리 정의해 두지 않으면 의견 차이가 끝나지 않으므로, 시작 전 합의된 기준을 두는 게 가장 중요했습니다.
실시간 자막 파이프라인의 전체 흐름을 설명해 주세요.
① TV 마이크/소스에서 PCM 오디오를 캡처해 일정 chunk 단위로 잘라 ② gRPC 클라이언트 스트리밍으로 STT 서버에 업로드, ③ 서버는 partial / final 텍스트를 같은 gRPC 채널로 양방향 스트리밍해 돌려주고, ④ 안드로이드 측 자막 렌더러는 partial은 임시 표시·final은 확정 표시로 갱신했습니다. 오디오 캡처 스레드, 네트워크 송신 스레드, 자막 갱신 UI 스레드가 분리되어 있어 공유 버퍼 접근 동기화가 핵심 이슈였습니다.
Race Condition이 정확히 어떤 상황에서 발생했고, 어떻게 해결했나요?
오디오 캡처 스레드가 링버퍼에 PCM 데이터를 쓰는 동안 네트워크 스레드가 같은 버퍼에서 chunk를 읽어 전송하는 구조였습니다. 둘이 동시에 인덱스(write_idx / read_idx)를 갱신해 chunk가 일부 누락되거나 중복 전송되는 사고가 있었습니다. 해결은 ① 공유 인덱스 갱신을 한정된 임계영역으로 묶고(Java의 synchronized 블록 / Kotlin의 Mutex), ② chunk 단위는 불변 객체로 복사해 일단 큐에 넣고, ③ 큐는 thread-safe한 BlockingQueue로 두는 식의 producer-consumer 패턴으로 재설계했습니다. 그 뒤로는 동기화 범위가 좁아지면서 사고가 사라지고 지연도 개선됐습니다.
"협력사 모델만 써야 한다"는 전제를 깨는 데 부담은 없었나요? 어떤 위험을 검토했나요?
세 가지를 검토했습니다 — ① 음성 데이터가 외부로 나간다는 점에서 개인정보·법무 검토가 먼저 필요했고(고객사 컨펌 필요), ② API 의존이 늘면 비용·SLA·장애가 외부에 종속되므로 인터페이스 추상화로 교체 가능성을 남기고, ③ 협력사 관계 측면에서 "PoC는 임시 우회"라는 점을 명시했습니다. 결과적으로 "교체 가능한 임시 구현 + 추상화"라는 형태로 만들었기 때문에, 협력사 모델 입고 후에도 어댑터만 바꾸면 되게 설계했습니다.

🧠 연관 CS 개념 질문

gRPC와 REST의 차이는 무엇이고, 왜 자막 파이프라인에 gRPC를 선택했나요?
REST는 텍스트 JSON과 HTTP/1.1을 기반으로 한 요청-응답 모델이라 실시간 양방향 스트림에는 적합하지 않습니다. gRPC는 HTTP/2 위의 binary protobuf 프로토콜로 multiplexing·스트리밍·strong typing을 1급 시민으로 지원합니다. 자막처럼 오디오 chunk를 끊임없이 보내고 부분 텍스트를 동시에 받아야 하는 경우 gRPC의 양방향 스트리밍이 자연스러운 선택이었습니다. protobuf의 IDL로 클라이언트·서버 코드가 자동 생성되어 인터페이스 어긋남 사고도 줄어듭니다.
gRPC의 4가지 통신 모드를 설명해 주세요.
① Unary — 1 요청 / 1 응답 (REST의 일반 호출과 유사), ② Server streaming — 1 요청 / N 응답 (대용량 결과를 청크로 받기), ③ Client streaming — N 요청 / 1 응답 (파일 업로드처럼 청크로 보내기), ④ Bidirectional streaming — N 요청 / N 응답 (채팅·실시간 변환). STT는 오디오를 청크로 보내고 partial 텍스트를 받아야 하므로 양방향 스트리밍이 가장 자연스럽습니다.
Race Condition을 막는 도구로 mutex, semaphore, atomic 변수가 있는데 각각 어떤 차이가 있나요?
뮤텍스(mutex)는 임계영역에 한 번에 한 스레드만 들어가게 하는 잠금이고, 락을 잡은 스레드가 반드시 해제해야 합니다. 세마포어(semaphore)는 동시 진입 가능한 수를 N개로 제한하는 카운팅 잠금으로, 자원 풀 제한이나 producer-consumer에서 유용합니다. atomic 변수는 CPU의 CAS 같은 명령으로 락 없이 짧은 갱신을 보장하는 도구로, 카운터·플래그처럼 단순 연산에는 atomic이, 여러 변수를 함께 보호해야 하면 뮤텍스가 적합합니다.
"임계영역(critical section)"이란 무엇이고, 잘못 설정하면 어떤 문제가 생기나요?
공유 자원에 동시에 접근하면 안 되는 코드 구간을 임계영역이라고 합니다. 잘못 설정하면 두 가지 문제가 생깁니다 — ① 영역을 너무 좁게 잡으면 race condition을 못 막고, ② 너무 넓게 잡으면 동시성을 잃어 처리량이 떨어지고 최악의 경우 데드락이 생깁니다. 그래서 "정확성을 보장하는 가장 좁은 범위"를 잡는 게 원칙이고, 잠금 순서를 일관되게 정해 데드락을 예방합니다.
Producer-Consumer 패턴은 무엇이고, 자막 파이프라인에 어떻게 적용됐나요?
한쪽이 데이터를 생산하고 다른 쪽이 소비하는 두 역할을 큐로 분리해, 속도 차이를 흡수하고 동기화를 큐 한 곳으로 모으는 패턴입니다. 자막 파이프라인에서는 오디오 캡처 스레드가 producer, 네트워크 송신 스레드가 consumer가 되고, BlockingQueue(또는 channel)가 그 사이의 버퍼였습니다. 큐 자체가 thread-safe하므로 양쪽 코드에서 직접 락을 잡을 일이 거의 사라지고, 자연스럽게 back-pressure(소비가 느리면 producer가 대기)도 구현됩니다.
HTTP/2가 HTTP/1.1과 비교해 gRPC에 적합한 이유는 무엇인가요?
HTTP/1.1은 한 연결에서 동시에 하나의 요청만 진행할 수 있어(헤더 블로킹), 양방향 스트리밍이 사실상 불가능합니다. HTTP/2는 한 TCP 연결 위에 여러 스트림을 다중화(multiplexing)할 수 있고, 헤더 압축(HPACK)과 binary framing으로 오버헤드가 작습니다. gRPC는 이 위에서 동작하므로 한 연결로 양방향 스트림을 자연스럽게 구성하고, 모바일·임베디드처럼 자원이 제한된 환경에서도 효율적입니다.
실시간 시스템에서 latency를 줄이려면 어떤 점을 고려하시겠어요?
① 네트워크 — 청크 크기와 전송 주기를 최적화(작으면 RTT 누적, 크면 첫 결과 지연), ② 직렬화 — JSON 대신 protobuf 같은 binary 포맷으로 크기·파싱 비용 감소, ③ 스레드 — UI/캡처/네트워크 스레드를 분리하고 임계영역을 짧게 유지, ④ 메모리 — 큰 객체 할당을 줄여 GC 휴지 최소화, ⑤ 측정 — 단계별 지연을 측정해 잘못된 최적화를 피합니다. 실시간은 "어디에 시간이 가는지 측정 가능한 상태"를 만드는 것이 가장 중요합니다.
