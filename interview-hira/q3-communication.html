Q3. 소통 단점 — 기술 결정 사전 공유 습관화

🎯 한 줄 핵심 메시지

"팀 인프라 제약과 운영 맥락을 먼저 묻지 않고 Spring Batch부터 구현하던 단점을, 구현 전에 선택 이유와 대안을 짧게 정리해 먼저 논의하는 습관으로 바꾸었다. 그 결과로 BASS 게이미피케이션 순위 기능을 Crontab + Java 메모리 정렬 + 청크 INSERT로 인프라 제약 안에서 1초 이내 응답을 달성했다."

📄 자소서 원문 요약

BASS 학습 순위 집계 스케줄러를 받았을 때 확장성을 이유로 Spring Batch를 단독 결정해 구현. 그러나 PL은 "이 규모·이 인프라에서는 Crontab 기반 스케줄러가 적합하다"는 피드백을 줬고 전면 재작업이 필요. 이후부터는 구현 전에 선택 이유와 대안을 짧게 정리해 먼저 논의하는 습관으로 바꿔, 재작업 없이 일관된 흐름으로 개발.

💡 각 답변 아래 "🎤 면접 답변" 박스는 그대로 입으로 옮기는 30초 내외의 구어체 스크립트입니다.

🔥 경험 deep dive 질문

왜 Spring Batch가 적합하다고 판단하셨나요? 그 판단의 어떤 부분이 잘못됐다고 보시나요?
장기적으로 데이터가 늘었을 때 chunk 단위 처리·재시작·트랜잭션 관리가 필요할 것이라 보고 Spring Batch를 골랐습니다. Job/Step 모델, ItemReader/Processor/Writer 추상화, 재실행 시 마지막 처리 지점부터 이어가는 메타테이블 구조가 매력이었습니다. 잘못된 부분은 두 가지였습니다 — ① 현재 데이터 규모와 운영팀의 익숙한 도구를 고려하지 않았고, ② BASS는 별도 배치 프레임워크를 신규 도입할 수 없는 인프라 환경이었는데 그 제약을 사전에 확인하지 않은 채 구현부터 시작한 점입니다. 이 두 가지는 사전 30분의 대화로 충분히 알 수 있었던 정보였습니다.
🎤 면접 답변 장기적으로 데이터 늘었을 때 chunk 처리, 재시작, 트랜잭션 관리가 필요할 거라 보고 골랐습니다. 잘못된 부분은 두 가지였는데요, 첫째로 현재 데이터 규모와 운영팀의 익숙한 도구를 고려 안 했고, 둘째로 BASS는 별도 배치 프레임워크 신규 도입 불가 인프라였는데 그 제약을 사전에 확인 안 한 채 구현부터 시작한 점입니다. 사전 30분 대화로 충분히 알 수 있었던 정보였습니다.
PL의 피드백을 받은 뒤 어떻게 재구현하셨나요? 결과는 어땠나요?
Spring Batch 도입을 포기하고 Crontab + Java 애플리케이션 메모리 정렬로 다시 설계했습니다. 매일 새벽 비피크 시간에 누적 점수를 사전 계산해 별도 '순위 테이블'에 적재하고, 화면 조회 시에는 그 테이블을 그대로 읽도록 만들었습니다. DB가 이미 시스템의 주요 병목이었기에 ORDER BY·RANK() 같은 정렬 연산을 DB에 위임하지 않고 총점 → 미션 완료일 → 문제 풀이 수 기준의 다중 조건 정렬·동점자 처리를 Java 애플리케이션 레이어에서 직접 구현했습니다. OOM 방지를 위해 전체 랭킹 데이터를 500건 단위 청크로 나눠 INSERT했고, 결과적으로 사용자가 1초 이내에 일관되게 순위를 조회할 수 있게 됐습니다.
🎤 면접 답변 Crontab으로 스케줄링하고, 매일 새벽 비피크 시간에 누적 점수를 사전 계산해서 별도 순위 테이블에 적재했습니다. DB가 이미 시스템 병목이라 ORDER BY나 RANK()를 DB에 위임하지 않고 총점·미션 완료일·문제 풀이 수 기준의 다중 정렬을 Java 메모리에서 직접 처리했고요. OOM 방지 위해 500건 단위 청크로 나눠 INSERT했습니다. 결과적으로 사용자가 1초 이내에 일관되게 조회할 수 있게 됐습니다.
"의사결정 전 공유"를 어떤 형식으로 하시나요? 너무 무거우면 오히려 안 하게 되지 않나요?
가장 가벼운 형태로는 슬랙 3줄 메모입니다 — "할 것 / 후보 1·2 / 추천과 이유" 한 페이지로 정리해 30분 안에 의견을 받습니다. 조금 더 큰 결정에는 ADR(Architecture Decision Record) 한 장(맥락·결정·결과·대안)을 PR과 함께 남기고, 리뷰어가 결정 자체에도 코멘트를 달 수 있게 합니다. 무게는 되돌리기 비용에 비례하게 두는 것이 핵심입니다 — 작은 결정은 슬랙 한 줄, 큰 결정은 ADR. 너무 무겁게 만들면 자연스럽게 안 쓰게 되어 다시 단독 결정으로 돌아갑니다.
🎤 면접 답변 가장 가벼운 형태로는 슬랙 3줄 메모입니다. "할 것 / 후보 1·2 / 추천과 이유" 정도로 정리해서 30분 안에 의견 받습니다. 큰 결정은 ADR 한 장으로 PR과 함께 남기고요. 무게는 되돌리기 비용에 비례하게 두는 게 핵심이고, 너무 무겁게 만들면 자연스럽게 안 쓰게 되어 다시 단독 결정으로 돌아가니까요.
이 단점이 다시 나오지 않게 만드는 시스템적 장치는 무엇인가요?
① 새 라이브러리·프레임워크 도입은 PR 본문에 의무 섹션으로 "도입 이유 / 대안 / 운영 영향"을 적게 한다, ② 일정 규모 이상의 변경은 ADR을 별도 폴더에 남긴다, ③ 스프린트 시작 시 kick-off 30분 동안 큰 결정을 미리 합의한다, ④ 회고에서 "사전 합의 없이 진행한 결정"의 사례를 식별해 공유한다. 장치가 없으면 결국 개인 의지에 기대게 되므로, 팀 워크플로 자체에 단계를 끼워 넣는 게 중요합니다.
🎤 면접 답변 PR 본문에 "도입 이유 / 대안 / 운영 영향"을 의무 섹션으로 넣고, 큰 변경은 ADR을 별도 폴더에 남깁니다. 스프린트 시작 시 30분 kick-off로 큰 결정을 미리 합의하고, 회고에서 사전 합의 없이 진행한 사례를 식별해 공유합니다. 개인 의지에만 기대지 않고 팀 워크플로 자체에 단계를 끼워 넣는 게 중요합니다.
팀의 결정과 본인의 기술적 신념이 다를 때 어떻게 행동하시나요?
먼저 결정 근거를 한 번 더 확인합니다 — 팀 이유가 명확하면 동의하고 빠르게 따릅니다. 그래도 신념과 다르다면 ① 팀 결정을 일단 따라 일정·약속을 지키고, ② 결정의 결과(성능, 운영 사고 빈도)를 데이터로 모은 뒤, ③ 다음 회고에서 데이터를 근거로 재논의합니다. 의견 차이를 즉석 논쟁으로 풀지 않고 시간차 데이터로 풀어내는 것이 협업 비용을 낮춘다고 생각합니다.
🎤 면접 답변 먼저 팀 결정 근거를 한 번 더 확인하고, 명확하면 동의하고 빠르게 따릅니다. 그래도 다르면 일단 결정을 따라 일정·약속을 지키고, 결과를 데이터로 모은 뒤 다음 회고에서 데이터 근거로 재논의합니다. 의견 차이를 즉석 논쟁이 아니라 시간차 데이터로 푸는 게 협업 비용을 낮춘다고 봅니다.

🧠 연관 CS 개념 질문

Spring Batch의 핵심 구성요소를 설명해 주세요.
Job이 가장 큰 단위로, 그 안에 여러 개의 Step이 순서대로 실행됩니다. 각 Step은 chunk 지향이라면 ItemReader → ItemProcessor → ItemWriter의 세 단계로 데이터를 chunk 크기 단위로 처리합니다. Spring Batch는 JobRepository(메타테이블)에 실행 이력을 기록해 실패 시 마지막 처리 지점부터 재시작할 수 있고, chunk 단위로 트랜잭션이 묶여 부분 실패 시 해당 chunk만 롤백됩니다. 대용량·장기 실행 배치에는 큰 가치를 주지만, BASS처럼 인프라 제약과 운영 부담이 있는 환경에는 과한 도구일 수 있습니다.
🎤 면접 답변 Job이 가장 큰 단위고 그 안에 여러 Step이 순차 실행됩니다. 각 Step은 chunk 지향이면 ItemReader → ItemProcessor → ItemWriter 세 단계로 chunk 단위 처리하고요. JobRepository 메타테이블에 실행 이력 남겨서 실패 시 마지막 처리 지점부터 재시작 가능합니다. 대용량·장기 실행에는 큰 가치인데, BASS 같은 인프라 제약 환경엔 과한 도구일 수 있습니다.
Crontab의 표현식과 동작 원리를 설명해 주세요.
Crontab은 분/시/일/월/요일 5필드(또는 초까지 6필드)로 시간 패턴을 표현하고, cron 데몬이 매분 표현식을 평가해 일치하는 작업을 실행합니다. 예: 0 3 * * 1-5는 평일 오전 3시 정각, */10 * * * *는 10분 간격. 장점은 OS 표준이라 학습 곡선이 짧고 외부 의존이 없다는 것이고, 한계는 작업 간 의존성·실패 처리·재실행을 직접 구현해야 하고 단일 서버에 묶인다는 점입니다. BASS에서는 새벽 비피크 시간에 한 번 트리거되어 Java 애플리케이션이 실제 처리·청크 적재를 책임지는 구조였습니다.
🎤 면접 답변 분/시/일/월/요일 5필드로 시간 패턴을 표현하고 cron 데몬이 매분 평가해서 일치하는 작업을 실행합니다. OS 표준이라 학습 곡선이 짧고 외부 의존이 없는 게 장점이고, 작업 간 의존성·실패 처리·재실행은 직접 구현해야 하고 단일 서버에 묶이는 게 한계입니다. BASS에서는 새벽 비피크에 한 번 트리거되어 Java가 실제 처리·청크 적재를 책임지는 구조였습니다.
"DB에 ORDER BY와 RANK()를 위임하지 않고 Java 메모리에서 정렬"한 결정의 trade-off는 무엇인가요?
장점은 ① 이미 병목이던 DB의 CPU·메모리 부하를 WAS로 오프로딩, ② 총점 → 미션 완료일 → 문제 풀이 수 같은 다중 조건 + 도메인 동점자 처리 로직을 SQL보다 Java 코드로 표현하는 게 명확, ③ 새벽 비피크 배치라 WAS가 한가한 시간을 활용 가능. 단점은 ① 데이터 양이 더 커지면 OOM 위험이 생기므로 청크 분할이 필수, ② 정렬 알고리즘이 DB의 최적화된 sort보다 느릴 수 있다. BASS에서는 500건 청크 INSERT로 OOM을 방지해 trade-off를 관리했습니다.
🎤 면접 답변 장점은 DB 부하를 WAS로 오프로딩하고, 다중 조건과 동점자 처리 같은 도메인 로직을 Java로 표현하는 게 명확하다는 점입니다. 새벽 비피크라 WAS도 한가하고요. 단점은 데이터가 더 커지면 OOM 위험이 생기는 거라 청크 분할이 필수였고, BASS는 500건 청크로 OOM 방지하면서 trade-off를 관리했습니다.
분산 환경에서 Crontab만으로는 어떤 문제가 생기나요?
① 중복 실행 — 같은 cron이 여러 노드에서 동시에 실행되어 데이터가 두 번 처리될 수 있고, ② 리더 장애 — 한 노드만 cron을 가졌다면 그 노드 장애 시 작업이 누락, ③ 관측성 부족 — 분산 노드의 cron 로그를 한 곳에 모으기 어려움. 해결로는 분산 락(Redis·DB·Zookeeper)으로 단일 실행 보장, Kubernetes CronJob처럼 클러스터 차원의 스케줄러, Quartz의 클러스터 모드 같은 도구를 씁니다.
🎤 면접 답변 여러 노드에서 같은 cron이 동시 실행되어 데이터가 두 번 처리되는 중복 실행, 한 노드만 cron 갖고 있으면 그 노드 장애 시 작업 누락, 분산 노드의 cron 로그를 한 곳에 모으기 어려운 관측성 문제가 있습니다. 해결로는 분산 락이나 Kubernetes CronJob, Quartz 클러스터 모드 같은 도구를 씁니다.
YAGNI와 KISS 원칙은 무엇이고, 이번 사례에 어떻게 적용되나요?
YAGNI(You Aren't Gonna Need It)는 "지금 필요하지 않은 기능은 만들지 말라", KISS(Keep It Simple, Stupid)는 "가능한 한 단순하게 유지하라"는 원칙입니다. 이번 사례는 "미래의 확장성"이라는 가정에 무게를 두고 현재 규모와 인프라 제약에 비해 과한 도구(Spring Batch)를 선택한 YAGNI 위반에 가까웠습니다. 교훈은 "확장이 필요해질 때 도구를 들이는 비용 vs 지금 들이는 비용"을 먼저 비교하는 것이고, 보통은 후자가 훨씬 큽니다.
🎤 면접 답변 YAGNI는 지금 필요하지 않은 건 만들지 말라는 원칙, KISS는 가능한 한 단순하게 유지하라는 원칙입니다. 이번 사례는 미래의 확장성에 무게를 두고 현재 규모·인프라 제약에 비해 과한 도구를 선택한 YAGNI 위반에 가까웠고요. 교훈은 확장이 필요해질 때 도구 들이는 비용이랑 지금 들이는 비용을 먼저 비교해야 하고, 보통은 후자가 훨씬 큽니다.
ADR(Architecture Decision Record)이 무엇이고, 왜 도입할 가치가 있나요?
ADR은 중요한 아키텍처 결정을 한 페이지에 맥락·결정·결과·대안 형식으로 남기는 문서입니다. 가치는 ① 6개월 뒤 "왜 이렇게 결정했지?"를 빠르게 답할 수 있고, ② 신규 멤버에게 도메인 결정의 맥락을 빠르게 전달하며, ③ 결정 근거가 데이터화되어 재논의가 감정적이지 않아진다는 점입니다. 대신 너무 무겁게 만들면 안 쓰게 되므로 "한 페이지·간결" 규칙을 지키는 것이 핵심입니다.
🎤 면접 답변 맥락·결정·결과·대안을 한 페이지로 남기는 문서입니다. 6개월 뒤에 왜 이렇게 결정했는지 빠르게 답할 수 있고, 신규 멤버에게 도메인 결정 맥락을 빠르게 전달할 수 있고요. 결정 근거가 데이터화되어서 재논의가 감정적이지 않게 됩니다. 단 너무 무겁게 만들면 안 쓰게 되니까 한 페이지·간결 규칙이 핵심입니다.
학습 순위를 SQL 한 줄로 짠다면 어떻게 짜시겠어요?
윈도우 함수 RANK() 또는 DENSE_RANK()를 활용합니다 — 예를 들어 
SELECT user_id, total_score,
       RANK() OVER (PARTITION BY scope_id ORDER BY total_score DESC) AS rnk
FROM   user_scores
WHERE  period = :daily_key
결과를 일별 스냅샷 테이블에 적재해 변경 이력을 남깁니다. 다만 BASS에서는 DB가 이미 병목이었기에 이 정렬을 Java 메모리에서 처리하는 결정을 내렸고, 인덱스는 (period, scope_id, total_score DESC) 복합 인덱스를 적용해 조회 성능을 확보했습니다.
🎤 면접 답변 RANK나 DENSE_RANK 윈도우 함수로 PARTITION BY scope_id ORDER BY total_score DESC로 순위를 뽑고, 결과를 일별 스냅샷 테이블에 적재합니다. 다만 BASS는 DB가 병목이라 이 정렬을 Java 메모리에서 처리했고, 인덱스는 (period, scope_id, total_score DESC) 복합으로 잡았습니다.
