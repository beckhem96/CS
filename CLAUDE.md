# CS 개념 학습 웹페이지

## 프로젝트 개요

CS(Computer Science) 핵심 개념을 **시각화와 직관적인 설명**으로 학습할 수 있도록 정리한 **로컬 정적 웹사이트**입니다.

- **목적**: 개인 학습용 CS 개념 정리 및 복습
- **사용자**: 본인 (로컬 환경에서만 실행)
- **배포**: 없음. 브라우저로 `index.html`을 직접 열어서 사용

## 기술 스택

순수 정적 웹페이지로 작성합니다. **빌드 도구, 프레임워크, 패키지 매니저는 사용하지 않습니다.**

- **HTML5**: 시맨틱 마크업
- **CSS3**: Vanilla CSS (외부 프레임워크 X)
- **JavaScript**: Vanilla JS (필요한 경우에만 최소한으로)
- **시각화**: 인라인 SVG 우선, 복잡한 인터랙션은 Canvas, 다이어그램은 Mermaid CDN

> ❗️ React, Vue, Tailwind, Bootstrap, npm 등은 사용하지 않습니다. 로컬에서 파일을 더블클릭만 해도 동작해야 합니다.

## 디렉토리 구조

```
.
├── CLAUDE.md
├── index.html                    # 홈 (카테고리 선택)
├── assets/
│   ├── css/

│   │   └── concept.css           # 개념 페이지 전용 스타일
│   └── js/
│       └── common.js             # 공통 JS (네비, 다크모드 등)
├── data-structure/               # 자료구조
│   ├── index.html                # 카테고리 페이지
│   ├── array.html
│   ├── linked-list.html
│   └── ...
├── algorithm/                    # 알고리즘
│   ├── index.html
│   └── ...
├── computer-architecture/        # 컴퓨터구조
├── operating-system/             # 운영체제
```

각 카테고리 폴더는 영문 kebab-case로 작성합니다. 개념 페이지 파일명도 동일한 규칙입니다 (예: `binary-search-tree.html`).

## 페이지 구조 규칙

### 1. 홈 페이지 (`index.html`)

- 8개 카테고리를 **카드 그리드** 형태로 배치
- 각 카드에는 카테고리명 + 한 줄 설명 + 대표 아이콘(이모지 또는 인라인 SVG)
- 카드 클릭 시 해당 카테고리의 `index.html`로 이동

### 2. 카테고리 페이지 (`{category}/index.html`)

- 상단: 카테고리명, 한 문단 분량의 카테고리 소개
- 본문: 해당 카테고리에 속한 개념 페이지들의 **링크 목록**
  - 개념 그룹별로 섹션을 나눔 (예: 자료구조 → "선형 구조", "비선형 구조", "해시 기반")
  - 각 링크 옆에 한 줄 요약 표시
- 상단 네비: "← 홈으로" 링크 필수

### 3. 개념 페이지 (`{category}/{concept}.html`)

각 개념 페이지는 **다음 구조**를 따릅니다:

1. **breadcrumb 네비게이션** (홈 > 카테고리 > 개념)
2. **개념 한 줄 정의** (가장 위, 강조 박스)
3. **구체적인 개념 설명**(자세하고 이해하기 쉽게 설명)
4. **왜 필요한가?** (배경/문제 상황)
5. **어떻게 동작하는가?** (시각화 + 단계별 설명)
6. **시간/공간 복잡도** 또는 **trade-off** (해당되는 개념에 한해)
7. **실무에서 어디에 쓰이는가?** (구체적 예시)
8. **면접으로 나올 질문과 답변**
9. **관련 개념** (같은 카테고리 내 다른 페이지로 링크)
10. **하단 네비**: 이전 개념 / 다음 개념 / 카테고리로 돌아가기

## 콘텐츠 작성 가이드

### 톤 & 매너

- **모든 텍스트는 한국어**로 작성
- 학습자에게 설명하듯 **친근하고 구어체에 가까운 말투**
- 어려운 용어가 처음 등장할 때는 **반드시 풀어서 설명**
- 영문 원어 용어는 처음 등장 시 한국어 옆에 병기 (예: 해시 테이블(Hash Table))

### 설명 방식

- **결론 먼저**, 그다음 이유와 동작 원리
- **비유와 일상 예시**를 적극 활용 (배열 = 사물함, 큐 = 줄서기, 캐시 = 책상 위 자주 보는 책 등)
- 코드 예시는 **Java**으로 작성 (백엔드 개발자 관점에서 친숙한 언어)
- 한 페이지에 너무 많은 내용을 넣지 말고, 개념 하나에 집중

### 시각화 가이드

**시각화는 이 프로젝트의 핵심**입니다. 텍스트만으로 끝내지 마세요.

- **자료구조/알고리즘**: 인라인 SVG로 노드, 화살표, 상태 변화 표현
  - 단계별 상태가 있다면 여러 개의 SVG를 나열해서 변화를 보여줄 것
  - 가능하면 간단한 JS 애니메이션 추가 (선택)
- **시스템 다이어그램** (운영체제, 네트워크, 컴퓨터구조): Mermaid 또는 SVG
- **계층/플로우** (TCP/IP 4계층, OS 계층 등): SVG로 박스와 화살표
- **표/매트릭스** (시간복잡도 비교, 격리 수준 비교 등): 시맨틱 `<table>` + CSS

> SVG는 외부 파일로 빼지 말고 **HTML 안에 인라인**으로 작성하세요. 검색·수정이 쉬워야 합니다.

## 스타일 가이드

- **다크 모드 친화적**인 차분한 색상 (배경 #1a1a1a 또는 #fafafa 중 선택, 일관성 유지)
- **본문 가독성 우선**: 본문 폰트 16~17px, 줄 간격 1.7
- **한국어 폰트**: `system-ui, -apple-system, "Pretendard", "Noto Sans KR", sans-serif`
- **코드 폰트**: `"D2Coding", "JetBrains Mono", monospace`
- **최대 콘텐츠 너비**: 760~820px (긴 줄 가독성 저하 방지)
- 강조 박스, 인용, 경고 박스 등 **반복되는 컴포넌트는 `common.css`에 정의**해서 재사용

## 카테고리별 다룰 핵심 개념 (초안)

처음부터 다 만들 필요는 없고, 새로운 개념 페이지를 추가할 때 이 목록을 참고합니다.

- **자료구조**: 배열, 연결 리스트, 스택, 큐, 덱, 해시 테이블, 트리(이진/이진탐색/AVL/B-Tree), 힙, 그래프, 트라이
- **알고리즘**: 정렬(버블/병합/퀵/힙), 탐색(이진/DFS/BFS), 동적 계획법, 그리디, 분할정복, 백트래킹, 다익스트라, 유니온파인드
- **컴퓨터구조**: CPU 구성, 파이프라이닝, 캐시(L1/L2/L3), 가상 메모리, MMU, 명령어 사이클, 인터럽트
- **운영체제**: 프로세스/스레드, 컨텍스트 스위칭, 스케줄링, 동기화(뮤텍스/세마포어), 데드락, 페이징/세그먼테이션, 파일 시스템
- **네트워크**: OSI 7계층, TCP/IP 4계층, TCP/UDP, 3-way handshake, HTTP/HTTPS, DNS, 라우팅, NAT, CORS
- **데이터베이스**: 정규화, 인덱스(B+Tree), 트랜잭션 ACID, 격리 수준, 락, MVCC, 조인 알고리즘, 실행 계획
- **소프트웨어공학**: SDLC, 애자일/스크럼, 디자인 패턴, SOLID, 테스트 전략, 코드 리뷰, CI/CD
- **데이터통신**: 변조(AM/FM), 다중화(TDM/FDM), 오류 검출(패리티/CRC), 매체 접근(CSMA/CD), 신호 인코딩

## 새 페이지 추가 시 작업 순서

1. **카테고리 페이지(`{category}/index.html`)에 링크 먼저 추가** (한 줄 요약 포함)
2. 개념 페이지 HTML 파일 생성 (위의 "개념 페이지 구조" 8단계 준수)
3. 시각화는 **반드시 1개 이상** 포함 (SVG 우선)
4. 같은 카테고리 내 관련 개념 페이지가 있다면 **상호 링크** 추가
5. 작성한 페이지를 브라우저에서 직접 열어 깨지는 부분 확인

## 작업 시 주의사항

- **외부 CDN, 외부 폰트 다운로드는 최소화**합니다. 오프라인에서도 동작해야 합니다 (Mermaid 정도는 예외 허용).
- **코드 블록은 반드시 syntax highlighting** 없이도 읽기 좋게 들여쓰기만 정확히 맞춥니다.
- 개념 설명 시 **틀린 정보를 추측해서 쓰지 않습니다**. 애매하면 명시적으로 "확인 필요" 주석을 남겨주세요.
- **이미지 파일을 새로 만들지 말고**, 시각화는 SVG로 직접 그립니다 (수정·검색 가능성 때문).
- 한 PR/작업 단위에서 **너무 많은 페이지를 한 번에 만들지 말고**, 1~3개 개념씩 깊이 있게 작성합니다.
