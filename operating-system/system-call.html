시스템 콜 — 유저 모드와 커널 모드의 경계

시스템 콜(system call)은 응용 프로그램이 커널의 기능(파일 입출력, 네트워크, 메모리 할당 등)을 빌리기 위해 호출하는 약속된 통로입니다. CPU는 시스템 콜이 발생하면 유저 모드에서 더 큰 권한을 가진 커널 모드로 전환되어, 커널의 코드를 실행한 뒤 다시 유저 모드로 돌아옵니다.

구체적인 개념 설명

현대 OS는 응용 프로그램이 하드웨어를 직접 만지는 것을 허용하지 않습니다. 디스크 한 블록을 읽으려면 디스크 컨트롤러의 정확한 명령을 알아야 하고, 그것이 잘못되면 시스템 전체가 망가질 수 있기 때문입니다. 그래서 OS는 "이렇게 부르면 그 일을 대신 해 주겠다"는 약속된 진입점을 제공하고, 그것을 시스템 콜이라고 부릅니다.

이 진입점이 단순한 함수 호출과 다른 점은, 호출하는 순간 CPU의 권한 모드가 바뀐다는 점입니다. 유저 모드에서는 자기 프로세스의 메모리만 만질 수 있지만, 커널 모드에서는 모든 물리 메모리·하드웨어·다른 프로세스의 자원을 다룰 수 있습니다. 이 권한 차이를 하드웨어가 직접 지원해 주기에 시스템 콜은 안전성과 성능 사이의 경계가 됩니다.

"라이브러리 함수 호출"과 "시스템 콜"은 다른 층입니다

printf는 라이브러리 함수입니다. 내부적으로 결국 write()라는 시스템 콜을 호출하지만, 호출 자체는 사용자 모드에서 끝납니다. write()가 진짜 시스템 콜로 들어가는 순간에만 모드 전환과 커널 코드 실행이 일어납니다.

왜 필요한가?

"유저 모드만 있고 시스템 콜이 없다면" 어떻게 될지 상상해 봅시다. 응용 프로그램이 디스크에 직접 접근할 수 있다면 한 프로그램의 버그가 다른 프로그램의 파일을 망가뜨릴 수 있습니다. 그래서 OS는 두 가지를 보장해야 합니다.

  • 보호 — 응용은 자기 메모리·자기 권한 안에서만 동작해야 한다.
  • 중재 — 디스크·네트워크·메모리 할당 같은 자원은 OS가 중간에서 정책에 따라 분배해야 한다.

이 두 가지를 동시에 만족시키는 가장 간단한 방법이 "권한 모드를 둘로 나누고, 모드 전환은 정해진 진입점으로만 허용한다"는 설계입니다. 그 정해진 진입점이 시스템 콜입니다. 모드 전환의 비용 때문에 시스템 콜은 비싸지만, 그 비용이 곧 안전성의 가격입니다.

어떻게 동작하는가?

1) 유저 모드와 커널 모드 — 권한 링

x86은 4개의 링(0~3)을 정의하지만, 실제 OS는 보통 0(커널)과 3(유저)만 사용한다.

2) 시스템 콜의 흐름 — 트랩과 복귀

한 번의 read() 호출도 7단계의 모드 전환과 디스패치를 거친다.

3) 시스템 콜 진입 명령어의 변천

명령어등장 시기특징
int 0x80구식 (i386 Linux)소프트웨어 인터럽트로 트랩 — 비교적 느림
sysenter / sysexitPentium II~ (Intel)전용 명령으로 모드 전환 비용 절감
syscall / sysretx86-64 표준가장 빠른 진입, 64비트 리눅스의 기본
svcARMSupervisor Call — ARM의 트랩 명령

어떤 명령이든 핵심은 같습니다 — 유저 코드가 자발적으로 트랩을 일으켜 커널 진입점으로 점프하고, CPU 모드 비트가 0(커널)으로 바뀐다는 점입니다. 모드 비트는 하드웨어가 강제하므로 유저 코드가 임의로 바꿀 수 없습니다.

4) 라이브러리 래퍼 — 우리가 직접 syscall을 부르지 않는 이유

대부분의 언어에서 read()·open()·write()는 시스템 콜처럼 보이지만, 실제로는 libc(또는 그 언어의 런타임)의 함수입니다. 이 함수가 시스템 콜 번호를 레지스터에 넣고, 인자를 정해진 레지스터에 옮긴 뒤 syscall 명령을 실행합니다. libc 래퍼는 호출 규약 통일·errno 변환·시그널 안전 처리를 함께 해 주므로, 직접 syscall을 부르는 건 매우 드뭅니다. 

// glibc의 read 래퍼(개념적으로)
ssize_t read(int fd, void *buf, size_t n) {
    long ret = syscall(SYS_read, fd, buf, n);
    if (ret < 0) {
        errno = -ret;   // 음수 → errno로 옮기기
        return -1;
    }
    return ret;
}

5) vDSO — 시스템 콜이 아닌 시스템 콜

gettimeofday 같은 자주 쓰이지만 단순한 호출은, 매번 모드 전환을 하는 게 너무 비쌉니다. 그래서 리눅스는 vDSO(virtual Dynamic Shared Object)라는 특별한 공유 라이브러리를 모든 프로세스의 주소 공간에 매핑해 두고, 커널이 갱신해 두는 페이지를 유저 모드에서 그대로 읽도록 합니다. 겉모습은 시스템 콜처럼 호출하지만 실제로는 모드 전환이 일어나지 않습니다.

6) 시스템 콜은 얼마나 비싼가

  • 한 번의 시스템 콜은 보통 수십 ~ 수백 나노초의 모드 전환 비용을 발생시킵니다.
  • 모드 전환 자체보다 캐시·TLB·파이프라인 오염이 더 큰 비용일 때가 많습니다.
  • 그래서 시스템 콜은 적게, 한 번에 큰 일을 하도록 설계해야 합니다 — readv, writev, epoll, io_uring이 모두 같은 방향의 노력입니다.

trade-off

설계 결정장점비용
커널을 별도 모드로 격리응용 버그가 시스템을 망가뜨리지 못함호출마다 모드 전환 비용
시스템 콜을 잘게 (read, write …)API가 단순하고 직관적호출 횟수 증가 → 누적 비용 큼
벡터/배치 시스템 콜 (writev, io_uring)한 번에 여러 작업, 진입 비용 분산API 복잡도 증가, 학습 비용
vDSO로 일부 호출 우회매우 빠른 시간 조회제한된 호출만 가능, 커널-유저 데이터 동기화 주의

실무에서 어디에 쓰이는가?

  • strace — 프로세스가 호출하는 시스템 콜을 그대로 찍어 디버깅에 쓰입니다. "왜 이 프로그램이 느리지?"의 첫 단서입니다.
  • perf / bcc / bpftrace — 시스템 콜 단위로 호출 빈도·지연을 추적합니다.
  • 고성능 네트워크 서버 — accept/read/write를 잘게 부르지 않고, epoll·io_uring으로 진입 횟수를 줄입니다.
  • 컨테이너 보안 — seccomp는 허용된 시스템 콜만 통과시키고 나머지는 차단합니다. Docker·Kubernetes도 기본 시스템 콜 화이트리스트를 적용합니다.
  • Java/Python 등 런타임 — JVM의 GC·스케줄러는 가능한 한 syscall 없이 사용자 모드 안에서 일을 끝내려 노력합니다.

면접에 나올 질문과 답변

시스템 콜이란 무엇이고 왜 필요한가요?
시스템 콜은 응용 프로그램이 커널의 기능(파일 I/O, 메모리 할당, 네트워크 등)을 빌리기 위해 호출하는 약속된 진입점입니다. 하드웨어가 응용 코드의 직접 접근을 막고 있기 때문에, OS가 정해 놓은 통로를 통해서만 특권이 필요한 일을 할 수 있습니다. 이 분리가 시스템의 안전성·격리·자원 중재의 기반입니다.
유저 모드와 커널 모드의 차이는 무엇인가요?
CPU의 권한 모드입니다. 유저 모드(Ring 3)에서는 자기 프로세스의 가상 메모리만 접근 가능하고 특권 명령은 실행할 수 없습니다. 커널 모드(Ring 0)에서는 모든 물리 메모리·하드웨어를 다루는 명령을 쓸 수 있습니다. 모드 전환은 시스템 콜·인터럽트·예외 같은 정해진 사건에서만 일어나며, 하드웨어가 직접 강제합니다.
시스템 콜의 호출 흐름을 단계별로 설명해 주세요.
① 응용이 libc 래퍼를 호출하고, ② 래퍼가 시스템 콜 번호와 인자를 정해진 레지스터에 셋업한 뒤, ③ syscall 명령으로 트랩을 발생시킵니다. ④ CPU가 커널 모드로 바뀌고 시스템 콜 디스패처가 번호를 보고 핸들러를 선택합니다. ⑤ 핸들러가 실제 작업을 수행한 뒤, ⑥ sysret로 유저 모드로 복귀하고, ⑦ 래퍼가 errno 변환을 거쳐 응용에 결과를 반환합니다.
시스템 콜이 왜 비용이 큰가요?
모드 전환 자체에 들어가는 명령 사이클뿐 아니라, 캐시·TLB·파이프라인이 오염되어 직후 코드의 성능이 떨어집니다. 또한 커널이 자기 코드를 안전하게 실행하기 위해 일부 검사·잠금을 추가로 수행합니다. 그래서 짧은 시스템 콜을 자주 부르는 패턴은 누적 비용이 매우 커지고, batched API(writev, epoll, io_uring)가 등장한 이유가 됩니다.
vDSO가 무엇이고 왜 도입되었나요?
vDSO는 커널이 모든 프로세스의 주소 공간에 매핑해 두는 작은 공유 라이브러리입니다. gettimeofday·clock_gettime처럼 호출이 매우 잦지만 단순한 시스템 콜을 모드 전환 없이 사용자 모드에서 바로 처리할 수 있게 해 줍니다. 커널이 시간 정보를 페이지에 갱신해 두면 vDSO 함수가 그것을 읽기만 하므로 호출 비용이 크게 줄어듭니다.
"라이브러리 함수 호출"과 "시스템 콜"의 차이는 무엇인가요?
라이브러리 함수는 사용자 모드에서 끝나는 함수 호출이고, 시스템 콜은 커널 모드로 트랩하는 호출입니다. printf는 라이브러리 함수이지만 결국 내부적으로 write라는 시스템 콜을 부릅니다. 대부분의 응용은 라이브러리 함수만으로 충분하며, 진짜 모드 전환은 그 라이브러리 안에서 가끔만 일어납니다.
