컨텍스트 스위칭(Context Switching)

컨텍스트 스위칭은 CPU가 실행 중이던 프로세스나 스레드를 멈추고, 그 실행 상태를 저장한 뒤 다른 실행 단위의 상태를 복원해 이어서 실행하는 과정입니다.

구체적인 개념 설명

CPU 코어 하나는 한 순간에 하나의 실행 흐름만 처리합니다. 그런데 우리는 여러 프로그램이 동시에 실행되는 것처럼 느낍니다. 운영체제가 아주 짧은 시간 단위로 실행 대상을 바꿔가며 CPU를 나눠주기 때문입니다.

실행 대상을 바꾸려면 현재 작업의 문맥(context)을 저장해야 합니다. 여기에는 프로그램 카운터(PC), 레지스터 값, 스택 포인터, 프로세스 상태 같은 정보가 포함됩니다. 이 정보는 보통 PCB(Process Control Block) 또는 TCB(Thread Control Block)에 저장됩니다.

컨텍스트 스위칭은 공짜가 아닙니다

상태 저장/복원 자체도 비용이지만, 캐시와 TLB의 지역성이 깨지는 비용도 큽니다. 그래서 스레드를 무작정 많이 만들면 병렬성이 늘기보다 스위칭 비용 때문에 느려질 수 있습니다.

왜 필요한가?

한 프로그램이 CPU를 독점하면 다른 프로그램은 멈춘 것처럼 보입니다. 운영체제는 타이머 인터럽트, I/O 대기, 우선순위 변화 등을 계기로 실행 대상을 바꿔 사용자 응답성과 전체 처리량을 유지합니다.

예를 들어 웹 서버 스레드가 디스크 I/O를 기다리는 동안 CPU를 계속 붙잡고 있으면 낭비입니다. OS는 그 스레드를 대기 상태로 두고, 실행 가능한 다른 스레드에게 CPU를 넘깁니다.

어떻게 동작하는가?

1) 실행 상태 저장과 복원

컨텍스트 스위칭은 이전 실행 상태를 잃지 않고 다른 작업으로 CPU를 넘기는 과정이다.

2) PCB에 저장되는 대표 정보

정보의미필요한 이유
PC다음에 실행할 명령어 주소복귀 지점 저장
레지스터CPU 내부의 임시 값계산 중이던 값 보존
스택 포인터현재 호출 스택 위치함수 호출 흐름 복원
프로세스 상태ready/running/waiting 등스케줄러 판단에 사용
메모리 정보페이지 테이블, 주소 공간 정보올바른 가상 주소 공간 사용

3) 상태 전이와 스위칭 시점

타이머 인터럽트, I/O 대기, I/O 완료 같은 이벤트가 실행 상태 전이를 만든다.

4) Java 코드와 스위칭 관점

ExecutorService pool = Executors.newFixedThreadPool(4);

for (int i = 0; i < 100; i++) {
    pool.submit(() -> {
        // 작업이 I/O를 기다리면 이 스레드는 waiting 상태가 될 수 있다.
        // OS는 그동안 다른 runnable 스레드에 CPU를 배정한다.
        doWork();
    });
}

// 스레드가 너무 많으면 실제 병렬 처리보다 컨텍스트 스위칭 비용이 커질 수 있다.

시간/공간 복잡도 또는 trade-off

요소좋은 점비용
짧은 타임 슬라이스응답성 향상스위칭 횟수 증가
긴 타임 슬라이스스위칭 비용 감소대화형 작업 응답성 저하 가능
스레드 수 증가I/O 대기 숨기기 가능스케줄링, 스택 메모리, 캐시 오염 비용 증가

실무에서 어디에 쓰이는가?

  • 서버 튜닝 — 스레드 풀 크기와 CPU 코어 수를 맞출 때 필요합니다.
  • 성능 분석 — context switch가 과도하면 CPU 사용률은 높은데 처리량이 낮을 수 있습니다.
  • 비동기 I/O — 블로킹 스레드를 줄여 불필요한 스위칭을 줄입니다.
  • 운영체제 스케줄링 이해 — ready, running, waiting 상태 전이의 기반입니다.

면접에 나올 질문과 답변

컨텍스트 스위칭이란 무엇인가요?
CPU가 현재 실행 중인 프로세스나 스레드의 상태를 저장하고, 다른 실행 단위의 상태를 복원해 실행 대상을 바꾸는 과정입니다. PC, 레지스터, 스택 포인터 같은 정보가 저장·복원됩니다.
컨텍스트 스위칭 비용은 왜 발생하나요?
레지스터 저장/복원, 커널 모드 진입, 스케줄러 실행뿐 아니라 캐시와 TLB 지역성이 깨지는 비용도 있습니다. 그래서 스위칭이 너무 잦으면 실제 작업보다 관리 비용이 커질 수 있습니다.
프로세스 스위칭과 스레드 스위칭 중 무엇이 더 무거운가요?
일반적으로 프로세스 스위칭이 더 무겁습니다. 주소 공간 전환, 페이지 테이블 관련 정보, TLB 영향이 더 크기 때문입니다. 같은 프로세스 안의 스레드 전환은 주소 공간을 공유하므로 상대적으로 가볍습니다.
