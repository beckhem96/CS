IPC — 프로세스 간 통신(Inter-Process Communication)

IPC는 메모리가 분리된 프로세스끼리 데이터를 주고받기 위한 메커니즘입니다. OS는 같은 머신 위에서 협력하는 프로세스들이 안전하게 협업할 수 있도록 파이프·메시지 큐·공유 메모리·시그널·도메인 소켓 등 여러 통로를 제공합니다.

구체적인 개념 설명

프로세스는 기본적으로 가상 주소 공간이 분리되어 있어, 서로의 메모리를 직접 읽거나 쓸 수 없습니다. 이 격리는 안전성의 기반이지만, 동시에 협업을 하려면 OS가 정해 놓은 통로를 통해서만 데이터를 주고받을 수 있다는 뜻이기도 합니다. 이 통로들의 모음이 IPC입니다.

IPC는 단순한 한 가지 기술이 아니라, 상황에 따라 적합한 도구가 다른 도구 모음입니다. "한쪽이 일방적으로 데이터를 흘려보내는가, 양쪽이 메시지를 주고받는가, 큰 데이터를 빠르게 공유해야 하는가"에 따라 선택이 달라지고, 각각은 속도·동기화 부담·생명주기·확장성에서 trade-off가 분명합니다.

한 줄 요약 — IPC는 "공유"와 "복사"의 trade-off다

공유 메모리는 데이터를 복사하지 않아 빠르지만 동기화 책임을 사용자가 진다. 파이프·메시지 큐·소켓은 커널을 거쳐 복사가 일어나기에 느리지만, 동기화·격리는 OS가 책임진다. IPC를 고를 때 가장 중요한 질문은 "데이터를 한 번 더 복사해도 괜찮은가?"입니다.

왜 필요한가?

현대의 시스템은 한 프로그램이 모든 일을 하지 않습니다. 셸은 명령어 여러 개를 파이프로 연결해 협업하고, 웹 서버는 워커 프로세스로 일을 분산하며, 데이터베이스는 백그라운드 프로세스가 메인 프로세스와 정보를 주고받습니다. 이 모든 협업의 기반이 IPC입니다.

  • 분리된 신뢰 경계 — 보안상 격리되어야 하는 컴포넌트끼리(예: 브라우저의 렌더러 vs 메인 프로세스) 안전하게 통신해야 합니다.
  • 장애 격리 — 한 프로세스가 죽어도 다른 프로세스는 살아남도록 분리하면서 협업하려면 IPC가 필요합니다.
  • 언어 / 런타임 차이 — 서로 다른 언어로 만든 프로세스끼리 협업하려면 메모리를 공유하기보다 IPC로 메시지를 주고받는 편이 안전합니다.

어떻게 동작하는가?

1) 파이프 — 한 방향으로 흐르는 바이트 스트림

파이프는 가장 단순한 IPC입니다. 부모 프로세스가 pipe()로 생성한 뒤 fork()를 통해 자식과 양쪽 끝을 공유합니다. 한쪽 끝은 쓰기 전용, 반대쪽 끝은 읽기 전용으로 두면 한 방향 채널이 됩니다. 이름이 없는 익명 파이프는 부모-자식 간에만 쓸 수 있고, 파일 시스템에 이름을 가진 named pipe(FIFO)는 무관한 프로세스끼리도 공유할 수 있습니다.

파이프는 단방향, 바이트 스트림, 메시지 경계 없음 — 가장 단순한 IPC.

2) 메시지 큐 — 경계가 있는 메시지 단위 통신

메시지 큐는 OS가 관리하는 큐에 메시지(헤더 + 페이로드)를 넣고 꺼내는 IPC입니다. 파이프와 달리 메시지 경계가 보존되고, 우선순위·타입을 부여할 수 있어 "이 타입의 메시지만 골라 받기" 같은 패턴이 가능합니다. POSIX 메시지 큐(mq_*), System V 메시지 큐가 대표적입니다.

메시지 큐는 메시지 단위로 보존되고, 우선순위·타입 기반 선택 수신이 가능하다.

3) 공유 메모리 — 가장 빠르지만 동기화는 직접

공유 메모리는 두 개 이상의 프로세스가 같은 물리 메모리 페이지를 자기 가상 주소 공간에 매핑하는 방식입니다. 읽기/쓰기에 시스템 콜이 필요 없으므로 가장 빠른 IPC이지만, 동시에 두 프로세스가 같은 위치를 동시에 쓰면 경쟁 상태가 발생합니다. 그래서 공유 메모리는 거의 항상 세마포어·뮤텍스·atomic 연산과 함께 사용됩니다.

공유 메모리는 매핑만 한 뒤에는 일반 메모리 접근처럼 빠르지만, 안전성은 사용자가 책임진다. 
// POSIX 공유 메모리(개념적)
int fd = shm_open("/myshm", O_CREAT | O_RDWR, 0600);
ftruncate(fd, 4096);
void *p = mmap(NULL, 4096, PROT_READ | PROT_WRITE, MAP_SHARED, fd, 0);

// 두 프로세스 모두 같은 "/myshm"을 열면 같은 페이지를 보게 된다.
// p에 쓴 값은 즉시 다른 프로세스에서 보이지만, 동기화는 별도로 해야 한다.

4) 시그널 — 가장 가벼운 비동기 알림

시그널은 OS가 프로세스에게 보내는 짧은 비동기 알림입니다. 데이터를 전달하기보다 "사건이 일어났다"를 알리는 용도에 가깝습니다. SIGINT(Ctrl+C), SIGTERM(종료 요청), SIGCHLD(자식 종료) 같은 표준 시그널이 있고, kill -SIGUSR1 <pid>처럼 사용자 정의 시그널을 보낼 수도 있습니다. 핸들러 안에서는 비동기-시그널-안전 함수만 호출해야 한다는 까다로운 제약이 있습니다.

5) 유닉스 도메인 소켓(UDS) — 같은 머신 안의 소켓

도메인 소켓은 TCP 소켓과 같은 인터페이스(connect·send·recv)를 쓰지만, 네트워크를 거치지 않고 같은 머신 안에서만 동작합니다. 네트워크 스택을 우회하므로 빠르고, 인증을 위해 피어의 UID/PID·파일 디스크립터 전달까지 지원합니다. Docker, PostgreSQL, X11이 같은 머신 안 통신에서 자주 사용합니다.

실무에서 어디에 쓰이는가?

  • 셸 파이프라인 — cat file | grep ERROR | wc -l은 익명 파이프 두 개로 세 프로세스를 잇습니다.
  • 웹 서버 / 데이터베이스 — Nginx의 master-worker 구조, PostgreSQL의 백엔드와 백그라운드 프로세스 간 협업이 시그널·공유 메모리·소켓을 함께 씁니다.
  • Docker / containerd — 데몬과 CLI, 컨테이너 런타임 간 통신에 도메인 소켓을 사용합니다.
  • 고성능 서비스의 fork 모델 — Redis는 RDB 스냅샷을 저장할 때 fork 후 자식과 부모가 페이지를 공유하다가 변경 시점에만 복사합니다(Copy-on-Write — 사실상 IPC의 한 형태).
  • 프로세스 풀 / 워커 — Python multiprocessing은 큐·파이프·공유 메모리(multiprocessing.shared_memory)를 통해 워커와 부모를 잇습니다.
  • 브라우저 — Chrome의 멀티프로세스 구조에서 렌더러·GPU·네트워크 프로세스가 도메인 소켓·공유 메모리로 통신합니다.

면접에 나올 질문과 답변

IPC가 필요한 이유는 무엇인가요?
프로세스는 격리된 가상 주소 공간을 가지기 때문에 서로의 메모리에 직접 접근할 수 없습니다. 하지만 셸 파이프, 멀티프로세스 서버, 컨테이너 런타임처럼 협력해야 하는 상황은 많기 때문에, OS가 안전하게 데이터를 주고받을 수 있는 통로(파이프·메시지 큐·공유 메모리·시그널·도메인 소켓 등)를 제공합니다. 이 통로의 모음이 IPC입니다.
파이프와 메시지 큐의 차이는 무엇인가요?
파이프는 바이트 스트림이라 메시지 경계가 보존되지 않고 단방향입니다. 메시지 큐는 보낸 단위 그대로 메시지가 보존되고, 우선순위·타입을 부여해 선택 수신이 가능합니다. 구조화된 데이터를 양방향으로 주고받아야 한다면 메시지 큐가, 단순한 한 방향 스트림이면 파이프가 적합합니다.
공유 메모리가 가장 빠른 이유와 그 대가는 무엇인가요?
다른 IPC는 데이터를 보낼 때마다 커널을 거쳐 복사가 발생하지만, 공유 메모리는 처음 매핑만 하면 그 뒤로는 일반 메모리 접근처럼 동작해 복사가 없습니다. 대신 두 프로세스가 같은 위치를 동시에 만지면 경쟁 상태가 생기므로, 세마포어·뮤텍스·atomic 같은 동기화 도구를 사용자가 직접 사용해야 합니다. 속도와 안전성이 분리되는 셈입니다.
시그널이 데이터 전달 용도로 적합하지 않은 이유는 무엇인가요?
시그널은 짧은 비동기 알림이고, 페이로드가 거의 없으며(SIGRTMIN 계열은 약간의 데이터 전달을 지원하지만 매우 제한적), 핸들러 안에서는 비동기-시그널-안전 함수만 호출할 수 있다는 강한 제약이 있습니다. 또한 같은 시그널이 연속으로 오면 합쳐질 수 있어 "몇 번 받았는지"가 보장되지 않습니다. 이 때문에 시그널은 "사건 통보"에는 좋지만 데이터 통신 채널로는 부적합합니다.
유닉스 도메인 소켓을 TCP 소켓 대신 쓰는 이점은 무엇인가요?
같은 머신 안의 통신이라면 TCP 스택을 거치지 않아 더 빠르고, 추가로 피어의 UID·PID 같은 자격 정보를 OS가 제공해 인증에 활용할 수 있습니다. 파일 디스크립터를 다른 프로세스에 전달하는 기능도 도메인 소켓의 강점입니다(SCM_RIGHTS). 다만 같은 머신을 가정하므로, 분산 환경에서는 적용이 안 됩니다.
멀티스레드와 멀티프로세스 + IPC를 어떻게 비교할 수 있나요?
멀티스레드는 같은 주소 공간을 공유해 통신이 매우 빠르지만, 한 스레드의 메모리 오염이 전체 프로세스를 망가뜨릴 수 있습니다. 멀티프로세스 + IPC는 격리가 강해 안정성과 보안에서 유리하고, 한 프로세스가 죽어도 다른 프로세스는 살아남을 수 있습니다. 대신 IPC를 거쳐야 하므로 통신 비용이 더 큽니다. 브라우저의 사이트 격리, 컨테이너 런타임처럼 안정성이 중요한 영역에서 후자가 선호됩니다.
