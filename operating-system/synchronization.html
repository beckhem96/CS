동기화(Synchronization)

동기화는 여러 실행 흐름(스레드·프로세스)이 같은 자원에 접근할 때 충돌 없이 일관된 결과를 만들도록 순서와 접근 권한을 조정하는 기법입니다. 핵심 도구는 뮤텍스(Mutex), 세마포어(Semaphore), 모니터(Monitor)입니다.

구체적인 개념 설명

동기화 문제는 여러 사람이 한 화이트보드에 동시에 글을 쓰는 상황과 비슷합니다. 한 사람이 단어를 다 쓰기 전에 다른 사람이 끼어들면 글자가 섞여서 무슨 말인지 알 수 없게 됩니다. 임계 영역(Critical Section)이란 이렇게 "한 번에 한 사람만" 들어가야 하는 코드 구간입니다.

가장 흔한 사례는 잔액 갱신입니다. balance = balance + 100은 한 줄짜리 코드처럼 보이지만 실제로는 "값 읽기 → 더하기 → 다시 쓰기" 세 단계로 나뉘는데, 이 사이에 다른 스레드가 끼어들면 갱신이 사라집니다(lost update). 이런 비결정적 결과를 경쟁 상태(Race Condition)라고 합니다.

"한 번에 한 명만"이 동기화의 본질

동기화 도구는 종류가 많아 보여도 결국 두 가지 질문에 답하는 도구입니다: ① 지금 들어가도 되는가? ② 못 들어간다면 어떻게 기다릴까? 뮤텍스, 세마포어, 모니터는 이 질문에 답하는 방식이 조금씩 다를 뿐입니다.

왜 필요한가?

현대 서버는 멀티코어 위에서 수많은 스레드가 동시에 돌아갑니다. 동기화 없이 공유 자료(잔액, 카운터, 자료구조 노드)를 만지면 값이 깨지는 것은 물론, 운이 좋을 때만 통과하는 버그가 생깁니다. 운영 환경에서만 가끔 터지는 가장 골치 아픈 부류죠.

OS 입장에서도 동기화는 필수입니다. 페이지 테이블, 파일 디스크립터 테이블, I/O 버퍼 같은 커널 자료구조는 여러 프로세스가 동시에 건드릴 수 있어서 락이나 그에 준하는 메커니즘으로 보호해야 합니다.

어떻게 동작하는가?

1) 경쟁 상태가 발생하는 모습

read–modify–write 사이에 다른 스레드가 끼어들면 갱신이 누락된다.

2) 뮤텍스 — 한 번에 한 명, 소유권 개념

뮤텍스(Mutex, Mutual Exclusion)는 자물쇠 한 개와 같습니다. 자물쇠를 가진 스레드만 임계 영역에 들어가고, 나올 때 자물쇠를 반납합니다. 잠근 스레드만 풀 수 있다는 소유권 규칙이 핵심입니다.

뮤텍스는 자물쇠처럼 동작하며, 잠근 주체만 풀 수 있다.

3) 세마포어 — 카운팅 가능한 일반화

세마포어(Semaphore)는 정수 카운터를 가진 도구입니다. P()(또는 wait/acquire)는 카운터를 1 줄이고 0 이하면 대기, V()(또는 signal/release)는 1 늘려 대기자 하나를 깨웁니다. 카운터가 1이면 사실상 뮤텍스와 같지만(이진 세마포어), 카운터가 N이면 N개의 입장권을 의미합니다.

세마포어는 정수 카운터로 "동시에 N개까지" 같은 패턴을 표현한다.

4) 모니터 — 언어 차원의 동기화 단위

모니터(Monitor)는 공유 자료와 그것을 다루는 메서드를 한 객체에 묶고, 그 메서드 진입을 자동으로 상호 배제하는 추상화입니다. 자바의 synchronized와 wait/notify가 대표적인 모니터 구현입니다. 개발자는 락을 명시적으로 잡고 풀지 않아도 메서드만 동기화하면 됩니다.

5) Java 코드

public class Counter {
    private long value = 0;
    private final Object lock = new Object();

    // (1) 모니터: 메서드 진입 시 자동 락
    public synchronized void inc() {
        value++;
    }

    // (2) ReentrantLock: 더 세밀한 제어
    private final java.util.concurrent.locks.ReentrantLock m = new java.util.concurrent.locks.ReentrantLock();
    public void incExplicit() {
        m.lock();
        try { value++; }
        finally { m.unlock(); }   // 예외가 나도 반드시 풀린다
    }

    // (3) 세마포어: 동시에 N개만 작업하도록 제한
    private final java.util.concurrent.Semaphore sem = new java.util.concurrent.Semaphore(3);
    public void downloadFile() throws InterruptedException {
        sem.acquire();             // 카운터 -1, 0이면 대기
        try { /* ... 다운로드 ... */ }
        finally { sem.release(); } // 카운터 +1
    }
}

시간/공간 복잡도 또는 trade-off

도구특징적합한 상황주의점
뮤텍스소유권 있음, 한 명만 진입공유 자료 보호잠근 주체만 해제 가능
이진 세마포어소유권 없음, 0/1 카운터이벤트 신호, 한 스레드가 잠그고 다른 스레드가 풀기실수로 두 번 release 시 카운터 어긋남
카운팅 세마포어N개 입장권커넥션 풀, 동시 다운로드 제한음수 누적 시 대기자 많아짐
모니터객체 단위 자동 락객체 메서드 보호, 조건 변수 결합락 범위가 넓어져 성능 저하 가능
스핀락대기 시 CPU 소모하며 폴링매우 짧은 임계 영역, 멀티코어단일 코어에서 비효율

락은 안전하지만 비용이 있습니다. 임계 영역이 길수록 다른 스레드가 더 많이 기다리고 처리량이 떨어집니다. 그래서 락 범위를 가능한 한 좁게 잡고, 가능하면 java.util.concurrent의 동시성 자료구조나 AtomicLong 같은 락-프리 도구를 우선 고려합니다.

실무에서 어디에 쓰이는가?

  • 은행·결제 시스템의 잔액 갱신 — DB 레벨 락 + 애플리케이션 락이 함께 작동합니다.
  • 커넥션 풀 — 카운팅 세마포어로 동시에 사용할 수 있는 커넥션 수를 제한합니다.
  • 캐시 갱신(Cache Stampede 방지) — 한 번에 한 스레드만 원본 갱신을 수행하도록 락을 겁니다.
  • 로깅 / 메트릭 수집 — 카운터·버퍼 보호, 다만 핫 패스에서는 락-프리 자료구조 사용.
  • OS 커널 — 페이지 테이블, 인터럽트 핸들러, scheduler runqueue 등에서 스핀락·뮤텍스를 적절히 혼용합니다.

면접에 나올 질문과 답변

뮤텍스와 세마포어의 차이는 무엇인가요?
뮤텍스는 소유권 개념이 있어 잠근 스레드만 풀 수 있고 항상 한 명만 진입합니다. 세마포어는 소유권이 없는 정수 카운터라 다른 스레드가 release할 수 있고, 카운터를 N으로 두면 동시에 N개의 진입을 허용합니다.
경쟁 상태(Race Condition)란 무엇이고 어떻게 막나요?
여러 스레드의 실행 순서에 따라 결과가 달라지는 비결정적 상황입니다. 공유 자료에 접근하는 임계 영역을 락으로 보호하거나, 원자적 연산(CAS)·불변 객체·메시지 패싱 같은 방법으로 공유를 줄여 막을 수 있습니다.
자바의 synchronized는 어떤 동기화 모델인가요?
모니터 패턴입니다. 객체 헤더에 모니터(intrinsic lock)가 있고, synchronized 블록·메서드 진입 시 자동으로 잠그고 빠져나갈 때 풉니다. 예외가 나도 반드시 해제되므로 락 누수가 없습니다.
스핀락은 언제 좋고 언제 나쁜가요?
임계 영역이 매우 짧고 멀티코어라면 컨텍스트 스위칭 비용을 아낄 수 있어 유리합니다. 반대로 단일 코어이거나 임계 영역이 길면 CPU만 낭비하므로 일반 뮤텍스를 써야 합니다.
생산자-소비자 문제에서 세마포어를 어떻게 쓰나요?
보통 두 개의 카운팅 세마포어(빈 칸 수, 채워진 칸 수)와 버퍼 보호용 뮤텍스를 함께 씁니다. 생산자는 빈 칸 세마포어를 P, 소비자는 채워진 칸 세마포어를 P 한 뒤 뮤텍스로 버퍼를 조작하고 반대편 세마포어를 V 합니다.
