PCB / TCB — 프로세스·스레드 제어 블록

PCB(Process Control Block)는 운영체제가 프로세스 하나를 관리하기 위해 들고 있는 자료구조이고, TCB(Thread Control Block)는 그 안에서 스레드 하나를 관리하기 위한 자료구조입니다. 프로세스/스레드가 사라지기 전까지 OS는 이 블록 안에 지금까지 어디까지 실행했는지, 어떤 자원을 쓰고 있는지를 전부 적어둡니다.

구체적인 개념 설명

OS 입장에서 프로세스는 코드와 메모리만으로는 부족합니다. CPU에서 잠시 내려와 다른 프로세스에 자리를 양보했다가 다시 돌아왔을 때 "내가 어디까지 실행했지?", "어떤 파일을 열어뒀지?", "어떤 시그널을 처리하기로 했지?"를 OS가 대신 기억해 줘야 합니다. 이 정보를 모아 둔 커널 메모리 안의 구조체가 바로 PCB입니다.

현대 OS는 한 프로세스에 여러 스레드를 둘 수 있기 때문에, 실제로는 주소 공간·열린 파일·프로세스 권한 같은 공유 자원은 PCB가, 프로그램 카운터·스택 포인터·레지스터 같은 실행 흐름 정보는 TCB가 들고 있습니다. PCB와 TCB는 부모-자식처럼 연결되어 있어, 한 프로세스의 모든 TCB는 같은 PCB를 가리킵니다.

"프로세스" = 자원의 컨테이너, "스레드" = 그 안의 실행 흐름

그래서 PCB는 격리된 자원(메모리 맵·FD 테이블·시그널 핸들러·UID/GID 등)을 들고, TCB는 흐름 자체(레지스터·커널 스택·스케줄링 우선순위·상태)를 들고 있습니다. 이 분리를 알면 fork/clone, 스레드 안전성, 컨텍스트 스위칭 비용이 한 번에 이해됩니다.

왜 필요한가?

OS가 다중 작업을 지원하려면 "지금 CPU에 안 올라와 있는 프로세스의 상태"를 어딘가에 적어둬야 합니다. 그렇지 않으면 다시 그 프로세스에 CPU를 줬을 때 어디서부터 실행해야 할지를 알 수 없습니다. 이 "어딘가"가 바로 PCB이고, OS는 PCB를 통해 다음 세 가지를 동시에 해냅니다.

  • 실행 재개 — 컨텍스트 스위칭 시 CPU 레지스터를 PCB/TCB에 저장하고 복원합니다.
  • 스케줄링 — ready/waiting 큐는 사실 PCB 포인터들의 연결 리스트입니다. 우선순위·상태·CPU 친화도 같은 메타데이터가 모두 PCB에 있습니다.
  • 자원 회수 — 프로세스가 종료될 때 OS는 PCB를 보고 열린 파일·메모리·자식 프로세스를 정리합니다.

어떻게 동작하는가?

1) PCB 안에 들어가는 정보

PCB의 9개 핵심 카테고리 — 어떤 OS든 이름만 다를 뿐 거의 같은 정보를 담는다.

2) 프로세스 상태 모델과 PCB의 state 필드

PCB의 state 필드는 이 다이어그램의 한 노드 값을 그대로 담는다.

3) PCB는 큐로 묶여 스케줄러에 노출된다

OS의 스케줄러는 PCB를 직접 들여다보지 않고, "준비 큐(ready queue)"·"대기 큐(I/O wait queue)" 같은 PCB 포인터의 연결 리스트로 보관합니다. 준비 큐의 head를 꺼내 CPU에 올리고, 타이머 인터럽트가 오면 다시 큐 끝에 붙입니다. 그래서 PCB에는 큐 연결을 위한 다음 PCB 포인터가 거의 항상 들어있습니다.

"프로세스 테이블"은 PCB 배열, "준비 큐"는 PCB 포인터 리스트다.

4) Linux의 PCB — task_struct

리눅스에서는 PCB와 TCB의 구분이 없습니다. 모든 실행 단위(스레드 포함)는 task_struct 한 종류로 표현되고, "프로세스"는 같은 메모리 맵·FD 테이블을 공유하는 task_struct 그룹일 뿐입니다. clone() 시 어떤 자원을 공유할지 플래그로 결정합니다. 

// 핵심 필드만 추린 task_struct (Linux 커널 일부 발췌)
struct task_struct {
    pid_t pid;                 // 스레드 ID
    pid_t tgid;                // 같은 프로세스의 스레드들이 공유하는 ID
    long state;                // TASK_RUNNING, TASK_INTERRUPTIBLE 등

    struct mm_struct *mm;      // 메모리 맵 (스레드끼리 공유 → 같은 mm)
    struct files_struct *files;// FD 테이블
    struct signal_struct *signal;
    struct cred *cred;         // 보안 자격(UID, capabilities)

    struct sched_entity se;    // 스케줄러용 메타데이터
    struct list_head tasks;    // 전역 task 리스트 연결

    struct task_struct *parent;
    struct list_head children;
};

5) PCB와 TCB의 관계 — 스레드는 무엇을 따로 갖는가

자원프로세스(PCB) 수준스레드(TCB) 수준
주소 공간 / 페이지 테이블공유 — 같은 mm공유
FD 테이블공유공유
시그널 핸들러공유마스크는 별도
UID/GID, capabilities공유공유
레지스터 / PC / SP—각자
커널 스택—각자
스케줄링 상태·우선순위—각자
TLS (Thread-Local Storage)—각자

이 표가 그대로 스레드 컨텍스트 스위칭이 프로세스 스위칭보다 싼 이유입니다. 페이지 테이블이 같으니 TLB를 비우지 않아도 되고, FD 테이블도 그대로 쓰면 됩니다.

trade-off

설계 결정장점비용
PCB를 크게 (많은 정보)스케줄러·디버거·관측 도구가 모두 행복스위칭 시 캐시 오염, 메모리 사용 증가
PCB와 TCB를 분리 (전통 OS)구조가 명확스레드 모델이 PCB에 갇혀 유연성 부족
리눅스식 단일 task_structfork/clone을 같은 코드로 처리, 유연함"프로세스"의 정의가 코드가 아닌 플래그에 의존
PID 정수 vs 64비트 식별자좁은 범위는 캐시 친화·인덱스 효율장기 운영 서버에서 PID 재사용 충돌

실무에서 어디에 쓰이는가?

  • ps, top, htop — 모두 커널의 PCB 정보를 읽어 보여주는 사용자 도구입니다.
  • /proc/<pid>/ — 리눅스에서 PCB의 일부 필드(상태, 메모리 맵, 열린 FD)를 가상 파일로 노출합니다.
  • 좀비 프로세스 — 자식이 종료됐는데 부모가 wait() 하지 않으면 PCB가 회수되지 않고 남아 PID 슬롯을 차지합니다.
  • 성능 분석 — perf, eBPF는 PCB 필드(컨텍스트 스위칭 횟수, on-CPU 시간)를 읽어 동작합니다.
  • 보안 — UID, capabilities, namespaces 같은 격리 정보가 PCB에 들어있어 컨테이너 격리의 기반이 됩니다.

면접에 나올 질문과 답변

PCB가 왜 필요한가요?
OS가 멀티태스킹을 하려면, CPU에서 잠시 내려와 있는 프로세스의 실행 상태와 자원 정보를 어딘가에 저장해 둬야 합니다. PCB는 그 정보(레지스터·PC·메모리 맵·FD·상태·스케줄링 메타데이터 등)를 모아 둔 커널 메모리 안의 구조체로, 컨텍스트 스위칭·스케줄링·자원 회수가 모두 PCB를 거쳐 일어납니다.
PCB와 TCB는 어떤 점에서 다르고, 왜 분리할까요?
PCB는 격리되어야 할 자원(주소 공간·FD 테이블·시그널 핸들러·자격)을 담고, TCB는 실행 흐름 정보(레지스터·PC·SP·커널 스택)를 담습니다. 한 프로세스의 여러 스레드는 같은 PCB를 공유하면서 각자 TCB만 따로 가지므로, 스레드 스위칭이 프로세스 스위칭보다 가벼워집니다. 대표적으로 페이지 테이블이 같아 TLB를 비울 필요가 없습니다.
리눅스에서는 왜 PCB·TCB를 구분하지 않고 task_struct 하나로 처리하나요?
리눅스는 clone() 시스템 콜의 플래그(CLONE_VM, CLONE_FILES, CLONE_THREAD …)로 자원 공유 여부를 정합니다. 스레드는 메모리 맵·FD 테이블을 공유하는 task일 뿐이고, 프로세스는 그것들을 공유하지 않는 task일 뿐입니다. 이 일반화 덕분에 fork·thread create·컨테이너 namespace 분리가 같은 코드 경로에서 처리됩니다.
PCB의 state 필드가 왜 중요한가요?
스케줄러가 어떤 PCB를 어떤 큐에 넣을지 결정하는 기준이 state입니다. running에서 timer interrupt가 발생하면 ready로, I/O를 시작하면 waiting으로 바뀌고, I/O가 끝나면 다시 ready로 돌아옵니다. 이 전이 그래프가 5-state(또는 7-state) 모델이고, PCB는 그 상태를 단 한 필드로 표현합니다.
좀비 프로세스(zombie)와 PCB의 관계를 설명해 주세요.
자식 프로세스가 종료되면 자원은 거의 회수되지만, 종료 코드 같은 정보가 부모에게 전달되어야 하므로 PCB는 잠시 남아있습니다. 부모가 wait() 또는 waitpid()로 종료 상태를 거둬가야 PCB가 완전히 해제됩니다. 부모가 이를 게을리하면 PCB가 계속 쌓여 PID 슬롯을 소진할 수 있고, 이게 좀비 누적 문제입니다.
PCB는 어디에 저장되나요? 사용자 프로세스가 직접 읽을 수 있나요?
PCB는 커널 메모리 영역에 있어 사용자 프로세스가 직접 읽거나 수정할 수 없습니다. 다만 리눅스는 /proc/<pid>/라는 가상 파일 시스템을 통해 PCB의 안전한 일부(상태·메모리 맵·통계)를 노출하고, ps·top 같은 도구는 이 정보를 읽어 보여줍니다. 직접 PCB를 만지려면 시스템 콜을 통해 커널에 요청해야 합니다.
