페이징(Paging)과 세그먼테이션(Segmentation)

페이징과 세그먼테이션은 가상 메모리를 일정 단위로 잘라 물리 메모리에 흩어 놓는 두 가지 방식입니다. 페이징은 고정 크기 블록(페이지)으로 자르고, 세그먼테이션은 의미 단위(코드/데이터/스택)로 자릅니다. 현대 OS는 둘을 결합한 형태를 많이 씁니다.

구체적인 개념 설명

프로그램을 실행하려면 메모리에 올려야 합니다. 그런데 프로그램마다 크기가 다르고, 한 메모리를 여러 프로그램이 나눠 써야 하니 그냥 통째로 올리면 빈 공간이 생기고 조각나기 쉽습니다(외부 단편화). 페이징과 세그먼테이션은 이 문제를 다른 방식으로 해결합니다.

페이징은 메모리를 동일 크기(보통 4KB) 칸으로 쪼개고, 프로그램 메모리도 같은 크기 단위(페이지)로 잘라 어디 빈 칸이든 채워 넣습니다. 박스에 같은 크기 블록을 끼워 넣는 느낌입니다.

세그먼테이션은 프로그램을 의미 단위(코드, 데이터, 스택, 힙 등)로 잘라 각 세그먼트가 가변 길이로 자리잡습니다. 사람이 보기엔 자연스럽지만 메모리 위에서는 길이가 달라 빈 칸 모양이 들쑥날쑥합니다.

현대 OS는 결합형

x86 보호 모드, 리눅스, 윈도우 모두 사실상 페이징을 핵심으로 쓰고, 세그먼테이션은 코드/데이터/스택 같은 영역 구분 정도로만 남아 있습니다. "프로세스 주소 공간은 세그먼트로 의미 단위 보고, 그 안의 매핑은 페이지로 수행한다"가 현실의 그림입니다.

왜 필요한가?

가장 큰 이유는 외부 단편화를 피하면서 메모리를 효율적으로 쓰는 것입니다. 통짜로 올리면 큰 빈 공간이 부족해 새 프로세스를 못 올리는 일이 생깁니다. 또 가상 메모리(Virtual Memory)를 구현하려면 프로그램이 보는 주소와 실제 주소를 분리해야 하는데, 페이지 단위 매핑이 이 분리를 자연스럽게 만듭니다.

보호와 공유도 핵심입니다. 페이지 테이블에 권한 비트(읽기/쓰기/실행)를 두면 같은 페이지를 여러 프로세스가 안전하게 공유할 수 있고 (예: 공유 라이브러리), 다른 프로세스의 메모리에는 접근하지 못하도록 막을 수 있습니다.

어떻게 동작하는가?

1) 페이징 — 가상 페이지 → 물리 프레임 매핑

페이지 테이블이 가상 페이지를 물리 프레임에 흩뿌려 매핑한다. 외부 단편화가 사라진다.

가상 주소는 페이지 번호 + 페이지 내 오프셋으로 구성됩니다. CPU는 페이지 번호로 페이지 테이블을 조회해 프레임 번호를 얻고, 프레임 시작 주소 + 오프셋으로 실제 물리 주소를 만듭니다. 이 변환을 매번 빠르게 하기 위해 TLB가 자주 쓰는 매핑을 캐싱합니다.

2) 세그먼테이션 — 의미 단위로 자르기

세그먼트는 의미 단위로 자연스럽지만 크기가 달라 빈 공간이 들쭉날쭉해진다.

3) 비교 표

항목페이징세그먼테이션
분할 단위고정 크기(페이지)가변 크기(논리적 단위)
외부 단편화없음있음
내부 단편화마지막 페이지에서 발생없음(거의)
주소 변환페이지 번호 + 오프셋세그먼트 번호 + 오프셋
보호/공유 단위페이지 단위, 의미와 무관해 다소 거침의미 단위라 자연스러움(코드만 공유 등)
현대 OS 사용핵심으로 사용제한적, 페이징과 결합

4) 다단계 페이지 테이블이 필요한 이유

64비트 시스템에서 가상 주소 공간은 어마어마하게 큽니다. 한 번에 평면적인 페이지 테이블을 만들면 그 자체로 메모리가 부족합니다. 그래서 페이지 테이블 자체를 페이지 단위로 잘라 다단계(2~5단계) 트리로 만듭니다. 실제로 사용 중인 영역만 하위 테이블이 만들어지므로 메모리를 크게 절약할 수 있습니다.

5) Java/C 관점

// Java/C 어느 쪽이든 개발자가 보는 주소는 모두 가상 주소다.
// 다음 코드의 변수 a, arr는 가상 주소를 가진다.
int a = 10;
int[] arr = new int[1024];

// CPU가 arr[i]에 접근할 때:
//   1) 가상 주소를 페이지 번호 + 오프셋으로 분해
//   2) TLB 조회 → 미스면 페이지 테이블 조회
//   3) 프레임 번호로 물리 주소 산출
//   4) 페이지가 메모리에 없으면 page fault → 디스크에서 로드(스왑)

시간/공간 복잡도 또는 trade-off

  • 페이지 크기 trade-off — 페이지가 크면 페이지 테이블 크기가 줄지만 내부 단편화가 늘어납니다. 작으면 반대.
  • 주소 변환 비용 — 매 접근마다 페이지 테이블 조회는 비싸므로 TLB로 캐싱합니다. TLB 미스율이 곧 성능에 큰 영향을 줍니다.
  • 다단계 페이지 테이블 — 메모리는 절약되지만 변환 시 트리를 여러 번 따라가야 합니다. TLB가 그래서 더 중요해집니다.
  • 세그먼테이션 단독은 외부 단편화 때문에 압축(compaction)이 필요할 수 있고, 이는 매우 비쌉니다.

실무에서 어디에 쓰이는가?

  • 모든 현대 운영체제가 페이징을 기반으로 가상 메모리를 제공합니다(리눅스 4KB 기본, huge page 2MB·1GB).
  • 공유 라이브러리 — 같은 코드 페이지를 여러 프로세스가 매핑해 한 부 적재로 공유.
  • Copy-on-Write(fork) — fork 직후 부모·자식이 같은 페이지를 가리키다 쓰기 발생 시에만 복제.
  • 메모리 매핑 파일(mmap) — 파일을 페이지 단위로 메모리에 투영해 입출력처럼 다룹니다.
  • JVM 힙 — JVM이 보는 힙은 가상 주소이고, OS 페이징을 통해 물리 메모리에 흩어져 있습니다.

면접에 나올 질문과 답변

페이징과 세그먼테이션의 차이를 말해보세요.
페이징은 고정 크기 블록으로 메모리를 자르고, 세그먼테이션은 의미 단위(코드·데이터·스택)로 가변 크기로 자릅니다. 페이징은 외부 단편화는 없지만 내부 단편화가 있고, 세그먼테이션은 반대입니다. 현대 OS는 페이징을 기반으로 두 가지를 결합해 씁니다.
내부 단편화와 외부 단편화는 무엇인가요?
내부 단편화는 할당된 블록 안에서 사용하지 않는 공간(페이지의 끝부분)이고, 외부 단편화는 빈 공간이 있긴 한데 흩어져 있어 큰 덩어리를 못 만드는 상황입니다. 페이징은 외부 단편화가 없고, 세그먼테이션은 외부 단편화가 발생합니다.
TLB가 왜 필요한가요?
모든 메모리 접근마다 페이지 테이블을 조회하면 너무 느립니다. TLB는 최근 사용된 가상→물리 페이지 매핑을 캐싱해 대부분의 접근을 한 사이클 안에 끝내도록 도와줍니다. 컨텍스트 스위치 시 TLB 무효화 비용이 커서 ASID 같은 보완책이 쓰입니다.
페이지 폴트(page fault)가 발생하면 OS는 어떻게 처리하나요?
페이지 테이블에 매핑이 없거나 페이지가 메모리에 없을 때 발생합니다. OS는 디스크(스왑/파일)에서 해당 페이지를 읽어 빈 프레임에 올리고, 페이지 테이블을 갱신해 명령어를 재실행합니다. 빈 프레임이 없다면 페이지 교체 알고리즘으로 희생자(victim)를 골라 내보냅니다.
왜 다단계 페이지 테이블을 쓰나요?
64비트 시스템에서 평면 페이지 테이블은 너무 크기 때문입니다. 다단계로 나누면 실제 사용하는 영역만 하위 테이블이 생기므로 메모리를 크게 절약합니다. 대신 변환에 트리 추적이 필요해 TLB의 중요성이 커집니다.
