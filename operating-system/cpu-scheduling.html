CPU 스케줄링(CPU Scheduling)

CPU 스케줄링은 실행 가능한 프로세스나 스레드 중에서 다음에 CPU를 사용할 대상을 고르는 운영체제의 정책입니다. 목표는 응답성, 처리량, 공정성, 대기 시간 사이의 균형을 잡는 것입니다.

구체적인 개념 설명

CPU는 빠르지만 동시에 실행하려는 작업은 많습니다. 운영체제는 준비 큐(ready queue)에 있는 작업 중 하나를 골라 CPU에 올립니다. 어떤 작업을 먼저 실행하느냐에 따라 사용자 체감 속도와 전체 처리량이 크게 달라집니다.

스케줄링 정책은 식당 줄 관리와 비슷합니다. 먼저 온 사람부터 받을 수도 있고, 금방 끝나는 주문을 먼저 처리할 수도 있고, VIP 우선순위를 둘 수도 있습니다. 어떤 방식이든 장점과 부작용이 함께 있습니다.

좋은 스케줄러는 하나의 지표만 보지 않습니다

평균 대기 시간이 짧아도 특정 작업이 계속 밀리면 공정하지 않습니다. 응답성이 좋아도 컨텍스트 스위칭이 너무 잦으면 처리량이 떨어집니다. 그래서 실제 OS 스케줄러는 우선순위, 시간 할당량, I/O 대기 패턴 등을 함께 고려합니다.

왜 필요한가?

여러 프로그램이 동시에 실행되려면 CPU 시간을 나눠야 합니다. 스케줄링이 없으면 한 작업이 CPU를 오래 붙잡아 다른 작업의 응답성이 크게 떨어질 수 있습니다. 특히 데스크톱에서는 클릭과 키보드 입력에 빠르게 반응해야 하고, 서버에서는 많은 요청을 안정적으로 처리해야 합니다.

CPU-bound 작업은 계산을 오래 하고, I/O-bound 작업은 자주 기다립니다. 스케줄러는 이런 특성을 고려해 CPU가 놀지 않도록 하면서도 사용자가 답답하지 않게 만들어야 합니다.

어떻게 동작하는가?

1) 준비 큐에서 실행 대상을 고른다

스케줄러의 선택 기준이 달라지면 같은 작업 목록도 전혀 다른 실행 순서를 가진다.

2) 대표 스케줄링 알고리즘

알고리즘방식장점단점
FCFS먼저 온 작업 먼저 실행단순하고 공정해 보임긴 작업이 앞에 있으면 convoy effect
SJF실행 시간이 짧은 작업 먼저평균 대기 시간 감소긴 작업 starvation 가능, 실행 시간 예측 어려움
Round Robin정해진 시간 조각만큼 번갈아 실행응답성 좋음, 시분할에 적합타임 퀀텀 설정이 중요
Priority우선순위 높은 작업 먼저중요 작업 빠르게 처리낮은 우선순위 starvation 가능

3) 라운드 로빈 타임라인

라운드 로빈은 각 작업에 짧은 시간 조각을 주고, 끝나지 않으면 다시 큐 뒤로 보낸다.

4) Java 코드와 스케줄링 관점

ExecutorService pool = Executors.newFixedThreadPool(4);

for (int i = 0; i < 20; i++) {
    final int taskId = i;
    pool.submit(() -> {
        // Java의 Executor는 애플리케이션 레벨에서 작업 큐를 관리한다.
        // 실제 스레드가 언제 CPU를 받을지는 OS 스케줄러가 결정한다.
        System.out.println("task " + taskId + " running");
    });
}

pool.shutdown();

시간/공간 복잡도 또는 trade-off

목표좋아지는 방향충돌할 수 있는 목표
응답 시간짧은 작업을 빨리 실행, 선점 사용컨텍스트 스위칭 증가
처리량CPU가 놀지 않게 유지개별 작업 응답성
공정성각 작업에 CPU 기회 제공중요 작업 우선 처리
대기 시간SJF 계열이 평균을 줄임긴 작업 starvation

실무에서 어디에 쓰이는가?

  • 서버 스레드 풀 설정 — CPU-bound 작업은 코어 수에 가깝게, I/O-bound 작업은 대기 시간을 고려해 더 크게 잡습니다.
  • 운영체제 지표 해석 — load average, run queue, context switch 수를 이해할 수 있습니다.
  • 배치 작업과 사용자 요청 분리 — 우선순위와 큐를 나눠 응답성을 보호합니다.
  • 컨테이너 운영 — CPU quota, nice 값, cgroup 스케줄링 영향을 이해하는 데 필요합니다.

면접에 나올 질문과 답변

CPU 스케줄링의 목적은 무엇인가요?
실행 가능한 작업 중 다음에 CPU를 사용할 대상을 정해 CPU 사용률, 처리량, 응답 시간, 공정성 사이의 균형을 잡는 것입니다.
FCFS의 convoy effect란 무엇인가요?
실행 시간이 긴 작업이 큐 앞에 있으면 뒤의 짧은 작업들이 모두 오래 기다리는 현상입니다. 단순하지만 평균 응답 시간이 나빠질 수 있습니다.
라운드 로빈에서 타임 퀀텀이 너무 작거나 크면 어떻게 되나요?
너무 작으면 컨텍스트 스위칭이 자주 발생해 오버헤드가 커집니다. 너무 크면 FCFS처럼 동작해 대화형 작업의 응답성이 떨어질 수 있습니다.
starvation은 무엇이고 어떻게 줄일 수 있나요?
특정 작업이 계속 선택되지 못해 오래 기다리는 현상입니다. 오래 기다린 작업의 우선순위를 점진적으로 높이는 aging 같은 기법으로 줄일 수 있습니다.
