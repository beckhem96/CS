프로세스 메모리 레이아웃 — code / data / heap / stack

프로세스 메모리 레이아웃은 한 프로세스가 자신의 가상 주소 공간을 어떻게 영역별로 나눠 쓰는지에 대한 약속입니다. 코드와 상수, 전역 변수, 동적 할당, 함수 호출, 공유 라이브러리가 각각 정해진 자리에 놓이고, OS와 컴파일러는 이 약속에 따라 메모리를 만들고 보호합니다.

구체적인 개념 설명

프로세스 하나에는 0번지부터 끝번지까지 자기만의 가상 주소 공간이 주어집니다. 이 공간은 단순한 평면이 아니라 역할이 다른 영역들의 합입니다. 프로그램 코드는 어디에, 전역 변수는 어디에, malloc으로 잡은 동적 메모리는 어디에, 함수 호출 시 만들어지는 지역 변수는 어디에 놓일지가 미리 정해져 있습니다.

OS는 이 영역을 읽기 전용·읽기쓰기·실행 가능처럼 권한 단위로 보호하기도 하고, 필요할 때 동적으로 늘리거나 줄이기도 합니다. 이 약속을 알면 segfault, 스택 오버플로우, 메모리 누수 같은 문제가 어느 영역의 사고인지 한 번에 파악할 수 있습니다.

주소가 낮은 쪽부터 — text · data · bss · heap → ↓ ↑ ← stack

리눅스/유닉스의 전형적인 배치는 위와 같습니다. heap은 위로 자라고, stack은 아래로 자랍니다. 그 사이의 큰 빈 공간을 mmap 영역이 채우고, 공유 라이브러리·익명 매핑·큰 malloc이 모두 거기에 놓입니다.

왜 필요한가?

한 프로세스 안의 모든 데이터를 무작위로 섞어 두면 OS가 보호도, 회수도, 디버깅도 할 수 없습니다. 영역을 나누는 이유는 명확합니다.

  • 권한 분리 — 코드는 실행만 가능하고, 데이터는 실행 불가능하게 막아야 코드 주입 공격을 어렵게 합니다.
  • 수명 구분 — 함수 호출 단위로 사라지는 값(스택), 명시적으로 해제되는 값(힙), 프로그램이 끝날 때 사라지는 값(전역)은 관리 방식이 달라야 합니다.
  • 공유 라이브러리 — 여러 프로세스가 같은 libc를 메모리에서 공유하려면 그 영역이 분리되어 있어야 합니다.
  • 디버깅·관측 — 코어 덤프, /proc/<pid>/maps, 메모리 프로파일러는 모두 이 영역 구분 위에서 동작합니다.

어떻게 동작하는가?

1) 전형적인 가상 주소 공간 배치

한 프로세스의 가상 주소 공간 — 영역마다 권한과 성장 방향이 다르다.

2) heap과 stack은 서로를 향해 자란다

heap은 brk/sbrk 시스템 콜로 위쪽으로 늘리고, stack은 함수 호출이 깊어질 때마다 아래로 늘립니다. 과거에는 둘이 맞닿으면 메모리 부족이었지만, 지금은 사이에 mmap 영역이 끼어들어 있어 그렇게 단순하지 않습니다. 대신 stack은 리미트(예: 8MB)를 두고 그걸 넘으면 segfault를 발생시킵니다 — 이게 스택 오버플로우입니다.

heap은 위로, stack은 아래로 — 그 사이의 mmap 영역이 동적으로 채워진다.

4) heap을 키우는 두 가지 길 — brk와 mmap

malloc은 직접 OS에 매번 메모리를 요청하지 않습니다. libc의 메모리 할당자(예: ptmalloc, jemalloc)가 큰 덩어리를 미리 받아두고 쪼개서 나눠 줍니다. 이때 큰 덩어리를 받아오는 통로가 두 개입니다.

  • brk / sbrk — heap 끝을 위로 밀어 올리는 시스템 콜. 작은 할당에 유리하지만, 한 번 늘어난 heap은 중간 부분만 비어도 OS에 돌려주기 어렵습니다.
  • mmap (anonymous) — mmap 영역에 새 페이지를 익명으로 매핑. 큰 할당에 유리하고, 해제 시 즉시 OS에 반환할 수 있습니다.

malloc 한 번이 어느 길을 타는지는 보통 임계값(예: glibc 기본 128KB)으로 결정됩니다. 그래서 같은 양을 한 번에 할당하느냐 잘게 쪼개 할당하느냐가 메모리 반환 행동을 다르게 만듭니다.

5) 보안 메커니즘 — 레이아웃 위에 덧씌워지는 보호

  • NX bit (W^X) — data·heap·stack에서 명령어 실행을 금지합니다. 코드 주입 공격을 막는 가장 기본적인 장치입니다.
  • ASLR (Address Space Layout Randomization) — text·heap·stack·라이브러리 시작 주소를 매 실행마다 무작위로 바꿉니다. ROP/공격에 사용되는 주소 추측을 어렵게 합니다.
  • Stack canary — 함수 호출 시 스택 프레임에 무작위 값을 박아두고, 반환 직전 그 값이 그대로인지 검사해 스택 버퍼 오버플로우를 잡습니다.
  • Guard page — 스택 끝과 다른 영역 사이에 접근 금지 페이지를 둬, 스택 오버플로우 시 즉시 segfault로 잡히게 합니다.

6) 코드로 보는 메모리 레이아웃

// C 코드 — 각 변수가 어느 영역에 들어가는지
#include <stdlib.h>

int g_init = 5;       // data
int g_zero;           // bss
const char *msg = "hello";  // 포인터 g_init 위치, "hello"는 rodata

void f(int n) {
    int local = n;          // stack
    int *buf = malloc(64);  // 포인터는 stack, 실제 64바이트는 heap
    free(buf);
}

int main() {
    f(3);
    return 0;
}

스택을 크게재귀·큰 자동 배열에 안전스레드당 메모리 비용 증가

실무에서 어디에 쓰이는가?

  • segfault 디버깅 — 잘못된 주소에 접근했을 때, 그 주소가 어느 영역인지(또는 어느 영역에도 없는지)가 첫 번째 단서입니다.
  • 메모리 누수 추적 — heap 영역만 무한히 자라면 누수, stack이 자라면 재귀 폭주, mmap 영역이 자라면 큰 객체 누수입니다.
  • 스택 크기 튜닝 — 서버에서 스레드 수가 수천 개면 스택 1MB만 해도 GB 단위가 됩니다. ulimit -s, pthread_attr_setstacksize로 조절합니다.
  • /proc/<pid>/maps 읽기 — 운영 중인 프로세스의 실제 영역 배치를 그대로 볼 수 있습니다.
  • JIT / 동적 코드 생성 — JVM·V8 같은 JIT는 mmap 영역에 R+W로 코드를 쓰고, 권한을 R+X로 바꿔 실행합니다(W^X 위반 회피).

면접에 나올 질문과 답변

프로세스의 메모리 영역은 어떻게 나뉘나요?
text(코드), rodata(상수), data(초기화 전역), bss(0 초기화 전역), heap(동적 할당), stack(자동 변수), mmap(공유 라이브러리·큰 매핑)으로 나뉩니다. text·rodata는 읽기 전용 또는 실행 가능, data·bss·heap·stack은 읽기/쓰기로 구분되며, OS가 페이지 단위 권한으로 보호합니다.
heap과 stack의 차이는 무엇인가요?
스택은 함수 호출 단위로 자동으로 늘었다 줄어드는 영역으로, 지역 변수가 놓이고 함수 반환 시 자동 해제됩니다. 힙은 malloc/new로 명시적으로 잡고 free/delete로 명시적으로 해제하는 영역입니다. 스택은 빠르고 단순하지만 크기에 한계가 있고, 힙은 유연하지만 단편화·해제 누락이 발생할 수 있습니다.
스택 오버플로우는 왜 발생하나요?
함수 호출이 너무 깊거나 큰 자동 배열을 잡아 스택 사용량이 OS가 정한 스택 크기 한도를 넘으면, 가드 페이지에 접근해 segfault가 발생합니다. 대표적으로 종료 조건이 빠진 재귀, 수십 MB짜리 지역 배열 선언이 원인입니다. 해결은 재귀 → 반복 변환, 큰 데이터를 heap으로 옮기기, 스택 크기 자체를 늘리기입니다.
data와 bss는 왜 분리되어 있나요?
둘 다 전역/정적 변수가 들어가지만, data는 0이 아닌 초기값이 있는 변수, bss는 0으로 초기화될 변수입니다. bss는 실행 파일에 실제 0을 적어둘 필요 없이 "이만큼의 영역을 0으로 만들어 달라"고만 적어두면 되므로, 실행 파일 크기를 줄이는 효과가 있습니다. 로딩 시 OS가 그만큼의 페이지를 0으로 채워 매핑합니다.
ASLR과 NX bit가 막아주는 공격은 무엇인가요?
NX bit는 데이터 영역(stack/heap)의 코드 실행을 금지해, 버퍼에 셸코드를 넣고 실행하는 전통적인 코드 주입 공격을 차단합니다. ASLR은 영역들의 시작 주소를 매 실행마다 무작위화해, 공격자가 라이브러리 함수나 가젯의 주소를 미리 알 수 없게 합니다. 둘이 함께 쓰이면 단순 버퍼 오버플로우만으로 임의 코드 실행에 도달하기가 매우 어려워집니다.
malloc은 항상 시스템 콜인가요?
아닙니다. libc의 메모리 할당자가 OS로부터 brk나 mmap으로 큰 덩어리를 받아두고, 그 안을 쪼개 사용자에게 나눠 줍니다. 대부분의 malloc 호출은 사용자 모드에서 끝나고 시스템 콜이 발생하지 않습니다. 다만 임계값(예: 128KB)을 넘는 큰 할당이나 캐시가 부족할 때만 brk/mmap이 호출됩니다.
