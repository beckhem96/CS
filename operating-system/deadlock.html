데드락(Deadlock)

데드락은 두 개 이상의 스레드(또는 프로세스)가 서로가 가진 자원을 기다리며 영원히 진행하지 못하는 상태입니다. 네 가지 조건(상호 배제·점유와 대기·비선점·순환 대기)이 모두 성립할 때 발생합니다.

구체적인 개념 설명

데드락은 좁은 외길에서 두 차가 정면으로 마주친 상황과 같습니다. 둘 다 후진하지 않으면 영원히 못 지나갑니다. 차이는 OS에서는 신호도 없고 운전자도 없다는 것입니다 — 시스템은 멈춘 줄도 모르고 그대로 정지합니다.

가장 흔한 패턴은 두 개의 락을 반대 순서로 잡으려는 경우입니다. Thread A가 락 1을 쥔 상태로 락 2를 기다리고, 동시에 Thread B가 락 2를 쥔 채 락 1을 기다리면 둘 다 멈춥니다. 이 단순한 패턴이 운영 환경에서는 호출 깊이가 깊은 코드 사이에 숨어 있어 잘 보이지 않습니다.

데드락 ≠ 라이브락 ≠ 기아

데드락은 모두 멈춘 상태, 라이브락(livelock)은 서로 양보만 하느라 일은 안 되는 상태, 기아(starvation)는 시스템은 돌지만 특정 스레드만 계속 자원을 못 받는 상태입니다. 셋 다 진행이 안 되지만 원인과 해결법이 다릅니다.

왜 필요한가?

데드락 자체가 필요한 게 아니라, 피하는 방법을 알아야 합니다. 서버 한 곳의 데드락이 커넥션 풀을 잠그고, 잠긴 풀이 요청을 막고, 막힌 요청이 다시 의존 서비스를 마비시키는 식으로 장애가 번질 수 있습니다. DB·분산 시스템·동시성 코드 어디에서도 가능성이 있어 면접 단골 주제이기도 합니다.

어떻게 동작하는가?

1) 데드락 발생의 4가지 조건(Coffman conditions)

조건의미깨면 어떻게 되나
상호 배제(Mutual Exclusion)한 자원은 한 번에 한 명만 쓸 수 있음공유 가능한 자원으로 바꾸면 데드락 불가
점유와 대기(Hold & Wait)이미 쥔 자원이 있는 채로 다른 자원을 기다림한 번에 모든 자원 요청 → 점유 채로 대기 X
비선점(No Preemption)다른 스레드가 강제로 빼앗을 수 없음OS가 빼앗아 회수 가능하면 진전 발생
순환 대기(Circular Wait)대기 그래프에 사이클 형성자원에 전역 순서를 정하고 순서대로만 잡으면 사이클 X

네 가지가 모두 성립해야 데드락이 발생합니다. 하나만 깨도 막을 수 있다는 뜻입니다.

2) 두 스레드 데드락 시각화

자원 할당 그래프에 사이클이 있으면 데드락이다.

3) 대처 방법 — 4가지 전략

전략방식비용/한계
예방(Prevention)4가지 조건 중 하나를 구조적으로 깬다 (예: 자원 순서 강제)설계 제약이 강함
회피(Avoidance)자원 요청 시 안전 상태인지 검사 후 허용 (Banker's algorithm)사전 정보 필요, 오버헤드 큼
탐지·복구(Detection & Recovery)주기적으로 사이클 탐지 후 한 스레드 종료/롤백탐지 비용·작업 손실
무시(Ostrich)거의 안 일어난다 보고 그냥 둠리눅스·일반 OS의 현실적 선택

4) 가장 실용적인 해결책 — 락 순서 강제

실무에서 가장 자주 쓰는 방법은 전역 락 순서를 정하는 것입니다. 모든 스레드가 락을 항상 같은 순서(예: A → B)로만 잡으면 사이클이 만들어질 수 없습니다.

자원에 전역 순서를 정하면 순환 대기 조건이 무너진다.

5) Java 코드

// (1) 데드락 발생 코드 — 두 스레드가 락을 반대 순서로 잡는다
class Account {
    final Object lock = new Object();
    long balance = 1000;
}

void transferBad(Account from, Account to, long amount) {
    synchronized (from.lock) {
        synchronized (to.lock) {       // 다른 스레드가 to → from 순서면 데드락
            from.balance -= amount;
            to.balance   += amount;
        }
    }
}

// (2) 락 순서 강제 — 항상 id 작은 쪽부터
void transferSafe(Account a, Account b, long amount) {
    Account first  = System.identityHashCode(a) < System.identityHashCode(b) ? a : b;
    Account second = (first == a) ? b : a;
    synchronized (first.lock) {
        synchronized (second.lock) {
            // ...
        }
    }
}

// (3) tryLock + 타임아웃 — 일정 시간 못 잡으면 포기하고 재시도
boolean transferTry(java.util.concurrent.locks.Lock l1,
                    java.util.concurrent.locks.Lock l2) throws InterruptedException {
    if (l1.tryLock(100, java.util.concurrent.TimeUnit.MILLISECONDS)) {
        try {
            if (l2.tryLock(100, java.util.concurrent.TimeUnit.MILLISECONDS)) {
                try { /* ... */ return true; }
                finally { l2.unlock(); }
            }
        } finally { l1.unlock(); }
    }
    return false;  // 데드락 의심 시 호출자가 재시도
}

시간/공간 복잡도 또는 trade-off

  • 예방은 가장 안전하지만 시스템 설계가 경직됩니다. 자원 순서를 강제하면 일부 알고리즘이 어색해집니다.
  • 회피는 이론적으로 우아하지만 모든 자원 요구를 미리 알아야 해서 일반 OS에서는 비현실적입니다.
  • 탐지·복구는 DBMS에서 흔합니다. 주기적으로 wait-for 그래프를 검사해 사이클이 있으면 한 트랜잭션을 강제 롤백합니다.
  • 무시는 데스크톱·일반 OS의 실용적 선택. 정말 드물게 일어나면 재부팅으로 해결합니다.

실무에서 어디에 쓰이는가?

  • RDBMS의 트랜잭션 락 — MySQL/PostgreSQL은 wait-for 그래프로 데드락을 탐지하고 한 트랜잭션을 자동 롤백합니다(Deadlock found).
  • 분산 락(Redis Redlock, ZooKeeper) — TTL과 fencing token으로 무한 대기를 방지합니다.
  • 커넥션 풀 + 외부 API 호출 — 풀에서 커넥션을 받고 그 안에서 또 풀을 호출하면 풀 데드락이 흔히 생깁니다.
  • JVM 진단 — jstack이나 thread dump의 "Found one Java-level deadlock" 메시지로 즉시 확인 가능합니다.

면접에 나올 질문과 답변

데드락 발생 조건 4가지를 말해보세요.
상호 배제, 점유와 대기, 비선점, 순환 대기입니다. 네 가지가 동시에 성립해야 발생하므로 하나만 깨도 막을 수 있습니다.
실제 서비스에서 데드락을 어떻게 예방하시나요?
가장 흔하게는 락 획득 순서를 전역적으로 강제합니다. 락 범위를 좁히고, tryLock에 타임아웃을 두어 무한 대기를 막고, 가능하면 락 자체를 줄여 동시성 자료구조나 메시지 패싱으로 대체합니다.
데드락과 라이브락은 어떻게 다른가요?
데드락은 모든 스레드가 정지한 상태, 라이브락은 스레드가 계속 무언가를 시도하지만 서로 양보만 해서 진전이 없는 상태입니다. 두 사람이 복도에서 같은 방향으로 비키다가 계속 서로 가로막는 상황이 라이브락입니다.
DBMS는 데드락을 어떻게 처리하나요?
대부분 탐지·복구 방식입니다. 트랜잭션 간 wait-for 그래프에서 사이클을 발견하면 비용이 가장 적은(보통 가장 최근, 가장 작은) 트랜잭션을 롤백합니다. 애플리케이션은 데드락 예외를 받아 재시도하도록 작성합니다.
은행원 알고리즘(Banker's algorithm)은 무엇인가요?
자원 요청 시 그것을 허용했을 때 시스템이 여전히 모든 프로세스를 끝낼 수 있는 "안전 상태"인지 검사하는 회피 알고리즘입니다. 총 자원량과 각 프로세스의 최대 요구량을 미리 알아야 하므로 실무 OS에는 잘 쓰이지 않습니다.
