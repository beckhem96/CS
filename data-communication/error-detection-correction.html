오류 검출 / 정정(Error Detection & Correction)

오류 검출은 전송 중 비트가 바뀌었는지 알아내는 기술이고, 오류 정정은 일부 오류를 재전송 없이 고치는 기술입니다. 대표 방식은 패리티, 체크섬, CRC와 해밍 코드이며, 핵심은 원본 데이터에 여분의 비트(redundancy)를 붙여 검증 가능하게 만드는 것입니다.

구체적인 개념 설명

실제 통신 채널에는 잡음, 감쇠, 간섭이 있습니다. 그래서 송신자가 1011001을 보냈어도 수신자는 1010001을 받을 수 있습니다. 오류 검출/정정은 이런 상황을 대비해 데이터에 규칙적인 중복 정보를 추가합니다. 수신자는 받은 데이터가 그 규칙을 만족하는지 확인하고, 만족하지 않으면 오류로 판단합니다.

검출만 하는 방식은 오류를 발견하면 재전송(ARQ: Automatic Repeat reQuest)이 필요합니다. 정정 방식은 수신자가 직접 고칠 수 있지만 더 많은 중복 비트를 필요로 합니다(FEC: Forward Error Correction).

비유 — 주민등록번호 검증 숫자

주민등록번호나 카드 번호에는 단순히 숫자만 있는 게 아니라 오타를 잡기 위한 검증 규칙이 들어갑니다. 데이터통신의 패리티·CRC도 비슷합니다. 데이터 뒤에 "검사용 숫자"를 붙여 수신자가 틀렸는지 확인하게 만듭니다.

왜 필요한가?

  • 신뢰성 확보 — 물리 채널은 완벽하지 않다. 오류를 모르면 잘못된 데이터를 정상으로 처리한다.
  • 재전송 판단 — 검출 결과가 있어야 프레임을 버리고 다시 요청할 수 있다.
  • 지연 감소 — 우주 통신·무선처럼 왕복 지연이 큰 곳은 정정 코드로 재전송을 줄인다.
  • 상위 계층 보호 — 링크 계층에서 오류를 줄이면 TCP 같은 상위 계층의 부담이 줄어든다.

어떻게 동작하는가?

1) 패리티 비트(Parity Bit)

데이터 비트의 1 개수가 짝수 또는 홀수가 되도록 1비트를 추가합니다. 짝수 패리티는 전체 1의 개수를 짝수로, 홀수 패리티는 홀수로 만듭니다.

패리티는 단일 비트 오류는 잡지만, 두 비트가 동시에 바뀌면 못 잡을 수 있다.

2) 체크섬(Checksum)

데이터를 일정 크기 단위로 나눠 더한 뒤, 그 결과를 함께 보냅니다. 수신자도 같은 방식으로 더해 결과가 맞는지 확인합니다. IP/TCP/UDP 체크섬이 대표적입니다. 단순하고 빠르지만 비트 위치가 바뀌어 합이 같아지는 오류에는 약합니다. 

// 아주 단순화한 8비트 체크섬 예시
blocks = [0x12, 0xA0, 0x03]
sum = (0x12 + 0xA0 + 0x03) & 0xFF
checksum = ~sum & 0xFF

3) CRC(Cyclic Redundancy Check)

CRC는 데이터를 이진 다항식으로 보고, 미리 정한 생성 다항식(generator)으로 나눈 나머지를 붙이는 방식입니다. 수신자는 받은 전체 비트열을 같은 생성 다항식으로 나누고, 나머지가 0이면 오류가 없다고 봅니다. 하드웨어로 빠르게 구현 가능하고 burst error(연속 비트 오류)에 강해 이더넷, 저장장치, 압축 포맷에서 널리 사용됩니다.

CRC는 산술 나눗셈이 아니라 XOR 기반의 다항식 나눗셈이다.

4) 해밍 코드(Hamming Code)

해밍 코드는 여러 패리티 비트를 특정 위치(1, 2, 4, 8, ...)에 배치해 어느 위치의 비트가 틀렸는지까지 찾아냅니다. 단일 비트 오류를 정정할 수 있고, 확장 해밍 코드는 2비트 오류 검출도 가능합니다.

패리티 검사 실패 패턴(syndrome)을 이진수로 읽으면 오류 위치가 나온다.

5) 검출 vs 정정 — ARQ와 FEC

  • ARQ — 검출 후 재전송 요청. 신뢰성은 높지만 왕복 지연이 듭니다(TCP, 링크 계층 재전송).
  • FEC — 수신자가 중복 정보로 직접 복구. 지연은 줄지만 대역폭을 더 씁니다(위성, 무선, 저장장치 ECC).
  • Hybrid ARQ — FEC로 먼저 복구를 시도하고 실패하면 재전송. LTE/5G에서 사용됩니다.

실무에서 어디에 쓰이는가?

  • 이더넷 프레임 — FCS(Frame Check Sequence)로 CRC-32 사용.
  • TCP/UDP/IP — 헤더와 데이터 일부에 체크섬 사용.
  • ECC 메모리 — 단일 비트 오류 정정, 2비트 오류 검출.
  • SSD/HDD/광디스크 — 강력한 오류 정정 코드로 저장 매체 오류 보정.
  • 무선 통신 — LDPC, Turbo code, Polar code 같은 FEC로 재전송 비용 절감.

면접에 나올 질문과 답변

오류 검출과 오류 정정의 차이는?
검출은 비트가 틀렸다는 사실만 알아내고 재전송이 필요합니다. 정정은 중복 정보를 이용해 수신자가 직접 원래 비트를 복구합니다. 정정은 지연을 줄이지만 더 많은 중복 비트와 계산 비용이 필요합니다.
패리티 비트의 한계는?
단일 비트 오류처럼 1의 개수가 홀수 개 바뀌는 오류는 잡지만, 두 비트가 동시에 바뀌면 전체 1의 짝/홀성이 그대로라 놓칠 수 있습니다. 오류 위치도 알 수 없어 정정은 불가능합니다.
CRC가 체크섬보다 강한 이유는?
CRC는 단순 덧셈이 아니라 생성 다항식 기반의 XOR 나눗셈을 사용해 연속 비트 오류(burst error)에 강합니다. 생성 다항식을 잘 고르면 특정 길이 이하의 burst error를 높은 확률로 검출할 수 있어 링크 계층에서 많이 사용합니다.
해밍 코드는 어떻게 오류 위치를 찾나요?
패리티 비트를 1,2,4,8 같은 2의 거듭제곱 위치에 배치하고 각 패리티가 담당하는 위치 집합을 검사합니다. 실패한 패리티 번호를 이진수처럼 합친 syndrome 값이 오류 위치가 되어 해당 비트를 뒤집으면 단일 비트 오류를 정정할 수 있습니다.
ARQ와 FEC는 언제 선택하나요?
왕복 지연이 작고 재전송이 쉬우면 ARQ가 효율적입니다. 위성·무선처럼 재전송이 비싸거나 지연이 큰 환경에서는 FEC가 유리합니다. 실제 이동통신은 둘을 결합한 Hybrid ARQ를 많이 씁니다.
TCP 체크섬이 있는데 이더넷 CRC가 또 필요한 이유는?
계층별 보호 범위가 다르기 때문입니다. 이더넷 CRC는 링크 한 구간의 프레임 오류를 빠르게 검출하고, TCP 체크섬은 종단 간 데이터 손상을 검출합니다. 링크 계층에서 버려야 할 프레임을 빨리 버리면 상위 계층 부담이 줄어듭니다.
