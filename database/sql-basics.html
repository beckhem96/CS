SQL 기본 — DDL / DML / DCL / TCL

SQL(Structured Query Language)은 관계형 데이터베이스와 대화하기 위한 표준 언어입니다. "이런 결과를 원해"라고 선언하면 DBMS가 어떻게 가져올지 결정합니다. 하는 일에 따라 DDL(스키마), DML(데이터), DCL(권한), TCL(트랜잭션) 네 가지 카테고리로 묶입니다.

구체적인 개념 설명

SQL은 1974년 IBM에서 만들어진 언어로, "어떻게 가져올지" 대신 "무엇을 원하는지"를 적는 선언형(declarative) 언어입니다. SELECT name FROM users WHERE age > 30이라고 쓰면, DBMS가 인덱스를 쓸지 풀스캔할지, 어떤 순서로 조인할지를 옵티마이저가 알아서 결정합니다.

SQL이 다루는 작업은 크게 네 부류로 나뉩니다. 이 네 부류는 각각 스키마를 다루는가, 데이터를 다루는가, 권한을 다루는가, 트랜잭션을 다루는가에 대응합니다. 이 분류를 알면 어떤 명령이 어떤 결과를 만드는지 직관적으로 떠올릴 수 있습니다.

한 줄 요약 — 4가지 카테고리

  • DDL (Data Definition) — 표·인덱스 같은 구조를 정의 (CREATE, ALTER, DROP, TRUNCATE)
  • DML (Data Manipulation) — 행을 읽고 쓰고 바꾸기 (SELECT, INSERT, UPDATE, DELETE)
  • DCL (Data Control) — 권한 부여·회수 (GRANT, REVOKE)
  • TCL (Transaction Control) — 트랜잭션 경계 (BEGIN, COMMIT, ROLLBACK, SAVEPOINT)

왜 필요한가?

SQL이 없다면 우리는 디스크 블록 번호를 직접 지정해 데이터를 읽고 쓰거나, 각 DB마다 다른 API를 배워야 합니다. 관계형 DB가 30년 넘게 표준이 된 핵심 이유 중 하나가 SQL입니다.

  • 표현 통일 — Oracle·MySQL·PostgreSQL이 방언은 다르지만 SQL이라는 공통 언어를 공유합니다. 한 번 배우면 어느 DB에서든 시작할 수 있습니다.
  • 옵티마이저에 위임 — "이 결과가 필요하다"만 적으면 어떻게는 DBMS가 결정합니다. 데이터가 늘어나도 같은 SQL을 그대로 쓸 수 있습니다.
  • 집합 연산 — for 루프 대신 한 번의 JOIN·GROUP BY로 집합 단위 사고를 할 수 있어 코드와 성능이 모두 단순해집니다.

어떻게 동작하는가?

1) DDL — 구조를 정의·변경

DDL은 테이블·인덱스·뷰·제약조건처럼 스키마 자체를 다룹니다. 대부분 자동 커밋되어 한 번 실행하면 이전 상태로 되돌릴 수 없는 DBMS도 있고(MySQL), 트랜잭션 안에서 롤백 가능한 DBMS도 있습니다(PostgreSQL). 

-- 테이블 생성
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    email VARCHAR(255) NOT NULL UNIQUE,
    age INT CHECK (age >= 0),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 컬럼 추가
ALTER TABLE users ADD COLUMN nickname VARCHAR(50);

-- 인덱스 생성
CREATE INDEX idx_users_email ON users(email);

-- 표 자체 삭제(스키마까지) vs 데이터만 비우기
DROP TABLE users;        -- 스키마 + 데이터 모두 사라짐
TRUNCATE TABLE users;    -- 스키마는 유지, 데이터만 빠르게 비움

2) DML — 데이터를 읽고 쓰기

가장 자주 쓰이는 카테고리입니다. SELECT는 데이터를 가져오기, 나머지는 데이터를 바꾸기에 쓰입니다. SELECT의 절(clause) 실행 순서는 작성 순서와 다릅니다 — 이게 SQL을 처음 배울 때 가장 많이 헷갈리는 지점입니다.

SELECT 절은 마지막에 실행되기에 SELECT에서 만든 별칭(alias)을 WHERE에서는 못 쓰고 ORDER BY에서는 쓸 수 있다. 
-- 조회
SELECT u.email, COUNT(*) AS post_count
FROM users u
JOIN posts p ON p.user_id = u.id
WHERE u.created_at >= '2025-01-01'
GROUP BY u.email
HAVING COUNT(*) >= 3
ORDER BY post_count DESC
LIMIT 10;

-- 삽입 / 변경 / 삭제
INSERT INTO users (email, age) VALUES ('a@x.com', 28);
UPDATE users SET age = age + 1 WHERE email = 'a@x.com';
DELETE FROM users WHERE age < 18;

3) DCL — 권한 부여와 회수

누가 어떤 테이블에 어떤 작업을 할 수 있는지 통제합니다. 보안의 1차 방어선이지만, 실무에선 OS·애플리케이션 레벨 인증과 함께 다층 방어를 구성합니다. DBA가 주로 다루지만, 백엔드 개발자도 "왜 이 계정으로는 이 쿼리가 안 되는가"를 디버깅할 때 알아야 합니다. 

-- 권한 부여
GRANT SELECT, INSERT ON orders TO app_user;

-- 권한 회수
REVOKE INSERT ON orders FROM app_user;

-- 역할 기반(많은 DB가 지원)
CREATE ROLE readonly;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO readonly;
GRANT readonly TO analytics_user;

4) TCL — 트랜잭션 경계

여러 DML을 하나의 단위로 묶어 "전부 적용되거나, 전부 무효" 둘 중 하나로 만드는 명령입니다. ACID의 'A(원자성)'를 SQL 차원에서 통제하는 통로입니다. 자세한 메커니즘은 트랜잭션 ACID 페이지에서 다룹니다. 

BEGIN;                  -- 또는 START TRANSACTION
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;                 -- 둘 다 반영 (실패 시 ROLLBACK)

-- 부분 롤백
BEGIN;
INSERT INTO logs (msg) VALUES ('start');
SAVEPOINT sp1;
UPDATE accounts SET balance = balance - 9999 WHERE id = 1;
ROLLBACK TO sp1;        -- UPDATE만 취소, INSERT는 살아있음
COMMIT;

5) SQL 한 문장이 실행되는 흐름

SQL은 선언형이지만, 그 안에서 옵티마이저라는 명령형 결정이 일어난다.

6) 자주 헷갈리는 SQL 함정

  • NULL 비교 — x = NULL은 항상 UNKNOWN입니다. IS NULL·IS NOT NULL을 써야 합니다.
  • COUNT(*) vs COUNT(col) — 후자는 NULL을 빼고 셉니다. 전자는 행의 존재만 셉니다.
  • INNER JOIN vs LEFT JOIN — INNER는 양쪽에 모두 있는 행만, LEFT는 왼쪽 모든 행 + 매칭이 없으면 오른쪽은 NULL.
  • GROUP BY + 비집계 컬럼 — SELECT에 들어간 비집계 컬럼은 GROUP BY에 모두 있어야 합니다(표준).
  • WHERE vs HAVING — 그룹화 전 필터는 WHERE, 그룹 결과 필터는 HAVING.
  • 정렬 안정성 — ORDER BY 없이 LIMIT만 쓰면 결과 순서가 보장되지 않습니다.

trade-off

설계 결정장점비용
선언형 SQL읽기 쉽고, 옵티마이저가 자동 최적화"왜 이렇게 느린가?"가 옵티마이저 선택에 의존 → EXPLAIN 필수
표준 SQL 고수이식성, 학습 비용 절감DB별 강력한 기능(RETURNING, JSON 연산자 등)을 못 씀
ORM 사용 (SQL 자동 생성)객체 모델과 일치, 보일러플레이트 감소N+1 쿼리, 복잡 쿼리에서 SQL 직시(直視) 필요
저장 프로시저로 로직 이전네트워크 왕복 감소, 권한 통제버전 관리·테스트가 어려워지고, 벤더 종속성 증가

실무에서 어디에 쓰이는가?

  • 스키마 마이그레이션 — Flyway·Liquibase는 DDL 스크립트를 버전 관리해 배포 자동화에 씁니다.
  • 분석 쿼리 — BI 도구·노트북에서 SELECT … GROUP BY … HAVING …로 집계 리포트를 만듭니다.
  • ORM 디버깅 — JPA·SQLAlchemy가 만든 SQL을 읽고 N+1, 카르테시안 곱을 식별합니다.
  • 데이터 보정 — 운영 사고 시 한 번의 UPDATE … WHERE …로 잘못된 행을 일괄 수정합니다(트랜잭션으로 감싸야 함).
  • 권한 격리 — 운영용·읽기 전용·배치용 계정을 GRANT로 분리해 사고 범위를 줄입니다.

면접에 나올 질문과 답변

SQL은 어떤 언어이고, 왜 선언형이라고 부르나요?
SQL은 관계형 DB와 대화하기 위한 표준 언어로, 데이터를 어떻게 가져올지가 아니라 무엇을 원하는지를 적습니다. 같은 결과를 만드는 방법이 여러 가지일 때, 옵티마이저가 통계와 인덱스를 바탕으로 가장 빠른 방법을 자동으로 고릅니다. 그래서 같은 SQL이 데이터 양이 늘어도 옵티마이저 선택에 따라 빠르게 동작할 수 있습니다.
DDL, DML, DCL, TCL의 차이는 무엇인가요?
DDL은 테이블·인덱스 같은 스키마를 정의·변경하는 명령(CREATE/ALTER/DROP), DML은 행 단위로 데이터를 읽고 쓰는 명령(SELECT/INSERT/UPDATE/DELETE), DCL은 권한을 주고 회수하는 명령(GRANT/REVOKE), TCL은 트랜잭션 경계를 통제하는 명령(BEGIN/COMMIT/ROLLBACK)입니다. 다루는 대상이 달라서 분리되어 있습니다.
WHERE와 HAVING은 어떻게 다른가요?
WHERE는 그룹화(GROUP BY) 전에 개별 행을 필터링하고, HAVING은 그룹화 후 집계 결과를 필터링합니다. 예를 들어 WHERE age > 30은 30살 초과 행만 그룹에 들어가게 하고, HAVING COUNT(*) >= 3은 그룹 중 행이 3개 이상인 그룹만 남깁니다. 실행 순서로 봐도 WHERE가 GROUP BY보다 먼저, HAVING은 더 나중입니다.
DELETE, TRUNCATE, DROP의 차이는 무엇인가요?
DELETE는 DML로, 조건을 걸어 행을 지우고 트랜잭션 안에서 롤백할 수 있으며 트리거가 동작합니다. TRUNCATE는 DDL에 가깝고 테이블의 모든 행을 빠르게 비우며, 보통 트랜잭션 롤백·트리거가 제한됩니다. DROP은 테이블 자체(스키마 + 데이터)를 삭제합니다. 속도와 가역성이 다르므로 의도에 맞게 골라야 합니다.
INNER JOIN과 LEFT JOIN의 차이는 무엇인가요?
INNER JOIN은 양쪽 테이블에서 조인 조건을 만족하는 행만 결과에 포함합니다. LEFT JOIN은 왼쪽 테이블의 모든 행을 보존하고, 오른쪽에 매칭이 없으면 NULL로 채웁니다. "왼쪽 모든 사용자와 그들이 쓴 글(없을 수도 있음)"을 한 번에 가져올 때 LEFT JOIN을 씁니다.
NULL 비교는 왜 조심해야 하나요?
SQL의 NULL은 "값이 없음"을 의미하며, 어떤 값과의 비교(=, <>, >)도 결과가 TRUE/FALSE가 아닌 UNKNOWN이 됩니다. 그래서 WHERE col = NULL은 어떤 행도 매칭되지 않습니다. 반드시 IS NULL·IS NOT NULL을 사용해야 하고, 집계 함수도 NULL을 어떻게 다루는지(예: COUNT(col)은 NULL 제외) 정확히 알아야 합니다.
