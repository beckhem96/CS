인덱스(Index)

인덱스는 특정 컬럼 값을 기준으로 정렬해 둔 별도 자료구조로, "이 값이 어느 행에 있는지"를 빠르게 찾도록 합니다. 대부분의 RDBMS는 B+Tree를 기본으로 사용하며, 이 자료구조의 특성이 곧 SQL의 인덱스 동작 규칙(범위 검색·정렬·복합 인덱스 순서)을 결정합니다.

구체적인 개념 설명

인덱스는 책 뒤의 색인과 비슷합니다. 단어가 등장하는 페이지를 쪽 번호 순으로 찾는 대신, 가나다순으로 정렬된 색인에서 단어 → 쪽 번호로 점프할 수 있죠. DB의 인덱스도 컬럼 값 → 데이터 위치를 빠르게 찾기 위해 정렬된 트리로 따로 보관합니다.

인덱스의 본질은 "값이 정렬되어 있다"는 것입니다. 그래서 동등 비교(=), 범위(>, <, BETWEEN), 정렬(ORDER BY), 그룹핑(GROUP BY)에서 강력합니다. 반대로 함수로 감싸거나 앞이 와일드카드인 LIKE처럼 정렬 순서를 깨뜨리는 연산은 인덱스를 못 씁니다.

비유 — 도서관 책

책은 보통 청구기호 순으로 꽂혀 있습니다(이게 클러스터드 인덱스의 개념). 같은 책을 저자명·제목으로도 빠르게 찾고 싶으면 별도의 저자 색인·제목 색인을 만들고, 거기서 청구기호로 점프합니다(논클러스터드 인덱스). 하나의 책에 여러 색인이 있을 수 있는 것처럼, 한 테이블에도 여러 인덱스를 만들 수 있습니다.

왜 필요한가?

  • 풀 테이블 스캔 회피 — 1억 행 중 1행을 찾으려고 1억 행을 모두 읽는 건 너무 비싸다.
  • 정렬·집계 비용 절감 — 이미 정렬된 트리에서 읽으면 별도 정렬이 불필요.
  • 유일성 보장 — UNIQUE 인덱스는 중복 키 삽입을 차단.
  • 외래 키 검사 가속 — 부모 행 존재 여부 확인을 빠르게.

대신 모든 인덱스는 공짜가 아닙니다 — 쓰기 비용과 저장 공간을 잡아먹습니다.

어떻게 동작하는가?

1) B+Tree의 구조

B+Tree는 모든 데이터 키를 리프에만 두고, 리프끼리 연결되어 범위 스캔에 강하다.

B+Tree의 핵심 성질 세 가지:

  • 모든 리프가 같은 깊이 — 어떤 키든 같은 횟수의 디스크 I/O로 도달.
  • 리프 노드만 실제 데이터(또는 데이터 포인터)를 가진다 — 내부 노드는 라우팅용.
  • 리프끼리 양방향 링크 — BETWEEN 30 AND 60 같은 범위 검색은 한 번 내려간 뒤 옆으로 쭉.

한 노드에 100~200개의 키를 담을 수 있어, 1억 행이라도 트리 높이는 보통 3~4단계입니다. 즉, 디스크 I/O 3~4번이면 어떤 키든 찾을 수 있다는 뜻이죠.

2) 클러스터드 vs 논클러스터드(보조)

클러스터드는 데이터 자체를 정렬하므로 한 번에 도달, 논클러스터드는 리프에서 한 번 더 점프해야 한다.

MySQL InnoDB는 PK가 곧 클러스터드 인덱스이고, 보조 인덱스는 PK 값을 가리킵니다. 그래서 InnoDB에서 PK가 너무 길거나 자주 바뀌면 모든 보조 인덱스가 같이 영향을 받습니다. PostgreSQL은 모든 인덱스가 보조 인덱스이고 데이터는 heap에 저장됩니다(클러스터드 X).

3) 복합 인덱스(Composite Index) — 컬럼 순서가 전부

복합 인덱스 (a, b, c)는 a로 정렬된 안에서 b로 정렬되고 그 안에서 c로 정렬됩니다. 그래서 다음만 효율적으로 사용할 수 있습니다:

  • WHERE a = ? ✅
  • WHERE a = ? AND b = ? ✅
  • WHERE a = ? AND b = ? AND c = ? ✅
  • WHERE b = ? ❌ (a 없이는 정렬 의미가 사라짐)
  • WHERE a = ? AND c = ? 🔶 (a까지만 인덱스, c는 별도 필터)

범위 조건이 들어가면 그 뒤 컬럼은 인덱스로 정렬을 활용할 수 없습니다. WHERE a = 1 AND b > 10 AND c = 5이면 c는 정렬 활용 불가입니다. 그래서 복합 인덱스 설계는 "동등 → 범위 → 정렬" 순서로 컬럼을 배치하는 게 보통입니다.

4) 커버링 인덱스(Covering Index)

쿼리에 필요한 모든 컬럼이 인덱스 안에 들어 있으면 테이블을 읽지 않고 인덱스만 읽고 끝냅니다. 이를 커버링 인덱스라고 하고, EXPLAIN에서 "Using index"로 표시됩니다. 

-- 인덱스: (user_id, created_at, status)
SELECT status FROM orders
WHERE user_id = 100 AND created_at >= '2026-01-01';
-- 인덱스만 읽고 끝 → 매우 빠름

5) 인덱스를 쓰지 못하는 패턴

  • 컬럼에 함수/연산 — WHERE DATE(created_at) = '2026-05-08' → created_at >= ... AND < ...로 바꾸기.
  • 앞 와일드카드 LIKE — LIKE '%abc'는 사용 불가, LIKE 'abc%'는 가능.
  • 타입 불일치 암묵적 형변환 — VARCHAR 컬럼을 숫자로 비교하면 함수 적용과 같은 효과.
  • OR로 묶인 다른 컬럼들 — 옵티마이저가 합치기 어려워 풀스캔할 수 있다(UNION ALL로 분리하면 풀릴 때 있음).
  • NOT, <> — 선택도가 너무 커서 옵티마이저가 인덱스를 포기하는 경우.
  • 선택도가 낮은 컬럼 — 성별처럼 값이 2~3개뿐이면 인덱스보다 풀스캔이 나을 수 있다.

6) 인덱스 쓰기 비용

INSERT/UPDATE/DELETE 시 모든 관련 인덱스도 함께 갱신됩니다. 인덱스가 N개면 쓰기 비용도 그만큼 증가하고, 페이지 분할(Split)이 발생하면 더 비싸집니다. "읽지 않을 컬럼에 인덱스를 만들지 마라"가 기본 원칙입니다.

7) Java 예시 — 인덱스를 의식한 쿼리 작성

// 좋지 않은 예 — created_at에 함수 적용
String bad = "SELECT * FROM orders WHERE DATE(created_at) = ?";

// 좋은 예 — 범위로 변환 (인덱스 사용 가능)
String good = """
    SELECT * FROM orders
    WHERE created_at >= ? AND created_at < ?
    """;

PreparedStatement ps = conn.prepareStatement(good);
ps.setTimestamp(1, Timestamp.valueOf("2026-05-08 00:00:00"));
ps.setTimestamp(2, Timestamp.valueOf("2026-05-09 00:00:00"));

시간/공간 복잡도 또는 trade-off

연산풀 스캔인덱스왜 그런가?
단일 키 조회O(N)O(log N)B+Tree 높이 비례
범위 조회O(N)O(log N + k)리프 링크드 리스트로 옆으로 스캔
INSERTO(1)*O(log N) × 인덱스 수인덱스마다 정렬 위치 갱신
저장 공간데이터만데이터 + Σ인덱스인덱스도 트리 구조라 작지 않음
선택도가 낮을 때나음나쁨(랜덤 I/O 많음)대부분 행을 읽으니 순차가 유리

* 인덱스가 없을 때의 INSERT만. 인덱스가 있다면 그만큼 비용이 추가된다.

실무에서 어디에 쓰이는가?

  • PK / FK / UNIQUE — 자동으로 인덱스가 생성된다.
  • WHERE / JOIN / ORDER BY에 자주 쓰는 컬럼 — 통계 기반으로 추가.
  • 커버링 인덱스로 핫 쿼리 최적화 — 자주 호출되는 좁은 컬럼셋을 통째로 인덱스에 포함.
  • 전문 검색 — Full-text 인덱스(MySQL FULLTEXT, PostgreSQL GIN+tsvector).
  • JSON / 배열 / 지리좌표 — GIN, GiST 같은 특수 인덱스.

면접에 나올 질문과 답변

왜 RDBMS는 해시 테이블이 아니라 B+Tree를 기본으로 쓰나요?
해시 테이블은 동등 비교는 O(1)로 빠르지만 범위 검색과 정렬에 약합니다. B+Tree는 모든 키가 정렬되어 있고 리프끼리 연결돼 있어, BETWEEN·ORDER BY·GROUP BY가 효율적입니다. 디스크 블록 단위 I/O와도 잘 맞아 한 노드에 수백 개 키를 담아 트리 높이를 매우 낮게 유지할 수 있습니다.
복합 인덱스의 컬럼 순서는 왜 중요한가요?
(a, b, c) 인덱스는 a로 먼저 정렬되고 그 안에서 b로, 그 안에서 c로 정렬됩니다. 그래서 a 없이 b만으로 검색하면 정렬을 활용할 수 없어 사실상 인덱스를 사용할 수 없습니다. 설계 가이드는 "동등 비교 → 범위 비교 → ORDER BY 컬럼" 순서로 배치하는 것입니다.
MySQL InnoDB의 클러스터드 인덱스가 왜 PK 설계에 영향을 주나요?
InnoDB는 PK가 곧 클러스터드 인덱스이고, 보조 인덱스는 PK 값을 함께 저장합니다. 그래서 PK가 길수록 모든 보조 인덱스의 크기도 같이 커지고, PK가 자주 바뀌면 보조 인덱스도 같이 갱신됩니다. 보통 짧고 변하지 않는 정수 PK(또는 ULID/Snowflake)를 권장합니다.
커버링 인덱스가 왜 빠른가요?
쿼리에서 SELECT/WHERE/ORDER BY에 쓰이는 모든 컬럼이 인덱스에 포함되어 있으면, 옵티마이저가 테이블 행을 읽지 않고 인덱스만 스캔해 결과를 만듭니다. 인덱스는 보통 행보다 작고 정렬되어 있어 I/O 양이 줄고 정렬도 생략됩니다. EXPLAIN에서 "Using index"로 확인할 수 있습니다.
인덱스를 추가했는데도 풀스캔하는 경우가 있나요?
있습니다. 컬럼에 함수가 적용되거나, 타입이 다르거나, 앞 와일드카드 LIKE처럼 정렬을 못 쓸 때입니다. 또 옵티마이저가 통계상 인덱스 사용보다 풀스캔이 빠르다고 판단하기도 합니다 — 선택도가 너무 낮을 때(예: 성별), 행이 적을 때, OR로 묶인 다른 컬럼이 인덱스가 없을 때 등입니다.
인덱스를 많이 추가하면 어떤 문제가 생기나요?
쓰기 비용과 저장 공간이 인덱스 수에 비례해 증가합니다. INSERT/UPDATE/DELETE마다 모든 관련 인덱스를 정렬 유지하면서 갱신해야 하므로, 페이지 분할이 잦아지면 단편화도 심해집니다. 그래서 "자주 쓰이지 않는 인덱스는 제거"가 정기 점검 항목입니다.
