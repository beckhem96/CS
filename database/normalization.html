정규화(Normalization)

정규화는 테이블을 더 작은 단위로 쪼개 중복을 제거하고, 결과적으로 삽입·수정·삭제 시 데이터가 어긋나는 이상(anomaly)을 막는 설계 절차입니다. 판단 기준은 "어떤 열이 어떤 열을 결정하는가" — 즉 함수 종속(Functional Dependency)입니다.

구체적인 개념 설명

한 테이블에 너무 많은 정보를 넣으면 같은 사실이 여러 행에 중복되어 저장됩니다. 이 상태에서 어떤 행을 고치면 다른 행과 어긋나고, 어떤 행을 지우면 멀쩡한 정보까지 함께 사라지죠. 정규화는 "한 테이블은 한 가지 주제에 대한 사실만 담는다"는 원칙을 단계적으로 적용해 이를 방지합니다.

각 단계는 이전 단계를 만족한 위에서 더 강한 조건을 추가합니다. 현실 설계는 보통 3NF 또는 BCNF까지 진행한 뒤, 성능을 위해 일부를 의도적으로 비정규화합니다.

비유 — 학원 출석부

한 종이에 학생 정보·강의 정보·수강 정보를 다 적으면, 학생이 강의를 5개 들으면 학생 이름·연락처도 5번 적게 됩니다. 연락처가 바뀌면 5번을 다 고쳐야 하고, 한 군데만 고치면 어떤 곳은 옛날 번호, 어떤 곳은 새 번호로 어긋나게 됩니다. 학생 표 / 강의 표 / 수강 표로 나누면 이런 일이 사라지죠. 정규화의 핵심도 이것입니다.

왜 필요한가? — 이상 현상(Anomaly)

중복을 그냥 두면 다음 세 가지 이상이 발생합니다.

  • 삽입 이상(Insertion anomaly) — 학생이 아직 어떤 강의도 안 듣는다면, 학생 정보를 넣을 자리가 없다.
  • 수정 이상(Update anomaly) — 학생 연락처가 바뀌면 그 학생이 등장하는 모든 행을 다 고쳐야 하고, 한 군데라도 빠지면 데이터가 어긋난다.
  • 삭제 이상(Deletion anomaly) — 학생이 마지막 강의를 취소하면 행 자체가 사라져 학생 자체의 존재까지 같이 사라진다.

어떻게 동작하는가?

1) 함수 종속(FD)이란?

"X가 정해지면 Y도 하나로 정해진다"를 X → Y로 표기하고 X는 결정자, Y는 종속자라고 부릅니다. 예: 학번 → 학생이름, (학번, 강의코드) → 성적. 정규화는 본질적으로 이 종속 관계를 깔끔하게 정리하는 작업입니다.

2) 단계별 변화 시각화

정규화는 "한 테이블 = 한 결정자 = 한 가지 사실"이 되도록 단계적으로 분리하는 작업이다.

3) 1NF — 원자값(Atomic)

모든 컬럼이 더 이상 쪼갤 수 없는 단일 값이어야 합니다. "DB, OS, 알고리즘" 같은 다중값이나 반복 그룹이 있으면 1NF가 아닙니다. 이걸 깨면 WHERE 강의 = 'DB' 같은 자연스러운 검색이 어려워집니다. 

-- 1NF 위반: 한 칸에 콤마로 여러 값
| 학번 | 강의                  |
| 100  | DB, OS, 알고리즘      |

-- 1NF 만족: 한 행 = 한 사실
| 학번 | 강의      |
| 100  | DB        |
| 100  | OS        |
| 100  | 알고리즘  |

4) 2NF — 부분 함수 종속 제거

복합 키에서 키의 일부만으로 결정되는 비키 컬럼이 있으면 2NF 위반입니다. 앞 예에서 (학번, 강의코드) → 성적은 키 전체에 의존하지만, 학번 → 이름은 키의 일부에만 의존합니다. 이 경우 학생 테이블로 분리합니다.

5) 3NF — 이행 종속 제거

비키 컬럼이 다른 비키 컬럼을 통해 키에 종속되는 경우(키 → A → B)를 막습니다. 예: 학번 → 학과 → 학과사무실이면 학과사무실은 학과를 통해서만 결정되므로, 학과 테이블로 분리해야 합니다. 

-- 3NF 위반: 학과사무실은 학과에 의존, 학번에는 이행적으로 의존
학생(학번 PK, 이름, 학과, 학과사무실)

-- 3NF 만족
학생(학번 PK, 이름, 학과 FK)
학과(학과 PK, 학과사무실)

6) BCNF — 모든 결정자가 후보키

3NF를 만족해도 결정자가 후보키가 아닐 때가 있습니다. 예: (학생, 과목) → 교수이면서 교수 → 과목인 경우, 교수는 과목을 결정하지만 후보키는 아닙니다. 이때 BCNF로 분해해 모든 결정자가 키가 되도록 만듭니다.

7) 4NF / 5NF — 다치 종속과 조인 종속

한 키에 두 개의 독립적인 다중값이 함께 들어가면 4NF 위반입니다(예: 학생의 취미와 자격증을 한 테이블에 함께 두면 행이 곱으로 늘어남). 5NF는 더 일반적인 조인 종속을 다루는데, 실무에서는 거의 등장하지 않습니다.

8) 비정규화(Denormalization)

정규화로 테이블이 잘게 쪼개지면 조인이 늘어 읽기 비용이 커집니다. 대규모 읽기 위주 서비스나 분석 쿼리에서는 일부러 중복을 허용해 조인을 줄입니다 — 단, 일관성을 어떻게 유지할지를 함께 설계해야 합니다(트리거, 캐시 무효화, 배치 갱신 등).

시간/공간 복잡도 또는 trade-off

관점고도화된 정규화(BCNF~)비정규화왜 그런가?
저장 공간적음(중복 제거)큼(복제)같은 사실이 여러 군데 저장
쓰기 일관성쉬움(한 곳만 수정)어려움(여러 곳 갱신)중복된 행을 모두 동기화해야
읽기 성능조인 多 → 느릴 수 있음조인 少 → 빠름JOIN은 정렬·해시 비용 동반
설계 복잡도스키마 명확중복 위치를 추적해야비정규화는 의도적 중복 관리 비용을 만든다
대표 사용처OLTP, 마스터 데이터OLAP, 캐시 테이블업무 패턴이 쓰기 vs 읽기 중심인지

실무에서 어디에 쓰이는가?

  • OLTP 서비스 — 결제·회원·주문처럼 일관성이 중요한 시스템은 보통 3NF/BCNF로 시작.
  • 분석/리포팅(OLAP) — 스타 스키마는 의도적 비정규화. 팩트 테이블에 차원 일부를 복제해 조인 비용을 줄임.
  • 캐시 테이블 / 머티리얼라이즈드 뷰 — 정규화된 원본은 그대로 두고, 자주 조회되는 집계만 별도로 비정규화해 보관.
  • 도메인 모델링 — DDD의 애그리거트 경계를 잡을 때 함수 종속 분석이 출발점이 된다.

면접에 나올 질문과 답변

정규화가 왜 필요한가요?
한 테이블에 여러 주제의 사실을 섞어 두면 같은 정보가 중복되어 삽입·수정·삭제 시 이상이 발생합니다. 한 컬럼이 다른 컬럼에 종속되는 관계(함수 종속)를 기준으로 테이블을 분리해 "한 테이블은 한 가지 주제만 담는다"를 만드는 절차가 정규화입니다.
2NF와 3NF의 차이를 예시로 설명해 주세요.
2NF는 복합 키의 일부에만 의존하는 비키 컬럼을 제거합니다. 예: (학번, 강의코드) → 이름에서 이름은 학번에만 의존하므로 학생 테이블로 분리. 3NF는 비키 컬럼이 다른 비키 컬럼을 통해 결정되는 경우를 제거합니다. 예: 학번 → 학과 → 학과사무실에서 학과사무실은 학과에 의존하므로 학과 테이블로 분리합니다.
3NF와 BCNF의 차이는 뭔가요?
3NF까지는 후보키가 아닌 결정자가 존재할 수 있지만, BCNF는 모든 결정자가 후보키여야 합니다. 예: (학생, 과목) → 교수이면서 교수 → 과목인 경우 3NF는 만족할 수 있지만, 교수가 후보키가 아니어서 BCNF 위반입니다. BCNF로 분해하면 종속 관계가 더 깔끔해지지만, 일부 함수 종속이 보존되지 않을 수 있어 항상 분해가 가능한 것은 아닙니다.
언제 비정규화를 정당화할 수 있나요?
읽기가 압도적으로 많고, 조인 비용이 응답 시간의 병목이 되며, 일관성 갱신을 어떻게 유지할지가 명확할 때 정당화됩니다. 대표 사례는 OLAP 스타 스키마, 자주 조회되는 집계 컬럼 캐싱, 검색용 머티리얼라이즈드 뷰입니다. 무작정 조인이 싫어서 합치는 건 부채로 돌아옵니다.
정규화와 NoSQL의 임베디드 문서 모델은 어떻게 다른가요?
관계형의 정규화는 중복 제거가 우선이고, NoSQL의 문서 모델은 한 번에 읽을 단위가 우선입니다. 예를 들어 주문 안에 상품 라인을 임베드하면 한 쿼리로 다 읽지만, 상품 정보가 변경되면 모든 주문을 갱신해야 합니다. 결국 둘 다 같은 trade-off(쓰기 일관성 vs 읽기 비용)를 다루는 다른 출발점일 뿐입니다.
