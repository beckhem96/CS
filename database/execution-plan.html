실행 계획(Execution Plan / EXPLAIN)

실행 계획은 옵티마이저가 SQL을 어떻게 실행할지 결정한 연산 트리입니다. 각 노드에는 사용한 접근 방식(Seq/Index/Bitmap Scan, NLJ/Hash/Sort-Merge), 예상 비용(cost), 예상 행 수(rows)가 적혀 있어 "이 쿼리가 왜 느린가?"의 단서가 모두 들어 있습니다.

구체적인 개념 설명

DB는 SQL을 받으면 파서 → 옵티마이저 → 실행기 순으로 처리합니다. 옵티마이저는 통계와 가능한 접근 방식들을 조합해 비용이 가장 낮은 계획을 만들고, 실행기는 그 트리를 따라 데이터를 가져옵니다. EXPLAIN은 그 트리를 출력하는 명령으로, 튜닝의 출발점입니다.

"쿼리가 느리다"는 결국 예상과 실제가 어긋났다는 뜻이고, 그 어긋남은 거의 항상 EXPLAIN을 보면 보입니다. 통계가 오래되었거나, 인덱스를 못 쓰는 표현식이 끼었거나, 잘못된 조인 알고리즘이 골라졌거나입니다.

비유 — 길 안내 앱의 경로 옵션

같은 출발-도착이라도 길 안내 앱은 여러 경로를 비교합니다 — 거리·교통량·통행료를 가중해 최저 비용을 고르죠. DB의 옵티마이저도 똑같이 통계를 바탕으로 여러 후보를 비교해 가장 싼 계획을 고릅니다. EXPLAIN은 그 결과물을 보여주는 화면입니다.

왜 필요한가?

  • 병목 진단 — 어느 노드에서 행이 부풀거나 시간이 걸리는지 확인.
  • 인덱스 효과 검증 — 인덱스를 추가했을 때 옵티마이저가 실제로 쓰는지 확인.
  • 통계 정합성 점검 — 예상 행 수와 실제 행 수의 차이가 크면 ANALYZE 필요.
  • 회귀 방지 — 새 쿼리가 운영에 들어가기 전 plan을 검토해 풀스캔·카티션을 미연에 차단.

어떻게 동작하는가?

1) 옵티마이저의 큰 그림

옵티마이저는 통계를 기반으로 여러 후보 중 비용이 가장 낮은 계획을 선택한다.

2) PostgreSQL EXPLAIN 출력 예

EXPLAIN ANALYZE
SELECT u.name, COUNT(*) AS cnt
FROM users u JOIN orders o ON o.user_id = u.id
WHERE u.created_at >= '2026-01-01'
GROUP BY u.name;

-- ─────────────────────────────────────────────────────────────
-- HashAggregate  (cost=1234.5..1240.0 rows=200 width=24)
--                (actual time=45.2..45.6 rows=187 loops=1)
--   Group Key: u.name
--   -> Hash Join  (cost=120.0..1100.0 rows=10000 width=20)
--                 (actual time=2.1..40.0 rows=9876 loops=1)
--        Hash Cond: (o.user_id = u.id)
--        -> Seq Scan on orders o  (cost=0..800 rows=80000 width=12)
--        -> Hash  (cost=100..100 rows=2000 width=12)
--             -> Index Scan using users_created_idx on users u
--                  Index Cond: (created_at >= '2026-01-01')
-- Planning Time: 0.5 ms
-- Execution Time: 46.0 ms

읽는 순서는 안에서 바깥으로입니다. 들여쓰기가 깊은 노드가 먼저 실행되고, 결과가 위로 흘러 올라가 합쳐집니다.

3) 핵심 숫자 — cost, rows, actual time

필드의미주의할 점
cost=시작..총옵티마이저가 추정한 상대 비용시간 단위 X. 같은 쿼리 안에서 비교용
rows=N예상 출력 행 수실제와 차이가 크면 통계 불량
width=W행 평균 바이트SELECT * 줄이면 작아짐
actual time실제 실행 시간(ms)EXPLAIN ANALYZE에서만 등장
loops그 노드가 반복 실행된 횟수NLJ 안쪽 노드는 loops > 1
Buffers읽은 블록 수(공유/임시)BUFFERS 옵션 켜야 보임. I/O 진단

4) 자주 보는 접근 방식(Access Path)

Seq Scan (Full Table Scan)

테이블을 처음부터 끝까지 훑는다. 행이 적거나, 인덱스가 없거나, 인덱스 사용보다 풀스캔이 빠를 때 선택된다.

Index Scan

인덱스를 따라가며 필요한 행만 가져옴. 동등 비교/좁은 범위에 적합.

Index Only Scan (커버링)

쿼리에 필요한 컬럼이 모두 인덱스에 있어 테이블을 안 읽고 끝남. 가장 빠른 형태.

Bitmap Index Scan / Bitmap Heap Scan

여러 인덱스 조건을 비트맵으로 합쳐 한 번에 테이블을 방문. 큰 결과 집합·OR 조건에 효과적.

Index Range Scan / Skip Scan

범위 검색에 인덱스 적용. 일부 RDBMS는 복합 인덱스의 선두 컬럼이 빠진 경우에도 skip scan을 시도하지만 비용이 큼.

5) MySQL EXPLAIN 형식

EXPLAIN FORMAT=TRADITIONAL
SELECT * FROM orders WHERE user_id = 100;

-- id | select_type | table  | type   | possible_keys | key            | rows | Extra
--  1 | SIMPLE      | orders | ref    | idx_user      | idx_user       |   12 | Using where
  • type — 가장 중요한 컬럼. const > eq_ref > ref > range > index > ALL 순으로 좋음.
    • const/eq_ref: PK·UNIQUE로 1행 확정.
    • ref: 일반 인덱스의 동등 비교.
    • range: 인덱스 범위 검색.
    • index: 인덱스 풀스캔.
    • ALL: 테이블 풀스캔. 거의 항상 빨간불.
  • Extra — Using index(커버링), Using filesort(메모리/디스크 정렬), Using temporary(임시 테이블), Using join buffer 등.

6) 진단 체크리스트

  • 예상 rows ≫ 실제 rows 또는 그 반대 — 통계 갱신(ANALYZE / ANALYZE TABLE).
  • Seq Scan인데 인덱스가 있다 — 함수 적용·타입 불일치·선택도 낮음 의심.
  • Hash Join에 큰 spill — work_mem(MySQL은 join_buffer_size) 부족 또는 build 측이 너무 큼.
  • Nested Loop의 loops가 매우 큼 — 외부 행이 폭발한 상태. 안쪽 인덱스 점검.
  • Using filesort — ORDER BY가 인덱스로 처리되지 않음. 정렬 컬럼 인덱스 또는 LIMIT 활용.

7) Java 프로파일링 예시

// PostgreSQL — 운영 진단용 EXPLAIN ANALYZE
String sql = "EXPLAIN (ANALYZE, BUFFERS, VERBOSE) " + targetSql;
try (PreparedStatement ps = conn.prepareStatement(sql)) {
    setParams(ps, params);
    ResultSet rs = ps.executeQuery();
    while (rs.next()) {
        log.info("{}", rs.getString(1));
    }
}
// 운영 DB에선 ANALYZE가 실제 쿼리를 실행하므로 INSERT/UPDATE는 트랜잭션 후 ROLLBACK으로 감싸기.

8) 자주 등장하는 옵티마이저 힌트

  • PostgreSQL — 표준 힌트는 없음. SET enable_seqscan = off 같은 세션 변수, 또는 pg_hint_plan 확장.
  • MySQL — USE INDEX, FORCE INDEX, STRAIGHT_JOIN.
  • Oracle — /*+ INDEX(t idx_x) */, /*+ USE_HASH(a b) */.
  • 힌트는 통계가 시간이 지나면 잘못된 강제가 될 수 있어, 가능하면 인덱스/쿼리 형태로 풀어내는 게 우선.

시간/공간 복잡도 또는 trade-off

  • EXPLAIN(설명) — 비용만 추정. 즉시 반환.
  • EXPLAIN ANALYZE(실행) — 실제 실행하며 측정. UPDATE/DELETE는 영향 발생 — 트랜잭션으로 감싸 ROLLBACK.
  • BUFFERS 옵션 — I/O 패턴 진단에 매우 유용. cache hit인지 disk read인지 보임.
  • plan stability — 통계가 갱신되거나 행 분포가 변하면 plan이 달라질 수 있음. plan baseline/hint plan으로 고정 가능.

실무에서 어디에 쓰이는가?

  • 슬로우 쿼리 진단 — slow_query_log → 의심 SQL → EXPLAIN.
  • 릴리스 전 회귀 검사 — 새 쿼리가 인덱스를 쓰는지 PR 단계에서 확인.
  • 인덱스 후보 결정 — 어떤 컬럼·복합 순서가 효과 있는지 plan으로 검증.
  • 통계 점검 — 예상 vs 실제 차이를 추적해 ANALYZE 주기 결정.

면접에 나올 질문과 답변

EXPLAIN의 cost는 시간(ms)인가요?
아닙니다. cost는 옵티마이저가 정의한 상대 비용 단위로, 같은 쿼리의 다른 후보 plan을 비교할 때 의미가 있습니다. 실제 시간은 EXPLAIN ANALYZE의 actual time에서 확인할 수 있습니다. cost가 낮은 plan이 늘 빠른 것은 통계가 정확할 때만 성립합니다.
예상 rows와 실제 rows가 크게 다르면 어떻게 해야 하나요?
통계가 오래되었거나 분포가 한쪽으로 치우친(skewed) 신호입니다. ① ANALYZE로 통계 갱신, ② 컬럼에 히스토그램이 없는 경우 추가, ③ 그래도 차이가 크면 옵티마이저가 잘못 판단할 가능성이 있어 인덱스/쿼리 형태로 우회하거나 힌트를 검토합니다.
MySQL의 type=ALL과 type=index의 차이는?
ALL은 테이블 자체를 풀스캔하는 것이고, index는 인덱스를 처음부터 끝까지 풀스캔하는 것입니다. 둘 다 비싸지만 index는 인덱스 폭만 읽으니 ALL보다 적게 읽습니다. 대부분은 type을 ref/range 이상으로 끌어올리는 게 목표입니다.
"Using filesort"가 보이면 항상 나쁜가요?
반드시 그렇진 않습니다. 작은 결과의 정렬은 메모리에서 빠르게 끝나 무시할 수준입니다. 문제는 결과가 크고 정렬 메모리가 부족해 디스크 임시 파일로 spill 되는 경우입니다. ORDER BY에 맞는 인덱스가 있다면 filesort 자체를 없앨 수 있고, LIMIT을 함께 쓰면 부분 정렬로 충분할 수도 있습니다.
Index Only Scan(커버링)은 왜 가장 빠른가요?
쿼리에 필요한 모든 컬럼이 인덱스 안에 있어서 테이블 행을 전혀 읽지 않습니다. 인덱스는 보통 행보다 작고, 정렬돼 있어 정렬·집계도 추가 비용 없이 끝나는 경우가 많습니다. SELECT 컬럼을 좁히고 INCLUDE 절(또는 컬럼을 인덱스 키에 추가)로 커버 범위를 넓히는 식으로 유도합니다.
EXPLAIN ANALYZE를 운영 DB에서 막 써도 되나요?
SELECT는 안전하지만, UPDATE/DELETE/INSERT가 포함된 쿼리에는 실제 변경이 발생합니다. 반드시 트랜잭션으로 감싸 BEGIN; ... ROLLBACK; 하거나, 별도 스테이징 환경에서 분석하세요. 무거운 쿼리는 여전히 부하를 만들기 때문에 운영 영향을 고려해 시간대를 골라 사용합니다.
