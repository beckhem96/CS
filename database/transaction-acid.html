트랜잭션과 ACID

트랜잭션(Transaction)은 "하나의 논리적 작업 단위" — 여러 SQL을 묶어 전부 성공하거나 전부 취소되도록 합니다. ACID는 그 트랜잭션이 보장해야 하는 4가지 성질 — Atomicity(원자성), Consistency(일관성), Isolation(격리성), Durability(지속성)입니다. 각 성질은 서로 다른 메커니즘(언두 로그, 제약조건, 락/MVCC, WAL+fsync)으로 실현됩니다.

구체적인 개념 설명

"A 계좌에서 1만원을 빼서 B 계좌에 넣는다"는 작업을 생각해 봅시다. 이 두 단계 사이에 시스템이 죽으면 돈이 사라지거나 두 배가 되어버릴 수 있죠. 트랜잭션은 이런 일을 막기 위해 두 작업을 묶고, "성공이면 commit, 실패면 rollback"의 약속을 보장합니다.

ACID는 트랜잭션이 만족해야 하는 4가지 약속을 한 번에 모은 이름입니다. 각 글자는 서로 독립적이지만, 실제로는 함께 동작해야 의미가 있습니다.

비유 — 송금 영수증

은행 창구에서 돈을 보내고 받을 때, "출금"과 "입금"이 따로따로 처리되는 것 같지만 결국 영수증 한 장에 묶여 있습니다. 중간에 실수가 생기면 둘 다 취소되고, 정상이면 둘 다 처리되죠. 트랜잭션이 정확히 그 역할을 합니다.

왜 필요한가?

  • 중간 실패에 대한 안전망 — 네트워크 단절, 전원 차단, OOM 같은 상황에서도 데이터가 어긋나지 않도록.
  • 동시성 환경의 일관성 — 여러 사용자가 같은 행을 동시에 수정해도 결과가 깨지지 않도록.
  • 비즈니스 불변식 유지 — "총합이 변하면 안 된다", "재고 ≥ 0" 같은 규칙을 항상 만족.
  • 복구의 단위 — 장애 후 재시작했을 때 미완료 트랜잭션은 되돌리고, 커밋된 트랜잭션은 다시 적용해야 한다.

어떻게 동작하는가?

1) 트랜잭션의 라이프사이클

트랜잭션은 BEGIN → ACTIVE → (Partially Committed → COMMITTED) 또는 (FAILED → ABORTED)로 끝난다.

2) Atomicity(원자성) — All or Nothing

트랜잭션 안의 모든 변경은 하나의 묶음입니다. 일부만 적용된 상태로 끝나는 일은 없어야 합니다. 구현은 언두 로그(Undo Log)로 합니다 — 변경 전 값을 따로 기록해 두고, 실패 시 그 기록을 거꾸로 적용해 되돌립니다. 

BEGIN;
UPDATE account SET balance = balance - 10000 WHERE id = 'A';  -- (1)
-- 여기서 서버 다운!
UPDATE account SET balance = balance + 10000 WHERE id = 'B';  -- (2) 미실행

-- 재시작 후 복구:
-- (1)의 언두 로그를 적용해 A의 잔액을 원복 → 부분 적용 상태가 사라진다.

3) Consistency(일관성) — 불변식 유지

트랜잭션의 시작과 끝 모두에서 DB의 무결성 제약을 만족해야 합니다. 여기서 일관성은 "데이터가 늘 의미 있는 상태에 있다"는 약속입니다.

  • 데이터 무결성 — PK 유일성, FK 참조 무결성, CHECK 제약, NOT NULL.
  • 비즈니스 불변식 — 트리거나 애플리케이션이 보장(예: 출금 후 잔액 ≥ 0).

일관성은 다른 ACID 성질이 함께 보장되어야 비로소 만족됩니다 — Atomicity가 깨지면 한쪽만 적용돼 불변식이 깨질 수 있고, Isolation이 부족하면 동시 갱신으로 깨질 수 있습니다.

4) Isolation(격리성) — 동시 트랜잭션 사이의 가림막

여러 트랜잭션이 동시에 실행돼도, 마치 차례로 실행된 것처럼 보여야 합니다. 현실에서는 성능 때문에 격리 수준을 단계적으로 완화하고, 그에 따라 발생 가능한 이상 현상(Dirty Read, Phantom 등)이 달라집니다. 구현 메커니즘은 보통 락이거나 MVCC입니다.

자세한 내용은 다음 페이지에서 다룹니다 → 격리 수준, 락, MVCC.

5) Durability(지속성) — 커밋된 결과는 살아남는다

커밋이 끝났다고 응답한 트랜잭션의 결과는 이후 어떤 장애가 와도 보존되어야 합니다. 이를 위해 대부분의 RDBMS는 WAL(Write-Ahead Logging)을 사용합니다.

변경은 먼저 WAL에 기록되고 fsync로 영속화된 뒤 커밋이 응답된다. 데이터 페이지는 나중에 천천히 디스크로 내려간다.

WAL의 핵심 규칙은 "데이터 페이지를 디스크로 내리기 전에 그 변경을 기록한 로그가 먼저 디스크에 영속화되어야 한다"입니다. 덕분에 도중에 시스템이 죽어도 WAL을 다시 읽어 커밋된 트랜잭션은 재적용(redo), 미완료 트랜잭션은 되돌릴(undo) 수 있습니다.

6) Java 트랜잭션 예시

// JDBC — 명시적 트랜잭션
conn.setAutoCommit(false);
try {
    PreparedStatement out = conn.prepareStatement("UPDATE account SET balance = balance - ? WHERE id = ?");
    out.setInt(1, 10000); out.setString(2, "A"); out.executeUpdate();

    PreparedStatement in = conn.prepareStatement("UPDATE account SET balance = balance + ? WHERE id = ?");
    in.setInt(1, 10000); in.setString(2, "B"); in.executeUpdate();

    conn.commit();    // 여기서 WAL fsync까지 끝나고 응답
} catch (Exception e) {
    conn.rollback();  // 언두 로그로 원복
}

// Spring — 선언적 트랜잭션
@Transactional
public void transfer(String from, String to, int amount) {
    accountRepo.withdraw(from, amount);
    accountRepo.deposit(to, amount);
    // 메서드가 정상 종료되면 commit, 예외가 나면 rollback (기본은 RuntimeException 한정)
}

시간/공간 복잡도 또는 trade-off

속성구현비용왜 그런가?
Atomicity언두 로그쓰기 1회당 로그 1회변경 전 값 저장 필요
Consistency제약조건 + 트리거제약 검증 시간각 갱신마다 무결성 검사
Isolation락 / MVCC락 대기, 버전 GC동시성 충돌 시 대기 발생
DurabilityWAL + fsync커밋마다 fsync I/O디스크 영속화는 느림. 그룹 커밋·SSD로 완화
throughput—커밋 빈도와 반비례큰 트랜잭션 한 번이 작은 것 여러 번보다 fsync 비용은 적음

실무에서 어디에 쓰이는가?

  • 금융·결제·재고 — 부분 적용이 절대 안 되는 도메인은 트랜잭션이 필수.
  • 여러 테이블 동시 수정 — 회원가입(회원 + 권한 + 약관 동의)처럼 한 번에 묶일 작업.
  • 스프링 @Transactional — 메서드 단위로 트랜잭션 경계를 선언적으로 표현.
  • 분산 환경의 Saga·Outbox 패턴 — 여러 서비스에 걸친 트랜잭션을 ACID로 보장하기 어려울 때 보상 트랜잭션 방식으로 대체.

면접에 나올 질문과 답변

ACID를 한 줄씩 설명해 보세요.
Atomicity: 트랜잭션의 모든 변경은 전부 적용되거나 전부 취소된다(언두 로그). Consistency: 트랜잭션 시작과 끝 모두 무결성 제약과 비즈니스 불변식을 만족한다. Isolation: 동시에 실행되는 트랜잭션이 서로의 중간 상태를 보지 않는다(락/MVCC). Durability: 커밋된 결과는 이후 장애가 발생해도 사라지지 않는다(WAL + fsync).
Atomicity는 어떻게 구현되나요?
대부분의 RDBMS는 변경 전 값을 별도의 언두 로그에 기록합니다. 롤백 시 언두 로그를 거꾸로 적용해 변경 전 상태로 복원하고, 장애로 미커밋 트랜잭션이 남으면 재시작 시 자동으로 되돌립니다.
Durability는 왜 fsync가 필요한가요?
OS는 쓰기 성능을 위해 데이터를 페이지 캐시에 모아뒀다가 나중에 디스크에 내립니다. 그 사이에 전원이 나가면 데이터가 사라질 수 있어, DB는 커밋 직전 fsync로 WAL을 디스크로 내려보내고 응답합니다. 그래서 fsync 성능이 곧 트랜잭션 처리량을 좌우합니다.
Consistency는 무결성 제약과 같은 말인가요?
같지 않습니다. 무결성 제약은 DB가 강제하는 일부 규칙일 뿐이고, Consistency는 그것을 포함해 "어떤 시점이든 데이터가 의미 있는 상태"라는 더 넓은 약속입니다. 예를 들어 "주문 합계 = 주문 라인 합계"는 DB가 자동으로 못 잡는 비즈니스 불변식이지만, 이 또한 일관성의 일부입니다.
트랜잭션을 너무 길게 잡으면 어떤 문제가 있나요?
락이 오래 유지돼 다른 트랜잭션이 대기하고, MVCC에서는 옛 버전이 GC되지 못해 디스크가 부풀어 오릅니다(PostgreSQL의 long-running 트랜잭션 → bloat 문제). 장애 시 롤백 시간도 길어집니다. 그래서 트랜잭션은 가능한 한 짧게, 사용자 입력 대기 같은 외부 I/O는 트랜잭션 밖으로 빼는 게 원칙입니다.
분산 시스템에서 ACID는 어떻게 다루나요?
한 DB 안에서 ACID가 쉽지만, 여러 서비스에 걸친 작업은 2PC가 비용이 크고 가용성을 해치기 때문에 실무에선 Saga 패턴(보상 트랜잭션)이나 Outbox 패턴(이벤트와 DB 변경을 한 트랜잭션으로 묶고 비동기 발행)으로 우회합니다. 결과는 최종 일관성이지만 운영적으로 더 견고합니다.
