조인 알고리즘(Join Algorithm)

조인 알고리즘은 두 테이블의 행을 결합하는 세 가지 대표 방식 — Nested Loop / Hash / Sort-Merge — 입니다. 옵티마이저는 통계와 인덱스 상태에 따라 어떤 알고리즘을 쓸지 결정하며, 각 방식의 비용 모델이 다르기 때문에 "왜 이 쿼리는 인덱스를 안 쓰지?"의 답이 종종 여기에 있습니다.

구체적인 개념 설명

JOIN의 SQL 문법은 같아도, DB 내부에서는 데이터 양·인덱스·메모리 상황에 따라 완전히 다른 알고리즘으로 실행됩니다. 세 가지를 알아두면 EXPLAIN을 봤을 때 왜 그 계획이 선택됐는지를 이해할 수 있고, 인덱스를 어떻게 추가해야 효율이 올라갈지가 보입니다.

비유 — 학생 명단과 출석부 합치기

학생 명단(작은 표)과 출석부(큰 표)를 합치려면 세 가지 방식이 있습니다. 한 학생씩 출석부에서 찾아보기(Nested Loop), 학생 명단을 색인 카드처럼 만들고 출석부를 한 번 훑어 매칭(Hash), 둘 다 학번으로 정렬한 뒤 위에서부터 비교(Sort-Merge). 데이터의 크기와 정렬 상태에 따라 무엇이 빠를지가 달라집니다.

왜 필요한가?

  • 관계형 데이터의 본질 — 정규화된 데이터는 결국 조인으로 합쳐 사용된다.
  • 최악의 쿼리 패턴 파악 — 잘못된 조인 알고리즘은 N×M 비용으로 폭발한다.
  • 인덱스 설계의 출발점 — Nested Loop을 효율적으로 만들려면 내부 테이블의 조인 키에 인덱스가 필수.
  • 옵티마이저 이해 — 통계가 틀리면 알고리즘 선택이 어긋나 성능이 무너진다.

어떻게 동작하는가?

1) Nested Loop Join — 한 행씩 짝짓기

바깥 행이 N개, 안쪽 인덱스 검색이 O(log M)이면 전체 비용은 O(N · log M). 
// 의사 코드
for r in Outer:
    for s in Inner where Inner.key = r.key:   // 인덱스가 있으면 O(log M)
        emit (r, s)
  • 강점 — 바깥이 작고 안쪽에 인덱스가 있을 때 매우 빠름. 메모리 거의 안 씀.
  • 약점 — 바깥이 크거나 안쪽에 인덱스가 없으면 O(N×M)으로 폭발.
  • 변형 — Block Nested Loop은 바깥 행을 블록 단위로 묶어 스캔 횟수를 줄임.

2) Hash Join — 해시 테이블로 한 번에 매칭

Hash Join은 동등 조인에 매우 강하지만 build 단계에 메모리가 필요하다. 
// 의사 코드
H = {}
for r in Build:                  // 작은 쪽
    H[r.key].append(r)
for s in Probe:                  // 큰 쪽
    for r in H.get(s.key, []):
        emit (r, s)
  • 강점 — 인덱스가 없어도 큰 테이블끼리의 동등 조인을 O(N+M)에 처리.
  • 약점 — 동등 조인(=)에만 사용. 메모리에 build 측이 안 들어가면 디스크 spill 발생해 느려짐.
  • 분할 해시 조인(Grace Hash Join) — 메모리에 다 못 들어가면 양쪽을 같은 해시로 분할해 짝끼리 처리.

3) Sort-Merge Join — 정렬해서 위에서부터 합치기

한쪽이 작거나 같으면 그쪽 포인터를 옮기고, 같으면 출력하면서 양쪽을 옮긴다. 
// 의사 코드 (정렬되어 있다고 가정)
i, j = 0, 0
while i < |A| and j < |B|:
    if A[i].key < B[j].key: i += 1
    elif A[i].key > B[j].key: j += 1
    else:
        emit (A[i], B[j])
        # 같은 키가 여러 개면 두 쪽의 동일 키 블록을 카티션 곱
  • 강점 — 정렬되어 있거나 인덱스가 정렬을 제공하면 O(N+M)에 끝남. 큰 데이터 스트리밍에 강함.
  • 약점 — 정렬 비용 O(N log N)이 추가될 수 있음. 정렬할 메모리·tmp 공간 필요.
  • 활용 — 양쪽이 이미 같은 키로 인덱스되어 있을 때 종종 선택된다.

4) 옵티마이저는 어떻게 고를까?

옵티마이저는 다음을 종합해 비용(cost)을 추정하고 가장 싼 알고리즘을 선택합니다.

  • 통계 — 행 수, 컬럼 카디널리티, 분포(히스토그램).
  • 인덱스 존재 여부 — 내부 테이블 조인 키에 인덱스가 있으면 NLJ 비용이 급감.
  • 예상 결과 행 수 — 작으면 NLJ 유리, 크면 Hash/Sort-Merge 유리.
  • 가용 메모리 — Hash Join의 build 측이 메모리에 들어가는지(work_mem).
  • 정렬 활용 — ORDER BY/GROUP BY가 함께 있으면 Sort-Merge가 추가 정렬을 안 해도 될 수 있음.

5) 어떤 인덱스가 어떤 조인에 유리한가

알고리즘유리한 상황인덱스 권장왜 그런가?
Nested Loop바깥이 작고 결과도 적음안쪽 조인 키에 B+Tree안쪽을 인덱스로 빠르게 점프
Hash큰 테이블끼리 동등 조인인덱스 무관해시 테이블이 인덱스를 대신
Sort-Merge큰 테이블이 정렬되어 있거나 ORDER BY 동반조인 키 인덱스(정렬 활용)정렬을 추가로 안 해도 됨
모든 조인—FK 컬럼에 인덱스외래키 검사 + 조인 모두 가속

6) 안티패턴 — 조인이 폭발하는 경우

  • 조인 키에 함수 적용 — 인덱스 사용 불가 → NLJ 인덱스 검색이 풀스캔으로 변질.
  • OR 조건이 두 테이블에 걸쳐 — 옵티마이저가 합치기 어려워 카티션 곱 위험.
  • WHERE 없이 LEFT JOIN 다중 — 결과 폭발. 필터를 ON절로 끌어올려야.
  • 통계가 너무 오래된 경우 — ANALYZE 주기적 실행 필요.

7) Java + EXPLAIN 예시

// EXPLAIN을 통해 알고리즘 확인 (PostgreSQL)
String sql = "EXPLAIN (ANALYZE, BUFFERS) " +
             "SELECT u.name, o.amount " +
             "FROM users u JOIN orders o ON o.user_id = u.id " +
             "WHERE u.created_at >= ?";

// 결과 예
// Hash Join  (cost=... rows=10000 width=...)
//   Hash Cond: (o.user_id = u.id)
//   -> Seq Scan on orders o
//   -> Hash
//        -> Index Scan using users_created_idx on users u

시간/공간 복잡도 또는 trade-off

알고리즘시간메모리전제조건왜 그런가?
Nested Loop (인덱스)O(N · log M)거의 없음안쪽 키 인덱스각 외부 행마다 트리 탐색
Nested Loop (인덱스 X)O(N · M)거의 없음없음풀 스캔의 반복
Hash JoinO(N + M)build 측 크기동등 조인해시 O(1) 조회
Sort-MergeO((N+M) log)정렬 메모리이미 정렬 또는 정렬 가능정렬+선형 스캔

실무에서 어디에 쓰이는가?

  • OLTP의 단일 사용자 페이지 — 주로 NLJ. 작은 결과를 인덱스로 빠르게.
  • 대시보드 / 집계 쿼리 — Hash Join이 많이 선택됨.
  • ETL / 보고서 — 큰 테이블끼리는 Hash/Sort-Merge.
  • 스타 스키마 — 팩트 테이블 ↔ 차원 테이블은 NLJ + 인덱스 조합이 일반적.

면접에 나올 질문과 답변

세 가지 조인 알고리즘의 핵심 차이를 말해보세요.
Nested Loop은 한 행씩 짝짓는 방식으로 작은 결과·인덱스 환경에 강하고, Hash Join은 build/probe 두 단계로 큰 동등 조인을 O(N+M)에 끝내며, Sort-Merge는 양쪽을 정렬해 합치는 방식으로 이미 정렬된 데이터나 ORDER BY가 함께 있을 때 유리합니다.
Nested Loop이 폭발하는 원인은?
안쪽 테이블의 조인 키에 인덱스가 없어 매 외부 행마다 풀 스캔하거나, 외부 테이블이 예상보다 너무 큰 경우입니다. 옵티마이저의 행 수 추정이 틀리면 NLJ로 잘못 잡혀 O(N×M)이 되어버립니다. 해결은 인덱스 추가, 통계 갱신(ANALYZE), 또는 조인 힌트입니다.
Hash Join이 디스크 spill을 일으킬 때 어떻게 해야 하나요?
build 측이 work_mem보다 커서 디스크 임시 파일에 분할되는 상황입니다. ① 작은 쪽이 build가 되도록 통계가 정확한지 확인, ② 필요시 work_mem(MySQL은 join_buffer_size 등) 임시 상향, ③ 인덱스가 있다면 Nested Loop으로 유도하는 힌트도 검토합니다.
FK에 인덱스가 없으면 어떤 일이 생기나요?
부모 행 변경이나 삭제 시 자식 테이블 검사 비용이 풀스캔이 되고, 부모-자식 조인도 NLJ에서 인덱스를 못 써 풀스캔으로 변질됩니다. InnoDB는 자동으로 인덱스를 만들어 주지만 PostgreSQL은 그렇지 않으므로, FK 컬럼에는 명시적으로 인덱스를 만드는 것이 일반적인 권장 사항입니다.
EXPLAIN에서 "Hash Join"이 보이는데 더 빠르게 만들 방법은?
① build 측이 더 작도록 WHERE를 미리 적용해 행을 줄입니다. ② 동등 조인 컬럼에 함수 적용이 있다면 표현식 인덱스나 컬럼 변환으로 풀어냅니다. ③ 가용 메모리를 늘려 spill을 막습니다. 그래도 NLJ + 인덱스가 더 빠르다면 옵티마이저가 통계 기반 결정을 잘못한 것이므로 ANALYZE 후 재확인합니다.
Sort-Merge Join이 NLJ보다 좋은 경우는?
한쪽이 매우 크고 인덱스가 없어 NLJ가 풀스캔의 반복이 되는 경우, 또는 양쪽이 같은 키 인덱스로 이미 정렬되어 있어 추가 정렬 없이 머지만 하면 되는 경우입니다. ORDER BY나 GROUP BY가 같은 키로 함께 걸려 있다면 정렬 비용을 다시 쓰지 않을 수 있어 추가로 유리합니다.
