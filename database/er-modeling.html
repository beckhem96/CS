ER 모델링과 DB 설계

ER(Entity-Relationship) 모델링은 도메인을 엔티티(개체)와 그들 사이의 관계로 그려내는 설계 기법입니다. 코드를 짜기 전에 "이 시스템에는 어떤 데이터가 있고, 그것들은 서로 어떻게 연결되는가"를 그림으로 합의하는 단계입니다. 이 그림이 잘 그려지면 정규화·인덱스·쿼리가 자연스럽게 따라옵니다.

구체적인 개념 설명

DB 설계의 출발은 SQL 문법이 아니라 도메인 분석입니다. "사용자가 게시글을 쓴다", "주문 하나에 여러 상품이 담긴다" 같은 사실을 그림으로 정리하는 것이 ER 다이어그램입니다. 도메인 전문가와 개발자가 같은 그림을 두고 대화할 수 있어, 의사소통 비용을 크게 줄여 줍니다.

ER 모델은 1976년 P. Chen이 제안했고, 이후 IE(Information Engineering) 표기와 UML 클래스 다이어그램 등 여러 표기법으로 확장되었습니다. 표기법은 달라도 핵심 개념은 같습니다 — 엔티티 · 속성 · 관계 · 카디널리티 · 키입니다.

ER 모델 → 관계형 스키마 변환

ER 다이어그램은 그 자체로 실행되지 않습니다. 정해진 변환 규칙을 따라 테이블·외래키로 옮기는 단계가 필요하고, 그 결과가 우리가 CREATE TABLE로 만드는 스키마입니다. ER을 잘 그리면 변환 단계가 거의 기계적이고, ER이 어수선하면 정규화 단계에서 통째로 다시 그리게 됩니다.

왜 필요한가?

코드부터 짜다가 "어, 이 정보가 어디에 들어가야 하지?"를 매번 물으면, 같은 데이터가 여러 테이블에 흩어지고 외래키 방향이 꼬이며 마이그레이션이 쌓입니다. ER 모델링은 이 혼란을 코드 작성 전에 도식화로 끝내려는 시도입니다.

  • 도메인 정합성 — 비즈니스가 인식하는 개체와 DB의 테이블이 1:1로 맞아야 변경이 단순해집니다.
  • 중복·누락 방지 — 같은 정보가 두 군데 적히지 않도록, 또는 정보가 들어갈 자리가 빠지지 않도록 검증할 수 있습니다.
  • 의사소통 — 기획·DBA·개발자가 같은 그림을 보고 회의할 수 있는 공통 언어가 됩니다.
  • 변경 영향 파악 — "이 엔티티에 컬럼 하나 추가하면 어떤 관계가 바뀌나?"를 그림에서 시각적으로 확인합니다.

어떻게 동작하는가?

1) 엔티티와 속성

엔티티(Entity)는 시스템이 관리해야 하는 실체입니다 — 사용자·주문·상품·게시글 같은 명사형 개념입니다. 각 엔티티는 자기를 설명하는 속성(Attribute)을 가지고, 그중 하나(또는 조합)는 인스턴스를 유일하게 식별하는 기본키(PK)가 됩니다.

Chen 표기 — 엔티티는 직사각형, 속성은 타원, 기본키는 밑줄.

2) 관계와 카디널리티

관계(Relationship)는 엔티티 사이의 연결입니다. 그 연결이 양쪽에서 몇 개를 허용하는지를 카디널리티(Cardinality)로 표시합니다. 대표 카디널리티는 1:1, 1:N, N:M 세 가지이고, 각각 변환 규칙이 다릅니다.

카디널리티에 따라 외래키를 어디에 둘지 또는 새 테이블이 필요한지가 결정된다.

3) ER → 관계형 스키마 변환 규칙

ER 패턴변환 결과외래키 위치
강한 엔티티독립 테이블, PK는 자기 식별자—
1:1 관계한 쪽 테이블에 외래키(NOT NULL UNIQUE)참여 강제(total)인 쪽
1:N 관계N쪽 테이블에 외래키 1개"많은" 쪽 (예: posts.user_id)
N:M 관계교차 테이블 신설(양쪽 외래키 + 관계 속성)교차 테이블에 양쪽 FK
약한 엔티티강한 엔티티 PK + 부분 식별자를 합쳐 PK강한 엔티티 PK가 FK이자 PK 일부
다중값 속성 (예: 전화번호 여러 개)별도 테이블로 분리원래 엔티티의 PK가 FK

4) 약한 엔티티 (Weak Entity)

혼자서는 식별이 안 되고, 다른 엔티티에 의존해야만 의미가 있는 엔티티입니다. 대표 예가 "주문(Order) 안의 주문 항목(OrderItem)"입니다. OrderItem은 자기만의 ID(예: line_no)가 있어도 그것만으로는 어떤 주문의 몇 번째 줄인지 알 수 없습니다. 그래서 식별자는 강한 엔티티의 PK + 부분 식별자의 조합이 됩니다. 

-- 약한 엔티티 OrderItem의 변환 예시
CREATE TABLE orders (
    order_id BIGINT PRIMARY KEY,
    customer_id BIGINT NOT NULL,
    ordered_at TIMESTAMP NOT NULL
);

CREATE TABLE order_items (
    order_id BIGINT NOT NULL,                 -- 강한 엔티티 PK
    line_no  INT    NOT NULL,                 -- 부분 식별자
    product_id BIGINT NOT NULL,
    quantity INT NOT NULL,
    PRIMARY KEY (order_id, line_no),          -- 합쳐서 PK
    FOREIGN KEY (order_id) REFERENCES orders(order_id) ON DELETE CASCADE
);

5) 표기법 비교 — Chen vs IE(까마귀발)

Chen 표기법은 학술적이고 직관적이지만 다이어그램이 커지면 산만해집니다. 그래서 실무에서는 IE(Information Engineering) 표기, 일명 까마귀발(crow's foot) 표기를 더 많이 씁니다. 한쪽 끝의 모양으로 카디널리티(0/1/N)를 한 번에 보여 주는 방식입니다.

  • —‖ 정확히 1 (mandatory one)
  • —○ 0 또는 1 (optional one)
  • —< N (mandatory many — 까마귀발)
  • —○< 0 이상 (optional many)

6) 실패하기 쉬운 모델링 패턴

  • 모든 것을 한 테이블에 — "user_data" 한 테이블에 컬럼 50개. 변경마다 락이 길어지고, 의미 단위 분리가 어렵습니다.
  • 1:N을 N:M으로 잘못 잡기 — "사용자가 한 번에 여러 회사에 속할 수 있다"고 가정해 교차 테이블을 만들었는데 실제 도메인은 1:N이었던 경우.
  • 부울로 표현해야 할 것을 별도 테이블로 — "활성 여부"를 별도 테이블로 빼면 조회 비용만 늘어납니다.
  • 외래키를 미루고 나중에 — 운영에서 데이터가 쌓이면 외래키 추가가 거의 불가능해집니다. 처음부터 잡아야 합니다.
  • 자연키를 PK로 (예: 이메일) — 이메일이 바뀌면 모든 외래키가 함께 바뀌어야 합니다. 보통 surrogate key(자동 증가 ID)를 권장합니다.

trade-off

설계 결정장점비용
surrogate key (자동 증가 ID)변경에 안전, JOIN이 가볍다의미가 없어 디버깅 시 추가 조회 필요
natural key (이메일·SSN 등)의미가 명확, 추가 컬럼 불필요값이 바뀔 때 외래키 캐스케이드 비용 큼
교차 테이블에 추가 속성 두기관계 자체의 메타정보(가입일 등) 표현 가능모델이 복잡해지고 조인 비용 증가
약한 엔티티로 분리도메인 의도가 명확조회마다 강한 엔티티와 함께 JOIN 필요
NoSQL식 비정규 모델 (참고)읽기 패턴에 최적화 가능중복·일관성 비용 증가, 변경 어려움

실무에서 어디에 쓰이는가?

  • 신규 시스템 설계 — 가장 먼저 도메인 인터뷰를 토대로 ER을 그리고, 거기서 테이블·외래키를 도출합니다.
  • 레거시 분석 — 기존 DB의 ER을 역공학(예: SchemaSpy, DBeaver)으로 그려, 현재 구조를 한눈에 파악합니다.
  • 마이그레이션 계획 — "이 컬럼을 어디로 옮길까?", "이 두 엔티티를 합칠까?"를 그림으로 합의한 뒤 DDL을 짭니다.
  • API 설계 — REST 리소스 모델은 ER의 엔티티에 자주 1:1로 대응합니다.
  • 도메인 주도 설계(DDD) — 애그리거트 경계, 엔티티/값 객체 식별의 출발점이 ER 모델입니다.

면접에 나올 질문과 답변

엔티티와 속성, 관계가 무엇인지 설명해 주세요.
엔티티는 시스템이 관리해야 하는 실체(사용자·주문·상품 같은 명사)이고, 속성은 그 엔티티를 설명하는 정보(이름·가격·날짜)입니다. 관계는 엔티티들 사이의 연결을 표현하며, 양쪽에서 몇 개를 허용하는지를 카디널리티로 적습니다. ER 다이어그램은 이 셋을 도식으로 표현해 도메인을 합의 가능한 그림으로 만들어 줍니다.
1:N 관계와 N:M 관계의 차이는 무엇이고, 각각 어떻게 변환하나요?
1:N은 한 쪽이 여러 개를 가질 수 있는 관계로, N쪽 테이블에 외래키 하나를 두면 표현됩니다(예: posts.user_id). N:M은 양쪽 모두 여러 개를 가질 수 있는 관계로, 외래키만으로는 표현이 안 되어 교차 테이블(연결 엔티티)을 만들고 양쪽 외래키를 둡니다. 교차 테이블에는 가입일·역할 같은 관계 자체의 속성을 함께 저장할 수도 있습니다.
약한 엔티티는 무엇이고 어떻게 식별하나요?
혼자서는 식별이 안 되고 강한 엔티티에 의존해야 의미가 있는 엔티티입니다. 예를 들어 OrderItem은 line_no만으로는 어떤 주문의 항목인지 알 수 없으므로, 강한 엔티티 Order의 PK와 자신의 부분 식별자(line_no)를 합쳐 PK로 사용합니다. 강한 엔티티가 삭제될 때 함께 삭제되는 ON DELETE CASCADE를 자주 적용합니다.
자연키(natural key)와 대리키(surrogate key) 중 무엇을 PK로 쓰는 게 좋을까요?
대부분의 실무에선 자동 증가 ID 같은 대리키를 권장합니다. 자연키는 의미가 있어 디버깅엔 좋지만, 값이 바뀌었을 때 그 키를 참조하는 모든 외래키가 함께 바뀌어야 해 운영 비용이 큽니다. 대리키는 값이 절대 바뀌지 않으므로 외래키가 안정적이고, 인덱스 사이즈도 작아 조인이 가볍습니다. 다만 의미가 없어 자연키 컬럼은 별도로 두고 UNIQUE 제약을 걸어 검색에 활용합니다.
ER 모델링과 정규화는 어떤 관계인가요?
ER 모델링은 도메인을 엔티티·관계로 그리는 설계 단계, 정규화는 그렇게 만들어진 스키마에서 함수적 종속성과 중복을 다듬는 후속 단계입니다. ER이 잘 그려지면 변환 결과가 거의 3NF에 가깝고, ER이 어수선하면 정규화로 큰 폭의 분해가 필요해집니다. 둘은 직렬 관계가 아니라 서로 점검 도구로 함께 작동합니다.
실무에서 ER 다이어그램을 어떻게 활용하시나요?
신규 시스템에서는 도메인 인터뷰를 거쳐 ER을 먼저 그리고, 그것을 기반으로 DDL과 ORM 매핑을 도출합니다. 레거시 시스템에서는 SchemaSpy·DBeaver 같은 도구로 ER을 역공학으로 추출해 현재 구조를 파악하고, 변경 영향 범위를 확인하는 데 씁니다. DDD에서 애그리거트 경계를 정할 때, REST API의 리소스 모델을 잡을 때도 ER이 출발점이 됩니다.
