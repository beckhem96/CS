복제 (Replication) — 같은 데이터를 여러 노드에

복제(Replication)는 같은 데이터를 여러 DB 노드에 복사해 두는 기법입니다. 한 노드가 죽어도 서비스가 살아남고(가용성), 읽기 부하를 여러 대가 나눠 받고(확장성), 지리적으로 가까운 곳에서 응답을 제공할 수 있습니다(지연 감소). 대신 같은 데이터를 동기화해야 하므로 일관성이라는 새 비용이 생깁니다.

구체적인 개념 설명

한 대의 DB로는 두 가지 한계가 있습니다. 첫째, 그 한 대가 죽으면 전체 서비스가 멈춥니다. 둘째, 트래픽이 늘면 한 대가 감당하지 못합니다. 복제는 이 두 문제를 동시에 노립니다. 같은 데이터를 N개의 노드에 두고, 한 대가 죽으면 다른 대로 페일오버하고, 읽기는 여러 대로 분산합니다.

복제는 "누가 쓰기를 받는가"로 크게 나뉩니다. 한 노드만 쓰기를 받는 리더-팔로워(single-leader)가 가장 흔하고, 여러 노드가 쓰기를 받는 multi-leader, 모든 노드가 쓰기·읽기를 받는 leaderless(예: Cassandra/Dynamo)가 있습니다. 구조에 따라 충돌·일관성·페일오버 동작이 크게 달라집니다.

복제는 일관성과의 거래다

리더에 쓴 직후 팔로워에서 읽으면 그 값이 아직 안 보일 수 있습니다 — 이게 리플리카 지연(replication lag)입니다. 지연을 0으로 만들려면 리더가 모든 팔로워의 응답을 기다려야 하고(동기 복제), 그러면 가용성과 응답 속도가 희생됩니다. 결국 "얼마나 강한 일관성을 원하느냐"에 따라 동기 ↔ 비동기 사이의 한 점을 골라야 합니다.

왜 필요한가?

  • 가용성 — 단일 노드가 죽어도 다른 노드가 즉시 서비스를 이어받습니다(failover).
  • 읽기 확장 — 분석·검색·대시보드처럼 읽기 비중이 큰 워크로드를 팔로워에 분산해 리더 부담을 줄입니다.
  • 지리적 분산 — 사용자에 가까운 리전에 팔로워를 두면 읽기 지연이 줄어듭니다.
  • 백업 / 분석 격리 — 운영 트래픽을 막지 않고 팔로워에서 백업·DW ETL을 돌릴 수 있습니다.
  • 업그레이드 / 장애 훈련 — 무중단 마이너 업그레이드, 카오스 테스트의 기반이 됩니다.

어떻게 동작하는가?

1) 리더-팔로워 구조 — 가장 흔한 모델

리더만 쓰기를 받고, 팔로워는 변경 로그를 받아 자기 데이터를 갱신해 읽기를 담당한다.

리더는 모든 쓰기를 받아 자기 디스크에 적용하고, 그 변경을 복제 로그(WAL/binlog)에 기록합니다. 팔로워는 그 로그를 받아 그대로 자기 디스크에도 적용합니다. 결과적으로 팔로워는 (지연이 있을지언정) 리더와 같은 데이터를 갖게 됩니다.

2) 동기 vs 비동기 vs 반동기 복제

동기는 안전하지만 느리고, 비동기는 빠르지만 손실 위험이 있다. 반동기는 그 사이의 절충점.

3) 페일오버 — 리더가 죽었을 때

페일오버의 4단계 — 자동화 도구(Patroni, Orchestrator 등)가 이 흐름을 책임진다.

4) 리플리카 지연과 읽기 일관성 패턴

비동기 복제에서는 리더에 쓴 직후 팔로워에 그 값이 도달하기까지 수 ms ~ 수 초의 지연이 생길 수 있습니다. 이 지연이 사용자 경험에 어떻게 새어 나오는지가 일관성 문제의 본질입니다.

  • read-your-writes (RYW) — 사용자가 자기 글을 쓰자마자 자기 페이지에 안 보이면 큰 혼란. "본인의 최근 작성 직후 N초간은 리더에서 읽기"로 해결.
  • monotonic read — 같은 사용자가 새로고침할 때마다 데이터가 과거로 돌아가면 안 됨. 같은 사용자는 같은 팔로워에 라우팅.
  • consistent prefix read — 인과 관계가 있는 글들이 순서가 뒤바뀌어 보이면 안 됨. 파티션 라우팅으로 해결.
  • 강한 일관성이 꼭 필요한 쿼리 — 결제·잔액 조회처럼 정확성이 결정적인 경우 명시적으로 리더에서 읽기(FOR UPDATE, read=primary 힌트).

5) 동작 원리 — 무엇을 보내는가

리더가 팔로워에게 보내는 변경 로그는 구현마다 형태가 다릅니다.

  • Statement-based — SQL 문 자체를 그대로 보냄. 가볍지만 비결정적 함수(NOW(), RAND()) 때문에 결과가 다를 수 있음.
  • Row-based — 변경된 행의 before/after 값을 보냄. 가장 안전하고 일반적. (MySQL binlog_format=ROW, PostgreSQL logical replication)
  • WAL / 물리 복제 — 디스크 페이지의 변경을 그대로 보냄. 매우 빠르지만 동일 버전·동일 아키텍처 필수. (PostgreSQL streaming replication)

구현체마다 트레이드오프가 다릅니다. PG는 WAL 기반 streaming이 표준이고, MySQL은 binlog row-based가 일반적입니다. Aurora·Spanner 같은 클라우드 DB는 분산 합의 프로토콜을 결합한 자체 구조를 씁니다.

6) Multi-leader와 Leaderless — 짧게

Multi-leader는 여러 리더가 동시에 쓰기를 받는 구조입니다. 지리적 분산에 유리하지만 충돌(같은 행을 두 리더에서 동시에 수정)을 누가 어떻게 해결할지가 핵심 난점입니다(타임스탬프, CRDT, 사용자 개입 등). Leaderless는 모든 노드가 쓰기를 받고 quorum(W + R > N)으로 일관성을 맞추는 구조로, Cassandra·DynamoDB가 대표입니다. 대부분의 일반 OLTP 워크로드에서는 single-leader면 충분하므로, 이 두 가지는 도메인 요구가 명확할 때만 고려합니다.

실무에서 어디에 쓰이는가?

  • 읽기 분산 — Rails의 connects_to(role: :reading)처럼 ORM이 트랜잭션이 없는 SELECT를 팔로워로 라우팅합니다.
  • HA 구성 — MySQL Group Replication, PostgreSQL + Patroni, MongoDB Replica Set이 자동 페일오버를 제공합니다.
  • 지리적 분산 — 글로벌 서비스에서 각 리전에 팔로워를 두어 사용자에 가까운 곳에서 읽기 처리.
  • 분석/CDC — Debezium·AWS DMS는 binlog/WAL을 구독해 데이터 웨어하우스에 실시간 동기화합니다.
  • 온라인 마이그레이션 — 새 DB를 팔로워로 띄워 동기화한 뒤, 짧은 절체 시간 안에 리더로 승격해 다운타임을 최소화합니다.
  • 장애 훈련 — 의도적으로 리더를 죽이고 페일오버 시간을 측정(GameDay, Chaos Engineering).

면접에 나올 질문과 답변

DB 복제는 왜 사용하나요?
같은 데이터를 여러 노드에 복사해 두면, 한 노드가 죽어도 다른 노드가 서비스를 이어받을 수 있고(가용성), 읽기 트래픽을 여러 대가 나눠 받아 부하를 분산할 수 있으며(확장성), 사용자에 가까운 리전에서 응답해 지연을 줄일 수 있습니다. 대신 데이터를 동기화해야 하므로 일관성이라는 새 비용이 발생합니다.
동기·비동기·반동기 복제의 차이는 무엇인가요?
동기는 모든 팔로워가 ACK를 보낼 때까지 리더가 응답을 미루는 방식으로, 데이터 손실은 거의 없지만 응답 지연이 큽니다. 비동기는 리더가 자기 쓰기를 마치자마자 응답하고 팔로워에는 나중에 보내는 방식으로, 빠르지만 리더 장애 시 일부 손실이 가능합니다. 반동기(semi-sync)는 최소 1대의 ACK만 받으면 응답하는 절충안으로, MySQL·Aurora 등 실무에서 자주 쓰입니다.
리플리카 지연(replication lag)이 왜 문제이고 어떻게 다루나요?
리더에 쓴 직후 팔로워에서 같은 데이터를 읽으면 아직 반영이 안 돼 옛 값이 보일 수 있습니다. 사용자가 자기 글을 안 보이는 것처럼 느끼는 read-your-writes 문제, 새로고침할 때마다 데이터가 과거로 돌아가는 monotonic read 문제 등이 발생합니다. 해결책으로는 본인 작성 직후엔 리더에서 읽기, 같은 사용자는 같은 팔로워에 라우팅, 결정적인 쿼리는 명시적으로 리더에서 읽기 같은 패턴이 사용됩니다.
페일오버에서 split-brain은 왜 위험하고 어떻게 막나요?
네트워크 분단 등으로 새 리더가 선출됐는데 옛 리더도 자기를 리더라고 믿으면, 두 리더가 동시에 쓰기를 받아 데이터가 갈라집니다(split-brain). 이를 막기 위해 펜싱(fencing) — 옛 리더의 쓰기 권한을 강제로 차단하거나, STONITH로 노드를 정지시키는 — 기법을 사용하고, 리더 선출에 합의 알고리즘(Raft, Paxos)을 도입해 한 시점에 단 하나의 리더만 인정하도록 합니다.
복제는 어떤 형태로 데이터를 전송하나요?
크게 statement-based(SQL 문 그대로), row-based(변경 행의 전후 값), WAL 기반 물리 복제 세 가지가 있습니다. statement-based는 가볍지만 NOW(), RAND() 같은 비결정적 함수에서 결과가 달라질 위험이 있고, row-based는 가장 안전해 MySQL의 기본 권장이며, WAL 물리 복제는 PostgreSQL streaming replication처럼 디스크 페이지 변경을 그대로 보내 매우 빠릅니다.
복제와 샤딩(파티셔닝)은 어떻게 다른가요?
복제는 같은 데이터를 여러 노드에 두는 것이고, 샤딩은 데이터를 여러 노드로 나눠 각 노드가 일부만 갖는 것입니다. 복제는 가용성과 읽기 확장에 유리하지만 쓰기 처리량은 결국 한 리더가 한계입니다. 샤딩은 쓰기 처리량까지 수평 확장할 수 있지만 트랜잭션·조인이 노드 경계를 넘으면 복잡해집니다. 실무에서는 보통 둘을 함께 씁니다 — 각 샤드가 자기 안에서 다시 리더-팔로워 복제를 하는 형태.
