MVCC(Multi-Version Concurrency Control)

MVCC는 하나의 행에 대해 여러 버전을 동시에 보관해, 읽기 트랜잭션이 쓰기 트랜잭션을 잠그지 않도록 만드는 동시성 제어 기법입니다. "읽기는 쓰기를 막지 않고, 쓰기는 읽기를 막지 않는다(Readers don't block writers; writers don't block readers)"가 핵심 약속입니다.

구체적인 개념 설명

락만 사용하는 DB는 누가 갱신 중이면 다른 사람이 그 행을 못 읽습니다. 그러면 보고서·검색 같은 읽기 작업이 갱신 트랜잭션에 자주 막히죠. MVCC는 행을 갱신할 때 이전 값을 따로 보관하고, 각 트랜잭션이 자신의 시작 시점에 맞는 버전을 골라서 봅니다. 덕분에 읽기와 쓰기가 서로를 기다리지 않아도 되고, 모든 트랜잭션이 일관된 스냅샷을 갖게 됩니다.

MVCC는 격리 수준의 구현 수단입니다. PostgreSQL과 Oracle은 처음부터 MVCC 중심으로 설계됐고, MySQL InnoDB도 MVCC + 락의 하이브리드입니다.

비유 — 위키 문서의 버전 히스토리

누군가 위키를 수정하는 중에도 다른 사람이 이전 버전을 그대로 읽을 수 있죠. 수정이 끝나야 비로소 새 버전이 "현재"가 됩니다. MVCC도 똑같이 옛 버전과 새 버전을 동시에 보관하면서, 트랜잭션마다 자신의 시점에 보이는 버전을 골라 보여줍니다.

왜 필요한가?

  • 읽기 성능 보존 — 읽기가 갱신을 기다리지 않으므로 OLTP 부하에서도 보고서가 가능.
  • 일관된 스냅샷 — 트랜잭션 동안 같은 행을 다시 읽어도 같은 결과(REPEATABLE READ).
  • 긴 읽기 트랜잭션 — 통계 쿼리가 갱신을 막지 않게.
  • 롤백 비용 절감 — 옛 버전이 그대로 남아 있으니 언두 자체가 자연스럽게 가능.

어떻게 동작하는가?

1) 핵심 아이디어 — 트랜잭션 ID와 가시성 판정

각 트랜잭션은 시작 시 단조 증가하는 트랜잭션 ID(xid/txid)를 받습니다. 행에는 그 행을 만든 트랜잭션의 ID(xmin)와 그 행을 지운/갱신한 트랜잭션의 ID(xmax)가 함께 저장됩니다. 트랜잭션 T가 행을 볼 수 있는지 여부는 보통 다음 규칙으로 정합니다:

  • 행의 xmin이 이미 커밋됐고 T보다 이전에 시작했다 → 보임 후보
  • 행의 xmax가 비어있거나 아직 커밋되지 않았다 → 보임
  • 그 외에는 보이지 않음(다른 트랜잭션의 미커밋 변경, T 시작 이후 시작된 트랜잭션의 변경 등)

2) PostgreSQL — 행 자체에 버전을 추가(append-only)

PostgreSQL은 갱신마다 새 튜플을 추가하고, 옛 튜플은 VACUUM이 정리한다.

PostgreSQL은 UPDATE를 "새 튜플을 만들고 옛 튜플의 xmax를 채우는" 동작으로 처리합니다. 옛 버전이 힙 안에 남아 있어 추가 자료구조 없이 MVCC가 동작합니다. 대신 VACUUM이 옛 튜플을 회수해야 디스크가 부풀지 않고, 트랜잭션 ID가 모자라기 전에 wraparound 방지를 위해 정기적으로 freeze해야 합니다.

→ 부작용: 갱신이 잦은 테이블은 bloat가 발생해 인덱스 효율이 떨어짐. autovacuum 튜닝이 운영 포인트.

3) Oracle / MySQL InnoDB — Undo 영역에 옛 버전 보관

Oracle/InnoDB는 데이터 블록은 항상 최신값을 보관하고, 옛 값은 Undo 영역에서 역추적해 재구성한다.

Oracle과 InnoDB는 데이터 페이지에는 항상 최신값만 두고, 갱신 전 값은 언두 영역에 기록합니다. 옛 버전이 필요한 트랜잭션은 언두 체인을 거꾸로 따라가 그 시점의 값을 재구성합니다. 이 방식은 데이터 페이지가 깔끔하게 유지되지만, 장시간 트랜잭션이 옛 언두를 붙잡고 있으면 새 트랜잭션이 ORA-1555("snapshot too old") 같은 에러를 만나거나 InnoDB의 history list가 길어져 성능이 떨어집니다.

5) MVCC와 격리 수준

  • READ COMMITTED — 각 SQL 실행 시점마다 새 스냅샷. 같은 트랜잭션 안에서 같은 행을 다시 읽으면 다른 값이 보일 수 있음(Non-repeatable Read 가능).
  • REPEATABLE READ — 트랜잭션 시작 시점의 스냅샷을 끝까지 사용. PG에서는 충돌이 감지되면 직렬화 실패로 abort, MySQL InnoDB는 next-key 락과 결합.
  • SERIALIZABLE — PG는 SSI로 충돌 사이클을 감지해 abort, MySQL은 자동 공유 락으로 직렬화.

6) MVCC 특유의 함정

  • Long-running 트랜잭션 — 옛 버전 GC가 안 돼서 디스크가 부풀고 옵티마이저 통계가 흐려진다.
  • VACUUM이 따라가지 못하는 갱신 — autovacuum 임계치 튜닝이 필요.
  • 인덱스 hot-update 실패 — 인덱스된 컬럼이 자주 바뀌면 인덱스 단편화 → REINDEX가 가끔 필요.
  • "snapshot too old" — Oracle/InnoDB에서 언두 보관 한계를 넘으면 발생.

7) Java 예시 — 일관된 스냅샷

// PostgreSQL, REPEATABLE READ
conn.setAutoCommit(false);
conn.setTransactionIsolation(Connection.TRANSACTION_REPEATABLE_READ);

ResultSet rs1 = stmt.executeQuery("SELECT balance FROM account WHERE id='A'");
// (그 사이 다른 세션이 balance를 갱신·커밋해도)
ResultSet rs2 = stmt.executeQuery("SELECT balance FROM account WHERE id='A'");
// rs1과 rs2는 같은 값을 본다 — 트랜잭션 시작 시점의 스냅샷
conn.commit();

시간/공간 복잡도 또는 trade-off

  • 읽기 처리량 ↑ — 잠그지 않고 읽으니 OLTP·보고서 동시 운영에 강함.
  • 저장 공간 ↑ — 옛 버전을 어디든 보관해야 함(힙 또는 언두).
  • GC 비용 — VACUUM/언두 정리가 정기적으로 필요.
  • UPDATE 비용 ↑ — 단순 in-place 수정보다 무거움(특히 PG에서 인덱스가 따라옴).

실무에서 어디에 쓰이는가?

  • OLTP 메인 DB — 동시 사용자 환경에서 읽기 일관성과 처리량 동시 확보.
  • 보고/분석 동시 운영 — 정산 쿼리가 결제 갱신을 안 막음.
  • 장기 트랜잭션 모니터링 — bloat, history list 길이를 운영 지표로 추적.
  • HOT 업데이트와 fillfactor 튜닝 — 갱신이 잦은 테이블에서 PG의 페이지 내 갱신 비율을 높이는 운영 기법.

면접에 나올 질문과 답변

MVCC가 락 기반 동시성 제어보다 좋은 점은?
읽기가 쓰기를 막지 않고 그 반대도 성립합니다. 갱신 중에도 다른 트랜잭션이 옛 버전을 그대로 읽을 수 있어 읽기 처리량이 떨어지지 않고, 장기 보고 쿼리가 갱신 트랜잭션을 차단하지도 않습니다. 대신 옛 버전 저장과 GC 비용이 추가로 발생합니다.
PostgreSQL과 Oracle/InnoDB의 MVCC 구현 차이는?
PostgreSQL은 갱신할 때마다 새 튜플을 힙에 추가하고 옛 튜플은 VACUUM이 정리합니다(롤백이 매우 빠름). Oracle/InnoDB는 데이터 블록에는 최신값만 두고 옛 값은 Undo 영역에 보관해 필요 시 역추적합니다(언두 체인이 길어지면 성능 저하). 둘 다 MVCC지만 옛 버전을 어디에 둘지의 선택이 다른 것입니다.
PostgreSQL의 VACUUM이 왜 중요한가요?
VACUUM은 더 이상 보이지 않는 옛 튜플을 회수해 공간을 재사용하고, 인덱스 단편화를 줄이며, xid wraparound를 막습니다. autovacuum이 따라가지 못하면 테이블이 bloated 되어 쿼리 성능이 떨어지므로, 갱신이 많은 테이블에서는 임계치(autovacuum_vacuum_scale_factor 등) 튜닝이 필수입니다.
"snapshot too old"는 왜 발생하나요?
Oracle/InnoDB는 옛 값을 Undo 영역에 보관하는데, 이 영역은 회전(circular) 구조라 오래된 항목이 덮어 써집니다. 장기 읽기 트랜잭션이 필요한 옛 버전이 이미 덮였을 때 ORA-1555가 발생합니다. 해결은 언두 보관 시간을 늘리거나 장기 트랜잭션을 짧게 쪼개는 것입니다.
MVCC가 Lost Update를 자동으로 막아주나요?
막지 않습니다. MVCC는 일관된 읽기 스냅샷을 제공하지만, 두 트랜잭션이 같은 행을 각자의 스냅샷으로 읽고 갱신하면 늦게 쓴 쪽이 먼저 쓴 쪽을 덮어버립니다. 그래서 Lost Update가 우려되는 흐름은 SELECT ... FOR UPDATE로 비관적 락을 걸거나, 버전 컬럼으로 낙관적 락을 사용해야 합니다.
REPEATABLE READ에서 PG와 MySQL은 충돌을 어떻게 다르게 처리하나요?
PostgreSQL은 트랜잭션 시작 시점의 스냅샷을 사용하다가 직렬화 충돌이 감지되면 트랜잭션을 abort하고 직렬화 실패 에러를 던집니다(애플리케이션이 재시도). MySQL InnoDB는 next-key 락으로 범위까지 잠가 갱신·삽입을 차단해 직렬화를 만들어냅니다(대기 발생).
